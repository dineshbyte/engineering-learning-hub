# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Walk the LSM write path from a client write to a key on disk: memtable, WAL, SSTable.'
      a: >-
          <b>Hit these points:</b> a write appends to the <b>WAL</b> for durability and inserts into
          the in-RAM <b>memtable</b> (a sorted map — skip list / RB-tree) &rarr; no in-place disk
          write happens, which is why writes are fast &rarr; when the memtable fills it is frozen and
          flushed as a new immutable <b>SSTable</b>: an on-disk file sorted by key &rarr; SSTables are
          never modified, only merged away by compaction &rarr; updates and deletes are also just
          writes &mdash; a newer value or a tombstone is appended and shadows the old one &rarr; the
          WAL exists so a crash before flush doesn't lose the memtable's contents.
    - level: 'core'
      q: 'What is an SSTable, and why does its sortedness make merging cheap?'
      a: >-
          <b>Hit these points:</b> an SSTable is an immutable on-disk file of <code>key,value</code>
          entries <em>sorted by key</em>, with a sparse index so a key can be found with one seek into
          a block &rarr; immutability means readers and compaction never contend over in-place edits
          &mdash; you only ever create new files and delete old ones &rarr; because every SSTable is
          already sorted, merging N of them is a linear <em>k-way merge</em> in one pass, like
          merge-sort's merge step: no random I/O, no re-sorting &rarr; during the merge you keep the
          newest value per key and drop shadowed/tombstoned ones &rarr; that "two sorted files merge in
          one sequential pass" is the property the whole engine is built on.
    - level: 'core'
      q: 'Name the three amplifications and say who pays for each.'
      a: >-
          <b>Hit these points:</b> <b>write amp</b> = disk bytes ÷ app bytes &mdash; pays in flash wear
          and write throughput &rarr; <b>read amp</b> = SSTables (disk reads) touched per logical read
          &mdash; pays in read latency and p99 &rarr; <b>space amp</b> = disk bytes ÷ live data bytes
          &mdash; pays the storage bill &rarr; the iron law (RUM conjecture): lowering one
          amplification raises another, so there is no free strategy &rarr; every compaction policy is
          just a chosen point in this trade-off space.
    - level: 'core'
      q: 'What does a Bloom filter do in an LSM read path, and what guarantee does it give?'
      a: >-
          <b>Hit these points:</b> each SSTable carries a Bloom filter over its keys; before opening
          the file a read probes the filter &rarr; the guarantee is <em>one-sided</em>: "definitely
          absent" is never wrong, "probably present" can be a false positive &rarr; so a "definitely
          absent" answer lets the read safely <em>skip</em> that SSTable &mdash; turning an absent-key
          read that would touch every run into one (rare) false-positive open &rarr; cost is small:
          ~10 bits/key for a ~1% false-positive rate (illustrative) &rarr; it attacks read amp
          specifically for negative/point lookups; it does nothing for range scans, which must still
          consult overlapping runs.
    - level: 'senior'
      q: 'Compare size-tiered and leveled compaction. When do you pick each?'
      a: >-
          <b>Hit these points:</b> <b>STCS</b> groups SSTables by size and merges a batch of same-size
          tables into one of the next size up; tiers grow geometrically so a key is rewritten only
          ~<code>log N</code> times &rarr; that buys <em>low write amp</em> but costs space (a big
          merge needs ~2× the dataset, and stale copies linger across un-merged tiers) and med&ndash;high
          read amp &rarr; <b>LCS</b> keeps levels L1+ non-overlapping, each ~10× the last, merging one
          table down into the few it overlaps &rarr; that buys <em>low read amp</em> (≤1 file per level
          + L0) and <em>low space amp</em> (~1.1×) but pays write amp ~T·log_T(N) ≈ 10&ndash;30× &rarr;
          pick STCS for write-heavy ingest with disk to spare; LCS for read-heavy, space-constrained
          data.
    - level: 'senior'
      q: 'Why can leveled compaction guarantee a point read touches at most one SSTable per level?'
      a: >-
          <b>Hit these points:</b> the LCS invariant: in every level <em>except L0</em>, SSTables have
          non-overlapping key ranges &rarr; so each level below L0 is effectively one sorted run with
          disjoint ranges, and any given key can fall in exactly one table there &rarr; L0 is the
          exception because it takes raw memtable flushes that <em>do</em> overlap, so a read still
          scans all of L0 &rarr; total files consulted ≈ all of L0 + one per level ≈ <code>log_T(N)</code>
          &rarr; that bounded read amp is exactly what you pay the 10&ndash;30× write amp to get.
    - level: 'senior'
      q: 'Walk me through the tombstone resurrection bug and how engines prevent it.'
      a: >-
          <b>Hit these points:</b> a delete can't erase data in immutable SSTables, so it writes a
          <b>tombstone</b>: "this key is dead as of this sequence number" &rarr; the value still lives
          in an older SSTable; the tombstone shadows it on read &rarr; the bug: if the tombstone is
          purged before the older SSTable holding the value is compacted away, the value reappears with
          nothing to mask it &mdash; a deleted record comes back to life &rarr; prevention: only drop a
          tombstone once compaction has carried it past every older copy of that key (the bottom level
          / past all older tiers) &rarr; distributed twist: Cassandra holds tombstones for
          <code>gc_grace_seconds</code> (default 10d) so the delete propagates to every replica before
          GC, or a node that missed the delete re-replicates the stale value.
    - level: 'senior'
      q: 'What triggers compaction, and why does an LSM under heavy ingest start stalling writes?'
      a: >-
          <b>Hit these points:</b> compaction is triggered by structural thresholds, not a timer: STCS
          fires when enough same-size SSTables accumulate in a tier; LCS fires when a level exceeds its
          size target; engines also track L0 file count and pending-compaction bytes &rarr; under
          sustained ingest, writes can outrun compaction's drain rate, so L0 / pending-compaction-bytes
          crosses a slowdown then stop trigger and the engine applies back-pressure &mdash; that's the
          stall &rarr; what you do <em>not</em> do is "write faster" &rarr; the fixes give compaction
          more throughput (more threads, faster disk), switch to a cheaper-write strategy, or
          smooth/shard the ingest &rarr; the underlying truth: sustainable write rate is bounded by
          compaction throughput.
    - level: 'staff'
      q: 'Should one compaction strategy apply to the whole database? Make the principal-level call.'
      a: >-
          <b>Hit these points:</b> no &mdash; a single global setting forces one corner of the
          read/write/space triangle on every table regardless of its access pattern &rarr; the senior
          move is to choose per table / column family, the same way you choose indexes per query &rarr;
          a write-once-read-rarely audit log wants tiered or time-windowed; a hot read-mostly profile
          table wants leveled; a TTL'd time-series wants TWCS &rarr; trade-off to name: per-table
          tuning adds operational surface and reasoning cost, so default sensibly and only override
          where the workload data justifies it &rarr; the discipline is matching each table's dominant
          amplification to a measured access pattern, not picking one strategy by reputation.
    - level: 'staff'
      q: '"Leveled is the modern default, just use it everywhere." Steelman it, then take it apart.'
      a: >-
          <b>Steelman:</b> leveled gives bounded read amp (≤1 file/level) and low space amp (~1.1×), so
          reads are predictable and you don't risk running out of disk during a merge &mdash; for
          read-mostly, space-constrained, SSD-backed data that's genuinely the right corner. <b>Take it
          apart:</b> leveled pays 10&ndash;30× write amplification, so on a write-heavy or
          ingest-firehose table it burns write bandwidth and flash endurance and is the first to hit
          write stalls &rarr; "default everywhere" forces that write tax onto tables that are mostly
          appended and rarely read &rarr; for append-only TTL'd time-series, TWCS beats it on all three
          by dropping whole old buckets at once. <b>Synthesis:</b> there is no globally-best strategy
          &mdash; leveled is one well-chosen point; match the corner to each table's workload and disk
          profile, and watch write-amp/stall metrics to know when you chose wrong.
    - level: 'staff'
      q: "Where do TWCS and RocksDB's universal compaction sit relative to STCS and LCS, and why do hybrids exist?"
      a: >-
          <b>Hit these points:</b> STCS and LCS are the write-optimized and read/space-optimized
          corners; real workloads often sit between them, which is why hybrids exist &rarr; RocksDB
          <em>universal</em> compaction is tiered-leaning &mdash; write-friendly, fewer rewrites than
          leveled, accepting more space/read amp &rarr; Cassandra <em>TWCS</em> buckets SSTables by
          time window for append-only, TTL'd time-series, so whole old buckets expire and drop at once
          &mdash; achieving low write + low space + low read <em>within a window</em> by exploiting the
          access pattern rather than beating RUM &rarr; the staff signal is naming the workload that
          justifies the hybrid (time-series with TTLs, immutable events) instead of treating compaction
          as one global knob &rarr; the lesson: you escape the triangle only by changing the workload's
          shape, not by tuning harder.
    - level: 'design'
      q: 'Design the compaction strategy for a write-heavy ingest table vs a read-heavy lookup table in the same cluster.'
      a: >-
          <b>A strong answer covers:</b> reject one global setting &mdash; choose per table / column
          family by access pattern &rarr; <b>write-heavy ingest</b> (firehose, mostly appended, rarely
          point-read): size-tiered, because ~<code>log N</code> rewrites keep write amp low and protect
          the sustainable write rate; accept the ~2× space spikes and med&ndash;high read amp, and
          provision disk headroom so a big merge can't exhaust the volume &rarr; if events carry TTLs,
          prefer TWCS so whole old buckets drop at once &mdash; lowest write + space + read within a
          window &rarr; <b>read-heavy lookup</b> (point reads, latency-sensitive, space-constrained):
          leveled, paying 10&ndash;30× write amp to buy ≤1 file/level read amp and ~1.1× space; lean on
          Bloom filters and block cache for negative/point lookups &rarr; name the trade-offs: STCS
          risks running out of disk during merges and slower reads; LCS risks write stalls if the
          "read-heavy" table actually takes heavy writes &rarr; monitor L0/pending-compaction and p99
          read per table, and re-tune when a table's real workload diverges from the assumption &mdash;
          sustainable write rate is bounded by compaction throughput, so size compaction I/O to the
          ingest table's rate.
    - level: 'design'
      q: 'Design deletes + compaction for a Cassandra time-series store that must honor data-retention (TTL) and GDPR-style erasure.'
      a: >-
          <b>A strong answer covers:</b> the workload is append-only, time-ordered, with TTL expiry and
          occasional hard deletes &mdash; that points at <b>TWCS</b>, bucketing SSTables by time window
          so whole expired windows drop in one operation instead of rewriting live data &rarr; model
          the partition/clustering key on time so a window maps cleanly to a bucket and old data ages
          out as a unit &rarr; deletes/expiry produce <b>tombstones</b>; the resurrection rule still
          binds &mdash; a tombstone must outlive every older copy, so set <code>gc_grace_seconds</code>
          long enough for the delete to reach every replica (default 10d) before GC, balanced against
          not letting tombstones pile up &rarr; avoid tombstone-heavy access (no
          queue/range-scan-over-deleted patterns) since reads scan tombstones and hit
          <code>tombstone_failure_threshold</code> &rarr; for GDPR erasure, ensure the delete
          propagates and is actually purged after the older SSTables compact away &mdash; TTL alone
          shadows but doesn't immediately reclaim, so verify space returns after compaction &rarr; name
          the trade-offs: TWCS gives near-free retention drops but is wrong if the table takes
          out-of-window updates or random deletes; tuning <code>gc_grace_seconds</code> trades
          resurrection safety against tombstone read cost and erasure latency.

        <h2>The job &amp; the three amplifications</h2>
    </div>

    <div class="facts">
        <div class="fact">
            <b>The job</b> Reclaim space from dead records (overwrites, deletes) and consolidate a

        </div>
        <div class="fact">
            <b>Why it exists</b> The LSM path is append-only — you can't update in place, so garbage

            flash endurance.
        </div>
        <div class="fact blue">
            <b>Read / Space amp</b> Files touched per logical read; and disk bytes ÷ live bytes. One
            read latency, one storage bill.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Compaction is just background cleanup; tune it later"</div>
        <div class="t">✓ It is THE policy choice — it sets read latency, write ceiling, and disk cost</div>
    </div>

    <div class="rule">
        <span>⚖️ Iron law:</span> Lowering one amplification raises another — there is no free
        strategy, only one matched to your workload.{' '}

<section class="lvl" id="l2">
    <div class="lvltop">
        <span class="num">2</span>
        <h2>Size-tiered (STCS) — the write-optimized corner</h2>
    </div>
    <p class="tagline">
        Bigtable's original, Cassandra's historical default. Group SSTables by size; merge a batch of
        same-size tables into one of the next size up.
    </p>

    <Fig3 />

    <div class="facts">
        <div class="fact">
            <b>Write amp · LOW</b> A key is rewritten once per tier it passes, tiers grow geometrically
            → ~<code>log(N)</code> total. Writes are cheap.
        </div>
        <div class="fact">
            <b>Space amp · HIGH</b> A merge needs room for inputs + output at once (~2× worst case);
            stale copies linger across un-merged tiers.
        </div>
        <div class="fact blue">
            <b>Read amp · MED–HIGH</b> A point lookup may consult one (or more) SSTable per tier; a key
            that exists can be chased through several.
        </div>
        <div class="fact blue">
            <b>Reach for it</b> Write-heavy ingest, disk to spare. The write-optimized corner of the
            triangle.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "STCS is the safe default — it's cheap"</div>
        <div class="t">
            ✓ A single big merge can momentarily double disk; clusters have run out of disk because of
            compaction
        </div>
    </div>

    <div class="rule">
        <span>📥 Memory rule:</span> Size-tiered = merge same-size tables upward; cheap writes (~log
        N), but high space (~2× spikes) and duplicate keys across tiers.
    </div>

    <MemoryCheck items={mc2} />

</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>Leveled (LCS) — the read-and-space-optimized corner</h2>
    </div>
    <p class="tagline">
        LevelDB's design; RocksDB's &amp; modern Cassandra's choice for read-heavy data. Levels
        L0,L1,L2… each ~<code>T</code>× (≈10) the last.
    </p>

    <Fig4 />

    <figure>
        <img
            src="diagrams/dd-01-lsm-compaction-strategies.png"
            alt="Size-tiered vs leveled compaction. Left: STCS merges same-size tables into bigger tiers (cheap writes, lots of space, a key may live in several tiers). Right: LCS keeps each level non-overlapping and ~10× the last, merging one table down at a time (cheap reads + space, but each key is rewritten many times)."
        />
        <figcaption>
            <strong>Size-tiered vs leveled, side by side.</strong> Left: STCS merges same-size tables
            into bigger tiers (cheap writes, lots of space, a key may live in several tiers). Right:
            LCS keeps each level non-overlapping and ~10× the last, merging one table down at a time
            (cheap reads + space, but each key rewritten many times).
        </figcaption>
    </figure>

    <table>
        <tr>
            <th>Dimension</th>
            <th>Size-tiered (STCS)</th>
            <th>Leveled (LCS)</th>
        </tr>
        <tr>
            <td>Groups by</td>
            <td>Size of SSTable</td>
            <td>Level (each ~10× the last)</td>
        </tr>
        <tr>
            <td>Overlap</td>
            <td>Tables can overlap</td>
            <td>Non-overlapping below L0</td>
        </tr>
        <tr>
            <td>Write amp</td>
            <td>Low (~log N)</td>
            <td>High (10–30×)</td>
        </tr>
        <tr>
            <td>Read amp</td>
            <td>Med–high</td>
            <td>Low (≤1 file/level)</td>
        </tr>
        <tr>
            <td>Space amp</td>
            <td>High (~2× spikes)</td>
            <td>Low (~1.1×)</td>
        </tr>
    </table>

    <div class="myth">
        <div class="m">✗ "Leveled is strictly better — it's the modern default"</div>
        <div class="t">
            ✓ It buys low read + space by paying 10–30× write bandwidth; wrong for write-heavy ingest
        </div>
    </div>

    <div class="rule">
        <span>📤 Memory rule:</span> Leveled = non-overlapping levels, ~10× each. Read amp ≤1
        file/level, space ~1.1×, but write amp ~T·log_T(N) = 10–30×.
    </div>

    <MemoryCheck items={mc3} />

</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>The trade-off triangle — pick two of three</h2>
    </div>
    <p class="tagline">
        Put both strategies on one picture and RUM gets concrete: minimise any two of
        read/write/space and the third suffers.
    </p>

    <figure>
        <img
            src="diagrams/dd-02-amplification-triangle.png"
            alt="The amplification trade-off. Read, write, and space amplification form a triangle; a compaction strategy is a point inside it. Leveled sits near low-read/low-space (paying write); size-tiered sits near low-write (paying read and space). You cannot reach all three corners at once."
        />
        <figcaption>
            <strong>The amplification trade-off.</strong> Read, write, and space form a triangle; a
            strategy is a point inside it. Leveled sits near low-read/low-space (paying write);
            size-tiered sits near low-write (paying read and space). You cannot reach all three
            corners at once.
        </figcaption>
    </figure>

    <table>
        <tr>
            <th>Strategy</th>
            <th>Write</th>
            <th>Read</th>
            <th>Space</th>
            <th>Reach for it when</th>
        </tr>
        <tr>
            <td>Size-tiered</td>
            <td>low</td>
            <td>med–high</td>
            <td>
                <b>high</b>
            </td>
            <td>write-heavy, disk to spare</td>
        </tr>
        <tr>
            <td>Leveled</td>
            <td>
                <b>high</b>

        * within a time window. <b>Hybrids</b> sit between corners: RocksDB <em>universal</em>{' '}

    </p>

        <div class="m">✗ "One global compaction setting is fine for the whole DB"</div>
        <div class="t">
            ✓ Senior move: choose per table / column family by access pattern — like choosing indexes

        </div>
    </div>

    <div class="rule">
        <span>🔺 Memory rule:</span> RUM: you may minimise two of {'{read, write, space}'} — never all
        three. Match the corner to the table's workload.
    </div>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Tombstones &amp; the resurrection bug</h2>
    </div>
    <p class="tagline">
        A delete can't erase in place — the data is in immutable SSTables. So a delete writes a{' '}
        <strong>tombstone</strong>: "this key is dead as of this sequence number."
    </p>

    <Fig5 />

    <figure>
        <img
            src="diagrams/dd-03-tombstone-resurrection.png"
            alt="Tombstone resurrection. A delete writes a tombstone, but an older SSTable still holds the value. Purge the tombstone before that old SSTable is compacted away, and the value reappears. The tombstone must survive until it has been compacted past all older copies."
        />
        <figcaption>
            <strong>Tombstone resurrection.</strong> A delete writes a tombstone, but an older SSTable
            still holds the value. Purge the tombstone before that old SSTable is compacted away, and
            the value reappears. The tombstone must survive until compacted past all older copies.
        </figcaption>
    </figure>

    <div class="facts">
        <div class="fact">
            <b>The rule</b> Purge a tombstone only after compaction has carried it where no older data
            for that key can remain — bottom level (LCS), or merged past every older tier (STCS).
        </div>
        <div class="fact">
            <b>Distributed twist</b> Cassandra holds tombstones for <code>gc_grace_seconds</code>{' '}
            (default 10 days) — the delete must reach every replica first, or read-repair/hinted-handoff
            re-shares the old value cluster-wide.
        </div>
        <div class="fact blue">
            <b>Read amp from tombstones</b> Tombstones are <em>data</em> until purged; a range read
            scans them all. A queue-like (write/read/delete) workload buries a partition in them.
        </div>
        <div class="fact blue">
            <b>The symptom</b> "My queue table times out reads" is, 9 times in 10, tombstone overload —
            Cassandra aborts past <code>tombstone_failure_threshold</code> (default 100,000).
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "A delete frees the space and removes the key immediately"</div>
        <div class="t">
            ✓ A delete WRITES a marker; the key + tombstone are dropped together only after compaction
            passes every older copy
        </div>
    </div>

    <div class="rule">
        <span>🪦 Memory rule:</span> A tombstone must outlive every older copy of its key — premature
        GC is a real, shipped data-loss-in-reverse bug.
    </div>

    <MemoryCheck items={mc5} />

</section>

{/* ================= LEVEL 6 ================= */}

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>L0, write stalls &amp; back-pressure</h2>
    </div>
    <p class="tagline">
        The operational reality that pages you. L0 takes raw flushes (overlapping), and it's the
        pressure gauge for the whole tree.
    </p>

    <Fig6 />

    <div class="facts">
        <div class="fact">
            <b>Why L0 is special</b> It takes raw memtable flushes that overlap → a read scans all of
            L0, and its file count is the pressure gauge for the tree.
        </div>
        <div class="fact">
            <b>Back-pressure</b> If writes outrun the drain rate, the engine deliberately slows then
            stops writes so compaction can catch up — that's the stall, not a failure.
        </div>
        <div class="fact blue">
            <b>RocksDB knobs</b> <code>level0_slowdown_writes_trigger</code> (throttle),{' '}
            <code>level0_stop_writes_trigger</code> (halt), and soft/hard{' '}
            <code>pending_compaction_bytes_limit</code> (by debt).
        </div>
        <div class="fact blue">
            <b>The fix</b> Never "write faster": give compaction more I/O (threads, faster disks), pick
            a cheaper-write strategy (tiered), or smooth the write rate.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Sudden write latency spikes mean the DB is just slow / broken"</div>
        <div class="t">
            ✓ It's back-pressuring: compaction fell behind L0 and the engine is throttling on purpose
        </div>
    </div>

    <div class="rule">
        <span>🚦 Deepest LSM truth:</span> Your sustainable write rate is bounded by your{' '}
        <span>compaction throughput</span>, not your disk's raw write speed.
    </div>

    <MemoryCheck items={mc6} />

</section>
