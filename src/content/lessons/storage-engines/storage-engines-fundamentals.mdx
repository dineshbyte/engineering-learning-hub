# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Walk me through what a storage engine does to turn "find me row 42" into disk I/O.'
      a: >-
          <b>Hit these points:</b> the engine consults an <em>index</em> to convert a scan into a seek
          &rarr; the mechanism depends on the engine: a B-tree walk root&rarr;leaf, a hash-map lookup
          to a byte offset, or memtable-then-SSTables for an LSM &rarr; without an index a read is a
          linear scan whose cost grows with data size &rarr; the unit of I/O is the <em>page</em> (~4&ndash;8
          KB), not the row, so the engine reads whole pages and the index is a small structure kept off
          to the side &rarr; the punchline: every index turns "search the data" into "consult a
          structure, then seek."
    - level: 'core'
      q: 'What is a clustered index versus a secondary index, and where does the row actually live?'
      a: >-
          <b>Hit these points:</b> a <b>clustered</b> index stores the full row <em>in the index
          leaf</em>, ordered by the key &mdash; InnoDB clusters on the primary key, so the table
          <em>is</em> the PK B-tree &rarr; a <b>secondary</b> index is a separate structure keyed on
          another column whose leaf holds only the key plus a pointer back to the row &rarr; that
          pointer differs by engine: InnoDB stores the <em>primary key</em> (so a secondary lookup
          walks two B-trees, secondary&rarr;PK&rarr;clustered), Postgres uses a heap and every index
          holds a <code>ctid</code> heap pointer &rarr; consequence: a wide PK bloats every secondary
          index in a clustered engine &rarr; a secondary index is pure redundancy you build, pay for on
          every write, and keep in sync.
    - level: 'core'
      q: 'What is a composite index on <code>(a, b)</code>, and which queries can use it?'
      a: >-
          <b>Hit these points:</b> it stores entries sorted lexicographically by <code>a</code> then
          <code>b</code> &rarr; so it serves any <em>left-anchored prefix</em>: <code>a=…</code> and
          <code>a=… AND b=…</code> &rarr; it does <em>not</em> serve <code>b=…</code> alone, because
          there is no left-anchored prefix to seek on &rarr; rule of thumb: put the equality column
          first and the range column second, since a range on the leading column "uses up" the sort
          order for everything after it &rarr; verify with <code>EXPLAIN</code> rather than guessing
          what the planner picks.
    - level: 'core'
      q: 'What is a write-ahead log (WAL), and why does a B-tree need one?'
      a: >-
          <b>Hit these points:</b> a WAL is an append-only log of <em>intent</em> written and fsync'd
          <em>before</em> the change is applied to the data pages &rarr; a B-tree updates pages in
          place, and overwriting a page is not crash-atomic &mdash; a torn write mid-update (or
          mid-split) can corrupt the page &rarr; the WAL records the change first so recovery can
          replay it and reach a consistent state &rarr; it also turns many random page writes into one
          sequential log write for durability, and engines add full-page images / doublewrite to
          survive torn writes &rarr; the durability contract: a commit is acknowledged once its WAL
          record is on stable storage, even if the dirty page is still in memory.
    - level: 'senior'
      q: 'A teammate says "LSM-trees are just faster than B-trees." Correct them.'
      a: >-
          <b>Hit these points:</b> it's not "faster," it's a different corner of the amplification
          triangle &rarr; an LSM has <em>low write amp now</em> (sequential appends, work deferred into
          compaction) but higher read amp (a point lookup may probe many sorted runs) and space amp
          (stale versions and tombstones until compacted) &rarr; a B-tree has low, predictable read amp
          (bounded by tree height) but random write I/O plus WAL + whole-page rewrites &rarr; latency
          profile differs too: B-tree is predictable, LSM is spiky under compaction stalls &rarr; so the
          answer is "depends on the workload": ingest-heavy &rarr; LSM, read-heavy/tail-latency-sensitive
          OLTP &rarr; B-tree.
    - level: 'senior'
      q: "Define read, write, and space amplification, and explain why you can't drive all three to zero."
      a: >-
          <b>Hit these points:</b> <b>read amp</b> = pages/files touched per logical read; <b>write
          amp</b> = bytes written to disk per byte of user data; <b>space amp</b> = disk bytes ÷ live
          data bytes &rarr; they trade off against each other: a B-tree keeps read amp ≈ 1 (the leaf)
          but pays write amp on every in-place page rewrite + WAL; an LSM keeps write amp low by
          deferring it, but pays read amp (many runs) and space amp (stale versions) until compaction
          runs &rarr; compaction itself converts space/read amp back into write amp &mdash; you're
          moving the cost, not removing it &rarr; this is the RUM conjecture: optimize any two of
          read/write/space and the third suffers &rarr; the skill is reading the workload and choosing
          which currency to spend.
    - level: 'senior'
      q: 'An InnoDB secondary-index lookup is slow. Explain the mechanics and a fix.'
      a: >-
          <b>Hit these points:</b> InnoDB is clustered, so the secondary leaf stores the <em>primary
          key</em>, not a direct row pointer &rarr; a secondary lookup therefore walks two B-trees:
          secondary index &rarr; PK value &rarr; clustered index to fetch the row (the "bookmark
          lookup") &rarr; a wide PK makes this worse because it bloats every secondary index and every
          lookup &rarr; fix: a <em>covering</em> index that includes the selected columns so the query
          is answered from the index alone (Index Only Scan), avoiding the second walk &rarr; or narrow
          the PK &rarr; confirm with <code>EXPLAIN</code> that you actually got the covering/index-only
          plan.
    - level: 'senior'
      q: 'What does the page / buffer cache do, and how does it change how you reason about read cost?'
      a: >-
          <b>Hit these points:</b> the buffer pool / page cache holds recently-used pages in RAM, so a
          "disk read" is often actually a memory hit &rarr; the real read cost is the <em>miss</em>
          rate, not the raw page count: a B-tree's upper levels stay cached, so a billion-row lookup is
          usually one uncached leaf read, not four disk reads &rarr; an LSM's read amp also softens once
          hot SSTable blocks and index/Bloom-filter blocks are cached &rarr; writes go through it too:
          dirty pages are buffered and flushed later, which is why durability rides on the WAL, not on
          the page being written &rarr; the senior framing: amplification numbers are worst-case/cold;
          the working-set fit in the cache is what dominates steady-state latency &rarr; watch the cache
          hit ratio, not just the plan.
    - level: 'staff'
      q: 'How do you choose between a B-tree (row store) and an LSM-tree for a new service, and how would you defend reversing that choice later?'
      a: >-
          <b>Hit these points:</b> start from the access pattern and SLO, not the brand: write:read
          ratio, point vs range reads, p99 latency target, dataset vs RAM size, and storage budget
          &rarr; map them onto the triangle &mdash; write-heavy ingest with disk to spare &rarr; LSM
          (sequential writes, low write amp); read-heavy, tail-latency-sensitive OLTP &rarr; B-tree
          (predictable, low read amp) &rarr; name the second-order costs: LSM compaction stalls and
          tunable strategy vs B-tree random-write ceiling and page slack &rarr; make it reversible: hide
          the engine behind a repository boundary, keep the data model engine-agnostic, and define the
          metric that would trigger a switch (e.g. compaction can't keep up &rarr; write stalls, or p99
          read latency from probing too many runs) &rarr; the staff move is committing to a default
          <em>and</em> the measured trip-wire that says you chose wrong.
    - level: 'staff'
      q: '"Add an index, the query''s slow" — make the principal-level case for when <em>not</em> to add one.'
      a: >-
          <b>Hit these points:</b> every secondary index is redundant data you maintain on <em>every</em>
          write &mdash; it taxes write throughput, write amp, and disk, and on a clustered engine it
          carries the PK in every entry &rarr; so an index is a read optimization paid for by all
          writers, and on a write-heavy table that tax can dwarf the read it saves &rarr; before adding
          one, ask: is the query hot enough and selective enough to matter? would an existing composite
          index serve it via a left-anchored prefix? is the planner ignoring an index that already
          exists (stale stats, non-sargable predicate)? &rarr; alternatives: rewrite the predicate, a
          covering index instead of two, a different data layout, or denormalize against a measured read
          pattern &rarr; the principal framing: indexes aren't free reads, they're a standing write
          liability &mdash; justify each one against the workload and measure with <code>EXPLAIN</code>,
          don't cargo-cult.
    - level: 'staff'
      q: 'When is a column store the right call over a row store, and what exactly are you trading?'
      a: >-
          <b>Hit these points:</b> the discriminator is the access pattern: row stores keep a record's
          columns contiguous (great for OLTP point reads/writes of whole rows), column stores keep each
          column contiguous (great for OLAP scans that touch few columns over many rows) &rarr; columnar
          wins because a scan reads only the columns it needs and homogeneous data compresses hard
          (run-length, dictionary, bit-packing), so you trade space amp for huge scan throughput &rarr;
          what you give up: single-row writes and updates are expensive (a row is scattered across
          column segments), so column stores favor bulk/append loads, not OLTP mutation &rarr; the
          synthesis: don't pick one &mdash; run OLTP on a row store and ETL into a columnar warehouse on
          a star schema, refreshed so analytics scans don't starve user I/O &rarr; justify each side by
          its access pattern and the amplification it spends.
    - level: 'design'
      q: 'Design the storage for a product with live order pages AND an analytics dashboard.'
      a: >-
          <b>A strong answer covers:</b> two access patterns, two layouts &mdash; don't force one engine
          to do both &rarr; OLTP path: a row store (B-tree/InnoDB, or LSM if ingest-heavy), normalized,
          indexed on the actual query predicates, with covering indexes for the hot read paths and a
          sensibly narrow clustered key &rarr; OLAP path: ETL/CDC the data into a column store /
          warehouse on a star schema (narrow fact table + dimension tables), refreshed periodically so
          heavy scans run on isolated resources &rarr; justify each choice by its access pattern and the
          amplification it spends (row store pays write amp for predictable reads; column store trades
          single-row write cost for compressed, fast scans) &rarr; keep them decoupled so a dashboard
          query can't starve order-page I/O &rarr; name the freshness trade-off: live OLTP vs
          periodically-refreshed analytics, and how stale the dashboard is allowed to be.
    - level: 'design'
      q: 'Design the storage engine + indexing for a write-heavy event-ingest service that also needs fast recent-event reads.'
      a: >-
          <b>A strong answer covers:</b> start from the workload &mdash; very high sustained write rate,
          mostly append, reads skewed to recent data &rarr; that points at an LSM-tree: turn every write
          into a sequential append (WAL + memtable), flush immutable SSTables, defer cleanup to
          compaction &mdash; low write amp is exactly what an ingest firehose needs &rarr; protect read
          latency with Bloom filters (skip SSTables that definitely lack a key) and a
          partitioning/clustering key on time so recent events sit in the freshest, smallest runs &rarr;
          choose compaction by sub-pattern: size-tiered for the raw write-heavy table (cheap writes,
          accept space + read amp), or time-windowed if events carry TTLs so whole old buckets drop at
          once &rarr; size the buffer cache to the recent working set so hot reads stay in RAM &rarr;
          durability via WAL fsync policy and replication &rarr; the trip-wire to monitor: write stalls
          when L0/pending-compaction crosses the back-pressure trigger &mdash; sustainable write rate is
          bounded by compaction throughput, so the fix is more compaction I/O or a cheaper-write
          strategy, never "write faster" &rarr; name the trade-off: LSM buys ingest throughput by paying
          read + space amp on cold data, which is acceptable because reads target recent (hot) events.

        <h2>Log + hash index — the simplest real engine</h2>
    </div>
    <p class="tagline">
        Two lines of bash become a durable database — and reveal the tension that governs every
        index you'll ever choose.
    </p>
    <Fig2 />
    <figure>
        <img
            src="diagrams/01-log-hash-index.png"
            alt="An append-only log on disk with an in-memory hash index pointing into it. Reads follow the index straight to a record; writes append at the end and repoint the index."
        />
        <figcaption>
            The same idea from the book: writes stream to the tail; the hash map turns "search the
            data" into "consult a small structure, then seek."
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>Is</b> An append-only log of <code>key,value</code> records + an in-RAM map from each
            key to its latest byte offset.
        </div>
        <div class="fact">
            <b>Why it exists</b> Appends are the fastest write a disk offers — one position, then
            stream. The index converts a scan into a seek.
        </div>
        <div class="fact blue">
            <b>Compaction</b> Segments close at a size cap, become immutable, and a background merge
            keeps only the latest value per key.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> A <code>SELECT … LIMIT 1</code> over a tail-appended file, but the
            offset is precomputed in a hash map.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "An append-only log keeps every dead version forever"</div>
        <div class="t">
            ✓ Compaction rewrites a fresh segment, swaps it atomically, deletes the old — crash-safe
            by immutability
        </div>
    </div>
    <div class="rule">
        <span>📜 Memory rule:</span> A storage engine is a log you write fast and an index you read
        fast — and every index you add taxes every write.
    </div>
    <MemoryCheck items={mc1} />
</section>

{/* ================= LEVEL 2 ================= */}

<section class="lvl" id="l2">
    <div class="lvltop">
        <span class="num">2</span>
        <h2>B-tree — the read-optimized default</h2>
    </div>
    <p class="tagline">
        The structure behind every <code>CREATE INDEX</code> since 1972: a shallow, sorted,
        page-oriented tree you update in place.
    </p>
    <figure>
        <img
            src="diagrams/02-btree.png"
            alt="A three-level B-tree with the lookup path for key 215 highlighted. The root page splits the key space, internal pages narrow it, and a single leaf page holds the value — three page reads to reach any row."
        />
        <figcaption>
            Lookup of key 215: root <code>[200,400]</code> → internal <code>[210,230,260]</code> →
            leaf. Three page reads, each a binary search in memory.
        </figcaption>
    </figure>
    <Fig3 />
    <div class="facts">
        <div class="fact">
            <b>Is</b> A balanced tree of fixed-size pages (4–8 KB); each node is sorted keys + child
            pointers; all leaves at equal depth.
        </div>
        <div class="fact">
            <b>Why it exists</b> A high branching factor keeps the tree shallow, so a point lookup
            is O(log n) <em>pages</em> — and range scans walk the sorted leaves.
        </div>
        <div class="fact blue">
            <b>Update-in-place</b> One canonical location per key; a write overwrites the leaf page.
            A full leaf splits and pushes a separator up.
        </div>
        <div class="fact blue">
            <b>Crash safety</b> Overwriting a page isn't atomic, so the WAL records intent first;
            recovery replays it.
            <sup><a href="#s1" id="r1b">1</a></sup>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "More keys = a much deeper, slower tree"</div>
        <div class="t">
            ✓ Depth grows logarithmically — fan-out ~256 means 4 levels hold ~4 billion keys
        </div>
    </div>
    <div class="rule">
        <span>🌳 Memory rule:</span> A B-tree is sorted pages updated in place — cheap predictable
        reads, paid for with random write I/O and WAL.
    </div>
    <MemoryCheck items={mc2} />
</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>LSM-tree — write-optimized, log all the way down</h2>
    </div>
    <p class="tagline">
        Turn <em>every</em> write into a sequential append; defer the cleanup to a background merge.
        The engine behind RocksDB &amp; Cassandra.
    </p>
    <figure>
        <img
            src="diagrams/03-lsm-tree.png"
            alt="The LSM-tree write and read paths on one page. Writes enter the in-memory memtable and flush to a new immutable SSTable; compaction merges older SSTables and drops shadowed keys; a read checks the memtable, passes a Bloom-filter gate, then scans SSTables newest to oldest."
        />
        <figcaption>
            Write → WAL + memtable; flush a full memtable as a new immutable SSTable; compaction
            merges runs; reads go memtable → SSTables newest-to-oldest, gated by Bloom filters.
            <sup><a href="#s3" id="r3">3</a></sup>
        </figcaption>
    </figure>
    <Fig4 />
    <div class="facts">
        <div class="fact">
            <b>Memtable</b> In-RAM sorted map (skip list / RB-tree) that buffers writes; a WAL
            alongside makes it crash-safe.
        </div>
        <div class="fact">
            <b>SSTable</b> Immutable on-disk file sorted by key — never modified, only merged away.
            Two sorted files merge in one linear pass.
        </div>
        <div class="fact blue">
            <b>Updates &amp; deletes are writes too</b> An update appends a newer entry; a delete
            appends a <em>tombstone</em>. Old values are shadowed, not erased.
        </div>
        <div class="fact blue">
            <b>Compaction</b> Merges runs, keeps the newest value per key, drops tombstoned keys —
            reclaiming space. Leveled vs size-tiered is a tunable knob.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "A delete frees the space immediately"</div>
        <div class="t">
            ✓ A tombstone shadows the value; space returns only when compaction reaches the oldest
            SSTable (Cassandra: <code>gc_grace_seconds</code>)
        </div>
    </div>
    <div class="rule">
        <span>🪵 Memory rule:</span> An LSM-tree buffers in RAM, flushes immutable sorted runs, and
        merges later — sequential writes now, read &amp; merge work deferred.
    </div>
    <MemoryCheck items={mc3} />
</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>Amplification — the three currencies you trade</h2>
    </div>
    <p class="tagline">
        Read, write, space: a logical op costing more than one physical op. You can't drive all
        three to zero — pick a corner.
    </p>
    <figure>
        <img
            src="diagrams/04-amplification.png"
            alt="B-tree and LSM-tree compared across the three amplification axes. The grid shows where each engine spends and where each engine saves; the insets show why."
        />
        <figcaption>
            The amplification grid from the book: where each engine spends, where each saves. B-tree
            pays write; LSM pays read &amp; space-until-compacted.
            <sup><a href="#s1" id="r1c">1</a></sup>
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Axis</th>
            <th>B-tree</th>
            <th>LSM-tree</th>
        </tr>
        <tr>
            <td>Read amp</td>
            <td>low — bounded by tree height</td>
            <td>medium–high — may probe many runs</td>
        </tr>
        <tr>
            <td>Write amp</td>
            <td>medium–high — in-place, random, WAL + page</td>
            <td>
                low <em>now</em> — deferred into compaction
            </td>
        </tr>
        <tr>
            <td>Space amp</td>
            <td>low — one copy, page slack</td>
            <td>medium — stale versions until compacted</td>
        </tr>
        <tr>
            <td>Write I/O</td>
            <td>random</td>
            <td>sequential</td>
        </tr>
        <tr>
            <td>Latency</td>
            <td>predictable</td>
            <td>spiky (compaction stalls)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Read amp</b> Pages touched per logical read. B-tree ≈ 1 uncached (the leaf); LSM
            grows with the number of sorted runs.
        </div>
        <div class="fact">
            <b>Write amp</b> Bytes written per byte of user data. B-tree: WAL + whole-page rewrite +
            splits. LSM: defers &amp; batches into big sequential rewrites.
        </div>
        <div class="fact blue">
            <b>Space amp</b> Disk used vs logical data. B-tree: page slack (~60–70% full). LSM:
            stale versions + tombstones, often offset by compression.
        </div>
        <div class="fact blue">
            <b>Choose by workload</b> Write-heavy ingest → LSM. Read-heavy, tail-latency-sensitive
            OLTP → B-tree.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "LSM-trees have no write amplification"</div>
        <div class="t">
            ✓ They <em>defer and batch</em> it into compaction — a key may be rewritten across
            several levels over its life
        </div>
    </div>
    <div class="rule">
        <span>⚖️ Memory rule:</span> Read, write, space — every engine picks one corner of the
        triangle and pays in the other two. The skill is reading the workload.
    </div>
    <MemoryCheck items={mc4} />
</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Indexes — clustered vs secondary, and the write tax</h2>
    </div>
    <p class="tagline">
        One set of rows, many access paths. Every secondary index is pure redundancy you build, pay
        for, and keep in sync.
    </p>
    <figure>
        <img
            src="diagrams/05-secondary-index.png"
            alt="A table heap holds rows by row id; a clustered primary-key index stores full rows in its leaves, while a secondary email index stores only key plus a pointer back to the row."
        />
        <figcaption>
            Clustered (InnoDB): the row lives in the PK leaf, secondaries store the PK. Heap
            (Postgres): rows in a heap, every index holds a <code>ctid</code> pointer.
        </figcaption>
    </figure>
    <Fig5 />
    <div class="facts">
        <div class="fact">
            <b>Clustered</b> The PK B-tree leaf <em>is</em> the row (InnoDB). A secondary stores the
            PK value → secondary lookup walks <em>two</em> trees.
        </div>
        <div class="fact">
            <b>Heap</b> Rows sit unordered in a heap; every index leaf holds a <code>ctid</code>{' '}
            pointer (Postgres). PK lookup = index walk + heap fetch.
        </div>
        <div class="fact blue">
            <b>Composite</b> Keyed on several columns, sorted lexicographically. Usable only for a
            left-anchored prefix (leftmost-prefix rule).
        </div>
        <div class="fact blue">
            <b>Covering</b> Holds every column the query reads, so the answer comes from the index
            alone — no row fetch.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Add indexes freely — they only make queries faster"</div>
        <div class="t">
            ✓ A table with 8 indexes turns 1 logical insert into 9 physical writes; each index is a
            standing write tax
        </div>
    </div>
    <div class="rule">
        <span>🔎 Memory rule:</span> An index makes reads it covers faster and every write that
        maintains it slower — index what you actually filter and sort by, and measure with{' '}
        <code>EXPLAIN</code>.
    </div>
    <MemoryCheck items={mc5} />
</section>

{/* ================= LEVEL 6 ================= */}

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Data models — relational, document, graph</h2>
    </div>
    <p class="tagline">
        The most consequential, hardest-to-change backend decision. Each model makes a{' '}
        <em>different</em> relationship cheap.
    </p>
    <figure>
        <img
            src="diagrams/06-data-models.png"
            alt="The same data — a user, their posts, their friends — modeled three ways: three relational tables wired by foreign keys, one nested JSON document, and a property graph of nodes and directed edges."
        />
        <figcaption>
            The same user-posts-friends domain modeled three ways: relational tables joined by
            value, one embedded document, a property graph of traversable edges.
            <sup><a href="#s4" id="r4">4</a></sup>
        </figcaption>
    </figure>
    <Fig6 />
    <div class="facts">
        <div class="fact">
            <b>Relational</b> Rows relate <em>by value</em> (foreign key); a join assembles them at
            query time. No duplication, referential integrity, ad-hoc queries.
        </div>
        <div class="fact">
            <b>Document</b> One self-contained JSON/BSON tree embedding related data — great
            locality, eases the object-relational impedance mismatch.
        </div>
        <div class="fact blue">
            <b>Graph</b> Nodes + labeled directed edges; an edge is a traversable pointer
            (index-free adjacency), so a hop is O(local fan-out), not a fresh search.
        </div>
        <div class="fact blue">
            <b>Convergence</b> Postgres <code>jsonb</code> stores an indexable document in a
            relational row — the choice is increasingly per-field.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Documents are modern / graphs are cool, so default to them"</div>
        <div class="t">
            ✓ Each model is a locality bet; relational stays the safe default when access patterns
            are unknown (Codd separated logical from physical)
        </div>
    </div>
    <div class="rule">
        <span>🗂️ Memory rule:</span> Pick the model whose <em>cheap</em> operation is what your app
        reads and traverses most — not the trendiest one.
    </div>
    <MemoryCheck items={mc6} />
</section>

{/* ================= LEVEL 7 ================= */}

<section class="lvl" id="l7">
    <div class="lvltop">
        <span class="num">7</span>
        <h2>Normalization vs denormalization — where each fact lives</h2>
    </div>
    <p class="tagline">
        Store a fact once and the database protects it. Copy it for speed and you inherit the job of
        keeping every copy honest.
    </p>
    <figure>
        <img
            src="diagrams/07-normalization.png"
            alt="The same data, two layouts. Normalized stores each customer fact once and joins on read; denormalized embeds a copy of the customer into every order for fast reads but must fan-out updates."
        />
        <figcaption>
            Normalized: customer stored once, joined on read. Denormalized: customer copied into
            every order — single-row reads, but a change fans out.
        </figcaption>
    </figure>
    <Fig7 />
    <table>
        <tr>
            <th></th>
            <th>Normalized</th>
            <th>Denormalized</th>
        </tr>
        <tr>
            <td>Each fact</td>
            <td>stored once</td>
            <td>stored many times</td>
        </tr>
        <tr>
            <td>Read "order + customer"</td>
            <td>join</td>
            <td>single row</td>
        </tr>
        <tr>
            <td>Address change</td>
            <td>one update</td>
            <td>fan-out update</td>
        </tr>
        <tr>
            <td>Owns consistency</td>
            <td>the database</td>
            <td>you (or a materialized view)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Normalization</b> Each independent fact in exactly one row; everything else
            references by key. Consistency by construction (1NF→2NF→3NF).
        </div>
        <div class="fact">
            <b>Denormalization</b> Deliberately copy a fact into the rows that read it — join-free
            reads, paid for with fan-out write sync.
        </div>
        <div class="fact blue">
            <b>Schema-on-write</b> Relational: columns/types/constraints validated on the way in.
        </div>
        <div class="fact blue">
            <b>Schema-on-read</b> Document/lake: bytes written as-is, structure interpreted by the
            reading code — the schema just lives in that code.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Schema-on-read means schemaless"</div>
        <div class="t">
            ✓ The schema still exists — it's implicit in the reader and can silently drift between
            versions
        </div>
    </div>
    <div class="rule">
        <span>📐 Memory rule:</span> Start normalized (correct by construction); denormalize{' '}
        <em>specifically</em>, against a measured read pattern, knowing exactly which invariant you
        just took ownership of.
    </div>
    <MemoryCheck items={mc7} />
</section>

{/* ================= LEVEL 8 ================= */}

<section class="lvl" id="l8">
    <div class="lvltop">
        <span class="num">8</span>
        <h2>Encoding &amp; evolution — bytes that outlive their writer</h2>
    </div>
    <p class="tagline">
        An in-memory object is a graph of pointers meaningful only in one process. To store or send
        it, you flatten it — and then change its shape for years without breaking old readers.
    </p>
    <figure>
        <img
            src="diagrams/08-schema-evolution.png"
            alt="Both old and new code keep working across a schema change when the rules are followed. A v2 schema adds an optional field; the two crossing arrows show each version reading the other's data and succeeding."
        />
        <figcaption>
            The two crossing arrows: backward (new code reads old data) and forward (old code reads
            new data) compatibility — what makes rolling deploys safe.
            <sup><a href="#s5" id="r5">5</a></sup>
        </figcaption>
    </figure>
    <Fig8 />
    <table>
        <tr>
            <th></th>
            <th>Field names in bytes</th>
            <th>Size</th>
            <th>Needs schema to read</th>
        </tr>
        <tr>
            <td>JSON / XML</td>
            <td>yes, every record</td>
            <td>larger</td>
            <td>no</td>
        </tr>
        <tr>
            <td>Protobuf / Thrift</td>
            <td>no — tag numbers</td>
            <td>smaller</td>
            <td>yes</td>
        </tr>
        <tr>
            <td>Avro</td>
            <td>no identifiers at all</td>
            <td>smallest</td>
            <td>yes (writer + reader)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Encoding</b> Flatten a pointer-laced object graph into a self-contained byte
            sequence; decoding rebuilds it in a possibly different process, months later.
        </div>
        <div class="fact">
            <b>Text vs binary</b> JSON ships field names + decimal text (readable, schema-free).
            Protobuf/Avro ship tagged values (compact, fast, need a schema).
        </div>
        <div class="fact blue">
            <b>Backward compat</b> Newer code reads older data — new fields are absent, filled by
            default.
        </div>
        <div class="fact blue">
            <b>Forward compat</b> Older code reads newer data — it skips the unknown tag and parses
            the rest. This is why rolling deploys are safe.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "A new required field is fine if the schema declares it"</div>
        <div class="t">
            ✓ The moment one producer omits it, every old consumer breaks — proto3 removed{' '}
            <code>required</code> for exactly this
        </div>
    </div>
    <div class="rule">
        <span>🔁 Memory rule:</span> Encoding choices at write time constrain every reader for the
        data's lifetime — which far outlives any one service version. Keep both directions
        compatible.
    </div>
    <MemoryCheck items={mc8} />
</section>

<section class="lvl" id="l9">
    <div class="lvltop">
        <span class="num">9</span>
        <h2>Row vs column — match the layout to the workload</h2>
    </div>
    <p class="tagline">
        The capstone. OLTP wants whole rows by key; OLAP scans millions of rows but a handful of
        columns. The bytes reorganize entirely.
    </p>
    <figure>
        <img
            src="diagrams/09-row-vs-column.png"
            alt="The same six-column table stored row-wise versus column-wise. Row storage: each row is a contiguous run of all six cells; a scan of one column must touch every block. Column storage: six separate column blocks, so an analytic query reads only the two it names."
        />
        <figcaption>
            Row store: all of a row's cells adjacent — perfect for "fetch order #4821". Column
            store: all values of one column adjacent — scan reads only the columns named.
            <sup><a href="#s7" id="r7">7</a></sup>
        </figcaption>
    </figure>
    <Fig9 />
    <table>
        <tr>
            <th></th>
            <th>OLTP</th>
            <th>OLAP</th>
        </tr>
        <tr>
            <td>Read pattern</td>
            <td>few rows, by key</td>
            <td>millions of rows, scan</td>
        </tr>
        <tr>
            <td>Columns touched</td>
            <td>all (whole record)</td>
            <td>2–5 of many</td>
        </tr>
        <tr>
            <td>Write pattern</td>
            <td>random inserts/updates</td>
            <td>bulk load / append</td>
        </tr>
        <tr>
            <td>Bottleneck</td>
            <td>seek latency</td>
            <td>scan bandwidth</td>
        </tr>
        <tr>
            <td>Layout</td>
            <td>row store (B-tree / LSM)</td>
            <td>column store (Parquet, Vertica)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Row store</b> All of a row's values adjacent. The unit you store together is the unit
            you retrieve together — ideal OLTP locality.
        </div>
        <div class="fact">
            <b>Column store</b> All values of one column adjacent in its own block; a scan reads
            only the columns it names, bounded by columns-touched not table-width.
        </div>
        <div class="fact blue">
            <b>Compression</b> A column is one type from one domain → run-length / dictionary /
            bitmap encoding compress far harder than a heterogeneous row.
        </div>
        <div class="fact blue">
            <b>Star schema</b> Warehouses ETL into a column store: a narrow central fact table (one
            row per event) + small dimension tables radiating out.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "There's one best storage engine"</div>
        <div class="t">
            ✓ Only an engine that fits an access pattern — the same logical table can (and at scale
            should) live in both a row and a column store
        </div>
    </div>
    <div class="rule">
        <span>🏛️ Memory rule:</span> Match the storage layout to the workload — many small key reads
        → row; few wide scans → column. The senior skill is reading the workload, not memorizing
        engines.
    </div>
    <MemoryCheck items={mc9} />
</section>

<h2 class="sec" id="quiz">
    Retrieval practice — mix the levels

<h2 class="sec" id="cheat">
    ★ Cheat sheet — name → mechanism
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> Log = the fast write · Index = the fast read · B-tree =
        sorted pages in place · LSM = log + merge · Amplification = the three currencies · Clustered
        = row in the PK leaf · Heap = row + pointer · Covering = answer from the index alone ·
        Normalize = fact stored once · Denormalize = fact copied · Encoding = flatten the object ·
        Column store = layout for the scan.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Term</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>Storage engine</td>
            <td>The layer that lays bytes on disk and finds them again</td>
        </tr>
        <tr>
            <td>Compaction / VACUUM</td>
            <td>Background rewrite reclaiming stale versions &amp; tombstones</td>
        </tr>
        <tr>
            <td>Write amplification</td>
            <td>Bytes physically written per byte of user data</td>
        </tr>
        <tr>
            <td>Tombstone</td>
            <td>An appended delete marker; space returns at compaction</td>
        </tr>
        <tr>
            <td>Bloom filter</td>
            <td>Per-SSTable bit array: "definitely absent" / "probably present"</td>
        </tr>
        <tr>
            <td>Covering index</td>
            <td>Index holding every column the query reads → no row fetch</td>
        </tr>
        <tr>
            <td>Schema-on-read</td>
            <td>Structure interpreted by the reading code, not validated on write</td>
        </tr>
        <tr>
            <td>Forward compatibility</td>
            <td>Old code reads new data by skipping unknown fields</td>
        </tr>
        <tr>
            <td>Star schema</td>
            <td>Narrow fact table + small dimension tables, for column-store OLAP</td>
        </tr>
    </table>
    <h3>Three rules for the architect chair</h3>
    <p style="margin-bottom:0">
        ① There is no best engine — <strong>match the layout to the workload</strong> (read vs write
        vs scan). ② Every index is a <strong>read win bought with a write tax</strong>; index
        deliberately, measure with <code>EXPLAIN</code>. ③ Start <strong>normalized</strong>;
        denormalize against a measured read pattern, knowing which invariant you took on.
    </p>
</div>
