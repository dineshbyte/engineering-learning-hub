# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Why is evaluation called the real moat for an AI product?'
      a: >-
          <b>Hit these points:</b> the model is a <b>commodity</b> — any competitor can call the same
          API &rarr; what you can't copy is a curated dataset of what "good" means for <b>your</b> task
          and a measured baseline &rarr; <b>you cannot improve what you cannot measure</b>: without a
          score, every change is a guess and progress is unprovable &rarr; "vibes-based" eval works at 5
          examples and collapses at scale, where a small regression is thousands of bad answers no human
          will eyeball &rarr; evaluation turns model changes into measured before/after diffs — the only
          way to improve reliably.
    - level: 'core'
      q: 'What is the difference between offline and online evaluation?'
      a: >-
          <b>Hit these points:</b> <b>offline</b> = score the system against a FIXED dataset (the golden
          set) BEFORE shipping; reproducible, runs in CI, gates a deploy — same inputs every run, like
          an integration test suite &rarr; <b>online</b> = measure quality on LIVE production traffic
          AFTER shipping; real distribution, real users &rarr; online has two shapes: <b>A/B</b> (split
          real users, they see the variant, you compare outcomes) and <b>shadow</b> (run the new system
          in parallel but discard output — no user risk, no live signal) &rarr; they're complementary:
          offline is repeatable but only covers what you captured; online is real but can't gate a
          deploy that hasn't happened.
    - level: 'core'
      q: 'What is a golden set and how is it different from a benchmark?'
      a: >-
          <b>Hit these points:</b> a <b>golden set</b> is a versioned, curated set of (input,
          expected-output / acceptance-criteria) cases that is YOUR ground truth for offline eval —
          small, hand-checked, treated like test fixtures, changed only by deliberate review &rarr; a
          <b>benchmark</b> is a shared, public dataset + metric used to compare models against EACH
          OTHER (MT-Bench, HELM scenarios) &rarr; the benchmark is a standardized exam: good for
          cross-model comparison, weak as a proxy for your task &rarr; benchmarks rot via
          <b>contamination</b> (test data leaked into training = memorization, not capability) &rarr; a
          fresh private golden set beats a public leaderboard for trusting a result on your problem.
    - level: 'core'
      q: 'What are the three ways to score an eval case, and when do you use each?'
      a: >-
          <b>Hit these points:</b> <b>programmatic / exact-match</b> — deterministic checks (regex, JSON
          schema, equals-expected, tool-succeeded); cheapest and most reliable, use whenever the output
          is structured or has one right answer &rarr; <b>human grading</b> — a person applies a rubric;
          the gold standard and the source of your gold labels, but slow and expensive, so reserve it
          for the golden set and audits &rarr; <b>LLM-as-judge</b> — an LLM scores or compares against a
          rubric; scales cheaply for open-ended quality, but biased and must be calibrated against human
          labels first &rarr; the stack: programmatic where you can, judge where you must, human to
          anchor both.
    - level: 'senior'
      q: 'Offline eval passes but production users complain. Diagnose.'
      a: >-
          <b>Hit these points:</b> the golden set only covers the distribution you thought to capture;
          live traffic drifts and hits inputs you never fixtured — green offline + unhappy users is a
          <b>coverage gap</b> &rarr; confirm by sampling the complaints and checking whether those
          inputs even exist in the golden set (usually they don't) &rarr; add online signals:
          thumbs-down rate, regenerate rate, escalation-to-human, task completion, plus a shadow or A/B
          comparison of old vs new on real traffic &rarr; close the loop: harvest the failing live
          cases, label them, fold them into the golden set so the next offline run catches that class
          &rarr; the fix isn't "trust offline less" — offline and online cover different risks and you
          need both.
    - level: 'senior'
      q: 'When do you use shadow deployment vs an A/B test for online evaluation?'
      a: >-
          <b>Hit these points:</b> <b>shadow</b> = run the new system on real traffic in parallel and
          THROW THE OUTPUT AWAY; users never see it, so zero user risk — use it when the change is
          risky, hard to roll back, or you only need to compare behavior, not measure conversion &rarr;
          <b>A/B</b> = split real users between old and new; users SEE the variant, so you get a real
          outcome signal (conversion, completion, satisfaction) but you've exposed some users to a
          possibly-worse experience &rarr; the trade: shadow gives safety but no business signal; A/B
          gives the business signal at real user cost &rarr; common sequence: shadow first to confirm no
          regressions, then A/B to measure the real lift.
    - level: 'senior'
      q: 'Your LLM-as-judge prefers the new prompt. Should you trust it? Argue both sides.'
      a: >-
          <b>Hit these points:</b> the judge is a model with documented biases — <b>position,
          verbosity, self-preference</b> — so a preference can be an artifact, not real quality &rarr;
          trust it ONLY to the degree you've calibrated it: measure agreement against a human-labeled
          slice; a strong judge can reach roughly human-level agreement on <em>some</em> tasks, but
          that's dataset-dependent and must be re-verified &rarr; mitigations before trusting a gate:
          randomize order, force a rubric/CoT, use a different model family as judge, include reference
          answers &rarr; synthesis: the judge is a cheap noisy sensor that lets you scale human judgment
          to thousands of cases — use it to flag, confirm close calls with humans, never let an
          uncalibrated judge alone decide a ship.
    - level: 'senior'
      q: 'A teammate wants to grow the golden set by dumping in every production log line. Push back.'
      a: >-
          <b>Hit these points:</b> the golden set is <b>fixtures</b>, not a data lake — its value is
          that every case is representative, labeled, and hand-checked, so a score on it means something
          &rarr; raw logs are unlabeled, redundant, and noisy; dumping them in inflates the suite, slows
          every run, and dilutes the signal without adding coverage &rarr; it also breaks the "changed
          only by deliberate review" property — silent growth means you no longer know what your
          baseline measures &rarr; the right move: <b>sample</b> from logs, especially failures and edge
          cases, <b>label</b> them, and add the ones that cover a class you don't already test &rarr;
          quality and coverage of the set beat raw size every time.
    - level: 'staff'
      q: 'Map the full eval loop end to end: from a failing user report to a regression test that protects against it.'
      a: >-
          <b>Hit these points:</b> <b>collect</b> — gather representative cases, including failures
          harvested from production &rarr; <b>score</b> — apply the right grader per case (programmatic
          / human / LLM-judge) &rarr; <b>diff</b> — compare the aggregate score against the baseline to
          see better or worse &rarr; <b>decide</b> — ship if it's a measured win, roll back if it's a
          regression; offline gates the merge, online (A/B / shadow) confirms on live traffic &rarr;
          <b>feed back</b> — every production failure becomes a new labeled case in the golden set, so
          the next offline run catches that class &rarr; the loop is the point: it converts one-off
          failures into permanent regression coverage and makes quality a ratchet, not a coin flip
          &rarr; measure each stage (coverage, judge-human agreement, regression catch-rate) so the loop
          itself is trustworthy.
    - level: 'staff'
      q: 'One eval system, three forces — speed, trust, coverage — that pull against each other. How do you reason about the trade-offs?'
      a: >-
          <b>Hit these points:</b> name the three: programmatic/judge graders give <b>SPEED</b>, human
          labels give <b>TRUST</b>, online + golden-set growth give <b>COVERAGE</b> &rarr; the tension:
          the fast graders (judge) are the least trusted, the trusted grader (human) is the slowest, and
          the most realistic signal (online) can't gate a pre-ship decision &rarr; sequence it:
          gold-label a small golden set with humans (trust), automate the bulk with programmatic +
          calibrated judge (speed), grow coverage by folding online failures back in &rarr; never trade
          an unmeasured risk for a measured saving: a cheap but uncalibrated judge is worse than no gate
          because it ships regressions confidently &rarr; measure the loop itself — judge-human
          agreement, offline coverage of real traffic, regression catch-rate — so you know which force
          is weakest.
    - level: 'staff'
      q: >-
          "We hit 90% on the public benchmark, so we don't need our own eval." Tear this apart as a
          system design.
      a: >-
          <b>Hit these points:</b> two unsafe assumptions &rarr; <b>"benchmark = our task"</b> is false:
          a public benchmark measures a generic task on shared data — it's a capability proxy, not a
          measure of YOUR inputs and YOUR success criteria &rarr; <b>"a high score = real skill"</b> is
          suspect: <b>contamination</b> means leaderboard test data leaks into training, so the rank can
          be memorization; benchmarks also saturate and rot as new models arrive &rarr; the right
          design: a private golden set of your real inputs + your graders gates the ship; benchmarks
          only shortlist candidate models cheaply &rarr; then confirm with shadow/A/B on live traffic
          &rarr; the framing: the leaderboard says a model is generally capable; only your eval says
          it's right for the job — teach the method, not the rank.
    - level: 'design'
      q: 'Design an evaluation strategy for a new LLM feature, from zero to a CI gate plus production monitoring.'
      a: >-
          <b>A strong answer covers:</b> define success criteria per task first — measurable pass/fail,
          not "good" &rarr; build a small versioned <b>golden set</b> of real, representative (input,
          expected) cases, hand-labeled by humans &rarr; choose graders: programmatic where structured,
          calibrated LLM-as-judge where open-ended, human for the gold labels and audits &rarr; stand up
          the harness (dataset + runner + grader + report) and establish a baseline number &rarr; wire
          offline regression eval into CI so a drop below baseline blocks the merge &rarr; ship behind
          <b>shadow</b> first (no user risk), then <b>A/B</b> to measure real lift &rarr; instrument
          production signals (thumbs-down, regenerate, escalation, task success) and route failures back
          into the golden set &rarr; map risks: benchmark contamination, judge bias, coverage gaps;
          measure the loop, not just the model.
    - level: 'design'
      q: >-
          A leader says "just use the public leaderboard score to pick our model." Tear this apart and
          propose what you'd do instead.
      a: >-
          <b>A strong answer covers:</b> a public benchmark measures a generic task on shared data —
          it's a proxy for capability, not a measure of YOUR task &rarr; <b>contamination</b>:
          leaderboard test data leaks into training sets, so a high rank can be memorization, and
          leaderboards saturate and rot as new models arrive &rarr; the right move is a private golden
          set of your real inputs with your success criteria, scored by your graders — that's the number
          that predicts production &rarr; use benchmarks only to <b>shortlist</b> candidate models
          cheaply, then decide with your own offline eval and confirm with shadow/A/B on live traffic
          &rarr; the framing: the leaderboard says a model is generally capable; only your eval says
          it's right for the job — teach the method, not the rank.

                <td>An LLM scores / compares outputs against a rubric</td>
                <td>Open-ended quality at scale &mdash; cheap, but biased; calibrate first</td>
            </tr>
            <tr>
                <td>
                    <b>Human</b>
                </td>
                <td>A person applies a rubric</td>
                <td>Gold labels &amp; audits &mdash; the trusted source, but slow and costly</td>
            </tr>
        </tbody>
    </table>

    <h3>Golden set vs benchmark — don't confuse them</h3>
    <table>
        <thead>
            <tr>
                <th>Dimension</th>

                <th>Benchmark (public)</th>

    <div class="facts">
        <div class="fact">
            <b>Is</b> Offline eval = scoring the system against a fixed golden set before shipping.
            Reproducible, gated in CI. A golden set is a versioned set of (input, expected /
            acceptance-criteria) cases, changed only by deliberate review.
        </div>
        <div class="fact">
            <b>Why it exists</b> You need a repeatable, pre-ship signal that a change helped or hurt —
            one you can re-run on every commit and block a merge on, exactly like a unit test.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> A drug trial against a fixed cohort: same patients, same protocol, run
            again after each formulation change so the only variable is the change itself.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> Snapshot / golden-file tests: a curated set of fixtures with expected
            outputs; CI fails the build when the new output diverges from the approved snapshot.
        </div>
    </div>

    <div class="myth">
        <div class="m">
            &#10007; &quot;A bigger golden set is always better — dump in every log line.&quot;
        </div>
        <div class="t">
            &#10003; Small and hand-checked beats large and noisy. The golden set is fixtures:
            representative, labeled, versioned, changed only by review — not a raw traffic dump.
        </div>
    </div>
    <div class="myth">
        <div class="m">
            &#10007; &quot;We scored high on the public benchmark, so we're good.&quot;
        </div>
        <div class="t">
            &#10003; Benchmarks measure a generic task and rot via contamination (test data leaked into
            training = memorization, not skill). A private golden set predicts your production.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span> Offline = <span>fixed dataset, repeatable, pre-ship, gates
        the merge.</span> Build the harness once; <span>diff every change vs the baseline</span>.
    </div>

    <MemoryCheck items={mc2} />

    <details class="iq">
        <summary>M2 — Walk me through building an offline eval for a support assistant from scratch.</summary>
        <div class="a">
            <b>Hit these points:</b> define success criteria per task <em>before</em> writing cases —
            what's a pass, what's a fail, in measurable terms &rarr; collect a golden set of real,
            representative inputs paired with expected outputs or acceptance criteria; keep it small,
            hand-checked, versioned like test fixtures &rarr; pick a grader per case: exact-match /
            programmatic where structured, LLM-as-judge where open-ended, humans for the gold labels
            &rarr; run the harness (dataset &rarr; runner &rarr; grader &rarr; report) &rarr; establish a
            baseline number, then every change is a diff vs that baseline &rarr; wire it into CI so it
            gates the merge &rarr; name the limit: a fixed dataset can't catch what you didn't think to
            put in it — which is why you also need online eval.
        </div>
    </details>

        outcomes) and <b>shadow</b> (run the new system in parallel, discard its output, zero user

    <div class="facts">
        <div class="fact">
            <b>Is</b> Online eval = measuring quality on live traffic after shipping. A/B splits real
            users between old and new and compares outcomes; shadow runs the new system in parallel and
            discards its output.
        </div>
        <div class="fact">
            <b>Why it exists</b> A fixed dataset can't cover what you never imagined. Real traffic drifts
            and surfaces failure classes the golden set missed — online is the only honest signal of
            production quality.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> Crash-test dummies (offline, repeatable lab) vs real-world accident data
            (online, messy but true). You need both: the lab to gate the design, the road to find what
            the lab missed.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> Tests in CI (offline) vs production observability / canary releases
            (online). Green CI lets you deploy; metrics and canaries tell you it actually worked under
            load.
        </div>
    </div>

    <div class="myth">
        <div class="m">
            &#10007; &quot;Offline eval is green, so we're done — ship and move on.&quot;
        </div>
        <div class="t">
            &#10003; Green offline + unhappy users = a coverage gap, not a broken eval. Add online
            signals and fold the failing live cases back into the golden set.
        </div>
    </div>
    <div class="myth">
        <div class="m">
            &#10007; &quot;The LLM judge scored the new prompt higher, so it's better. Ship it.&quot;
        </div>
        <div class="t">
            &#10003; An uncalibrated judge has biases (position, verbosity, self-preference). Measure its
            agreement with humans first; otherwise it ships regressions confidently.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span> Offline gates the ship; online proves it worked. <span>Feed
        every production failure back into the golden set</span> &mdash; the loop closes.
    </div>

    <details class="iq">
        <summary>M3 — Offline eval is green but users complain after you ship. What happened and what do you add?</summary>
        <div class="a">
            <b>Hit these points:</b> offline eval only measures the distribution you captured in the
            golden set — real traffic drifts and hits inputs you never fixtured &rarr; green offline +
            unhappy users = a coverage gap, not a broken eval &rarr; add online evaluation: A/B split
            real users between old and new and compare outcomes, or run the new system in shadow (same
            traffic, output discarded) to compare safely with zero user risk &rarr; instrument
            production signals: thumbs-down, regenerate-rate, escalation-to-human, task completion &rarr;
            close the loop: harvest the failing live cases, label them, fold them into the golden set so
            the next offline run catches that class &rarr; the two are complementary — offline is
            repeatable and pre-ship, online is real and post-ship.
        </div>
    </details>

    <details class="iq">
        <summary>M3 — Your LLM-as-judge gives a new prompt a higher score, so you ship. The judge was wrong. How do you prevent this?</summary>
        <div class="a">
            <b>Hit these points:</b> an LLM judge is itself a model with biases — position (favors the
            first option), verbosity (favors longer answers), self-preference (favors its own family's
            text) &rarr; never trust a judge you haven't calibrated: measure its agreement against a
            human-labeled slice before letting it gate anything &rarr; a strong judge can reach roughly
            human-level agreement on <em>some</em> tasks, but that's dataset-dependent and must be
            re-verified &rarr; mitigations: randomize option order, force a rubric / chain-of-thought,
            use a different model family as judge, anchor with reference answers &rarr; treat the judge
            as a cheap noisy sensor that scales human judgment, never a replacement for the gold labels
            it was calibrated against.
        </div>
    </details>
