# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is an idempotency key and what problem does it solve?'
      a: >-
          <b>Hit these points:</b> a client-generated unique token attached to a request so the
          server can recognize a <em>retry</em> of the same logical operation rather than a new one
          &rarr; it exists because a network call can time out after succeeding, so the client must be
          able to safely retry without re-executing the effect &rarr; the server records the key on
          first execution and, on any later request with the same key, replays the original result
          instead of re-running the side effect &rarr; this turns a non-idempotent operation ("charge
          $10", "create order") into one that's safe to retry &rarr; it is per-operation, per-store
          dedup — not a delivery guarantee.
    - level: 'core'
      q: 'Beyond the key itself, what do you store in the idempotency record — and why is a bare key not enough?'
      a: >-
          <b>Hit these points:</b> the scoped claim <code>(account_id + key)</code>, a
          <code>request_fingerprint</code> (hash of the params), a <code>status</code>
          (in_progress/completed), the saved <code>response_code/body</code> to replay, and
          <code>created_at/locked_at</code> for TTL + lease &rarr; a bare key can't catch the
          same-key-different-body trap: the fingerprint lets you tell a genuine retry (params match →
          replay) from a key-reuse bug (params differ → reject 400) instead of returning a
          confidently-wrong cached result &rarr; the saved response lets a retry return the
          <em>original</em> outcome verbatim &rarr; scoping by account prevents one tenant's key from
          colliding with another's.
    - level: 'core'
      q: 'Why is "exactly-once delivery" impossible, and what''s the real guarantee?'
      a: >-
          <b>Hit these points:</b> an unacked send forces resend (risk duplicate) or no resend (risk
          loss) — no third option; that's the Two Generals problem &rarr; so exactly-once
          <em>delivery</em> over an unreliable network can't exist &rarr; the achievable guarantee is
          at-least-once delivery + an idempotent (or deduplicating) endpoint = exactly-once
          <em>effect</em> &rarr; the idempotency key is the most common way to make an endpoint that
          idempotent &rarr; always pin down "exactly-once across <em>which</em> boundary?" — the
          property is per receiver, not a magic end-to-end promise.
    - level: 'core'
      q: "Which operations don't need an idempotency key, and why?"
      a: >-
          <b>Hit these points:</b> operations that are <em>already</em> idempotent — an absolute write
          <code>SET balance = 100</code>, <code>DELETE /orders/42</code>, "mark invoice paid", and in
          general well-behaved <code>GET/PUT/DELETE</code> &rarr; re-applying them yields the same
          state, so a retry is harmless and the extra machinery is just noise &rarr; keys are for the
          <em>unsafe</em>, non-idempotent ops: relative mutations like "charge $10" or "create an
          order" (which would duplicate) &rarr; the design instinct: prefer making the operation
          naturally idempotent (absolute, not relative) before reaching for a key.
    - level: 'senior'
      q: 'Two retries with the same key hit two app servers at the same instant. Why does the naive design double-charge, and how do you fix it?'
      a: >-
          <b>Hit these points:</b> the naive design is <em>check-then-act</em>: both requests read the
          store, both see "not seen" before either writes, so both proceed and charge — a classic race
          &rarr; the fix is an <em>atomic claim</em>: a <code>UNIQUE(account_id, key)</code>
          <code>INSERT … status=in_progress</code> performed <em>before</em> any work, so exactly one
          racer wins the insert &rarr; the loser catches the unique-violation, reads the row, and
          branches: <code>completed</code> → replay the saved response; <code>in_progress</code> →
          return <code>409</code> (retry later) &rarr; alternatives: <code>SELECT … FOR UPDATE</code>
          or an advisory lock &rarr; the principle: replace check-then-act with a single atomic
          conditional write.
    - level: 'senior'
      q: "You've solved the race. The server still occasionally double-charges. Where's the remaining window?"
      a: >-
          <b>Hit these points:</b> a crash <em>between</em> performing the side effect and recording
          the <code>completed</code> result — the effect happened, no completion record exists, so the
          retry re-runs it &rarr; the fix: recording the result must be <em>atomic with the side
          effect</em>, not a second step &rarr; for a local effect, wrap the business mutation and the
          idempotency record in one DB transaction &rarr; red flag: "just save the key right after the
          charge" — that's two operations a crash can tear apart, which is the dual-write problem in
          miniature.
    - level: 'senior'
      q: "When can you wrap the side effect and the idempotency record in one transaction — and when can't you?"
      a: >-
          <b>Hit these points:</b> <b>Case A — local effect:</b> the business mutation and the
          idempotency record live in the same database, so a single transaction commits both or
          neither — clean and atomic &rarr; <b>Case B — external effect</b> (e.g. a Stripe charge): no
          transaction can span your DB and a remote API — that's the dual-write problem — so you
          <em>delegate</em> by passing your own key through to the downstream's idempotency mechanism
          &rarr; if the downstream has no idempotency support, fall back to reconciliation: detect
          duplicates after the fact and refund/compensate &rarr; the first question to ask of any
          payment integration: "is this API idempotent, and how do I pass a key?"
    - level: 'senior'
      q: 'A request claims the key (in_progress) then the process crashes. What happens to future retries, and how do you prevent a permanent outage?'
      a: >-
          <b>Hit these points:</b> the orphaned <code>in_progress</code> row blocks every future retry
          with a permanent <code>409</code> — the operation can never complete, a self-inflicted
          outage on that key &rarr; prevention is a recovery lease: store <code>locked_at</code> and
          treat an <code>in_progress</code> record older than a timeout as reclaimable, so a later
          retry can take it over and resume &rarr; ideally pair with recovery points (Stripe's model)
          so a retry resumes a half-finished request rather than restarting it blindly &rarr; the
          lease timeout must exceed the operation's worst-case legit duration, or you'll reclaim a
          still-running request and risk concurrent execution.
    - level: 'staff'
      q: "How do you choose the retention TTL, and what goes wrong if it's too short?"
      a: >-
          <b>Hit these points:</b> records can't live forever (storage growth), so they expire —
          Stripe keeps keys ≈ 24h (illustrative) &rarr; after expiry the same key is treated as brand
          new, so a <em>late</em> retry — a message stuck in a queue 25h, a long backoff, a DLQ
          replayed days later — re-executes and double-charges &rarr; so the TTL must comfortably
          exceed your worst-case <em>retry horizon</em>: max queue redelivery delay + retry-backoff
          cap + DLQ replay window, with margin &rarr; derive it from those actual bounds, not a round
          number &rarr; the staff framing: the TTL is a contract between your dedup window and every
          retry source upstream — if any source can outlive the window, the guarantee is silently
          void.
    - level: 'staff'
      q: 'A team says "we added idempotency keys, so the whole payment pipeline is exactly-once." Push back.'
      a: >-
          <b>Hit these points:</b> a key guards <em>one</em> operation against <em>one</em> store — it
          is not a distributed-transaction substitute and gives no exactly-once across a chain of
          services &rarr; three services in a row = three independent dedup guards, each with its own
          key, store, and TTL; a gap in any one re-opens the duplicate window &rarr; the underlying
          theorem: there is no exactly-once <em>delivery</em>; there is at-least-once delivery + an
          idempotent/deduplicating endpoint = exactly-once effect, scoped per endpoint &rarr; also
          name the TTL hole, the downstream-without-idempotency hole, and the dual-write hole &rarr;
          the correct claim is "each step is individually idempotent," verified end to end — not "the
          pipeline is exactly-once."
    - level: 'staff'
      q: 'Producer-side keys vs consumer-side dedup — how do you decide where to put the guarantee?'
      a: >-
          <b>Hit these points:</b> apply the end-to-end argument — put the guarantee at the layer that
          actually has the knowledge to enforce it &rarr; producer-side idempotency keys work when the
          caller can generate a stable key and the receiving service owns an atomic store to claim it
          &rarr; but if the producer is external, non-idempotent, and offers no hook (you can't make it
          send a stable key or pass one downstream), you can't fix it there — move dedup to the
          <em>consumer</em>, which sees every delivery and can dedup on a natural business key or
          content hash &rarr; trade-off: producer keys are precise but require cooperation; consumer
          dedup is robust to uncooperative producers but needs a good dedup key and a window sized to
          redelivery &rarr; the staff move: choose by who holds the stable identifier and atomic
          store, not by habit.
    - level: 'design'
      q: 'Design an idempotent payment API: charge a customer exactly once despite client retries.'
      a: >-
          <b>A strong answer covers:</b> the client generates an idempotency key per logical charge and
          sends it on every retry of that charge &rarr; on receipt, atomically claim the key with
          <code>INSERT … UNIQUE(account_id, key) status=in_progress</code> <em>before</em> doing work;
          the unique constraint makes exactly one concurrent racer win &rarr; store a request
          fingerprint and reject same-key-different-body with 400 &rarr; perform the charge, then record
          <code>completed</code> + the response <em>atomically with the effect</em>: one DB transaction
          for a local ledger, or pass the key through to the payment processor's own idempotency for an
          external charge (dual-write → delegate) &rarr; on a duplicate request, branch on status:
          completed → replay the saved response, in_progress → 409 &rarr; add a recovery lease so a
          crashed in_progress row is reclaimable, not a permanent 409 &rarr; set the TTL to exceed the
          worst-case retry horizon so a late retry can't re-charge &rarr; for the external leg with no
          idempotency support, add reconciliation (detect + refund) &rarr; name the trade-off: strong
          dedup needs a stable client key and an atomic store; without them you fall back to
          consumer-side reconciliation.
    - level: 'design'
      q: 'Design exactly-once consumption from an at-least-once message queue (e.g. payments off Kafka/SQS).'
      a: >-
          <b>A strong answer covers:</b> accept that the broker is at-least-once, so the consumer must
          be idempotent — there is no exactly-once delivery to lean on &rarr; pick a stable dedup key:
          a producer-supplied message/event ID, else a content hash or a natural business key &rarr; on
          each message, atomically claim that key in a dedup store before applying the effect (unique
          insert / conditional write), so reprocessing a redelivered message is a no-op replay &rarr;
          make "apply the effect" and "mark the message processed" atomic — same transaction for a
          local DB sink, or delegate to the sink's idempotency for an external one — so a crash between
          them doesn't double-apply &rarr; only ack/commit the offset <em>after</em> the effect is
          durably recorded; an ack-before-effect crash loses the message &rarr; size the dedup window
          (TTL) to exceed max redelivery delay + DLQ replay window, or duplicates slip through after
          expiry &rarr; handle poison messages with a DLQ and bounded retries with backoff + jitter
          &rarr; name the trade-off: a large dedup window costs storage but is the only thing that
          closes the late-redelivery hole, and the dedup store itself must be at least as
          available/consistent as the effect it guards.

        />
        <figcaption>
            <strong>Atomicity, two cases.</strong> When the side effect is in your own database,
            wrap the business write and the idempotency record in ONE transaction. When it's an
            external call, you can't — so delegate idempotency to the downstream API by passing the
            key through.
        </figcaption>
    </figure>
    <div class="rule">
        <span>⚛️ The lesson:</span> Recording "this key is done, here is the result" must be atomic
        with the side effect itself. If a crash can tear them apart, the key does not protect you.
    </div>
    <Fig3 />
    <div class="facts">
        <div class="fact">
            <b>Case A · local</b> Wrap the business mutation + the idempotency record in{' '}
            <em>one</em> transaction. A mid-way crash leaves it uncommitted; the retry re-runs from
            scratch — Part 2's all-or-nothing.
        </div>
        <div class="fact">
            <b>Case B · external</b> You can't make a network call atomic with a local write (the
            dual-write problem). So delegate: pass your key through to the downstream's idempotency
            mechanism.
        </div>
        <div class="fact blue">
            <b>The boundary rule</b> Make the thing at the boundary idempotent, because you can't
            wrap it in your transaction. Stripe's <code>Idempotency-Key</code> header is exactly
            this.
        </div>
        <div class="fact blue">
            <b>No downstream support?</b> You're forced into reconciliation — detect and refund the
            double. Hence "is this API idempotent?" is the first question a senior asks of any
            payment/messaging integration.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Do the work, then save the key in a second step"</div>
        <div class="t">
            ✓ A crash between them re-executes — record must commit atomically WITH the effect (or
            be delegated)
        </div>
    </div>
    <div class="rule">
        <span>⚛️ Memory rule:</span> Where the effect lives decides everything — local effect → one
        transaction; external effect → delegate idempotency to the downstream.
    </div>
    <MemoryCheck items={mc2} />
</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>What to store — and the same-key-different-body trap</h2>
    </div>
    <p class="tagline">
        A key alone is not enough state. Without a fingerprint, keys return{' '}
        <em>confidently wrong</em> results.
    </p>
    <table>
        <tr>
            <th>Field</th>
            <th>Why it's there</th>
        </tr>
        <tr>
            <td>
                <code>idempotency_key</code> + <code>account_id</code>
            </td>
            <td>the unique claim — scoped, never global</td>
        </tr>
        <tr>
            <td>
                <code>request_fingerprint</code>
            </td>
            <td>a hash of the request parameters (the trap-catcher)</td>
        </tr>
        <tr>
            <td>
                <code>status</code>
            </td>
            <td>
                <code>in_progress</code> / <code>completed</code> — the §1 state machine
            </td>
        </tr>
        <tr>
            <td>
                <code>response_code</code>, <code>response_body</code>
            </td>
            <td>the saved result to replay</td>
        </tr>
        <tr>
            <td>
                <code>locked_at</code>, <code>created_at</code>
            </td>
            <td>recovery lease (§1) and TTL (§4)</td>
        </tr>
    </table>
    <Fig4 />
    <div class="facts">
        <div class="fact">
            <b>The trap</b> Client reuses a key for a <em>different</em> request (new
            amount/recipient). The naive server returns the <em>first</em> response — confidently
            wrong.
        </div>
        <div class="fact">
            <b>The catch</b> Store <code>hash(params)</code>. On every repeat, compare. Match →
            genuine retry. Differ → reject with <code>400</code>.
        </div>
        <div class="fact blue">
            <b>Scope, not global</b> The claim is <code>(account_id, key)</code> — scoped to avoid
            cross-tenant collisions and bound the keyspace you store.
        </div>
        <div class="fact blue">
            <b>Replay payload</b> Save <code>response_code</code> + <code>response_body</code> so a
            retry-after-success replays the identical answer, byte for byte.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Same key → always return the saved response"</div>
        <div class="t">
            ✓ Only if the request fingerprint matches; a differing body is a client bug — reject it
        </div>
    </div>
    <div class="rule">
        <span>🧾 Memory rule:</span> Fingerprint the body — one key names one operation; same key +
        different params is a contract violation, not a retry.
    </div>
    <MemoryCheck items={mc3} />
</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>The key's lifecycle — scope, generation, and the TTL hazard</h2>
    </div>
    <p class="tagline">
        Pull it together as a lifecycle and two more decisions surface: how keys are generated, and
        how long they live.
    </p>
    <figure>
        <img
            src="diagrams/ik-03-key-lifecycle.png"
            alt="The idempotency-key lifecycle. The client generates one key per logical operation and reuses it on retries. The server claims it, executes, and stores the result for a retention window — after which the same key is treated as brand new, which is a hazard if a retry arrives late."
        />
        <figcaption>
            <strong>The idempotency-key lifecycle.</strong> The client generates one key per logical
            operation and reuses it on retries. The server claims it, executes, and stores the
            result for a retention window — after which the same key is treated as brand new, which
            is a hazard if a retry arrives late.
        </figcaption>
    </figure>
    <Fig5 />
    <div class="facts">
        <div class="fact">
            <b>Generation</b> One key per <em>logical operation</em> (a UUID), reused on every
            retry. A fresh key per attempt defeats dedup entirely.
        </div>
        <div class="fact">
            <b>Scope</b> Per <code>(account, endpoint)</code>, never global — avoids cross-tenant
            collisions and bounds storage.
        </div>
        <div class="fact blue">
            <b>Only for unsafe ops</b> Keys are for <code>POST</code>-style creates and charges.{' '}
            <code>GET</code>/<code>PUT</code>/<code>DELETE</code> are already idempotent — a key
            there is noise.
        </div>
        <div class="fact blue">
            <b>The TTL hazard</b> After the window expires the same key is brand new. A retry
            arriving late (queue stuck 25h, long backoff) re-executes — a double charge.
        </div>
    </div>
    <div class="myth">
        <div class="m">
            ✗ "Generate a fresh idempotency key on each retry" / "any 24h TTL is fine"
        </div>
        <div class="t">
            ✓ Same key across retries; retention must comfortably exceed your worst-case retry
            horizon
        </div>
    </div>
    <div class="rule">
        <span>⏳ Memory rule:</span> One key per operation, reused on retries, scoped per account —
        and retain it longer than your slowest possible retry.
    </div>
    <MemoryCheck items={mc4} />
</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Where idempotency keys stop helping</h2>
    </div>
    <p class="tagline">They are not magic. Knowing the edges is the senior part.</p>
    <Fig6 />
    <div class="facts">
        <div class="fact">
            <b>One op, one store</b> A key makes <em>one</em> call safe against <em>one</em> system.
            Not a distributed-transaction substitute; no exactly-once across a chain — each hop
            needs its own key/dedup.
        </div>
        <div class="fact">
            <b>Already idempotent?</b> <code>SET balance = 100</code>,{' '}
            <code>DELETE /orders/42</code>, "mark invoice paid" are naturally repeat-safe. Don't add
            machinery where the property already exists.
        </div>
        <div class="fact blue">
            <b>Producer can't be safe → dedupe at the consumer</b> If the effect is non-idempotent,
            external, and downstream offers no hook, move the guarantee to the receiver: process
            each event id once.
        </div>
        <div class="fact blue">
            <b>The unifying truth</b> Exactly-once is always built as at-least-once <em>plus</em> an
            idempotent/deduplicating endpoint — the key is the most common way to be that endpoint.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "An idempotency key gives me exactly-once across my whole pipeline"</div>
        <div class="t">
            ✓ It guards one boundary; a chain of three services is three places to get it right
        </div>
    </div>
    <div class="rule">
        <span>🧭 Memory rule:</span> Claim atomically, record atomically with the effect,
        fingerprint the request, retain longer than you retry — and the key only protects the one
        boundary it guards.
    </div>
    <MemoryCheck items={mc5} />
</section>

    The one idea behind all four edges
</h2>

<p>

        <th>Edge</th>
        <th>The failure if you skip it</th>
        <th>The discipline</th>
    </tr>

        <td>Two retries both pass the check → double charge</td>
        <td>Claim atomically (unique INSERT)</td>
    </tr>

        <td>Crash between work and record → re-execute</td>
