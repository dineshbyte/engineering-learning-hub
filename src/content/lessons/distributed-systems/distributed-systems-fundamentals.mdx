# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'State CAP precisely, then correct the most common misreading.'
      a: >-
          <b>Hit these points:</b> CAP = under a network <em>partition</em> you must choose Consistency
          (linearizability) or Availability — you cannot keep both &rarr; P is not optional, partitions
          happen, so the real lever is C-vs-A <em>during</em> a partition: CP blocks the minority side,
          AP serves possibly-stale reads &rarr; the misreading is "pick 2 of 3" as if a system is
          permanently one letter — CAP only speaks about partitions and says nothing about the healthy
          case &rarr; that's exactly the gap PACELC fills: Else (no partition), you still trade Latency
          vs Consistency &rarr; so a real label is two-part, e.g. PA/EL or PC/EC.
    - level: 'core'
      q: 'Name the consistency models from weakest to strongest and what each guarantees.'
      a: >-
          <b>Hit these points:</b> <b>eventual</b> — if writes stop, replicas converge; any stale read
          is legal meanwhile &rarr; <b>read-your-writes / session</b> — a client sees its own writes
          and monotonic reads within a session &rarr; <b>causal</b> — operations that are causally
          related are seen in order by everyone; concurrent ones may differ &rarr; <b>linearizable</b>
          — every operation appears to take effect atomically at one instant between invocation and
          response, as if on a single copy &rarr; rule of thumb: buy the weakest model your correctness
          actually needs, since each rung up costs latency and availability.
    - level: 'core'
      q: 'What is a quorum, and why does R + W &gt; N matter in a leaderless store?'
      a: >-
          <b>Hit these points:</b> in a leaderless (Dynamo-style) store with N replicas, a write waits
          for W acks and a read gathers R responses &rarr; if R + W &gt; N the read set and the most
          recent write set must share at least one node, so a read overlaps every acknowledged write
          and can return the freshest value &rarr; e.g. N=5, W=3, R=3 overlaps; W=2, R=2 does not
          &rarr; quorums tune the C-vs-A/latency trade: bigger W = more durable, slower writes; smaller
          R = faster, riskier reads &rarr; caveat: strict quorums still need read-repair / anti-entropy
          and don't by themselves give linearizability.
    - level: 'core'
      q: 'Name the three replication topologies and the price each pays.'
      a: >-
          <b>Hit these points:</b> <b>single-leader</b> — all writes to one node, replicate to
          followers; simple and linearizable-able, but the leader is a write bottleneck and a failover
          hazard &rarr; <b>multi-leader</b> — writes accepted at several leaders (e.g. multi-region);
          better write availability/latency, but concurrent writes create conflicts you must resolve
          &rarr; <b>leaderless</b> — any replica takes writes, quorums for correctness; great
          availability, but you own conflict detection (version vectors) and read-repair &rarr; the
          through-line: as you move away from one leader you buy availability and pay in conflict
          handling.
    - level: 'senior'
      q: 'A payment call times out with no reply. What do you do, and why is "just retry" dangerous?'
      a: >-
          <b>Hit these points:</b> a timeout has four indistinguishable causes — lost request, slow
          node, lost reply, or dead node — so you cannot tell "didn't happen" from "happened, ack lost"
          &rarr; the charge may already have succeeded, so a blind retry double-charges &rarr; the fix
          is to make the operation idempotent with a client-generated key: at-least-once delivery +
          idempotent effect = exactly-once <em>effect</em> &rarr; pair it with bounded retries,
          exponential backoff + jitter, and a retry budget so you don't amplify an overload &rarr; red
          flag: treating a timeout as proof of failure.
    - level: 'senior'
      q: 'Why is "exactly-once delivery" impossible, and what do we actually ship instead?'
      a: >-
          <b>Hit these points:</b> an unacked send forces a choice — resend (risk a duplicate) or don't
          (risk loss); there is no third door, it's the Two Generals problem &rarr; so "exactly-once
          delivery" over an unreliable network is unachievable &rarr; what we build is exactly-once
          <em>effects</em>: at-least-once delivery + an idempotent or deduplicating endpoint =
          effectively-once &rarr; always ask "exactly-once across <em>which</em> boundary?" — the
          guarantee is per consumer/store, not end-to-end magic &rarr; mechanisms: idempotency keys,
          dedup windows, transactional outbox to avoid the dual-write.
    - level: 'senior'
      q: 'Two replicas accepted writes; which one wins? Walk me through the clock issue.'
      a: >-
          <b>Hit these points:</b> wall-clock timestamps on different machines aren't comparable —
          clock skew can make the later write carry the smaller timestamp &rarr; so last-write-wins by
          physical time silently drops a genuinely-later write &rarr; you need a <em>logical</em>
          notion of order: happens-before via Lamport clocks, or version vectors to actually
          <em>detect</em> concurrency rather than impose a false order &rarr; once concurrency is
          detected you either merge (CRDT/app-level) or surface siblings to the application &rarr; if
          you must use LWW, scope its data-loss risk explicitly and use it only where losing a
          concurrent write is acceptable.
    - level: 'senior'
      q: 'You key a time-series table by timestamp and one node is melting. Why, and how do you fix it?'
      a: >-
          <b>Hit these points:</b> key-range partitioning by timestamp routes all <em>current</em>
          writes to the single partition holding the newest range — a moving write hot spot &rarr; fix
          options: hash-partition the key for even write spread (cost: range scans now scatter across
          partitions) or prefix/salt the key so recent writes fan out &rarr; never rebalance with
          <code>mod N</code> — adding a node remaps almost everything; use a fixed number of partitions
          or consistent hashing so adding a node moves only ~K/N keys &rarr; for read-heavy time
          ranges, consider a compound key (e.g. tenant + time bucket) so writes spread but scans stay
          local.
    - level: 'staff'
      q: 'Events are "coming out of order." Diagnose it, then decide how much ordering to actually buy.'
      a: >-
          <b>Hit these points:</b> first ask <em>which</em> order is required — FIFO (per source),
          causal, or total — because they cost wildly different amounts &rarr; FIFO ⊂ causal ⊂ total;
          pick the lowest rung that satisfies correctness, over-ordering is over-engineering &rarr;
          causal order is enforceable locally with vector clocks and no coordination; <b>total order
          needs consensus</b> (a single Kafka partition, a replicated log) and is the expensive,
          throughput-capping one &rarr; diagnose by tracing whether the disorder is cross-partition
          (you chose total order but partitioned for throughput) or genuine concurrency &rarr;
          synthesis: partition for throughput where order doesn't matter, reserve a totally-ordered
          lane only for the keys that need it.
    - level: 'staff'
      q: 'How does Raft avoid two leaders, and what does FLP have to do with it?'
      a: >-
          <b>Hit these points:</b> a node grants at most one vote per term and a leader needs a
          majority; any two majorities overlap in ≥1 node, so two leaders in the same term is
          impossible — that's the safety argument &rarr; FLP proves no <em>deterministic</em> protocol
          can <em>guarantee termination</em> in a purely asynchronous model with even one crash &rarr;
          Raft doesn't beat FLP; it sidesteps it by assuming partial synchrony (timeouts) for
          <em>liveness</em> and using <em>randomized</em> election timeouts to break symmetric split
          votes &rarr; the principal framing: <b>safety always, liveness only when the network is
          eventually calm</b> &rarr; so a permanently flaky network can stall progress without ever
          violating correctness.
    - level: 'staff'
      q: 'How would you choose a failure detector, and why is "perfect failure detection" the wrong goal?'
      a: >-
          <b>Hit these points:</b> in an async network you cannot distinguish a slow node from a dead
          one, so any timeout-based detector is a tunable guess trading completeness (always suspect
          real failures) against accuracy (never falsely suspect) &rarr; aggressive timeouts cause
          false positives — needless failovers, re-elections, duplicate work; lax timeouts delay
          recovery &rarr; prefer adaptive detectors (e.g. phi-accrual) that output a suspicion level
          from observed RTT history instead of a hard boolean &rarr; pair detection with fencing tokens
          so a wrongly-suspected-then-revived node can't act as a stale leader &rarr; the staff move:
          don't chase perfect detection — design so a <em>wrong</em> suspicion is safe (idempotent
          work, fencing, leader leases), because wrong suspicions are inevitable.
    - level: 'design'
      q: 'Design an order-checkout flow across payment, inventory, and shipping that survives partial failure.'
      a: >-
          <b>A strong answer covers:</b> idempotency keys on every external side effect so retries
          don't double-charge or double-ship &rarr; bounded retries with exponential backoff + jitter
          and a retry budget to avoid retry storms &rarr; the transactional outbox so the DB state
          change and the event publish commit atomically (kills the dual-write) &rarr; a saga with
          compensating transactions (refund, restock) because you cannot 2PC across independent
          services &rarr; dedup at each consumer with a window comfortably exceeding the worst-case
          redelivery delay &rarr; choose consistency per step: inventory reservation may need
          strong/conditional writes, order history can be eventual &rarr; the end-to-end argument —
          confirm the charge with the payment provider's own idempotency, don't infer success from a
          timeout &rarr; name the trade-off: saga gives availability and loose coupling but exposes
          intermediate states, so model compensations and "stuck saga" alerts.
    - level: 'design'
      q: 'Design a globally replicated key-value store and state its consistency contract under partition.'
      a: >-
          <b>A strong answer covers:</b> start from the contract — do callers need linearizable reads
          or is causal/eventual acceptable? that single choice drives everything &rarr; partition the
          keyspace with consistent hashing (virtual nodes) so adding capacity moves only ~K/N keys and
          avoids hot spots; salt known hot keys &rarr; replicate each partition N ways across failure
          domains/regions; expose tunable quorums (R + W &gt; N for read-overlap, lower for latency)
          &rarr; under a partition, declare the choice: CP (block the minority, stay linearizable) for
          a config/lock store, or AP (serve stale, reconcile later) for a cart &rarr; conflict handling
          for AP: version vectors to detect concurrency, CRDTs or app merge for resolution, read-repair
          + anti-entropy to converge &rarr; place it on PACELC and say it: e.g. PC/EC for a metadata
          store, PA/EL for a session cache &rarr; ops: detect partitions and replica lag, fence stale
          leaders with tokens, and test failover with fault injection &rarr; name the trade-off
          explicitly between availability and the strength of the read your callers can rely on.

            application layer
        </div>
    </div>
    <div class="rule">
        <span>🧠 Memory rule:</span> When a remote call goes silent you cannot tell{' '}
        <span>lost request · slow node · lost reply · dead node</span> apart — and the work may
        already be done.
    </div>
    <MemoryCheck items={mc1} />
</section>

<section class="lvl" id="l2">
    <div class="lvltop">
        <span class="num">2</span>
        <h2>Delivery guarantees &amp; idempotency</h2>
    </div>
    <p class="tagline">
        The fix for the double-charge: pick at-least-once, then make processing duplicate-proof.
    </p>
    <figure>
        <img
            src="diagrams/04-delivery-guarantees.png"
            alt="The three delivery guarantees. Send-and-forget may lose the message; retry-until-acked may duplicate it; exactly-once is the goal you can't get at the delivery layer."
        />
        <figcaption>
            <strong>The three delivery guarantees.</strong> Send-and-forget may lose the message;
            retry-until-acked may duplicate it; exactly-once is the goal you can't get at the
            delivery layer.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Guarantee</th>
            <th>Promise</th>
            <th>The price · where you see it</th>
        </tr>
        <tr>
            <td>At-most-once</td>
            <td>send &amp; forget; 0 or 1 times</td>
            <td>
                may be <b>lost</b>, never duplicated · metrics, telemetry
            </td>
        </tr>
        <tr>
            <td>At-least-once</td>
            <td>retry until acked</td>
            <td>
                never lost, may <b>duplicate</b> · Kafka, SQS, webhooks (the default)
            </td>
        </tr>
        <tr>
            <td>
                Exactly-once <em>delivery</em>
            </td>
            <td>once, period</td>
            <td>
                <b>impossible</b> over an unreliable network — see below
            </td>
        </tr>
    </table>
    <p class="tagline">
        An unacked send leaves you two moves: <b>resend</b> (risk a duplicate → at-least-once) or{' '}
        <b>don't</b> (risk a loss → at-most-once). There is no third door — it's Two Generals in
        work clothes. So stop chasing exactly-once <em>delivery</em> and build exactly-once{' '}
        <strong>effects</strong>.
    </p>
    <figure>
        <img
            src="diagrams/05-idempotency-key.png"
            alt="The idempotency key. The timed-out retry carries the same key; the server recognises it, returns the original result, and does not charge again. At-least-once + idempotency = exactly-once effect."
        />
        <figcaption>
            <strong>The idempotency key.</strong> The timed-out retry carries the same key; the
            server recognises it, returns the original result, and does not charge again.{' '}
            <b>At-least-once + idempotency = exactly-once effect.</b>
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>Idempotent</b> Doing it many times = doing it once. <code>balance=100</code>,{' '}
            <code>DELETE /orders/42</code>, "mark #7 paid."
        </div>
        <div class="fact">
            <b>Not idempotent</b> <code>balance=balance+10</code>, <code>POST /orders</code>,
            "increment counter," "append a row."
        </div>
        <div class="fact blue">
            <b>Idempotency key</b> Client mints one UUID per <em>logical</em> op, reuses it on every
            retry; server stores <code>key→result</code>, replays the saved result on repeats.
            (Stripe's <code>Idempotency-Key</code>.)
        </div>
        <div class="fact blue">
            <b>Dedup (receiver side)</b> Broker tags each message; consumer records processed IDs
            and drops repeats — exactly-once <em>within the dedup window</em>.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Idempotency also protects me from reordering"</div>
        <div class="t">
            ✓ Idempotency defends against duplicates only; reordering needs ordering (Lessons 3–4)
            or commutative ops
        </div>
    </div>
    <div class="rule">
        <span>🔁 Memory rule:</span> Exactly-once delivery is a myth; ship{' '}
        <span>at-least-once + idempotency</span> = exactly-once <em>effects</em>.
    </div>
    <MemoryCheck items={mc2} />
</section>

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>No global clock — logical time</h2>
    </div>
    <p class="tagline">
        Stop asking <em>when</em> things happened on a wall clock; ask which one <em>caused</em> the
        other.
    </p>
    <Fig3 />
    <table>
        <tr>
            <th>Clock kind</th>
            <th>Answers</th>
            <th>Failure mode</th>
        </tr>
        <tr>
            <td>Physical / time-of-day</td>
            <td>"what wall-clock instant?"</td>
            <td>skew, drift, backward jumps</td>
        </tr>
        <tr>
            <td>Monotonic</td>
            <td>"how much elapsed locally?"</td>
            <td>meaningless across machines</td>
        </tr>
        <tr>
            <td>Logical</td>
            <td>
                "what is the <em>order</em>?"
            </td>
            <td>carries no real-time meaning</td>
        </tr>
    </table>
    <figure>
        <img
            src="diagrams/06-happens-before.png"
            alt="Happens-before vs concurrent, with Lamport timestamps. A space-time diagram showing that B sending event m2 only after receiving m1 from A makes A's send causally before B's send, while C's event is concurrent with both."
        />
        <figcaption>
            Happens-before vs concurrent. B sends m2 only after receiving m1, so{' '}
            <code>send(m1) → recv(m1) → send(m2)</code>; C's event is reachable from neither —{' '}
            <em>concurrent</em>. (Lamport, 1978.)
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>Happens-before (a → b)</b> Same node &amp; earlier; or send→receive of one message;
            plus transitivity. A <em>partial</em> order.
        </div>
        <div class="fact">
            <b>Concurrent (a ∥ b)</b> Neither reaches the other — causally independent, <em>not</em>{' '}
            "same wall-clock time."
        </div>
        <div class="fact blue">
            <b>Lamport clock</b> One integer; <code>recv: L=max(L,t)+1</code>. Gives{' '}
            <code>a→b ⟹ L(a)&lt;L(b)</code> — but <em>not</em> the converse. Can't detect
            concurrency.
        </div>
        <div class="fact blue">
            <b>Vector clock</b> One counter per node; compare element-wise.{' '}
            <code>a→b iff V(a)&lt;V(b)</code>; incomparable = concurrent. Cost: O(N) per message.
            (Dynamo.)
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "NTP syncs clocks, so timestamps order events"</div>
        <div class="t">
            ✓ NTP error is tens of ms and can jump backward; use logical time for ordering
        </div>
    </div>
    <div class="rule">
        <span>⏱ Memory rule:</span> There is no global clock — for ordering, throw away time and
        track <span>cause: happens-before</span>.
    </div>
    <MemoryCheck items={mc3} />
</section>

    <div class="lvltop">
        <span class="num">4</span>
        <h2>Order &amp; causality — the broadcast ladder</h2>
    </div>
    <p class="tagline">
        Pick the cheapest rung that still satisfies the requirement — that choice is the whole art.
    </p>
    <figure>
        <img
            src="diagrams/07-broadcast-ordering.png"
            alt="The broadcast-guarantee ladder, and the causal violation it prevents. The reader should conclude: each rung adds exactly one ordering promise, and the causal rung is the one that stops the overtaking reply."
        />
        <figcaption>
            <strong>The broadcast-guarantee ladder, and the causal violation it prevents.</strong>{' '}
            Each rung adds exactly one ordering promise; the causal rung is the one that stops a
            reply from overtaking the question it depends on.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Rung</th>
            <th>Adds this promise</th>
            <th>Concretely</th>
        </tr>
        <tr>
            <td>Best-effort</td>
            <td>nothing — try once</td>
            <td>sender may crash mid-send; some get it, some don't</td>
        </tr>
        <tr>
            <td>Reliable</td>
            <td>all-or-nothing</td>
            <td>if any correct node delivers, every correct node eventually does</td>
        </tr>
        <tr>
            <td>FIFO</td>
            <td>per-sender order</td>
            <td>one sender's messages arrive in send order</td>
        </tr>
        <tr>
            <td>Causal</td>
            <td>happens-before order</td>
            <td>if A→B, every node delivers A before B</td>
        </tr>
        <tr>
            <td>Total order</td>
            <td>one global sequence</td>
            <td>
                all nodes deliver all messages in the <b>same</b> order
            </td>
        </tr>
    </table>
    <Fig4 />
    <div class="myth">
        <div class="m">✗ "The events came out of order — make it strictly ordered"</div>
        <div class="t">
            ✓ First ask: FIFO, causal, or total? Total order needs consensus; don't pay for it
            unless required
        </div>
    </div>
    <div class="rule">
        <span>📡 Memory rule:</span> FIFO ⊂ causal ⊂ total;{' '}
        <span>
            need every replica on one order → you need total-order broadcast → you need consensus
        </span>
        .
    </div>

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Replication — leader, multi-leader, leaderless</h2>
    </div>
    <p class="tagline">
        Keep copies useful-agreement-close despite the lost messages and timeouts you already
        accepted.
    </p>
    <table>
        <tr>
            <th>Architecture</th>
            <th>Who takes writes</th>
            <th>Key trade-off</th>
        </tr>
        <tr>
            <td>Single-leader</td>
            <td>one leader; followers replay its log</td>
            <td>
                no write conflicts (one serializer); leader is a SPOF →{' '}
                <b>failover &amp; split brain</b>
            </td>
        </tr>
        <tr>
            <td>Multi-leader</td>
            <td>a leader per DC / per device</td>
            <td>
                local fast writes; <b>conflicts</b> must be resolved after the fact
            </td>
        </tr>
        <tr>
            <td>Leaderless (Dynamo)</td>
            <td>any replica; client uses quorums</td>
            <td>
                no leader; freshness via <code>R + W &gt; N</code>; needs read-repair / anti-entropy
            </td>
        </tr>
    </table>
    <figure>
        <img
            src="diagrams/08-quorum-rw.png"
            alt="A read quorum and a write quorum must overlap by at least one node. With N=3 replicas, a write that lands on 2 nodes (W=2) and a read that contacts 2 nodes (R=2) are guaranteed to share at least one node, so the read sees the latest write."
        />
        <figcaption>
            A read quorum and a write quorum must overlap by ≥1 node. With <b>N=3, W=2, R=2</b>{' '}
            (R+W=4&gt;3), whichever two the read picks, at least one was in the write set — so the
            read sees the latest value. W and R are tunable knobs, not fixed laws. (Dynamo, 2007.)
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>Why replicate</b> Fault tolerance · latency (replica near the user) · throughput
            (parallel reads). Name which one you're buying.
        </div>
        <div class="fact">
            <b>Sync vs async</b> Synchronous = durable but one slow follower blocks; async = fast
            but a leader crash loses acked writes. Most run <em>semi-sync</em>.
        </div>
        <div class="fact blue">
            <b>Conflict resolution</b> LWW (silently loses data — clocks lie) · app merge (more
            code, no loss) · CRDTs (deterministic merge, limited shapes).
        </div>
        <div class="fact blue">
            <b>Replication lag anomalies</b> read-your-writes · monotonic reads · consistent-prefix
            — each with a standard bounded fix (route to leader / pin replica).
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "R + W &gt; N gives me linearizability"</div>
        <div class="t">
            ✓ It only guarantees a read <em>sees</em> a recent write; concurrent/partial writes
            still leak stale values
        </div>
    </div>
    <div class="rule">
        <span>🗂 Memory rule:</span> One serializer = no conflicts; many writers = resolve later; no
        leader = <span>R + W &gt; N</span> for overlap.
    </div>
    <MemoryCheck items={mc5} />
</section>

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Consistency models — eventual → causal → linearizable</h2>
    </div>
    <p class="tagline">
        A contract for what a read may return. Stronger = fewer surprises, more coordination, more
        latency.
    </p>
    <figure>
        <img
            src="diagrams/09-consistency-spectrum.png"
            alt="Bold caption: consistency is a ladder, not a switch. A vertical ladder of replica-consistency models from eventual (weak, bottom) to linearizable (strong, top), with a two-client read example showing eventual lets a read see the old value just after a write while linearizable does not."
        />
        <figcaption>
            <strong>Consistency is a ladder, not a switch.</strong> From eventual (weak) up to
            linearizable (strong). A stronger model permits fewer behaviours — easier to program
            against, more expensive to provide. (Jepsen, "Consistency Models".)
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Model</th>
            <th>Promise</th>
            <th>Cost / note</th>
        </tr>
        <tr>
            <td>Eventual</td>
            <td>if writes stop, replicas converge</td>
            <td>any past/stale value meanwhile; cheapest, stays available (AP)</td>
        </tr>
        <tr>
            <td>Session (RYW, monotonic reads)</td>
            <td>
                per-<em>your</em>-client promises
            </td>
            <td>kills the anomalies users notice; pin to an up-to-date replica</td>
        </tr>
        <tr>
            <td>Causal</td>
            <td>cause-and-effect seen in order everywhere</td>
            <td>
                strongest model that stays available under partition; concurrent writes unordered
            </td>
        </tr>
        <tr>
            <td>Linearizable</td>
            <td>acts like one copy; recency, real-time order</td>
            <td>no staleness window — but needs coordination (consensus/quorum) → latency</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>"Consistency" is overloaded</b> CAP's C (recency of single-object reads) ≠ ACID's C
            (transaction invariants) ≠ serializability (multi-object). Keep them apart.
        </div>
        <div class="fact blue">
            <b>The rule of thumb</b> Buy the strongest model your correctness actually needs and not
            a rung more. Reserve linearizability for locks, leader election, unique constraints.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Linearizable = serializable"</div>
        <div class="t">
            ✓ Linearizable is recency for single objects; serializable is isolation across
            transactions — different words
        </div>
    </div>
    <div class="rule">
        <span>📈 Memory rule:</span> Climb only as high as correctness demands —{' '}
        <span>eventual → session → causal → linearizable</span>, each rung costs coordination.
    </div>
    <MemoryCheck items={mc6} />
</section>

<section class="lvl" id="l7">
    <div class="lvltop">
        <span class="num">7</span>
        <h2>CAP &amp; PACELC — the real trade-offs</h2>
    </div>
    <p class="tagline">
        CAP isn't "pick 2 of 3." P is forced on you; the only choice is C-vs-A <em>during</em> a
        partition.
    </p>
    <figure>
        <img
            src="diagrams/10-cap-pacelc.png"
            alt="The PACELC decision tree. If there is a Partition, choose Availability or Consistency; Else — the normal case — choose lower Latency or stronger Consistency."
        />
        <figcaption>
            <strong>The PACELC decision tree.</strong> A partition forces the CAP choice (PA vs PC);
            but the far more common no-partition state still forces a Latency-vs-Consistency choice
            (EL vs EC). Most systems are PA/EL or PC/EC. (Abadi, IEEE Computer 2012.)
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>C / A / P</b> C = linearizability · A = every non-failed node answers (not "fast") ·
            P = partition tolerance — <em>not optional</em>, partitions happen.
        </div>
        <div class="fact">
            <b>Under partition</b> CP refuses/blocks on the minority side (etcd, ZooKeeper, Mongo
            default). AP answers with possibly-stale data (Cassandra, DynamoDB, Riak).
        </div>
        <div class="fact blue">
            <b>PACELC</b> if-Partition: A vs C. Else: <b>L vs C</b> — strong consistency costs
            round-trips to a quorum even on a healthy network.
        </div>
        <div class="fact blue">
            <b>Tunable per-request</b> Cassandra/Dynamo let one cluster be EL for feeds and EC for
            balances (consistency level / <code>ConsistentRead</code>).
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "You pick two of C, A, P" / "CAP describes my system always"</div>
        <div class="t">
            ✓ P is mandatory → only choose C-or-A under partition; CAP is silent on healthy networks
            — that's the hole PACELC fills (Brewer 2012)
        </div>
    </div>
    <div class="rule">
        <span>⚖️ Memory rule:</span>{' '}
        <span>if Partition → A or C; Else → Latency or Consistency.</span> Coordination buys
        consistency and always costs latency.
    </div>
    <MemoryCheck items={mc7} />
</section>

<section class="lvl" id="l8">
    <div class="lvltop">
        <span class="num">8</span>
        <h2>Consensus — FLP impossibility + Raft</h2>
    </div>
    <p class="tagline">
        Get nodes to agree on one value despite crashes — using a majority quorum that always
        overlaps.
    </p>
    <Fig5 />
    <figure>
        <img
            src="diagrams/11-raft-election.png"
            alt="A 5-node Raft cluster electing a leader. One node's election timeout fires first; it becomes a candidate in a new term, gathers a majority of votes, and becomes leader."
        />
        <figcaption>
            A 5-node Raft cluster electing a leader: one node's <em>randomized</em> election timeout
            fires first, it becomes a candidate in a new <b>term</b>, gathers a <b>majority</b> of
            votes, and becomes leader. Random timeouts dissolve split votes. (Ongaro &amp;
            Ousterhout, 2014.)
        </figcaption>
    </figure>
    <figure>
        <img
            src="diagrams/12-raft-log.png"
            alt="A leader replicating a log entry. The entry is committed only after a majority of nodes have persisted it; it is then applied to each node's state machine."
        />
        <figcaption>
            A leader replicating a log entry: it is <b>committed</b> only after a <em>majority</em>{' '}
            persists it, then <b>applied</b> to each node's state machine. The leader replies to the
            client only after commit — so an acked write can never be lost.
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>Consensus = 3 properties</b> Agreement (no two decide differently) · Validity
            (decided value was proposed) · Termination (everyone eventually decides).
        </div>
        <div class="fact">
            <b>Majority quorum</b> ⌈(N+1)/2⌉. Two majorities always share a node → two conflicting
            decisions impossible. N=3 tolerates 1, N=5 tolerates 2.
        </div>
        <div class="fact blue">
            <b>Why odd-sized</b> A 4-node cluster needs 3 for majority, tolerates 1 — same as 3
            nodes. The extra even node buys nothing.
        </div>
        <div class="fact blue">
            <b>Log Matching</b> AppendEntries carries the prior entry's index+term; a mismatch is
            rejected and the divergent tail overwritten — committed history never lost.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Raft/Paxos tolerate any kind of node failure"</div>
        <div class="t">
            ✓ They assume <em>crash-stop</em>, not lying; malicious nodes need Byzantine consensus
            (PBFT, 3f+1) (Castro &amp; Liskov 1999)
        </div>
    </div>
    <div class="rule">
        <span>🗳 Memory rule:</span>{' '}
        <span>elect by randomized timeout · commit on a majority · majorities overlap</span> → two
        leaders / two decisions can't coexist.
    </div>
    <MemoryCheck items={mc8} />
</section>

<section class="lvl" id="l9">
    <div class="lvltop">
        <span class="num">9</span>
        <h2>Partitioning / sharding — splitting without hot spots</h2>
    </div>
    <p class="tagline">
        Replication raises availability; <em>partitioning</em> raises the storage and write ceiling.
    </p>
    <figure>
        <img
            src="diagrams/13-hash-vs-range.png"
            alt="Two ways to map keys to partitions, and what each one costs you. Left: key-range partitioning keeps keys sorted (A–H, I–P, Q–Z) so range scans are cheap, but a burst of writes to one range overloads one node. Right: hash partitioning scatters keys evenly across nodes so load is balanced, but a range scan must hit every node."
        />
        <figcaption>
            <strong>Two ways to map keys to partitions, and what each costs.</strong> Key-range
            keeps keys sorted (cheap range scans, but a write burst to one range overloads a node);
            hash scatters keys evenly (balanced load, but a range scan must hit every node).
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Property</th>
            <th>Key-range</th>
            <th>Hash</th>
        </tr>
        <tr>
            <td>Range scan</td>
            <td>cheap — few adjacent partitions</td>
            <td>expensive — scatter to all</td>
        </tr>
        <tr>
            <td>Load distribution</td>
            <td>uneven → hot spots</td>
            <td>even by design</td>
        </tr>
        <tr>
            <td>Example systems</td>
            <td>HBase, Bigtable</td>
            <td>Cassandra, Dynamo</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Rebalancing — never mod N</b> <code>hash(key) mod N</code> moves almost every key
            when N changes. Use a fixed partition count, or <b>consistent hashing</b> (relocates
            ~K/N).
        </div>
        <div class="fact">
            <b>Hot spot / skew</b> One celebrity key melts one partition; hashing can't help (same
            key hashes the same). Fix: random suffix splits the hot key across 100 sub-keys.
        </div>
        <div class="fact blue">
            <b>Request routing</b> Forward via any node (gossip) · a routing tier · partition-aware
            client. The map is kept consistent via ZooKeeper/etcd.
        </div>
        <div class="fact blue">
            <b>Rebalance deliberately</b> Auto-rebalance triggered by a node that merely{' '}
            <em>looks</em> dead (Lesson 1!) can start a data-movement storm.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Add read replicas to fit more data"</div>
        <div class="t">
            ✓ Every replica holds the <em>whole</em> dataset; to raise the storage ceiling you must
            partition
        </div>
    </div>
    <div class="rule">
        <span>🧩 Memory rule:</span>{' '}
        <span>range = cheap scans, hot spots · hash = even load, no scans</span>; grow with fixed
        partitions or consistent hashing, never mod N.
    </div>
    <MemoryCheck items={mc9} />
</section>

{/* ================= LEVEL 10 ================= */}

    <div class="lvltop">

        <h2>Putting it together — the resilience toolkit</h2>
    </div>
    <p class="tagline">
        Every pattern is one trade: at-least-once delivery + idempotent effects =
        "effectively-once."
    </p>
    <Fig6 />
    <figure>
        <img
            src="diagrams/14-outbox-pattern.png"
            alt="The outbox pattern turns a dual write into one atomic write plus an idempotent publish. The service commits the business row and an outbox row in a single database transaction; a separate relay reads unsent outbox rows, publishes them to the broker at-least-once, and marks them sent."
        />
        <figcaption>
            <strong>The outbox pattern</strong> turns a dual write into one atomic write + an
            idempotent publish. The service commits the business row <em>and</em> an outbox row in
            one DB transaction; a relay publishes unsent rows at-least-once and marks them sent.
        </figcaption>
    </figure>
    <Fig7 />
    <div class="facts">
        <div class="fact">
            <b>Dual-write problem</b> Writing DB <em>and</em> publishing as two steps: a crash in
            the gap leaves them disagreeing. The outbox collapses it into one atomic write.
        </div>
        <div class="fact">
            <b>Effectively-once</b> There's no exactly-once <em>delivery</em>. Ship at-least-once
            delivery + idempotent effects. Always ask: exactly-once across <em>which boundary</em>?
        </div>
        <div class="fact blue">
            <b>End-to-end argument</b> A correctness guarantee (dedup, idempotency) can only be
            enforced by the <em>endpoints</em>; the network is a best-effort accelerator. (Saltzer,
            Reed &amp; Clark, 1984.)
        </div>
        <div class="fact blue">
            <b>Retry budgets</b> Independent retries multiply down a chain (3×3×3 = 27 calls). Cap
            retries (~10% of volume) and retry at one layer. (Google SRE.)
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "TCP / the broker gives me exactly-once, so my handler is safe"</div>
        <div class="t">
            ✓ Only your consumer knows it already processed event abc-123 — put dedup at the
            endpoint
        </div>
    </div>
    <div class="rule">
        <span>🛡 Memory rule:</span>{' '}
        <span>
            retry idempotent ops with backoff+jitter · outbox the dual write · saga + compensate ·
            enforce correctness at the endpoints.
        </span>
    </div>

<h2 class="sec" id="cheat">
    ★ Cheat sheet — buzzword → engineering
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> can't tell slow from dead · timeout = a guess · exactly-once
        = at-least-once + idempotency · no global clock → cause not time · FIFO ⊂ causal ⊂ total ·
        R+W&gt;N overlaps · climb consistency only as far as needed · P is mandatory (C-or-A under
        partition) · majorities overlap · range vs hash · outbox kills the dual write · enforce
        correctness at the endpoints.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Buzzword</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>Eventual consistency</td>
            <td>if writes stop, replicas converge — meanwhile any stale read is legal</td>
        </tr>
        <tr>
            <td>Quorum</td>
            <td>R + W &gt; N so read and write sets share a node</td>
        </tr>
        <tr>
            <td>CP / AP</td>
            <td>under partition: block the minority / serve stale data</td>
        </tr>
        <tr>
            <td>PACELC</td>
            <td>if Partition: A-vs-C; Else: Latency-vs-Consistency</td>
        </tr>
        <tr>
            <td>Consensus</td>
            <td>agree on one value despite crashes; ≈ total-order broadcast ≈ a replicated log</td>
        </tr>
        <tr>
            <td>Raft commit</td>
            <td>entry persisted on a majority, then applied to the state machine</td>
        </tr>
        <tr>
            <td>Consistent hashing</td>
            <td>nodes &amp; keys on a ring; add a node, move only ~K/N keys</td>
        </tr>
        <tr>
            <td>Exactly-once</td>
            <td>marketing for at-least-once delivery + idempotent effects</td>
        </tr>
        <tr>
            <td>Outbox / saga</td>
            <td>one atomic local write + relay / local txns + compensations</td>
        </tr>
    </table>
    <h3>Three instincts to install</h3>
    <p style="margin-bottom:0">
        ① At every network boundary ask <strong>"what if this times out but succeeded?"</strong> ②
        Buy the <strong>weakest guarantee</strong> (delivery, ordering, consistency) your
        correctness actually needs. ③ Put each correctness guarantee at the{' '}
        <strong>layer that has the knowledge</strong> to enforce it (the end-to-end argument).
    </p>
</div>
