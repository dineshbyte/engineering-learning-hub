# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is a domain agent, and how is it different from a general assistant?'
      a: >-
          <b>Hit these points:</b> a domain agent is an <b>LLM using tools in a loop</b>, scoped to one
          business domain (billing, support, SRE) with a defined goal, a fixed tool surface, and an
          owner &rarr; a general assistant has an open-ended goal and an unbounded surface; the domain
          agent does <b>one job</b> &rarr; the design artifact is the <b>boundary</b>, not the model
          &rarr; because the job is narrow you can actually evaluate it, secure it, and size its blast
          radius &rarr; the model is interchangeable; the boundary, tools, and policies are what you
          own.
    - level: 'core'
      q: 'List the 9 parts of the design template in order, and say which one comes first.'
      a: >-
          <b>Hit these points:</b> (1) responsibilities &mdash; the boundary, drawn first &rarr; (2)
          tools &mdash; the surface &rarr; (3) context &rarr; (4) memory &rarr; (5) runtime &rarr; (6)
          evaluation &rarr; (7) security &rarr; (8) approval gates &rarr; (9) ROI / build-vs-buy &rarr;
          <b>responsibilities first</b> because the boundary defines the surface, the surface defines
          the risk, the risk defines the gates, and value decides whether you ship at all.
    - level: 'core'
      q: 'What is blast radius, and how does it relate to approval gates?'
      a: >-
          <b>Hit these points:</b> blast radius = the <b>worst-case scope of damage</b> if one action
          is wrong, malicious, or repeated &rarr; a read-only query has near-zero blast radius; an
          irreversible write or a fan-out to many records has large blast radius &rarr; the rule: the
          more an agent can do <b>unsupervised</b>, the more an approval gate matters &rarr; so you gate
          <b>by action</b> &mdash; auto-allow the safe reads, pause the high-blast writes for a human.
    - level: 'core'
      q: "What's the default build-vs-buy answer for a domain agent, and why?"
      a: >-
          <b>Hit these points:</b> split the system into <b>commodity plumbing</b> (model, runtime
          loop, tool protocol, eval harness) and <b>domain glue</b> (boundary, tools, policies, gates)
          &rarr; the plumbing is the same for every domain &mdash; <b>buy it</b> &rarr; the glue
          encodes your business rules and risk tolerance, which no vendor can know &mdash; <b>build
          it</b> &rarr; the call is really "where is our differentiation," and it's in the boundary and
          policies, not the runtime.
    - level: 'senior'
      q: 'A teammate says "we''ll add a system-prompt rule: never refund over $50." Push back.'
      a: >-
          <b>Hit these points:</b> a prompt rule is a <b>request to the model, not an enforced
          control</b> &mdash; it can be ignored, overridden, or jailbroken via injected input &rarr; a
          refund over $50 is a high-blast, irreversible write; that's exactly where the boundary must
          be <b>enforced in code</b> &rarr; enforce it in the tool: the refund tool caps the amount, or
          requires an approval gate above the threshold &rarr; keep the prompt rule as defense-in-depth
          UX, but the gate is the boundary &rarr; this mirrors authorization in the data layer &mdash;
          never in a comment asking the query to behave.
    - level: 'senior'
      q: "What goes into an agent's ROI calculation, and what's the most common trap?"
      a: >-
          <b>Hit these points:</b> ROI = value returned vs <b>all-in cost</b> &rarr; all-in cost =
          tokens + tool calls + <b>human-review time</b> + failure/cleanup cost, not just the model
          bill &rarr; the trap: an agent that needs review on <b>every</b> action often costs more than
          the human it replaced &mdash; you pay the model AND the reviewer &rarr; ROI is coupled to
          gate design: more gates means more safety but lower ROI &rarr; it's the build-or-kill test
          &mdash; if deflected work doesn't clear all-in cost, don't ship it.
    - level: 'senior'
      q: 'Your agent occasionally loops for dozens of steps, burning budget without resolving. Which template parts failed?'
      a: >-
          <b>Hit these points:</b> the symptom is a <b>runaway loop</b>, so the primary gap is part 5
          (runtime) &mdash; no step cap, no cost cap &rarr; add a hard step limit and a per-session cost
          ceiling so it terminates and escalates instead of spinning &rarr; it's also an <b>eval
          gap</b> (part 6): no resolution metric means you didn't catch it &rarr; possibly a <b>context
          gap</b> (part 3): if the right info never lands in the window, the model keeps retrying &rarr;
          fix the runtime to stop the bleeding, then instrument eval, then check whether retrieval
          feeds the loop what it needs.
    - level: 'senior'
      q: 'Why is the tool surface the hinge that everything downstream turns on?'
      a: >-
          <b>Hit these points:</b> the surface is the agent's <b>reach</b> &mdash; it defines what the
          agent can do, so it defines what can go wrong &rarr; a read-only surface has near-zero blast
          radius: no gates, light security, easy ROI &rarr; add one irreversible write and you now need
          a gate on it (part 8), a tighter security review (part 7: can injected input trigger it?), an
          eval that checks it fires correctly (part 6), and higher all-in cost from review (part 9)
          &rarr; so parts 6&ndash;9 are all <b>functions of part 2</b> &rarr; keep the surface as small
          as the job allows, because every tool cascades cost and risk through the rest of the
          template.
    - level: 'staff'
      q: "You're standing up agents for five business domains at once. How do you avoid five bespoke designs?"
      a: >-
          <b>Hit these points:</b> the whole point of a template is that the <b>method is shared and
          only the instantiation varies</b> &rarr; standardize the commodity layer once: one runtime
          (loop, retries, limits), one tool protocol (e.g. MCP), one eval harness, one gate mechanism
          &mdash; buy or build it a single time &rarr; per domain, fill in only parts 1, 2, 8, 9: the
          boundary, the tool surface, where gates fire, and the ROI test &mdash; the parts that encode
          that domain's rules &rarr; this gives you consistent security and observability across all
          five and lets one platform team own the plumbing while domain teams own the glue &rarr; the
          failure mode to avoid is five teams each rebuilding a loop and a gate &mdash; that's where
          bespoke creeps in and security drifts.
    - level: 'staff'
      q: 'Two proposed agents are identical except one has a read-only surface and the other can also write. Compare the full design implications.'
      a: >-
          <b>Hit these points:</b> the surface difference cascades through every later part &rarr;
          read-only: near-zero blast radius &mdash; no approval gates, security is mostly "don't leak
          data," eval checks answer quality, ROI is clean because there's no cleanup cost &rarr; the
          writer: every write is an irreversible action &mdash; now you need a gate sized to its blast
          radius (part 8), a security review for whether injected input can trigger it (part 7), an
          eval that the write fires correctly and only when intended (part 6), and a higher all-in cost
          from human review (part 9) &rarr; the design lesson: <b>capability is not free</b> &mdash; the
          moment you cross from read to write you've signed up for the gate, the review, and the
          cleanup-cost line in the ROI &rarr; so justify each write against the deflection it buys, and
          gate it.
    - level: 'staff'
      q: "How does this track build on AI Agents and Context Engineering, and what's genuinely new here?"
      a: >-
          <b>Hit these points:</b> AI Agents taught the <b>primitives</b> &mdash; LLM, tool, agent
          loop, runtime, memory, MCP &mdash; and introduced business-agent archetypes &rarr; Context
          Engineering taught how to <b>curate the window</b> &mdash; retrieval, assembly, caching,
          ordering, security &rarr; this track turns those into a <b>design discipline</b>: a
          repeatable template you instantiate per domain &rarr; parts 3&ndash;4 (context, memory) lean
          on Context Engineering; parts 2, 5 (tools, runtime) lean on AI Agents &rarr; genuinely new
          here is the <b>boundary, the approval gates, and the ROI/build-vs-buy test</b> &mdash; the
          parts specific to a business domain that no general track can hand you.
    - level: 'design'
      q: 'Design a Support agent end to end using the 9-part template. Where do the gates go and why?'
      a: >-
          <b>A strong answer covers:</b> <b>responsibilities</b> &mdash; answer account/order questions,
          draft replies, issue small refunds; refuse legal/medical advice and anything outside support
          &rarr; <b>tools</b> &mdash; read order, read account, search KB, issue refund, escalate;
          smallest set that covers the job &rarr; <b>context</b> &mdash; the question + retrieved KB +
          the order record, scoped to the customer; <b>memory</b> &mdash; prior tickets retrieved, not
          dumped &rarr; <b>runtime</b> &mdash; bounded loop with retries and a step/cost limit &rarr;
          <b>evaluation</b> &mdash; deflection, resolution quality, escalation correctness on a labelled
          set &rarr; <b>security</b> &mdash; ticket text and KB are untrusted, scope retrieval by
          customer, least-privilege tools &rarr; <b>gates</b> &mdash; reads auto-allow (zero blast);
          refund above a cap and escalate-to-billing pause for a human (high blast) &rarr; <b>ROI</b>
          &mdash; deflected agent-hours vs tokens + review time; if every refund needs review the gate
          eats the savings, so set the auto-approve cap where the blast radius is acceptable.
    - level: 'design'
      q: "Design an SRE incident-triage agent. Contrast its gates and ROI with the Support agent's."
      a: >-
          <b>A strong answer covers:</b> <b>responsibilities</b> &mdash; read alerts/logs/metrics,
          correlate, summarize, propose remediations; refuse to apply changes it can't reason about
          &rarr; <b>tools</b> &mdash; query metrics, read logs, read deploy history, (gated) restart
          service, (gated) roll back; the writes are the high-blast ones &rarr; <b>context/memory</b>
          &mdash; the firing alert + recent deploys + prior similar incidents retrieved &rarr;
          <b>runtime</b> &mdash; bounded loop, step/cost cap &rarr; <b>evaluation</b> &mdash;
          time-to-correct-diagnosis and false-remediation rate, not just deflection &rarr;
          <b>security</b> &mdash; treat log/alert content as untrusted (it can carry injected text),
          least-privilege infra tools &rarr; <b>gates vs Support</b> &mdash; SRE's blast radius is far
          higher: a wrong restart or rollback can take down prod, so far more should be propose-only
          with a human gate; the Support agent can safely auto-resolve most tickets, the SRE agent
          should mostly recommend &rarr; <b>ROI</b> &mdash; the value is faster MTTR and analyst time
          saved, but if every remediation is gated the deflection is in the <em>diagnosis</em>, not the
          action &mdash; which is fine, because the cleanup cost of an unsupervised bad remediation
          dwarfs the saving.

    <div class="lvltop">
        <span class="num">1</span>
        <h2>The Boundary &amp; the Surface</h2>
    </div>
    <p class="tagline">
        A <b>domain agent</b> is an LLM using tools in a loop, scoped to one business domain with a
        defined goal, a fixed tool surface, and an owner. The design artifact is the <b>boundary</b>,
        not the model &mdash; and you draw it before you wire a single tool.
        <sup>
            <a href="#s2" id="r2">2</a>
        </sup>
    </p>

    <h3>Definition &amp; why it exists</h3>
    <div class="facts">
        <div class="fact">
            <b>Is</b> A domain agent does one job: answer billing questions, triage an incident, draft
            a research brief. Its <em>responsibility boundary</em> is its API contract &mdash; inputs it
            accepts, actions it owns, things it must defer, escalate, or refuse.
        </div>
        <div class="fact">
            <b>Why it exists</b> A general assistant has an open goal and an unbounded surface &mdash;
            you can't evaluate it, secure it, or size its risk. Narrow the job and all four become
            tractable.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> A job description. A teller cashes cheques and answers account
            questions; they don't approve mortgages or give tax advice. The role is the boundary
            &mdash; written before the desk is built.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> A service interface. You define the contract (accepted requests, owned
            operations, errors it throws) before you implement the handlers. Tools are the
            implementation; the boundary is the interface.
        </div>
    </div>

    <h3>Boundary &rarr; surface: every tool widens the reach</h3>
    <Fig2 />

    <div class="myth">
        <div class="m">&#10007; "Pick the tools first, then figure out what the agent should do."</div>
        <div class="t">
            &#10003; Boundary first. Choosing tools first decides the reach before the job &mdash; you
            end up with a wide surface and no contract for what's out of bounds.
        </div>
    </div>
    <div class="myth">
        <div class="m">
            &#10007; "More tools make the agent more capable, so add everything it might need."
        </div>
        <div class="t">
            &#10003; Every tool added widens capability <em>and</em> blast radius. The smallest surface
            that covers the job is the goal &mdash; unused reach is pure risk.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span> <span>Boundary before tools.</span> The boundary is the
        agent's <span>contract</span>; the tool surface is the smallest set that <span>fits</span> it.
    </div>

    <details class="mc">
        <summary>Memory check</summary>
        <ul>
            <li>
                What is a domain agent's "design artifact"?{' '}
                <span class="ans">
                    &rarr; the responsibility boundary &mdash; inputs accepted, actions owned, things
                    deferred/escalated/refused; not the model
                </span>
            </li>
            <li>
                Why draw the boundary before choosing tools?{' '}
                <span class="ans">
                    &rarr; the boundary tells you the smallest tool surface that covers the job;
                    tools-first decides the reach before the job
                </span>
            </li>
            <li>
                What does adding a tool cost you?{' '}
                <span class="ans">
                    &rarr; more capability AND more blast radius; an unused tool is risk you carry for
                    nothing
                </span>
            </li>
        </ul>
    </details>

    <details class="iq">
        <summary>
            M1 &mdash; A teammate wants to give the support agent a "run any SQL" tool "so it can
            answer anything." Talk them out of it.
        </summary>
        <div class="a">
            <b>Hit these points:</b> "run any SQL" is the widest possible surface &mdash; it can read
            every tenant's data, mutate rows, drop tables &rarr; that's enormous blast radius for a job
            that needs three read tools &rarr; the boundary says the agent answers <em>support</em>
            questions, so the surface is read-order, read-account, search-KB &mdash; scoped,
            parameterized, least-privilege &rarr; an open SQL tool also makes injection catastrophic:
            untrusted ticket text could steer a query &rarr; replace it with narrow tools that encode
            exactly the reads the boundary allows, and the agent loses no real capability while
            shedding most of the risk.
        </div>
    </details>

    <h3>Approval gates &mdash; sized to blast radius</h3>
    <Fig4 />

    <h3>ROI &mdash; and why the gate design decides it</h3>
    <div class="facts">
        <div class="fact">
            <b>Is</b> ROI = value returned (deflected human work) vs all-in cost: tokens + tool calls +
            <em>human-review time</em> + failure/cleanup cost. Not just the model bill.
        </div>
        <div class="fact">
            <b>Why it exists</b> An agent that needs review on every action often costs <em>more</em>
            than the human it replaced &mdash; you pay the model <em>and</em> the reviewer. ROI is the
            build-or-kill test.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> Hiring an assistant you must double-check on every task. If you re-do
            everything they do, you've added cost, not capacity.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> A cache with a 0% hit rate: all the machinery, none of the saving. If
            the gate fires on every action, the deflection never materializes.
        </div>
    </div>

    <h3>Build vs buy &mdash; commodity plumbing vs domain glue</h3>
    <div class="myth">
        <div class="m">&#10007; "Building an agent platform from scratch is our differentiation."</div>
        <div class="t">
            &#10003; The loop, model, tool protocol, and eval harness are commodity &mdash;
            <em>buy</em> them. Your differentiation is the boundary, tools, and policies no vendor can
            know &mdash; <em>build</em> those.
        </div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Buy the whole thing &mdash; the vendor knows agents better than we do."</div>
        <div class="t">
            &#10003; No vendor knows <em>your</em> business rules, refund limits, or risk tolerance.
            Buy the platform; build the domain glue. That glue is the agent's actual value.
        </div>
    </div>

            <b>Hit these points:</b> deflection and ROI aren't the same thing &mdash; auto-approving
            every refund maximizes deflection but exposes the full blast radius of the refund tool
            &rarr; a confused or injected agent can now issue unbounded refunds with no human in the
            path; one incident's cleanup cost can dwarf months of savings &rarr; the right move is a
            threshold: auto-allow refunds up to a bounded cap where the blast radius is acceptable, gate
            the rest &rarr; that keeps most of the deflection (most refunds are small) while bounding
