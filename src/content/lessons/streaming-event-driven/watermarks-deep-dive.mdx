# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Event time vs processing time — define each and why they diverge.'
      a: >-
          <b>Hits:</b> event time is when the thing actually happened (stamped at the source);
          processing time is when the record reaches the operator's wall clock. They diverge because
          of buffering, network delay, retries, and replay — arrival lags origin, unboundedly and out
          of order. Bucket by event time for reproducible, replay-stable results; processing time is
          only acceptable when you don't care about correctness under delay.
    - level: 'core'
      q: "Define a watermark precisely, and say why it's like a timeout."
      a: >-
          <b>Hits:</b> a single, monotonic, non-decreasing timestamp carried with the data; asserts
          <em>no future record will have event time &le; W</em>. It's a heuristic the system commits
          to because it can't wait forever for certainty — exactly like a timeout for failure
          detection. Red flag: confusing it with the processing-time clock.
    - level: 'core'
      q: 'Name the window types — tumbling, sliding, session — and what each is for.'
      a: >-
          <b>Hits:</b> tumbling = fixed-size, non-overlapping buckets (each event in exactly one) —
          periodic aggregates. Sliding = fixed-size but overlapping by a step, so an event lands in
          several windows — moving averages. Session = dynamic, data-driven windows bounded by a gap
          of inactivity, so window length varies per key — user activity bursts. The first two are
          aligned to the clock; sessions are defined by the data.
    - level: 'core'
      q: 'What is allowed lateness, and what happens to data past it?'
      a: >-
          <b>Hits:</b> allowed lateness extends how long a window's state is kept <em>after</em> the
          watermark passes its end, so late stragglers can still re-fire and correct the result. Once
          lateness is exceeded the state is dropped — events arriving after that are "truly late" and
          should be routed to a side output / dead-letter, never silently discarded. It is the dial
          trading memory/state lifetime against completeness.
    - level: 'senior'
      q: 'Perfect vs heuristic watermark — when can you have a perfect one, and what does it buy?'
      a: >-
          <b>Hits:</b> perfect requires perfect knowledge of timestamps — an in-order single
          partition or a hard, known max delay; then there is <em>never any late data</em>. Heuristic
          estimates (max-seen &minus; bound B) and is wrong in one of two directions. Bonus: most real
          sources are heuristic, so design for late data.
    - level: 'senior'
      q: 'Your pipeline is processing records but emitting nothing, with no error. Diagnose.'
      a: >-
          <b>Hits:</b> an operator's watermark is the <em>min</em> of its inputs; a silent/idle
          partition's stale watermark pins the min and freezes every downstream window. Fix: idleness
          detection (<code>withIdleness</code> / idle-source handling) to exclude the idle input,
          re-include on first record. Not a crash, not back-pressure.
    - level: 'senior'
      q: 'Why is the propagation rule a MIN and not a max or average?'
      a: >-
          <b>Hits:</b> the watermark is a <em>completeness</em> claim; you cannot assert "complete up
          to W" on the output if even one input might still deliver something older than W. The
          slowest input governs. Bonus: buffered state with pending event-time timers can hold it back
          further.
    - level: 'senior'
      q: '"But what about late data?" — answer it without dropping anything.'
      a: >-
          <b>Hits:</b> with late-firing triggers + an accumulating (or +retracting) mode, a straggler
          arriving after W re-fires the window and <em>corrects</em> the result. Allowed lateness
          bounds how long state is kept; past it, route truly-late events to a side output /
          dead-letter — never silently. The watermark stops being a guillotine and becomes a
          completeness estimate.
    - level: 'staff'
      q: 'Separate windowing, triggers, and accumulation. Where does the watermark fit?'
      a: >-
          <b>Hits:</b> windowing = <em>where</em> in event time data is grouped; triggers =
          <em>when</em> output is emitted (early/on-time/late, possibly many times); accumulation =
          <em>how</em> successive results relate (discarding / accumulating / +retracting). The
          watermark only <em>informs</em> the on-time firing — it does not decide when the window
          fires. This is the Dataflow Model's core insight.
    - level: 'staff'
      q: 'Argue that there is no "correct" watermark setting.'
      a: >-
          <b>Hits:</b> a watermark trades three axes that cannot all win — <em>completeness</em> (wait
          for stragglers), <em>latency</em> (fire early), and <em>cost</em> (state held open). An
          aggressive watermark fires fast but drops/corrects more; a conservative one is complete but
          slow and memory-hungry. So the "right" setting is a business choice about which failure you
          tolerate, not a property of the data. Staff signal: the candidate ties the setting to a
          downstream SLA, not to a default.
    - level: 'staff'
      q: 'How do triggers, allowed lateness, and accumulation mode interact to give correct-but-eventually-consistent output?'
      a: >-
          <b>Hits:</b> the trigger may fire early (speculative), on-time (at the watermark), and late
          (per straggler within allowed lateness); accumulation mode decides whether each firing
          replaces, adds to, or retracts-and-replaces the prior result. Together they let a window emit
          a fast approximate answer and converge to the complete one as late data arrives — the
          streaming analogue of eventual consistency. Allowed lateness is the cutoff after which
          convergence stops and state is reclaimed.
    - level: 'design'
      q: 'Design event-time windowing for an out-of-order stream (e.g. mobile clients with intermittent connectivity).'
      a: >-
          <b>A strong answer covers:</b> stamp event time at the source and key by user/device; pick
          the window type to the metric (tumbling for periodic counts, session for activity bursts); a
          heuristic watermark with a bound B sized to observed lateness; triggers that fire early for a
          live estimate, on-time at the watermark, and late within an allowed-lateness budget, with an
          accumulating-or-retracting mode so results converge; truly-late events sent to a side output
          for reconciliation; idleness detection so a quiet device doesn't freeze the min watermark;
          and explicit monitoring of watermark skew and late-drop rate. The candidate should frame the
          dials as a completeness/latency/cost choice tied to the SLA.
    - level: 'design'
      q: 'Design a monthly billing rollup vs a real-time fraud signal on the same event stream.'
      a: >-
          <b>A strong answer covers:</b> the same primitives, opposite dials. Billing: a conservative
          watermark, on-time firing only, long allowed lateness and durable state &rarr; completeness
          over speed, accepting latency and memory, with reconciliation against a side output before
          close. Fraud: an aggressive watermark, early + late firings, short state and an
          accumulating/retracting mode &rarr; low latency, tolerating corrections. The candidate names
          completeness vs latency vs cost as the governing trade-off and maps each workload to a point
          on it, rather than reaching for one "correct" configuration.

    </table>

    <div class="myth">
        <div class="m">✗ "The watermark fires the window, exactly once"</div>
        <div class="t">
            ✓ The <em>trigger</em> fires — early, on-time, and late; the watermark only informs the
            on-time firing
        </div>
    </div>

    <div class="rule">
        <span>🎯 Memory rule:</span> Windowing = where, triggers = when, accumulation = how — so late
        data isn't a guillotine; the window re-fires and corrects.
    </div>

    <MemoryCheck items={mc4} />

</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Allowed lateness — and the three-way trade-off</h2>
    </div>
    <p class="tagline">
        Late firings can't run forever. Allowed lateness is the horizon, and the third dial.
    </p>

    <Fig5 />

    <div class="facts">
        <div class="fact">
            <b>Allowed lateness</b> The extra duration a window keeps its state after <code>W</code>{' '}
            passes its end; late events within it re-fire.
        </div>
        <div class="fact">
            <b>After the horizon</b> State is garbage-collected; later events are <em>truly</em> late →
            drop or route to a side output / dead-letter.
        </div>
        <div class="fact blue">
            <b>Fire early</b> Lower latency, less complete. <b>Wait for W</b> → more complete, higher
            latency. <b>Hold state longer</b> → tolerate lateness, at memory cost.
        </div>
        <div class="fact blue">
            <b>Dials</b> Watermark, triggers, allowed-lateness set completeness vs latency vs cost. You
            can't max all three.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Late data is just dropped and lost"</div>
        <div class="t">
            ✓ Within allowed lateness it re-fires; past it, route to a side output — never silently, if
            you value your data
        </div>
    </div>

    <div class="rule">
        <span>🎚️ Memory rule:</span> Completeness vs latency vs cost — the three dials are watermark,
        triggers, and allowed lateness; pick the two your use case needs.
    </div>

    <MemoryCheck items={mc5} />

</section>

{/* ================= DIAL SETTINGS BY WORKLOAD ================= */}

<h2 class="sec" id="usecases">
    Same primitives, opposite dials — by workload
</h2>

<p>
    Grounding: the same windowing primitives, set to opposite ends, for different jobs. This
    per-workload judgement is the whole point.
</p>

<table>
    <tr>
        <th>Workload</th>
        <th>Wants</th>
        <th>Dial settings</th>
    </tr>
    <tr>
        <td>Fraud check</td>
        <td>Low latency; tolerates re-firing &amp; correction</td>
        <td>aggressive watermark · early + late firings · short state</td>
    </tr>
    <tr>
        <td>Billing rollup</td>
        <td>Completeness; accepts delay</td>
        <td>conservative watermark · on-time firing · long allowed lateness</td>
    </tr>
    <tr>
        <td>Live dashboard</td>
        <td>Fast, improving estimate</td>
        <td>frequent early firings · accumulating mode · modest lateness</td>
    </tr>
    <tr>
        <td>Exact financial sum</td>
        <td>Correctness across late updates</td>
        <td>accumulating + retracting · long allowed lateness · side output for the rest</td>
    </tr>
    <tr>
        <td>Multi-source join</td>
        <td>Not freezing on a quiet input</td>
        <td>
            <code>withIdleness</code> on every source · watch the min
        </td>
    </tr>
</table>

<p class="tagline">
    Pattern to notice: the dials don't change the <em>engine</em> — they change which of
    completeness, latency, and cost you pay for. Naming that trade-off out loud is the senior move.
    <sup><a href="#s2" id="r2">2</a></sup>
</p>

<h2 class="sec" id="cheat">
    ★ Cheat sheet — term → what it actually is
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> Watermark = a timeout for completeness · Perfect = never late,
        rare · Heuristic = a guess, too-fast drops / too-slow lags · Min rule = slowest input governs ·
        Idle stall = silent freeze, fix with idleness detection · Triggers = when (many times) ·
        Accumulation = how firings relate · Allowed lateness = how long state lives.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Term</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>Watermark W</td>
            <td>Monotonic lower bound: no future event ≤ W</td>
        </tr>
        <tr>
            <td>Skew</td>
            <td>How far W lags true event-time progress</td>
        </tr>
        <tr>
            <td>Perfect vs heuristic</td>
            <td>Exact (no late data) vs estimated (late data happens)</td>
        </tr>
        <tr>
            <td>Min rule</td>
            <td>Operator W = min(input watermarks); slowest governs</td>
        </tr>
        <tr>
            <td>Idle-partition stall</td>
            <td>Silent input pins the min → downstream freezes</td>
        </tr>
        <tr>
            <td>Trigger</td>
            <td>
                Decides <em>when</em> to emit: early / on-time / late
            </td>
        </tr>
        <tr>
            <td>Accumulation mode</td>
            <td>Discarding / accumulating / +retracting</td>
        </tr>
        <tr>
            <td>Allowed lateness</td>
            <td>How long window state survives for late firings</td>
        </tr>
        <tr>
            <td>Side output</td>
            <td>Where truly-late events go (a dead-letter)</td>
        </tr>
    </table>
    <h3>Three rules for the on-call chair</h3>
    <p style="margin-bottom:0">
        ① A watermark is a <strong>guess</strong>, not a fact — a timeout for completeness. ② An
        operator's watermark is the <strong>min of its inputs</strong>; one idle input freezes
        everything — reach for <strong>idleness detection</strong>. ③ <em>When</em> a window fires is
        the <strong>trigger</strong>, not the watermark — so late data corrects, it doesn't have to
        drop.
    </p>
</div>
