# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
# Only the 13 LEVELED disclosures from the #interview panel. The 5 un-leveled ones
# stay inline in the body (see the NOTE above).
interview:
    - level: 'core'
      q: 'Define agent systems engineering and name its core concerns.'
      a: >-
          <b>Hit these points:</b> agent engineering = make one agent work (prompt + tools + loop)
          &rarr; agent <b>systems</b> engineering = run it as a production system &rarr; core
          concerns: planning, state management, execution engine, loop control, human-in-the-loop,
          failure recovery, orchestration, long-running durability, cost, observability &rarr; the
          through-line: an agent run is a long-lived, non-deterministic distributed computation, and
          all of those follow from that.
    - level: 'core'
      q: 'Walk through the agent loop and say exactly when it stops.'
      a: >-
          <b>Hit these points:</b> observe (gather results/context) &rarr; think (model decides next
          action) &rarr; act (tool call) &rarr; repeat &rarr; <b>completion stops:</b> model emits a
          final answer with no tool call, or calls an explicit done/submit tool, ideally confirmed by
          an external verifier &rarr; <b>safety stops (runtime):</b> step cap, wall-clock timeout,
          token/cost budget, repetition detector, human interrupt &rarr; the model proposes stopping;
          the runtime decides &mdash; stops are enforced outside the prompt.
    - level: 'core'
      q: 'What is agent state, and why is the LLM called stateless?'
      a: >-
          <b>Hit these points:</b> state = goal/current task + the plan + message history + tool
          outputs/intermediate results &rarr; the LLM is a pure function f(input)&rarr;output with no
          memory between calls &rarr; the <b>runtime</b> makes it stateful by re-assembling and
          re-sending context every turn &rarr; the context window is the slice of state the model
          sees this call &rarr; externalize state and you can scale, checkpoint, and recover; trap it
          in process memory and you can't.
    - level: 'core'
      q: 'Name the four rungs of agent failure recovery and a one-line example of each.'
      a: >-
          <b>Hit these points:</b> <b>retry</b> &mdash; transient idempotent tool error, backoff +
          jitter, bounded &rarr; <b>fallback</b> &mdash; degrade to a smaller model / cache /
          alternate tool &rarr; <b>compensate</b> &mdash; irreversible side effect already happened,
          run a registered undo (Saga) &rarr; <b>escalate</b> &mdash; recovery exhausted, dead-letter
          + human &rarr; climb only as far as needed, and wrap tools in circuit breakers.
    - level: 'senior'
      q: 'Why is success detection hard, and how do you make stopping reliable?'
      a: >-
          <b>Hit these points:</b> the loop's natural stop is the model self-reporting "done," and
          the model is frequently wrong &mdash; it quits early on unfinished work or loops when
          finished &rarr; so never trust self-report alone: add an <b>external verifier</b> (tests
          pass, schema validates, artifact exists) &rarr; and add runtime <b>safety stops</b>
          independent of the model (step cap, wall-clock timeout, budget, repetition detector, human
          halt) &rarr; the senior framing: completion is a verification problem, termination is a
          control problem &mdash; solve them separately, both in the runtime.
    - level: 'senior'
      q: 'Why is multi-agent so often the wrong call, and how do you decide?'
      a: >-
          <b>Hit these points:</b> teams reach for it for prestige or to "add intelligence," but
          intelligence is a model/context lever, not a topology lever &rarr; each split buys
          coordination overhead, a context handoff that can drop information, harder debugging
          ("which agent broke?"), and N&times;M cost &rarr; decide like a microservice split: only
          when work is genuinely <b>independent &amp; parallel</b>, context is <b>separable</b> and
          exceeds a window, or subtasks need <b>distinct toolsets</b> &rarr; default to a single
          agent (or a router) and earn each split with a real boundary.
    - level: 'senior'
      q: 'Map agent failures onto distributed-systems patterns &mdash; and name the one agents get wrong.'
      a: >-
          <b>Hit these points:</b> tool failure = downstream service failure &rarr; retry + circuit
          breaker; model failure = bad RPC response &rarr; validate + repair + retry on another tier;
          context failure = cache pollution &rarr; compress/re-rank/re-inject; planning failure = bad
          query plan &rarr; re-plan; execution failure = partial write &rarr; saga / idempotency
          &rarr; the one agents get wrong is <b>compensation</b>: there's no rollback for a sent email
          or an issued refund, so every irreversible action needs a registered compensating action
          &mdash; plus idempotency keys upstream so retries don't double-apply.
    - level: 'senior'
      q: "An agent's bill 10&times;'d overnight. Diagnose and design the controls."
      a: >-
          <b>Hit these points:</b> first suspects &mdash; a runaway loop (no progress, repeated
          calls), context bloat (history re-sent and growing), or a multi-agent fan-out amplifying
          both &rarr; cost is super-linear because input grows each step, so the unit to watch is
          <b>cost per completed task</b>, attributed per tenant &rarr; controls: prompt caching on
          the stable prefix, history compaction, model tiering, per-task step + token budgets with a
          kill switch, and an anomaly alert at ~3&times; baseline so it trips an incident before the
          invoice &rarr; close the loop with the runtime guards from loop control so "runaway" can't
          happen silently.
    - level: 'staff'
      q: 'Design the execution + durability layer for an agent that runs for hours and survives crashes.'
      a: >-
          <b>Hit these points:</b> model the run as a <b>durable workflow</b>, not an in-memory loop
          &mdash; the agent loop is the non-deterministic step <em>inside</em> deterministic
          orchestration &rarr; <b>state</b> (work-list with per-item status, plan, cursor, artifacts)
          lives in a durable store; <b>checkpoint</b> after each meaningful item &rarr; on restart,
          <b>resume</b> from the last checkpoint via deterministic replay, skipping completed items
          &rarr; every replayed/ retried step is <b>idempotent</b> (keys) so resume can't
          double-charge or duplicate &rarr; layer recovery (retry&rarr;fallback&rarr;compensate&rarr;escalate),
          HITL gates before irreversible writes, per-item budgets, and per-item traces + an immutable
          audit log &rarr; name the through-line: borrow durable-execution discipline (Temporal-style)
          so a non-deterministic worker gets deterministic retries, checkpoints, and observability.
    - level: 'staff'
      q: 'An engineer wants to convert a stable, well-defined business process into an agent. Make the call.'
      a: >-
          <b>Hit these points:</b> if the steps are known and rarely change, an agent is the
          <b>wrong tool</b> &mdash; you'd pay non-determinism, latency, and token cost for control
          flow a workflow or plain code handles deterministically, debuggably, and testably &rarr;
          "agent for everything" is the dominant anti-pattern &rarr; the senior move: put it on the
          workflow&harr;agent spectrum &mdash; fixed path &rarr; workflow; open-ended path the model
          must decide &rarr; agent; hybrid &rarr; workflow with agent steps only where the path is
          genuinely unknown &rarr; reserve agency for the irreducible uncertainty and keep the rest
          deterministic &rarr; this is also cheaper to operate and far easier to get past a review.
    - level: 'staff'
      q: '"Ship it fully autonomous." Make the principal-level call on autonomy and reliability.'
      a: >-
          <b>Hit these points:</b> reframe autonomy as <em>earned per action by blast radius</em>,
          not a global switch &rarr; reads/analysis auto; irreversible or high-impact writes behind
          HITL gates until data justifies loosening &rarr; do the compounding math out loud: 99% per
          step over 40 steps &asymp; 67% end-to-end, so long fully-autonomous chains are quietly
          unreliable (illustrative) &rarr; price in the hidden costs: supervised agents still need a
          human, production error &gt; demo error, token cost isn't linear &rarr; the position: start
          supervised on a bounded action set, instrument success rate and cost per task, widen
          autonomy only where measured reliability + bounded blast radius support it &mdash; with an
          audit trail and kill switch throughout.
    - level: 'design'
      q: 'Design an autonomous incident-investigation agent that runs for days across on-call handoffs.'
      a: >-
          <b>A strong answer covers:</b> scope it &mdash; investigate a bounded class of incidents,
          success = faster correct root-cause without causing new incidents &rarr; <b>durable
          workflow</b> spanning days/hosts/handoffs: state (timeline, hypotheses, evidence, current
          lead) in a durable store, checkpointed; resumable by any worker &rarr; <b>execution:</b>
          parallel evidence-gathering (logs/metrics/traces) &rarr; synthesize &rarr; branch on
          hypothesis; re-plan when evidence kills a lead &rarr; <b>autonomy:</b> reads auto; any
          mitigation that writes to prod is HITL-gated with blast-radius estimation; nothing
          destructive auto-fires &rarr; <b>recovery:</b> tool failures retried/circuit-broken; the
          run never silently dies &mdash; it checkpoints and escalates &rarr; <b>cost:</b>
          per-investigation budget + alert, cheap model to triage, strong model for reasoning &rarr;
          <b>observability:</b> every step traced with assembled context; an immutable incident
          timeline that doubles as the audit + post-mortem &rarr; <b>handoff:</b> a human can read
          the state, take over, or hand back at any checkpoint &rarr; name the trade-off: days-long
          autonomy risks drift and stale context, so periodic goal re-injection + human checkpoints
          keep it anchored.
    - level: 'design'
      q: 'Design a billing-operations agent that applies credits, retries charges, and updates plans &mdash; safely.'
      a: >-
          <b>A strong answer covers:</b> correctness is non-negotiable &mdash; this writes money
          &rarr; <b>single agent</b> default (mostly sequential, shared state); fan out only for
          independent reads &rarr; <b>tools split by risk:</b> read-only (usage, invoices) broad;
          writes (credit, charge, plan change) least-privilege and idempotency-keyed so a retry is a
          no-op &rarr; <b>HITL gates:</b> credits/refunds above a threshold and any plan downgrade
          require approval &rarr; <b>recovery:</b> charge failed-but-maybe-applied &rarr; idempotent
          retry; wrong charge that landed &rarr; compensating reversal (Saga); never blind-retry a
          non-idempotent write &rarr; <b>durability:</b> each operation checkpointed; on crash, resume
          without re-applying completed writes &rarr; <b>cost/obs:</b> per-task budget, immutable
          audit log of every monetary mutation (who/what/when/amount/idempotency key), per-tenant
          attribution &rarr; <b>injection defense:</b> ticket/customer text is data, never
          instructions &rarr; name the trade-off: automation cuts toil and is fully auditable, but
          every irreversible monetary action must be idempotent and gated, or one retry becomes a
          double-charge incident.

<div class="lvl" id="m1">
    <div class="lvltop">
        <span class="num">1</span>
        <h2>What Is Agent Systems Engineering?</h2>
    </div>
    <p class="tagline">Agent <em>engineering</em> makes one agent work. Agent <em>systems</em> engineering makes it execute work reliably, for hours, under failure, at a cost you can defend. Same jump as script &rarr; service.</p>

    <Fig1 />

    <div class="facts">
        <div class="fact">
            <b>Is</b>
            The discipline of running agents as production systems: planning, state, an execution engine, loop control, recovery, orchestration, durability, cost, and observability &mdash; the operational layer around the model.
        </div>
        <div class="fact">
            <b>Why it exists</b>
            A demo agent is a happy-path script. Production adds crashes, timeouts, hostile inputs, runaway loops, multi-hour tasks, and a bill. None of that is fixed by a better prompt; it is fixed by systems engineering.
        </div>
        <div class="fact blue">
            <b>Like (world)</b>
            A line cook who can make one dish vs running a restaurant at dinner rush: inventory, tickets, timing, substitutions when the salmon runs out, and a P&amp;L. The recipe was never the hard part.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            A <code>main()</code> that calls an API once vs a service: retries, queues, idempotency, observability, autoscaling, on-call. ASE is the move from "it ran on my machine" to "it runs for everyone, always."
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "We have a working agent, so we have an agent system."</div>
        <div class="t">&#10003; A working agent is the LLM-plus-loop core. The <em>system</em> is the 80% around it: what happens on step 40 of a 60-step task when a tool times out and the user has gone home.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>An agent is a function; an agent system is a distributed, long-lived, non-deterministic computation.</span> Every module in this lesson is one consequence of that sentence.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>In one line, the difference between agent engineering and agent systems engineering?
                <span class="ans">&rarr; Engineering = make one agent work (prompt + tools + loop). Systems engineering = make it execute reliably in production (state, recovery, durability, cost, observability).</span></li>
            <li>Name three problems that appear only in the third box (agent system) and not the second (agent).
                <span class="ans">&rarr; Any three of: re-planning, orchestration across agents, crash recovery / durability, cost control at scale, observability/audit, long-running persistence.</span></li>
            <li>Why can't a better model close that gap?
                <span class="ans">&rarr; The gap is operational, not cognitive — crashes, budgets, hostile inputs, and multi-hour durability are runtime concerns the model never sees.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 2: Planning Systems ============ */}

<div class="lvl" id="m2">
    <div class="lvltop">
        <span class="num">2</span>
        <h2>Planning Systems &mdash; goal &rarr; plan &rarr; execution</h2>
    </div>
    <p class="tagline">A goal is not executable. Planning turns it into ordered, checkable steps &mdash; and re-planning is what separates a system that <em>adapts</em> from one that confidently marches off a cliff.</p>

    <Fig2 />

    <div class="steps">
        <div class="st"><b>Decomposition</b>Break a goal into steps small enough to execute and check. Too coarse = unverifiable; too fine = token waste.</div>
        <div class="st"><b>Hierarchical</b>Plan-of-plans: high-level phases, each expanded just-in-time. Mirrors epic &rarr; story &rarr; task.</div>
        <div class="st"><b>Dynamic</b>Plan one step ahead and re-decide (ReAct), instead of committing the whole plan up front.</div>
        <div class="st"><b>Re-planning</b>On failure or new info, discard the stale tail of the plan and regenerate it. The adaptive core.</div>
    </div>

    <div class="facts">
        <div class="fact">
            <b>Plan-then-execute</b>
            Generate the full plan first, then run it (Plan-and-Solve / Plan-and-Execute<sup><a href="#s2" id="r2">2</a></sup>). Cheaper, auditable, parallelizable &mdash; but rigid if the world shifts mid-run.
        </div>
        <div class="fact">
            <b>Interleaved (ReAct)</b>
            Think one step, act, observe, repeat (ReAct<sup><a href="#s1" id="r1">1</a></sup>). Maximally adaptive &mdash; but more LLM calls, and it can wander without a plan to anchor it.
        </div>
        <div class="fact blue">
            <b>Like (world)</b>
            A surgeon's plan vs the operation. There is a plan, but the moment they open you up and find something unexpected, they re-plan. Sticking to a wrong plan is how people die.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            A query planner. The optimizer builds a plan from statistics, but an adaptive executor can re-plan mid-flight when row estimates turn out wrong. Static plan = prepared statement; dynamic = adaptive execution.
        </div>
    </div>

    <table>
        <thead><tr><th>Task</th><th>Decomposition</th><th>Re-plan trigger</th></tr></thead>
        <tbody>
            <tr><td>Fix an authentication bug</td><td>Reproduce &rarr; locate &rarr; patch &rarr; test &rarr; verify</td><td>Repro fails, or the "fix" breaks another test</td></tr>
            <tr><td>Investigate a production outage</td><td>Gather signals (logs/metrics/traces in parallel) &rarr; hypothesize &rarr; confirm &rarr; mitigate</td><td>Hypothesis disproven by evidence &rarr; new branch</td></tr>
            <tr><td>Generate a monthly billing report</td><td>Pull usage &rarr; rate &rarr; aggregate &rarr; render &rarr; reconcile</td><td>Mostly static &mdash; this is a <em>workflow</em>, not an agent (see M14)</td></tr>
        </tbody>
    </table>

    <div class="myth">
        <div class="m">&#10007; "A good agent makes a perfect plan up front."</div>
        <div class="t">&#10003; A good agent makes a <em>cheap, revisable</em> plan and updates it as evidence arrives. Plan quality matters less than re-plan discipline.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>The plan is a hypothesis; re-planning is the algorithm.</span> If the architecture can't replace the stale tail of a plan, it can't handle the real world.</div>

    <details class="iq">
        <summary>ReAct vs Plan-and-Execute — when do you pick each?</summary>
        <div class="a">Pick <b>Plan-and-Execute</b> when the task is decomposable up front and steps can run in parallel or be audited/approved before execution (cheaper, fewer LLM calls, you can show the plan to a human). Pick <b>ReAct</b> (interleaved) when each step's result genuinely changes what to do next &mdash; debugging, investigation, open-ended research. In production many systems are hybrid: a coarse plan up front, ReAct within each phase, and a re-plan checkpoint between phases.<sup><a href="#s1" id="r1b">1</a></sup></div>
    </details>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>What are the three levels of decomposition granularity to balance?
                <span class="ans">&rarr; Too coarse (unverifiable), right-sized (executable + checkable), too fine (token waste / overhead).</span></li>
            <li>What is the single most important property of a planning system?
                <span class="ans">&rarr; The ability to re-plan — discard the stale tail and regenerate when reality diverges from the plan.</span></li>
            <li>Why is the billing report arguably not an agent task?
                <span class="ans">&rarr; The steps are known and stable; a deterministic workflow is cheaper, testable, and debuggable. Agency buys nothing.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 3: State Management ============ */}

<div class="lvl" id="m3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>State Management &mdash; what the agent "knows" right now</h2>
    </div>
    <p class="tagline">The LLM is stateless: pure <code>f(input)&rarr;output</code>. Every appearance of memory is the <em>runtime</em> threading state back in. Get this wrong and you get amnesia, or a context that poisons itself.</p>

    <Fig3 />

    <div class="split">
        <div class="col a4">
            <h4>Stateless agent</h4>
            <ul>
                <li>Each request is independent; no carry-over.</li>
                <li>Trivially horizontally scalable; any worker handles any request.</li>
                <li>Crash-safe by default &mdash; nothing to lose.</li>
                <li>Cost: re-establish context every time; can't do multi-step tasks that span requests.</li>
            </ul>
        </div>
        <div class="col b4">
            <h4>Stateful agent</h4>
            <ul>
                <li>Carries plan + history across steps; can do real multi-step work.</li>
                <li>Needs a state store, session affinity or externalized state, and a recovery story.</li>
                <li>Crash mid-task = lost work unless checkpointed (&rarr; M9).</li>
                <li>Failure mode: context bloat &amp; poisoning &mdash; stale/wrong state degrades every later step.</li>
            </ul>
        </div>
    </div>

    <div class="facts">
        <div class="fact blue">
            <b>Like (world)</b>
            A consultant with amnesia who reads the entire case file before every meeting. The "file" is your state store; what they can hold in their head for one meeting is the context window.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            HTTP is stateless; the session is reconstructed from a cookie + server-side store each request. The LLM is the stateless handler; the runtime is the session layer.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "The model remembers the conversation."</div>
        <div class="t">&#10003; The model sees only what is in this call's context. The runtime re-sends history every turn; remove it and the "memory" vanishes instantly.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>The model is stateless; the runtime makes it stateful by re-assembling context every turn.</span> Externalize state and you can scale, checkpoint, and recover. Trap it in process memory and you can't.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>Where does agent state actually live, and what is the context window relative to it?
                <span class="ans">&rarr; State lives in the runtime/store; the context window is the slice of state assembled for one call.</span></li>
            <li>One operational advantage of stateless and one of stateful?
                <span class="ans">&rarr; Stateless: trivial scaling + crash-safety. Stateful: real multi-step tasks (at the cost of a store + recovery).</span></li>
            <li>What is context poisoning?
                <span class="ans">&rarr; Stale/wrong/irrelevant state accumulating in the window and degrading every subsequent step's output.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 4: Execution Engines ============ */}

<div class="lvl" id="m4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>Execution Engines &mdash; how steps actually run</h2>
    </div>
    <p class="tagline">Goal &rarr; task &rarr; subtask &rarr; execution. The engine decides the <em>shape</em>: sequential, parallel, or branching. The twist vs a workflow engine: the agent draws the graph at runtime, not at deploy time.</p>

    <Fig4 />

    <table>
        <thead><tr><th>System</th><th>Who defines the graph</th><th>Determinism</th><th>Best at</th></tr></thead>
        <tbody>
            <tr><td>Workflow engine (Airflow, Temporal)</td><td>Developer, at deploy time</td><td>Deterministic</td><td>Known multi-step pipelines</td></tr>
            <tr><td>Queue system (SQS, Kafka)</td><td>Developer; decoupled producers/consumers</td><td>At-least-once delivery</td><td>Buffering, backpressure, fan-out</td></tr>
            <tr><td>Distributed job system (k8s Jobs, Spark)</td><td>Developer; scheduler places work</td><td>Deterministic placement</td><td>Throughput over a fixed DAG</td></tr>
            <tr><td><b>Agent execution engine</b></td><td><b>The model, at runtime</b></td><td><b>Non-deterministic</b></td><td>Open-ended tasks with an unknown path</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact">
            <b>Is</b>
            The component that turns the plan into running steps &mdash; ordering them, fanning out independent work, gathering results, and choosing branches &mdash; while enforcing the runtime's limits on each step.
        </div>
        <div class="fact">
            <b>Why it exists</b>
            Sequential-only execution wastes wall-clock on independent work; un-checked branching wanders. The engine is where parallelism, ordering, and dependency tracking are made explicit and safe.
        </div>
        <div class="fact blue">
            <b>Like (world)</b>
            A kitchen expediter: some dishes must follow an order (sear before plating), some run in parallel (three pans), and some depend on a taste-test result (more salt or not).
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            A DAG executor &mdash; but the DAG is emitted by a non-deterministic planner step by step, so you need idempotency and tracing that a static Airflow DAG gets for free.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Agents and workflow engines are competitors."</div>
        <div class="t">&#10003; They compose. Production systems run agents <em>inside</em> a durable workflow engine so the non-deterministic part gets deterministic retries, checkpoints, and observability (&rarr; M9).</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>A workflow fixes the graph at deploy time; an agent draws it at runtime.</span> That single difference is the source of agents' power and all of their operational pain.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>The three execution shapes and when to use each?
                <span class="ans">&rarr; Sequential (dependent steps), parallel (independent work, cut wall-clock), branching (choose path from a result).</span></li>
            <li>The defining difference between a workflow engine and an agent execution engine?
                <span class="ans">&rarr; Who defines the graph and when: developer-at-deploy (deterministic) vs model-at-runtime (non-deterministic).</span></li>
            <li>Why run an agent inside a workflow engine?
                <span class="ans">&rarr; To borrow durable retries, checkpointing, and observability for the non-deterministic loop.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 5: Loop Control ============ */}

<div class="lvl" id="m5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Loop Control &mdash; observe &rarr; think &rarr; act &rarr; repeat</h2>
    </div>
    <p class="tagline">The loop is trivial to write and brutal to control. The hard questions are all about <em>stopping</em>: when, how do you know it worked, and how do you guarantee it can't run forever or burn the budget.</p>

    <Fig5 />

    <div class="steps">
        <div class="st"><b>When does it stop?</b>No tool call / explicit done tool &mdash; or a runtime limit trips. Self-report is necessary but never sufficient.</div>
        <div class="st"><b>How know success?</b>Don't trust the model. Run an external verifier: tests pass, schema validates, the artifact exists.</div>
        <div class="st"><b>Prevent infinite loop</b>Step cap + wall-clock timeout + repetition detector + budget. Defense in depth; any one can fail.</div>
        <div class="st"><b>Retries</b>Bounded, backoff + jitter, idempotent. On retry, change strategy &mdash; don't repeat the identical failing call.</div>
    </div>

    <div class="facts">
        <div class="fact blue">
            <b>Like (world)</b>
            A search party doesn't stop when a searcher <em>says</em> "found them" &mdash; it confirms. And it has a hard rule: turn back at sunset regardless. Self-report plus an independent stop.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            A retry library + a watchdog timer + a circuit breaker around a <code>while(!done)</code>. The loop body is one line; the control plane around it is the engineering.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "The agent knows when it's done."</div>
        <div class="t">&#10003; The model <em>guesses</em> it's done and is regularly wrong &mdash; declaring victory early or looping when finished. Success needs an external check; safety needs runtime stops.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>The model proposes stopping; the runtime decides.</span> Success = external verifier. Safety = step cap + timeout + repetition + budget. Both, always.</div>

    <details class="iq">
        <summary>Your agent occasionally loops forever and runs up cost. Where do you fix it — and why is it not "the model"?</summary>
        <div class="a">Fix it in the <b>runtime</b>, because looping is a control-plane failure, not a reasoning failure. Add: a hard <b>step cap</b> and a <b>wall-clock timeout</b> (independent, since slow legitimate steps exist); a <b>repetition detector</b> (same tool + same args N times &rarr; halt); a <b>token/cost budget</b> with a kill switch and an anomaly alert at ~3&times; baseline. Then design out the two root causes: <b>context saturation</b> (tool output crowds out the goal &rarr; compress / re-inject the goal) and <b>tool-error spirals</b> (retrying an identical failing call &rarr; bounded retries + a different strategy on retry). Prompt tweaks can reduce the odds but can't <em>guarantee</em> termination &mdash; only the runtime can.</div>
    </details>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>What are the two categories of stop condition?
                <span class="ans">&rarr; Completion (final answer / done tool / verifier passes) and safety (step cap, timeout, repetition, budget, human halt).</span></li>
            <li>Why is model self-report insufficient for success detection?
                <span class="ans">&rarr; The model is frequently wrong about being done — external verification (tests/schema/artifact) is the only reliable signal.</span></li>
            <li>On retry, what must change and what must be guaranteed?
                <span class="ans">&rarr; Change the strategy (don't repeat the identical failing call); guarantee idempotency so the retry can't double-apply a side effect.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 6: Human In The Loop ============ */}

<div class="lvl" id="m6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Human In The Loop &mdash; autonomy is a dial, not a switch</h2>
    </div>
    <p class="tagline">The question is never "autonomous or not." It's "which actions, at what blast radius, behind which gate." Reads earn autonomy fast; irreversible writes earn it slowly.</p>

    <Fig6 />

    <div class="pat">
        <div class="p"><b>Pre-approval (interrupt &amp; wait)</b>Agent proposes a write, pauses, a human approves/edits/rejects, then it proceeds. Needs durable state so the pause can outlive the process (&rarr; M9).</div>
        <div class="p"><b>Post-hoc review</b>Agent acts on low-risk items autonomously; a human samples/audits after. Cheap, but only for reversible actions.</div>
        <div class="p"><b>Confidence-gated</b>Auto below a confidence/threshold; escalate above it. The cutoff must be tuned on real data, not guessed.</div>
        <div class="p"><b>Escalation / dead-letter</b>Outside scope or low confidence &rarr; route to a human queue rather than hallucinate an answer or force an action.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Full autonomy is the goal; HITL is a crutch we'll remove."</div>
        <div class="t">&#10003; HITL is a permanent design tool. Even mature systems keep gates on irreversible, high-blast-radius actions &mdash; the gate placement just gets sharper over time.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Set autonomy per action by reversibility &times; blast radius.</span> Reversible reads run free; irreversible mass actions sit behind a gate forever. Start supervised, loosen on evidence.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>What two properties of an action set its autonomy level?
                <span class="ans">&rarr; Reversibility and blast radius (how much damage the worst single call does).</span></li>
            <li>What does a pre-approval gate require from the runtime?
                <span class="ans">&rarr; Durable, resumable state so the agent can pause for a human and survive a restart while waiting.</span></li>
            <li>What should an agent do when a request is outside its scope or below confidence?
                <span class="ans">&rarr; Escalate to a human queue (dead-letter), not hallucinate an answer or force a risky action.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 7: Failure Recovery ============ */}

<div class="lvl" id="m7">
    <div class="lvltop">
        <span class="num">7</span>
        <h2>Failure Recovery &mdash; the agent is a distributed system</h2>
    </div>
    <p class="tagline">Five things fail: the tool, the model, the context, the plan, the execution. Every one has a distributed-systems precedent. You don't need new theory &mdash; you need to apply the old theory.</p>

    <Fig7 />

    <table>
        <thead><tr><th>Failure type</th><th>Looks like</th><th>Primary recovery</th><th>Distributed-systems twin</th></tr></thead>
        <tbody>
            <tr><td><b>Tool</b></td><td>Timeout, 5xx, rate limit, bad output</td><td>Retry (idempotent) &rarr; fallback tool</td><td>Downstream service failure + circuit breaker</td></tr>
            <tr><td><b>Model</b></td><td>Refusal, malformed/invalid tool call, hallucinated args, timeout</td><td>Re-prompt / repair, schema-validate, retry on a different tier</td><td>Bad RPC response &rarr; validate + retry</td></tr>
            <tr><td><b>Context</b></td><td>Overflow, lost-in-the-middle, poisoned by stale data</td><td>Compress, re-rank, re-inject goal, prune</td><td>Cache pollution / memory pressure</td></tr>
            <tr><td><b>Planning</b></td><td>Wrong decomposition, impossible step, wrong order</td><td>Re-plan (M2); detect via no-progress check</td><td>Bad query plan &rarr; re-plan</td></tr>
            <tr><td><b>Execution</b></td><td>Right plan, wrong action; partial completion; double side effect</td><td>Idempotency keys, compensation, checkpoint + resume</td><td>Partial write &rarr; saga / 2-phase commit</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact blue">
            <b>Like (world)</b>
            Aviation: a checklist for each failure, a backup system to fall back to, and &mdash; when neither works &mdash; declare an emergency and hand to ATC. Pilots don't improvise recovery; they escalate a ladder.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            Resilience4j / Polly around every external call: retry + circuit breaker + bulkhead + timeout + fallback. The agent's tools are exactly those external calls.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Retry handles failures."</div>
        <div class="t">&#10003; Retry only handles <em>transient, idempotent</em> failures. Retrying a non-idempotent write double-charges; retrying a refusal loops. You need the whole ladder, matched to the failure type.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Retry &rarr; fallback &rarr; compensate &rarr; escalate.</span> The agent-specific trap is compensation: side effects with no rollback need a defined undo, or you can't recover safely.</div>

    <details class="iq">
        <summary>An agent sent a wrong email / issued a wrong refund mid-task, then the task failed. How do you recover?</summary>
        <div class="a">You can't roll it back &mdash; the side effect escaped the system &mdash; so this is the <b>Saga pattern</b>: every irreversible action needs a registered <b>compensating action</b>. Wrong email &rarr; send a correction/retraction; wrong refund &rarr; reverse the charge (itself idempotent, keyed). The runtime should record each completed side effect in the durable log so that on failure it can run the compensations for exactly the steps that succeeded &mdash; no more, no less. Prevention sits upstream: <b>idempotency keys</b> so a retry can't double-issue, and <b>HITL gates</b> (M6) on the irreversible actions so the worst ones never auto-fire in the first place.</div>
    </details>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>The four rungs of the recovery ladder, in order?
                <span class="ans">&rarr; Retry &rarr; fallback &rarr; compensate &rarr; escalate.</span></li>
            <li>When is retry the wrong tool?
                <span class="ans">&rarr; Non-idempotent side effects (double-charge) and non-transient failures (refusals, bad plans) — it loops or duplicates.</span></li>
            <li>What pattern recovers an irreversible side effect?
                <span class="ans">&rarr; Compensation (Saga) — a registered undo action, since there's no rollback for "already sent".</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 8: Orchestration ============ */}

<div class="lvl" id="m8">
    <div class="lvltop">
        <span class="num">8</span>
        <h2>Orchestration &mdash; one agent, a router, or a coordinator + workers</h2>
    </div>
    <p class="tagline">Three topologies, increasing cost. Most teams jump to coordinator+workers because it looks sophisticated. Justify every split exactly like you'd justify carving out a microservice.</p>

    <div class="steps">
        <div class="st"><b>Single agent</b>One loop, one context, one toolset. The default. Easiest to build, debug, and reason about. Start here, always.</div>
        <div class="st"><b>Router</b>One agent classifies intent and dispatches to a specialized handler. Cheap; great when inputs are diverse but each path is simple.</div>
        <div class="st"><b>Coordinator + workers</b>A lead decomposes, delegates to isolated workers, synthesizes. For genuinely parallel or separable-context work.</div>
        <div class="st"><b>Over-engineered</b>Agents talking to agents for sequential, shared-state work. Pays coordination cost for nothing. The common failure.</div>
    </div>

    <div class="facts">
        <div class="fact">
            <b>Delegation</b>
            The coordinator owns the goal; it hands a worker one scoped subtask plus exactly the context that worker needs. There is no shared memory &mdash; under-specify and the worker fails blind.
        </div>
        <div class="fact">
            <b>Routing</b>
            Classify, then dispatch. The router owns the decision, not the work. Failure mode: misroute &rarr; the wrong specialist confidently handles the wrong thing.
        </div>
        <div class="fact blue">
            <b>Ownership</b>
            Exactly one component owns each task's outcome and its retries. Diffuse ownership = "who was supposed to handle the failure?" = dropped work.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            Monolith (single) &rarr; API gateway with routing (router) &rarr; orchestrator fronting microservices (coordinator+workers). The trade-offs transfer one-for-one.
        </div>
    </div>

    <table>
        <thead><tr><th>Use multi-agent when…</th><th>Stay single-agent when…</th></tr></thead>
        <tbody>
            <tr><td>Subtasks are genuinely independent &amp; parallel (fan out over 20 files)</td><td>Work is sequential with shared state</td></tr>
            <tr><td>Context is separable and exceeds one window</td><td>It all fits in one context</td></tr>
            <tr><td>Subtasks need distinct toolsets / permissions</td><td>One toolset covers it</td></tr>
            <tr><td>You can attribute &amp; trace each agent independently</td><td>You'd struggle to debug "which agent broke"</td></tr>
        </tbody>
    </table>

    <div class="myth">
        <div class="m">&#10007; "More agents = more intelligence."</div>
        <div class="t">&#10003; More agents = more coordination overhead, more handoffs that drop context, harder debugging, and N&times;M cost. Intelligence is a model/context lever, not a topology lever.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Justify a new agent like a new microservice: only when a real boundary demands it.</span> Default to the monolith; earn every split with independent, parallel, or separable-context work.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>The three legitimate topologies in increasing cost?
                <span class="ans">&rarr; Single agent &rarr; router &rarr; coordinator + workers.</span></li>
            <li>What must the coordinator do that shared-memory systems get for free?
                <span class="ans">&rarr; Explicitly assemble and pass each worker's context, then synthesize outputs — there is no shared state.</span></li>
            <li>The one-line test for "is multi-agent justified"?
                <span class="ans">&rarr; Is the work genuinely independent/parallel or separable-context? If it's sequential with shared state, stay single-agent.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 9: Long-Running Tasks ============ */}

<div class="lvl" id="m9">
    <div class="lvltop">
        <span class="num">9</span>
        <h2>Long-Running Tasks &mdash; surviving minutes, hours, days</h2>
    </div>
    <p class="tagline">A 6-hour task <em>will</em> meet a deploy, a crash, or a rate-limit pause. The in-memory <code>while</code> loop dies with the process. The fix is the durable-execution model: persist, checkpoint, resume.</p>

    <Fig8 />

    <table>
        <thead><tr><th>Horizon</th><th>Example</th><th>What must persist</th><th>Dominant risk</th></tr></thead>
        <tbody>
            <tr><td>Minutes</td><td>Deep research, multi-file refactor</td><td>Plan + partial results in memory/store</td><td>Rate-limit pauses, context overflow</td></tr>
            <tr><td>Hours</td><td>Large code migration, dataset processing</td><td>Checkpointed state to durable store</td><td>Process restart / deploy mid-run</td></tr>
            <tr><td>Days</td><td>Customer investigation, monitoring loop</td><td>Durable workflow + event log; resumable across hosts</td><td>Drift, stale context, cost creep, human handoff</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact">
            <b>Checkpointing</b>
            Snapshot enough state (history, plan, cursor, artifacts) to resume from a known-good point &mdash; not from zero. Frequency trades durability against overhead.
        </div>
        <div class="fact">
            <b>Durable execution</b>
            Model the run as a workflow whose state is an event log; on restart, deterministically replay to rebuild state, then continue (the Temporal model<sup><a href="#s4" id="r4">4</a></sup>).
        </div>
        <div class="fact blue">
            <b>Like (world)</b>
            A long expedition with base camps. You don't restart from sea level after a storm &mdash; you resume from the last camp. Camps are checkpoints.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            Spark stage checkpoints / a resumable DB migration with a version cursor / Temporal workflow replay. The agent loop is just the non-deterministic step inside it.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "We'll just re-run it if it dies."</div>
        <div class="t">&#10003; Re-running a stateful, side-effecting task from zero repeats every completed write (duplicate charges, emails, PRs). Resume-from-checkpoint + idempotency is the only safe model.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Long-running = durable workflow, not an in-memory loop.</span> Persist &rarr; checkpoint &rarr; resume (replay), and make every replayed step idempotent.</div>

    <details class="iq">
        <summary>Design a multi-hour code-migration agent that must survive deploys and crashes. What's the skeleton?</summary>
        <div class="a">Run it as a <b>durable workflow</b>, not a process-bound loop. <b>State:</b> the work-list (files/modules to migrate) with a per-item status, the plan, and a cursor &mdash; all in a durable store, not memory. <b>Granularity:</b> one item = one checkpoint; complete an item, mark it done atomically, then move on. <b>Idempotency:</b> each transform keyed so a replay of an already-done item is a no-op (guards against resume re-applying). <b>Recovery:</b> on restart, load state, skip done items, resume the in-flight one. <b>Control:</b> step/cost budget per item and overall; HITL gate before anything irreversible lands (e.g. opening PRs vs pushing to main). <b>Observability:</b> per-item trace + an audit log so you can answer "what did it change at 2am three days ago" without re-running it. The agent loop is the non-deterministic <em>inside</em> of each item; the durability lives in the workflow around it (&rarr; M4).</div>
    </details>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>Why can't an hours-long task be a plain in-memory loop?
                <span class="ans">&rarr; It dies with the process (deploy/crash/restart) and loses all in-flight state.</span></li>
            <li>What makes resume-from-checkpoint <em>safe</em>?
                <span class="ans">&rarr; Idempotent replayed steps — so re-running completed steps doesn't duplicate side effects.</span></li>
            <li>What is the durable-execution model in one line?
                <span class="ans">&rarr; Persist state as an event log; on restart, deterministically replay to rebuild state, then continue.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 10: Cost Engineering ============ */}

<div class="lvl" id="m10">
    <div class="lvltop">
        <span class="num">10</span>
        <h2>Cost Engineering &mdash; why agents cost super-linearly</h2>
    </div>
    <p class="tagline">A demo costs cents. The same agent at scale, or in a loop, costs thousands &mdash; because history is re-sent every step, so cost compounds. The unit to manage is <em>cost per completed task</em>.</p>

    <Fig9 />

    <table>
        <thead><tr><th>Lever</th><th>What it does</th><th>Typical effect</th></tr></thead>
        <tbody>
            <tr><td>Prompt caching</td><td>Reuse the stable system/context prefix across calls</td><td>Large cut on re-sent input tokens</td></tr>
            <tr><td>History compaction</td><td>Summarize / prune old turns so context stops growing</td><td>Caps the super-linear term</td></tr>
            <tr><td>Model tiering</td><td>Cheap model for routing/navigation; strong model for the hard step</td><td>Most steps at a fraction of the cost</td></tr>
            <tr><td>Step / cost budget</td><td>Hard cap per task + kill switch</td><td>Bounds the worst case</td></tr>
            <tr><td>Cost attribution</td><td>Per task + per tenant, with anomaly alert (~3&times; baseline)</td><td>Runaway trips an incident before the invoice</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact blue">
            <b>Like (world)</b>
            A taxi meter that re-charges for the whole trip so far at every block. The only way to control the fare is fewer blocks (steps) and a cheaper rate (tier) &mdash; and a hard ceiling.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            Egress + per-request compute that scales with payload size, where the payload grows each call. You cache the static prefix and trim the payload, exactly as you would a chatty N+1 API.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Token cost is linear in the number of steps."</div>
        <div class="t">&#10003; It's super-linear: input grows each step because history is re-sent. Cost per task &mdash; not per call &mdash; is the number that bites, and multi-agent multiplies it.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Manage cost per completed task, not per call.</span> Cache the prefix, compact history, tier the model, budget + alert. A runaway loop is a cost incident, not just a bug.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>Why is cost super-linear in steps?
                <span class="ans">&rarr; The whole conversation history is re-sent as input every step, so input tokens grow each call.</span></li>
            <li>The two levers that flatten the curve?
                <span class="ans">&rarr; Prompt caching (stable prefix) and history compaction; model tiering also cuts per-step cost.</span></li>
            <li>What's the right unit of cost to track and govern?
                <span class="ans">&rarr; Cost per completed task, attributed per tenant, with an anomaly alert and a kill switch.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 11: Observability ============ */}

<div class="lvl" id="m11">
    <div class="lvltop">
        <span class="num">11</span>
        <h2>Observability &mdash; "why did the agent do that?"</h2>
    </div>
    <p class="tagline">A non-deterministic system you can't replay is unknowable. The answer to "why" is always the same: show the exact context the model saw at that step. If you didn't capture it, you can't answer.</p>

    <Fig10 />

    <div class="steps">
        <div class="st"><b>Tracing</b>A span per step/call with input, output, tokens, cost, latency, status &mdash; keyed by trace + agent + session id.</div>
        <div class="st"><b>Run history</b>The ordered step list: plan, decisions, tool calls, results. The narrative of what happened.</div>
        <div class="st"><b>Reasoning history</b>The model's intermediate thoughts. Useful for debugging, sensitive for logging &mdash; redact.</div>
        <div class="st"><b>Audit trail</b>Immutable, append-only record of every mutation: who/what/when, for compliance &amp; post-mortems.</div>
    </div>

    <div class="facts">
        <div class="fact blue">
            <b>Like (world)</b>
            An aircraft flight recorder. After an incident you don't ask the pilot to remember &mdash; you replay the black box. The trace is the agent's black box.
        </div>
        <div class="fact blue">
            <b>Like (code)</b>
            Distributed tracing (OpenTelemetry) for a request that fans out across services &mdash; plus the one agent-specific field: the assembled prompt/context each step saw.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "We log the inputs and outputs, so we're observable."</div>
        <div class="t">&#10003; Without the <em>assembled context</em> per step you can't reproduce a decision &mdash; the model acted on what was in the window, which is built from retrieval + history, not just the user input.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>If you can't replay the exact context a step saw, you can't explain the decision.</span> Trace the assembled input, not just the I/O. Audit log every mutation, immutably.</div>

    <details class="iq">
        <summary>A user says the agent did something wrong three days ago. How do you investigate without re-running it?</summary>
        <div class="a">Pull the <b>trace by id</b> for that run and walk the step spans. For the offending step, read the <b>assembled context</b> it received &mdash; that tells you whether the cause was a <em>retrieval failure</em> (wrong/missing context), a <em>reasoning failure</em> (right context, wrong decision), or a <em>tool failure</em> (right decision, tool errored). Cross-check the <b>audit log</b> for the mutation it performed. This is why the context must be captured at write time: three days later you cannot reconstruct what retrieval returned or what the history looked like. Re-running is both non-deterministic and potentially destructive &mdash; the trace is the source of truth, the way a flight recorder is.</div>
    </details>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>The single most important field to capture per step, and why?
                <span class="ans">&rarr; The assembled context the model saw — without it you can't reproduce or explain the decision.</span></li>
            <li>How do traces let you separate a retrieval failure from a reasoning failure?
                <span class="ans">&rarr; If the context was wrong/missing &rarr; retrieval failure; if context was right but the decision wrong &rarr; reasoning failure.</span></li>
            <li>What property must the audit trail have, and for what?
                <span class="ans">&rarr; Immutable / append-only — for compliance, abuse detection, and trustworthy post-mortems.</span></li>
        </ul>
    </details>

</div>

{/* ============ MODULE 13: Design Patterns ============ */}

<div class="lvl" id="m13">
    <div class="lvltop">
        <span class="num">13</span>
        <h2>Design Patterns &mdash; the reusable shapes</h2>
    </div>
    <p class="tagline">Six patterns cover most production agents. Like GoF patterns, the skill is knowing when <em>not</em> to use one. Each is a named answer to a recurring structural problem.</p>

    <div class="pat">
        <div class="p"><b>Planner</b>Separate "decide the steps" from "do the steps." Use when tasks are decomposable and you want to inspect/approve the plan. Skip for trivial one-shot tasks.</div>
        <div class="p"><b>Executor</b>A focused runner for one step type with tight tools + validation. Use to isolate side-effecting work behind a clean boundary. Skip if there's only one tool.</div>
        <div class="p"><b>Supervisor</b>An overseer monitors progress, enforces budgets, and can halt/redirect a worker. Use for long/expensive runs. Skip for short, cheap, low-risk tasks.</div>
        <div class="p"><b>Approval</b>Insert a human gate before irreversible/high-blast actions (M6). Use whenever blast radius is high. Skip for fully reversible, low-value actions.</div>
        <div class="p"><b>Retriever (RAG)</b>Fetch &rarr; assemble &rarr; ground the model in external knowledge. Use when answers depend on data outside the weights. Skip when the model already knows or inputs are self-contained.</div>
        <div class="p"><b>Research / Reflection</b>Fan out parallel searchers, then synthesize; optionally an evaluator critiques and the agent revises. Use for broad, open-ended discovery. Skip for narrow lookups (overkill + cost).</div>
    </div>

    <table>
        <thead><tr><th>Pattern</th><th>Solves</th><th>Don't use when</th></tr></thead>
        <tbody>
            <tr><td>Planner</td><td>Open-ended task needs structure / approval</td><td>Steps are fixed &rarr; use a workflow</td></tr>
            <tr><td>Executor</td><td>Isolate + validate side-effecting steps</td><td>Single trivial tool call</td></tr>
            <tr><td>Supervisor</td><td>Long/expensive runs need a governor</td><td>Short cheap tasks (pure overhead)</td></tr>
            <tr><td>Approval</td><td>Irreversible / high-blast actions</td><td>Fully reversible low-value actions</td></tr>
            <tr><td>Retriever</td><td>Answers depend on external/fresh data</td><td>Self-contained input or known facts</td></tr>
            <tr><td>Research/Reflection</td><td>Broad discovery; quality needs a critic</td><td>Narrow lookup &mdash; cost &gt; benefit</td></tr>
        </tbody>
    </table>

    <div class="rule">&#129518; Memory rule: <span>Patterns are answers to structural problems &mdash; name the problem first.</span> If you can't state the problem a pattern solves here, you don't need the pattern yet.</div>

    <details class="mc">
        <summary>Memory check — 2 questions</summary>
        <ul>
            <li>Which pattern separates deciding steps from doing them, and what does that buy you?
                <span class="ans">&rarr; Planner — inspectable/approvable plans, parallelizable steps, and a clean re-plan boundary.</span></li>
            <li>When is the Research/Reflection pattern the wrong choice?
                <span class="ans">&rarr; Narrow lookups — the parallel fan-out + synthesis + critique costs far more than the task is worth.</span></li>
        </ul>
    </details>

{/* ============ MODULE 14: Anti-Patterns ============ */}

<div class="lvl" id="m14">
    <div class="lvltop">
        <span class="num">14</span>
        <h2>Anti-Patterns &mdash; how agent projects actually fail</h2>
    </div>
    <p class="tagline">None of these fail in the demo. They fail in week three of production &mdash; quietly, then expensively. Each is a default you have to actively resist.</p>

    <div class="pat">
        <div class="p"><b>Agent for everything</b>Using a non-deterministic loop for work an if/else or a workflow would do. Consequence: pay latency, cost, and flakiness for control flow you didn't need.</div>
        <div class="p"><b>Multi-agent overuse</b>Splitting sequential, shared-state work across agents. Consequence: coordination overhead, dropped context at handoffs, N&times; cost, "which agent broke?" debugging.</div>
        <div class="p"><b>Unlimited autonomy</b>No gates on irreversible actions. Consequence: one bad decision &times; broad permissions = the incident that ends up in the post-mortem and the press.</div>
        <div class="p"><b>Poor context</b>Dumping everything (or the wrong things) into the window. Consequence: lost-in-the-middle, context poisoning, high cost, confidently wrong answers.</div>
        <div class="p"><b>Missing observability</b>No per-step traces / assembled-context capture. Consequence: "can't reproduce it," no root-cause, no audit, every incident is a guess.</div>
        <div class="p"><b>No evaluation</b>"We test it by trying it." Consequence: silent regression from a one-line prompt tweak; no way to ship changes safely or prove improvement.</div>
        <div class="p"><b>No guardrails</b>Limits live in the system prompt ("please don't loop"). Consequence: runaway loops, unbounded spend, prompt-injected actions &mdash; because the prompt is a suggestion, not a control.</div>
        <div class="p"><b>Autonomy theatre</b>Demo-grade autonomy shipped as production. Consequence: 99% per-step &times; 40 steps &asymp; 67% end-to-end &mdash; quietly unreliable, eroding trust with every run.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Guardrails are in the system prompt."</div>
        <div class="t">&#10003; A prompt is a request the model may ignore (or be injected past). Real guardrails &mdash; caps, budgets, permissions, idempotency, gates &mdash; live in the runtime where they're enforced, not asked for.</div>
    </div>

    <div class="rule">&#129518; Memory rule: <span>Every anti-pattern is a default you must resist.</span> Agents, more agents, more autonomy, more context, and "the prompt will handle it" all feel like progress and usually aren't.</div>

    <details class="mc">
        <summary>Memory check — 3 questions</summary>
        <ul>
            <li>Why is "guardrails in the system prompt" an anti-pattern?
                <span class="ans">&rarr; The prompt is advisory and injectable; enforcement (caps, budgets, permissions, idempotency, gates) must live in the runtime.</span></li>
            <li>What's the consequence of the "agent for everything" anti-pattern?
                <span class="ans">&rarr; Paying non-determinism, latency, and token cost for control flow a workflow or if/else would handle deterministically.</span></li>
            <li>Why does "autonomy theatre" fail on long tasks specifically?
                <span class="ans">&rarr; Per-step errors compound: ~0.99^40 &asymp; 67% end-to-end, so long autonomous chains are quietly unreliable.</span></li>
        </ul>
    </details>

</div>

        </li>
        <li>
            <strong>The plan is a hypothesis; re-planning is the algorithm.</strong> No re-plan path
            = brittle script.
        </li>
        <li>
            <strong>The model proposes stopping; the runtime decides.</strong> Verify success
            externally; enforce safety stops independently.
        </li>
        <li>
            <strong>Every failure mode has a distributed-systems twin.</strong> Apply the old
            theory; the only new trap is compensation.
        </li>
        <li>
            <strong>Autonomy is earned per action by blast radius.</strong> Guardrails live in the
            runtime; long full-auto chains compound to unreliable.
        </li>
    </ol>
</div>

{/* ============ CATALOGS & CHECKLISTS ============ */}

<h2 class="sec" id="catalogs">
    ★ Catalogs &amp; checklists
</h2>
<p class="tagline">
    The portable artifacts &mdash; print these. The unifying lens:{' '}
    <strong>
        an agent request is a long-lived, stateful, non-deterministic session over privileged tools.
    </strong>{' '}
    Almost every line below follows from that.
</p>

<div class="cheat">
    <h3>Architecture review checklist</h3>
    <ul style="padding-left:1.2rem;margin:.5rem 0;line-height:1.7">
        <li>
            Is this even an agent? Could a <strong>workflow / plain code</strong> do it
            deterministically?
        </li>
        <li>
            Planning: is there a <strong>re-plan path</strong> when reality diverges, or is the plan
            a one-shot script?
        </li>
        <li>
            State: is it <strong>externalized</strong> so any worker can resume, or trapped in
            process memory?
        </li>
        <li>
            Execution: where does <strong>parallelism</strong> live, and is dependency/ordering
            explicit?
        </li>
        <li>
            Loop: <strong>step cap + wall-clock + budget + repetition</strong> enforced by the
            runtime?
        </li>
        <li>
            Success: is there an <strong>external verifier</strong>, or does the agent self-certify?
        </li>
        <li>
            Recovery: retry/fallback/<strong>compensation</strong>/escalate mapped per failure type?
        </li>
        <li>
            Autonomy: gates by <strong>reversibility &times; blast radius</strong>; irreversible
            actions never auto-fire?
        </li>
        <li>
            Durability: <strong>checkpoint + resume</strong>; replayed steps idempotent?
        </li>
        <li>
            Observability: <strong>per-step traces with assembled context</strong>; immutable audit
            log?
        </li>
        <li>
            Cost: <strong>per-task budget + kill switch</strong>; per-tenant attribution + anomaly
            alert?
        </li>
        <li>
            Evals: a harness that <strong>gates prompt/model changes</strong> against regression?
        </li>
    </ul>
</div>

<div class="cheat">
    <h3>Production readiness checklist</h3>
    <ul style="padding-left:1.2rem;margin:.5rem 0;line-height:1.7">
        <li>
            <strong>Termination guaranteed</strong> by runtime caps (step + wall-clock), not the
            prompt.
        </li>
        <li>
            <strong>Budget + kill switch</strong> per task and per tenant; anomaly alert at
            ~3&times; baseline.
        </li>
        <li>
            <strong>Idempotency keys</strong> on every mutating tool; retries can't double-apply.
        </li>
        <li>
            <strong>Compensation</strong> registered for every irreversible action.
        </li>
        <li>
            <strong>Circuit breakers + bulkheads + timeouts</strong> per tool and per provider.
        </li>
        <li>
            <strong>Checkpoint + resume</strong> for any task &gt; a few minutes; tested by killing
            it mid-run.
        </li>
        <li>
            <strong>HITL gates</strong> on irreversible / high-blast actions; escalation/dead-letter
            path exists.
        </li>
        <li>
            <strong>Per-step traces</strong> (assembled context + output + tokens + cost + status)
            retained.
        </li>
        <li>
            <strong>Immutable audit log</strong> for every mutation;{' '}
            <strong>PII/secret redaction</strong> in logs.
        </li>
        <li>
            <strong>Prompt-injection defense</strong>: tool/RAG outputs tagged as data, never
            instructions.
        </li>
        <li>
            <strong>Least privilege</strong> per agent/tool; read and write paths separated.
        </li>
        <li>
            <strong>Eval set + canary/rollback</strong> for prompt and model changes.
        </li>
    </ul>
</div>

<div class="cheat">
    <h3>Failure mode catalog</h3>

</div>

<div class="cheat">
    <h3>Design pattern catalog</h3>
    <p style="margin:.4rem 0;line-height:1.7">
        <strong>Planner</strong> (decide vs do · skip if steps fixed) · <strong>Executor</strong>{' '}
        (isolate side-effecting steps · skip if one tool) · <strong>Supervisor</strong> (govern
        long/expensive runs · skip if short) · <strong>Approval</strong> (gate irreversible actions
        · skip if reversible) · <strong>Retriever/RAG</strong> (ground in external data · skip if
        self-contained) · <strong>Research/Reflection</strong> (parallel discovery + critic · skip
        narrow lookups) · <strong>Router</strong> (classify &amp; dispatch · skip if one path) ·{' '}
        <strong>Coordinator+Workers</strong> (parallel/separable work · skip if sequential
        shared-state).
    </p>
</div>

<div class="cheat">
    <h3>Anti-pattern catalog</h3>
    <p style="margin:.4rem 0;line-height:1.7">
        <strong>Agent for everything</strong> (workflow would do) ·{' '}
        <strong>Multi-agent overuse</strong> (sequential shared-state work split into agents) ·{' '}
        <strong>Unlimited autonomy</strong> (no gates on irreversible actions) ·{' '}
        <strong>Poor context</strong> (dump-everything &rarr; poison + cost) ·{' '}
        <strong>Missing observability</strong> (no traces &rarr; "can't reproduce") ·{' '}
        <strong>No evaluation</strong> (silent regressions) · <strong>No guardrails</strong> (limits
        live in the prompt) · <strong>Autonomy theatre</strong> (demo-grade shipped as production).
    </p>
</div>

<div class="cheat">
    <h3>VP Engineering review questions</h3>
    <ol style="padding-left:1.3rem;margin:.5rem 0;line-height:1.8">
        <li>
            "Why is this an agent and not a workflow? What's the irreducible uncertainty agency buys
            us?"
        </li>
        <li>
            "What's the per-step success rate, and therefore the end-to-end rate over a full task?"{' '}
            <em>(0.99⁴⁰ &asymp; 67%.)</em>
        </li>
        <li>
            "What guarantees the loop terminates &mdash; show me the runtime cap, not a prompt
            sentence."
        </li>
        <li>
            "Enumerate every irreversible action. For each: what's the gate, and what's the
            compensation?"
        </li>
        <li>
            "Where does state live if a worker dies at step 40 of 60? Walk me through the resume."
        </li>
        <li>
            "What's cost per completed task, the kill switch, and the per-tenant anomaly alert?"
        </li>
        <li>"Show me how you debug a wrong action from three days ago without re-running it."</li>
        <li>
            "I change one prompt line &mdash; what automatically catches a 5% regression before it
            ships?"
        </li>
        <li>"Why this topology? Justify every agent split like a microservice boundary."</li>
    </ol>
</div>
