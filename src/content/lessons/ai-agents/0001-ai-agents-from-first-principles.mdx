# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: "What is an LLM, and what are the three things it can't do on its own?"
      a: >-
          <b>Hit these points:</b> an LLM is a stateless text-prediction function &mdash; <code>f(text)
          &rarr; text</code>, the reasoning model, the brain &rarr; it can't remember between calls,
          can't act on the world, and holds no state &rarr; a chatbot only "remembers" because the
          runtime re-sends the transcript every call.
    - level: 'core'
      q: 'Walk me through exactly what happens when an agent "uses a tool".'
      a: >-
          <b>Hit these points:</b> a tool is a callable capability &mdash; a typed function (name +
          param schema) wrapping an API or function &rarr; the model only <em>emits a request</em>
          (tool name + args), it never executes anything &rarr; the runtime runs the real function and
          feeds the result back into the next call &rarr; red flag: "the LLM ran the SQL" &mdash; it
          only asked.
    - level: 'core'
      q: 'Define an agent and its agent loop.'
      a: >-
          <b>Hit these points:</b> an agent is an LLM using tools to accomplish a goal &mdash; LLM +
          tools + loop + goal &rarr; the loop is observe (last result) &rarr; decide (LLM picks next
          step) &rarr; act (run a tool) &rarr; observe, repeating until the goal is met or a cap is hit
          &rarr; same model as a bare LLM; what's added is structural, not IQ.
    - level: 'core'
      q: 'What is the agent runtime, and what is memory in this stack?'
      a: >-
          <b>Hit these points:</b> the runtime is the orchestration layer &mdash; ordinary code that
          hosts the loop, holds state, calls the LLM, dispatches tools, enforces
          retries/timeouts/max-steps/permissions &rarr; memory is persisted state the model can't see
          until the runtime retrieves it back into the context window &rarr; short-term = the window
          (RAM, re-sent each call); long-term = an external store, pulled in via RAG.
    - level: 'senior'
      q: 'When would you NOT build an agent, and ship a workflow instead? Name the trade-off.'
      a: >-
          <b>Hit these points:</b> a workflow is a developer-defined path; an agent is a
          model-influenced path &rarr; if the steps are known and repeatable, write the workflow
          &mdash; you buy predictability, lower cost, lower latency, easier debugging &rarr; reach for
          an agent only when the path can't be predetermined and that flexibility is worth the autonomy
          tax &rarr; the named trade-off: predictability/cost vs flexibility/autonomy &rarr; the middle
          ground is an agentic workflow &mdash; a fixed path with a few model-chosen branches.
    - level: 'senior'
      q: 'Your agent occasionally loops forever and burns huge cost. Where do you fix it, and why is it not "the model"?'
      a: >-
          <b>Hit these points:</b> the fix lives in the runtime, not the model &mdash; max-steps,
          timeouts, token budgets, and guardrails are the runtime's job &rarr; the stateless LLM can't
          stop itself or host a loop; the layer that owns the loop owns the limits &rarr; blaming "the
          model" misreads the architecture &rarr; add observability per turn so a runaway run is a
          traced, capped event, not a surprise invoice.
    - level: 'senior'
      q: 'A team wants to add a second agent to "go faster." Talk them into it &mdash; or out of it.'
      a: >-
          <b>Hit these points:</b> same call as premature microservices &rarr; multi-agent costs
          coordination, latency, partial-failure handling, and roughly 15&times; the tokens
          (illustrative) &rarr; it only pays when the work is genuinely parallel and independent, or
          one context/role can't cope &rarr; start single-agent (the monolith); split with evidence,
          not vibes &rarr; "more agents" is not a capability, it's a bill.
    - level: 'senior'
      q: 'Explain MCP to a skeptical exec in one analogy, and name the problem it solves.'
      a: >-
          <b>Hit these points:</b> MCP is a protocol for exposing tools, resources, and prompts to AI
          clients &mdash; "USB-C / JDBC for tools" &rarr; without it, every agent-to-tool pair is
          bespoke glue: N&times;M integrations &rarr; expose a server once and any MCP client uses it
          &rarr; N+M &rarr; a server exposes the capability, a client connects from the host app &rarr;
          it is an open multi-vendor standard, not a Claude feature, and not the tool itself &mdash;
          it's the way to reach tools.
    - level: 'staff'
      q: 'A vendor pitches "fully autonomous agentic AI." Place it on the workflow&harr;agent spectrum and name the trade-off the pitch bought.'
      a: >-
          <b>Hit these points:</b> strip the buzzword to the engine &mdash; what tools, what loop, what
          guardrails? &rarr; "fully autonomous" sits at the far agent end: the model writes the path,
          so they bought flexibility and paid in predictability, cost, latency, and debuggability
          &rarr; ask what they gave up: can you reproduce a run, cap its spend, audit its actions?
          &rarr; for most real tasks the right answer is further left &mdash; an agentic workflow with
          autonomy only where the path truly can't be scripted &rarr; the senior read: autonomy is a
          cost you justify per use case, not a feature you brag about.
    - level: 'staff'
      q: 'How does a "coding agent" differ from a "support agent" architecturally &mdash; and what does that tell you about every agent product?'
      a: >-
          <b>Hit these points:</b> architecturally they barely differ &mdash; same engine: LLM + loop +
          runtime + memory &rarr; what changes is the tool registry, the system prompt, and the
          guardrails &mdash; same employee, different job description &rarr; so "AI agent product" is
          rarely a moat in the engine; the moat is the tools you wired, the data you retrieve, and the
          guardrails you can stand behind &rarr; evaluate any pitch on those three, not on the model
          name.
    - level: 'staff'
      q: 'Build vs buy: a team wants to adopt an off-the-shelf agent runtime/SDK instead of writing their own loop. How do you reason about it?'
      a: >-
          <b>Hit these points:</b> the loop itself is cheap to write; the hard, ongoing parts are state
          handling, retries, tool dispatch, permissions, observability, and evals &mdash; that's what a
          mature runtime buys you &rarr; buy when those are commodity and you want to spend your effort
          on tools, prompts, and domain guardrails &rarr; build when your control, latency, security,
          or audit requirements outgrow the SDK's seams &rarr; name the lock-in: how tools are defined,
          how memory is wired, how you trace a run &rarr; the reversible move is to keep your tools and
          guardrails portable and treat the runtime as swappable plumbing.
    - level: 'design'
      q: "Design a support agent that's safe to ship for a SaaS billing product."
      a: >-
          <b>A strong answer covers:</b> goal &mdash; resolve common billing questions and disputes,
          escalate the rest &rarr; tools: look up invoice/payments (read), issue refund and adjust
          subscription (write), notify customer, open a ticket &rarr; bounded loop in the runtime with
          max-steps and timeouts &rarr; guardrails: refund caps, human approval above a threshold,
          idempotent refunds so a retry never double-pays, full audit log, scoped permissions per tool
          &rarr; per-case memory plus retrieval of the customer's history &rarr; a workflow for the
          happy path with an agent escape-hatch for ambiguous disputes &rarr; eval on resolution rate
          and a hard zero-tolerance metric for wrong-amount refunds. <em>Want to run this live? Ask me
          to grade your whiteboard answer.</em>
    - level: 'design'
      q: 'Design an SRE incident-triage agent. What can it touch, and where does it stop?'
      a: >-
          <b>A strong answer covers:</b> goal &mdash; triage an alert, gather signal, propose (not
          silently execute) remediation &rarr; tools: read metrics/logs/traces and recent deploys
          (read-heavy), with write actions like rollback or scale gated behind explicit approval &rarr;
          observe&rarr;decide&rarr;act loop &mdash; an intent-level reconciliation loop &mdash; capped
          by max-steps so it can't thrash &rarr; guardrails: read-only by default, human-in-the-loop
          for any production-mutating action, blast-radius limits, idempotent and reversible
          operations, a full audit trail &rarr; memory of the current incident plus retrieval of
          similar past incidents/runbooks &rarr; workflow for the known runbook steps, agent autonomy
          only for the open-ended investigation &rarr; eval on time-to-diagnosis and false-action rate,
          and design so the worst failure is a no-op, never an outage.

    - >-
        <b>Goal &amp; scope</b> &mdash; what the agent is responsible for, and the explicit line where

    - >-
        <b>Tools</b> &mdash; the callable capabilities it needs (read vs write), each with a scoped
        permission boundary.
    - >-
        <b>Loop &amp; runtime</b> &mdash; observe&rarr;decide&rarr;act, with max-steps, timeouts, and
        token budgets in the runtime.
    - >-
        <b>Guardrails</b> &mdash; approval thresholds, idempotency on writes, dry-run/rollback, and
        hard limits on destructive actions.
    - >-
        <b>Memory</b> &mdash; per-task short-term context plus what's persisted to an external store
        and retrieved back.
    - >-
        <b>Workflow vs agent</b> &mdash; script the common path, keep an agent escape-hatch only for
        the messy cases.
    - >-
        <b>Observability &amp; eval</b> &mdash; per-run traces, an audit log of every action, and how
        you'd measure success/regressions.

        <div class="fact">
            <b>Why it exists</b> The next action depends on the last result — you must react, not
            pre-plan.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> Debugging an outage: symptom → hypothesis → try one thing → observe
            → adjust.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> A K8s reconciliation loop / a REPL / an event loop.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "The loop runs inside the model / it plans all steps first"</div>
        <div class="t">✓ The loop is code in the runtime; the model is called once per turn</div>
    </div>
    <div class="rule">
        <span>🔁 Memory rule:</span> The agent loop is a reconciliation loop for intent: observe,
        think, act, repeat until desired state.
    </div>
    <MemoryCheck items={mc4} />
</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Agent runtime — the app server around the brain</h2>
    </div>
    <p class="tagline">
        Plain code that hosts the loop. The layer leaders skip — and where reliability lives.
    </p>
    <Fig6 />
    <div class="facts">
        <div class="fact">
            <b>Is</b> The program running the loop: holds state, calls the LLM, runs tools, enforces
            guardrails.
        </div>
        <div class="fact">
            <b>Why it exists</b> The stateless LLM can't host a loop or stop itself; something must
            — and it's code you own.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> The office around the employee: building, phones, badges, the
            manager who says "stop".
        </div>
        <div class="fact blue">
            <b>Like (code)</b> <b>Laravel + web server + queue worker.</b> Claude Code, Cursor, the
            Agent SDK are runtimes.
            <sup><a href="#s4" id="r4">4</a></sup>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "The model manages its own memory &amp; limits"</div>
        <div class="t">✓ The runtime re-sends history and enforces every limit</div>
    </div>
    <div class="rule">
        <span>⚙️ Memory rule:</span> The runtime is the app server hosting the loop — it holds
        state, runs tools, and enforces the guardrails the model can't.
    </div>
    <MemoryCheck items={mc5} />
</section>

{/* ================= LEVEL 6 ================= */}

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Memory — because the brain forgets everything</h2>
    </div>
    <p class="tagline">
        Not a model feature — a runtime discipline. It's just what gets re-injected.
    </p>
    <Fig7 />
    <div class="facts">
        <div class="fact">
            <b>Is</b> Whatever the runtime puts back into the prompt. Short-term = context window;
            long-term = external store.
        </div>
        <div class="fact">
            <b>Why it exists</b> The model forgets between calls and the window is finite; relevant
            info must be re-injected.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> The employee's notebook (today) + the company wiki (looked up on
            demand).
        </div>
        <div class="fact blue">
            <b>Like (code)</b> Stateless HTTP + sessions + DB. Context = RAM payload; store = disk;
            RAG = a query injecting rows.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Bigger context window = memory"</div>
        <div class="t">✓ That's more RAM per call; durable memory is the external store</div>
    </div>
    <div class="rule">
        <span>💾 Memory rule:</span> The model is stateless; memory is what the runtime reloads into
        the prompt — context is RAM, the store is disk.
    </div>
    <MemoryCheck items={mc6} />
</section>

{/* ================= LEVEL 7 ================= */}

<section class="lvl" id="l7">
    <div class="lvltop">
        <span class="num">7</span>
        <h2>Workflow vs Agent — who writes the if/else?</h2>
    </div>
    <p class="tagline">
        The most important <em>decision</em> in the stack — and the one VPs over-engineer.
    </p>
    <Fig8 />
    <table>
        <tr>
            <th>Dimension</th>
            <th>Workflow</th>
            <th>Agent</th>
        </tr>
        <tr>
            <td>Controls flow</td>
            <td>You (developer)</td>
            <td>The model, at runtime</td>
        </tr>
        <tr>
            <td>Predictability</td>
            <td>High</td>
            <td>Lower (path emerges)</td>
        </tr>
        <tr>
            <td>Cost / latency</td>
            <td>Low</td>
            <td>Higher (many LLM calls)</td>
        </tr>
        <tr>
            <td>Debuggability</td>
            <td>Easy (fixed paths)</td>
            <td>Harder (per-run traces)</td>
        </tr>
        <tr>
            <td>Best for</td>
            <td>Known, repeatable tasks</td>
            <td>Open-ended, unpredictable tasks</td>
        </tr>
    </table>
    <p class="tagline">
        <b>Agentic workflow</b> = the middle: a fixed workflow with a few model-decided steps (a
        runbook with "use your judgment here").
    </p>
    <div class="myth">
        <div class="m">✗ "Agents are the advanced / better default"</div>
        <div class="t">
            ✓ Anthropic: start simple, use workflows, reach for agents only when flexibility pays
            <sup><a href="#s1" id="r1b">1</a></sup>
        </div>
    </div>
    <div class="rule">
        <span>🧭 Memory rule:</span> Workflow = you write the if/else; Agent = the model writes it;
        choose by how predictable the path is.
    </div>
    <MemoryCheck items={mc7} />
</section>

{/* ================= LEVEL 8 ================= */}

<section class="lvl" id="l8">
    <div class="lvltop">
        <span class="num">8</span>
        <h2>MCP — USB-C for tools</h2>
    </div>
    <p class="tagline">One standard so you don't hand-write glue for every external system.</p>
    <Fig9 />
    <div class="facts">
        <div class="fact">
            <b>MCP</b> An open standard to discover &amp; call tools/data from external systems.
            "USB-C for AI."
            <sup><a href="#s3" id="r3b">3</a></sup>
        </div>
        <div class="fact">
            <b>Why it exists</b> Kills the N×M bespoke-glue mess: expose a server once, any MCP
            agent uses it → N+M.
        </div>
        <div class="fact blue">
            <b>Server / Client</b> Server = exposes tools (device). Client = connects from the host
            app (port).
        </div>
        <div class="fact blue">
            <b>Tool registry</b> The catalog advertised to the model = service discovery for tools.
        </div>
    </div>
    <div class="myth">
        <div class="m">
            ✗ "MCP is a Claude feature / MCP <em>is</em> the tool"
        </div>
        <div class="t">
            ✓ Open multi-vendor standard; it's the <em>protocol to reach</em> tools (≈ JDBC)
        </div>
    </div>
    <div class="rule">
        <span>🔌 Memory rule:</span> MCP is USB-C for tools — server = device, client = port,
        registry = catalog of what's plugged in.
    </div>
    <MemoryCheck items={mc8} />
</section>

{/* ================= LEVEL 9 ================= */}

<section class="lvl" id="l9">
    <div class="lvltop">
        <span class="num">9</span>
        <h2>Single vs Multi-agent — monolith vs microservices</h2>
    </div>
    <p class="tagline">The same architecture call you've made a hundred times for services.</p>
    <Fig10 />
    <div class="facts">
        <div class="fact">
            <b>Single</b> One loop, one context, one tool registry, end to end. The monolith.
        </div>
        <div class="fact">
            <b>Multi</b> Orchestrator plans &amp; delegates to worker subagents (own context+tools),
            then synthesizes.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> Monolith vs microservices / map-reduce. Orchestrator = the
            dispatcher.
        </div>
        <div class="fact blue">
            <b>The cost</b> Coordination, latency, partial failure, observability — and ~15× tokens.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "More agents = better"</div>
        <div class="t">
            ✓ Premature-microservices trap; split only when one context/role can't cope
        </div>
    </div>
    <div class="rule">
        <span>🧩 Memory rule:</span> Single agent is a monolith; multi-agent is microservices —
        split only when one context/role can't cope.
    </div>
    <MemoryCheck items={mc9} />
</section>

{/* ================= LEVEL 10 ================= */}

<section class="lvl" id="l10">
    <div class="lvltop">
        <span class="num">10</span>
        <h2>Archetypes — one engine, many job descriptions</h2>
    </div>
    <p class="tagline">
        The payoff: every "AI agent product" is the same engine + a different toolset + a different
        prompt.
    </p>
    <Fig11 />
    <div class="myth">
        <div class="m">✗ "Each agent type is different technology"</div>
        <div class="t">✓ Identical architecture; only tools + instructions differ</div>
    </div>
    <div class="rule">
        <span>👔 Memory rule:</span> Every archetype is the same engine + a different tool registry
        + a different job description.
    </div>
    <MemoryCheck items={mc10} />
</section>

    ★ Cheat sheet — buzzword → engineering
</h2>
<div class="cheat">
    <p style="margin-top:0">

        Runtime = the app server · Memory = what's reloaded · Workflow = you choose the path · Agent

    </p>

    <table>
        <tr>
            <th>Buzzword</th>
            <th>What it actually is</th>
        </tr>

            <td>An LLM in a while-loop with tools + a goal</td>
        </tr>
        <tr>
            <td>Function calling / tools</td>
            <td>Typed RPC the model requests, you execute</td>
        </tr>
        <tr>
            <td>Orchestration framework</td>
            <td>The runtime (app server for the loop)</td>
        </tr>
        <tr>
            <td>Context window</td>
            <td>Bounded RAM, re-sent every call</td>
        </tr>
        <tr>
            <td>Memory / RAG</td>
            <td>External store + a query injecting text into the prompt</td>
        </tr>
        <tr>
            <td>Workflow / chain</td>
            <td>A hardcoded pipeline / DAG you wrote</td>
