# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is an agent runtime, and name the things it does that the LLM cannot do for itself?'
      a: >-
          <b>Hit these points:</b> the runtime is the orchestration layer wrapping a stateless LLM in
          a loop &rarr; it reads the model's output and parses any tool-call block &rarr; executes the
          real tool (file, API, DB) and feeds the result back &rarr; manages conversation state across
          turns &rarr; enforces retries/backoff, loop caps, token budget, timeouts and cost ceilings
          &rarr; the LLM only predicts text — it can't call tools, keep state, retry, or stop itself,
          so every one of those jobs is runtime code, not model behaviour.
    - level: 'core'
      q: 'Define the context window and explain what competes for it on a single agent call.'
      a: >-
          <b>Hit these points:</b> the context window is a fixed token budget the model accepts per
          call — a hard ceiling, the same for everyone on that model &rarr; everything shares that one
          budget: the system prompt, the tool/function definitions (grow with #tools), the
          conversation history (grows every turn — the quiet eater), the retrieved data you pull in,
          and the space reserved for the model's own answer &rarr; only what's left after the overhead
          is yours for useful context &rarr; so window size isn't the real constraint — what you choose
          to spend it on is.
    - level: 'core'
      q: 'Name the three memory tiers and what each one is for.'
      a: >-
          <b>Hit these points:</b> <b>short-term</b> = the conversation history living in the context
          window — like RAM, gone on reset &rarr; <b>working</b> = the task scratchpad/plan: current
          notes, intermediate results, in-flight tool outputs &rarr; <b>long-term</b> = a persisted
          knowledge store that survives across sessions and is retrieved on demand &rarr; the
          through-line: the LLM remembers nothing between calls, so each tier is just a different place
          state is stored and a different cost to read it back into the window.
    - level: 'core'
      q: >-
          Define memory in an agent system — why is "the agent remembers" a misleading phrase?
      a: >-
          <b>Hit these points:</b> memory is persisted state that lives <em>outside</em> the model and
          is only "known" if the runtime retrieves it back into the context window &rarr; the LLM is a
          stateless pure function — it recalls nothing from past calls on its own &rarr; so "the agent
          remembers a past conversation" really means "something stored a fact and something retrieved
          it into this call" &rarr; if it wasn't stored and re-injected, it does not exist to the model
          &rarr; the phrase hides the two steps (store, retrieve) where the real engineering — and the
          real bugs — live.
    - level: 'senior'
      q: 'Walk me through the lifecycle of one agent loop iteration from the moment the LLM returns output.'
      a: >-
          <b>Hit these points:</b> LLM returns text with a tool-call block &rarr; runtime parses it and
          extracts tool name + args &rarr; validates the tool name against the allowed list and
          validates the arg schema &rarr; executes the real function (file read, API call, DB query)
          &rarr; captures output or error &rarr; appends a <code>[tool_result]</code> message to the
          conversation history &rarr; calls the LLM again with updated context &rarr; repeat until the
          LLM returns a final answer (no tool call) or a guardrail (max-steps, budget, timeout) trips
          and the runtime halts.
    - level: 'senior'
      q: 'Why is context management often a bigger lever than which model you pick?'
      a: >-
          <b>Hit these points:</b> the model can only reason over what it's given — the right document
          missing means a wrong answer at any model size &rarr; right context on a smaller model
          routinely beats wrong context on the flagship, and costs less &rarr; cost and latency scale
          with tokens sent, so context selection is also your primary cost lever &rarr; context
          failures are silent: the model never says "you gave me the wrong files," it just answers
          confidently from what it has &rarr; a model upgrade is a flat tax on every call; better
          retrieval is an asset that compounds across features.
    - level: 'senior'
      q: 'What belongs in long-term memory and what should never go there?'
      a: >-
          <b>Hit these points:</b> store <em>derived, structured facts</em> — a customer's
          preferences, settled decisions, durable entities keyed for lookup &rarr; don't dump raw
          conversation transcripts: noisy, expensive to search, and they bury the one fact that
          mattered &rarr; don't store redundant or stale facts — version/timestamp so they can be
          pruned &rarr; keep volatile, high-stakes data (current balance, plan, open-ticket status) as
          live look-ups, not stale embeddings &rarr; avoid PII unless you genuinely need it and have a
          compliant retention policy &rarr; rule of thumb: persist the conclusion, not the
          conversation.
    - level: 'senior'
      q: >-
          "A bigger context window means we don't need context engineering anymore." Where does that
          break?
      a: >-
          <b>Hit these points:</b> a bigger window raises the ceiling, not the relevance — selection
          still decides what the model reasons over &rarr; real corpora (a multi-million-token
          monorepo) still exceed any window, so you're choosing a subset regardless &rarr; cost and
          latency scale with tokens sent, so a 5&times; bigger context is roughly a 5&times; bigger
          bill on every call (illustrative) &rarr; long-context recall is uneven — facts in the middle
          get lost while start and end are over-weighted &rarr; irrelevant tokens dilute attention and
          degrade the answer; they're not free padding &rarr; the fix is a better working context, not
          more raw capacity.
    - level: 'staff'
      q: "Design the runtime guardrails so an agent can't loop forever or burn $500 on one task."
      a: >-
          <b>Hit these points:</b> layer independent limits so no single failure is unbounded: a loop
          cap (max iterations per session) &rarr; a per-turn token budget so one call can't blow up
          &rarr; a per-session cumulative cost ceiling tracked in real dollars, checked before every
          model call, that hard-stops when crossed &rarr; a wall-clock timeout on the whole session
          &rarr; circuit breakers on tool failures (max retries + exponential backoff, then abort
          instead of retrying forever) &rarr; loop-detection on repeated identical tool calls/outputs
          &rarr; human-in-the-loop interrupts that pause before high-cost or irreversible actions
          &rarr; make every limit observable and alarmed, and fail closed — halt with a clear error
          rather than silently continuing.
    - level: 'staff'
      q: 'Design memory for an agent that must keep meaningful state across sessions and across many users.'
      a: >-
          <b>Hit these points:</b> separate the tiers by durability and backend — short-term in the
          live message list, working in fast scratch state (in-process/Redis), long-term in a persisted
          store &rarr; for cross-session state, write <em>derived facts</em> keyed by a stable id
          (user/account), not raw transcripts &rarr; scope every read and write to the authenticated
          user so one account can never retrieve another's memory &rarr; choose the store by access
          pattern: relational/key-value for exact keyed facts, a vector store for fuzzy semantic
          recall, and live look-ups for volatile data &rarr; define a write policy (what gets promoted
          to long-term, and when) and a retention/versioning policy so stale facts are pruned &rarr; on
          session start, retrieve the small relevant set and inject it as a compact structured block,
          never the whole history &rarr; instrument it: measure "had-the-right-fact" rate, and budget
          the slice of the window memory may consume.
    - level: 'staff'
      q: 'Answers are wrong. A teammate wants to spend the quarter upgrading the model. Make the principal-level call.'
      a: >-
          <b>Hit these points:</b> don't decide by opinion — instrument the failing cases and attribute
          them by class: was the needed fact even in the context, or was it present and the model still
          failed? &rarr; control test: hand-feed the correct context to the same model; if it now
          answers, the defect was retrieval, not reasoning, and a bigger model just reasons better over
          the wrong input &rarr; in most products retrieval/context is the dominant cause, so a model
          swap pays a flat per-token tax while leaving the real bug in place &rarr; frame the
          trade-off: model upgrade = recurring cost, one-time bump any competitor can also rent;
          retrieval + memory investment = an asset you own that compounds &rarr; sequence it — fix and
          evaluate context first, then pay for model capability only once data shows reasoning is the
          binding constraint, with a clear before/after metric and a kill criterion.
    - level: 'design'
      q: >-
          Design the runtime + context strategy for a long-running production agent that may run for
          hours over a 2M-token codebase with a 200k window.
      a: >-
          <b>A strong answer covers:</b> the loop is the spine — parse output, validate tool calls
          against an allow-list, execute, append <code>[tool_result]</code>, re-call until done or a
          guardrail trips &rarr; because a long run is unbounded by default, every limit must be
          explicit: loop cap, per-turn token budget, a cumulative session cost ceiling in real dollars
          checked before each call, a wall-clock timeout, tool retries with backoff, and loop-detection
          on repeated identical calls &rarr; you can show ~10% of the repo at most, so context
          selection is the product: retrieve by exact symbol match + semantic similarity, walk the
          import/dependency graph for structure, re-rank, then pack signatures/docstrings over full
          bodies with the query-named files first &rarr; reserve fixed budget for history and the
          answer, and compress or summarize old turns as the run grows so history doesn't crowd out
          fresh context &rarr; checkpoint progress (a durable plan/scratchpad) so a long task survives
          a restart &rarr; instrument assembled context + token/cost spend per call, alarm on ceilings,
          and fail closed with a clear halt rather than silently continuing &rarr; name the trade-off:
          pre-indexed embeddings (fast, can go stale) vs read-on-demand (fresh, more round-trips).
    - level: 'design'
      q: >-
          Design the memory architecture for a customer-support agent that must recall prior
          interactions across sessions under a fixed token budget.
      a: >-
          <b>A strong answer covers:</b> map the three tiers to backends — short-term is the live
          conversation in the window; working is the in-flight task state (current ticket, steps
          taken); long-term is a persisted store of derived facts (preferences, past resolutions,
          entitlements) keyed by <code>customer_id</code> &rarr; never persist raw transcripts to
          long-term: write structured, derived facts so retrieval is cheap and precise &rarr; partition
          the window into fixed slices (system prompt &amp; policy, retrieved facts, recent history,
          current question, reserved answer) so no source starves the others &rarr; on session start,
          look up the customer's facts by id and inject a compact structured block; pull volatile,
          exact data (current plan, open tickets, balance) live rather than from stale storage &rarr;
          define the write policy: after an interaction, promote new durable facts to long-term,
          version/timestamp them, and prune stale ones &rarr; scope every read/write to the
          authenticated customer so you never leak another account's memory &rarr; observability:
          measure "had-the-right-fact" rate and retrieval hit-rate, not just CSAT &rarr; failure modes:
          missing fact &rarr; ask or escalate rather than hallucinate; conflicting KB vs customer fact
          &rarr; prefer the customer-specific one; budget overflow &rarr; drop lowest-ranked history
          before identity/policy &rarr; name the trade-off: pre-stored facts (fast, can lag) vs live
          look-ups (fresh, exact, more latency), plus the privacy line that history is always retrieved
          per-customer.

<section class="lvl" id="m1">
    <div class="lvltop"><span class="num">1</span><h2>Agent Runtime — the OS around the LLM</h2></div>
    <p class="tagline">The LLM is stateless. Someone has to run the loop, manage tools, and know when to stop. That's the runtime.</p>

    <Fig1 />

    <div class="facts">
        <div class="fact"><b>Is</b> Orchestration code that wraps an LLM in a loop — reads output, executes tools, feeds results back, repeats until done or halted.</div>
        <div class="fact"><b>Why it exists</b> The LLM is a stateless pure function. It can't call tools, maintain state, retry failures, or stop itself. The runtime supplies all of that.</div>
        <div class="fact blue"><b>Like (world)</b> The office around a brilliant but absent-minded consultant. The consultant (LLM) only answers questions. The office (runtime) books the meetings, delivers results, tracks budget, and fires the consultant if they go over hours.</div>
        <div class="fact blue"><b>Like (code)</b> Laravel + php-fpm + Horizon (queue worker) combined. The LLM is one handler function. The runtime is the web server + queue + retry policy + middleware chain around it.<sup><a href="#s1" id="r1b">1</a></sup></div>
    </div>

    <div class="myth">
        <div class="m">✗ "The LLM decides when to stop and what tools to call"</div>
        <div class="t">✓ The LLM <em>requests</em> a tool call as text output; the runtime decides whether to execute it, execute it safely, and when the loop is done</div>
    </div>

    <div class="rule"><span>🏗️ Memory rule:</span> Runtime = Agent OS. The LLM is a CPU instruction set — powerful but inert. The runtime is the kernel that actually runs the process.</div>

<section class="lvl" id="m2">
    <div class="lvltop"><span class="num">2</span><h2>Context Management — the RAM budget</h2></div>
    <p class="tagline">A 200k-token window sounds huge. A real codebase is millions of tokens. Every coding agent is fighting this constraint every call.</p>

    <Fig3 />

    <div class="facts">
        <div class="fact"><b>Is</b> The process of deciding what text to put in the limited context window before each LLM call — selecting, ranking, and assembling the most relevant content from a larger source.</div>
        <div class="fact"><b>Why it exists</b> LLMs have a fixed token budget per call. A real codebase, conversation history, docs, and tool outputs together vastly exceed it. Something must choose what fits.</div>
        <div class="fact blue"><b>Like (world)</b> A consultant's pre-meeting briefing pack. You can't hand them 10,000 pages — someone curates the 20 most relevant pages. Wrong curation = wrong advice.</div>
        <div class="fact blue"><b>Like (code)</b> Redis working set vs full database. The context window is L1 cache. The codebase is the full DB. Context selection is the query + cache-fill strategy.<sup><a href="#s2" id="r2">2</a></sup></div>
    </div>

    <div class="myth">
        <div class="m">✗ "A bigger context window means you don't need context engineering"</div>
        <div class="t">✓ Bigger windows reduce pressure but don't eliminate it; irrelevant tokens dilute attention and raise cost — curation still wins</div>
    </div>

    <div class="rule"><span>📐 Memory rule:</span> Context window = RAM. You can't load the whole disk into RAM. Context engineering is the query that decides what loads.</div>

        </tr>
        <tr>
            <td>Weakness</td>
            <td>Embedding quality can miss structural/logical relationships</td>
            <td>Slower start; reads more tokens to understand project layout</td>
        </tr>
        </tbody>
    </table>

    <details class="iq">
        <summary>Why is context management often more important than the model itself?</summary>
        <div class="a">
            <b>Hit these points:</b> The model can only reason about what it's given — wrong or irrelevant context produces wrong answers regardless of model capability. <b>Right context with a smaller model often beats wrong context with the best model.</b> Budget is also a direct function of tokens sent — context selection is the primary cost lever. Finally, context errors are silent: the model doesn't say "you gave me the wrong files" — it just answers incorrectly based on what it received.
        </div>
    </details>

    <details class="iq">
        <summary>You are building a coding agent. The codebase is 2M tokens but your window is 200k. What is your retrieval strategy?</summary>
        <div class="a">
            <b>Hit these points:</b> embed + similarity search for relevant files by function/class/symbol name &rarr; supplement with structural context (file tree, import graph, dependency map) &rarr; prioritize files the user's query mentions by name — exact matches first &rarr; compress large files by showing only function signatures and docstrings, not bodies &rarr; reserve a fixed token budget for tool outputs and conversation history and enforce it &rarr; track what you've loaded to avoid refetching &rarr; instrument retrieval quality so you can measure and improve hit rate over time.
        </div>
    </details>

</section>

{/* ============ MODULE M3: MEMORY ============ */}

<section class="lvl" id="m3">
    <div class="lvltop"><span class="num">3</span><h2>Memory — because the LLM forgets everything</h2></div>
    <p class="tagline">The LLM has no memory. Every durable fact must be stored somewhere else and retrieved deliberately.</p>

    <Fig4 />

    <div class="facts">
        <div class="fact"><b>Is</b> The set of mechanisms an agent uses to store and retrieve information across turns — conversation history (short-term), task scratchpad (working), and a knowledge store (long-term) that persists across sessions.</div>
        <div class="fact"><b>Why it exists</b> The LLM itself remembers nothing between calls. Every fact the agent needs across more than one turn must be explicitly stored and fed back in.</div>
        <div class="fact blue"><b>Like (world)</b> Human memory: RAM (what you're thinking right now) + notepad (notes from this meeting) + company wiki (facts that outlive this meeting and are shared).</div>
        <div class="fact blue"><b>Like (code)</b> In-process variables (short-term) + Redis scratchpad (working) + PostgreSQL / vector store (long-term). Each tier has different latency, durability, and query model.</div>
    </div>

    <div class="myth">
        <div class="m">✗ "An agent 'knows' things from past conversations"</div>
        <div class="t">✓ An agent knows what the runtime injects into the context window. Past facts only appear if explicitly retrieved from long-term storage and included.</div>
    </div>

    <div class="rule"><span>🗄️ Memory rule:</span> LLM = stateless function. Memory = what the runtime injects. Nothing is remembered unless something stored it and something retrieved it.</div>

    <MemoryCheck items={frontmatter.mcMemory} />

    <details class="iq">
        <summary>Your support agent needs to remember a customer's product preferences across sessions. What memory tier handles this and what's your storage/retrieval design?</summary>
        <div class="a">
            <b>Hit these points:</b> long-term memory — persists across sessions by definition &rarr; store as <b>structured facts</b> in a relational DB keyed by <code>customer_id</code> (not raw transcripts) &rarr; retrieve by <code>customer_id</code> lookup at session start and inject into the system prompt as a short, structured block &rarr; update after each interaction where a new preference is expressed — write a derived fact, not the full transcript &rarr; version or timestamp facts so stale data can be pruned &rarr; avoid PII in long-term store unless you have a compliant data-retention policy for it.
        </div>
    </details>

    <details class="iq">
        <summary>What's the difference between working memory and context window?</summary>
        <div class="a">
            <b>Hit these points:</b> <b>Context window</b> is the physical token budget passed to the LLM each call — it's a technical constraint of the model API. <b>Working memory</b> is a higher-level concept: the scratchpad state the agent maintains for the current task — plans, intermediate results, tool outputs in progress. Working memory is often what fills the context window for a given turn, but short-term history and long-term retrieved facts also compete for that same budget. The distinction matters: context window is the constraint; working memory is one category of content that consumes it.
        </div>
    </details>

</section>
