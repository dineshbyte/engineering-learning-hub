# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: >-
          Snapshot isolation prevents dirty reads and lost updates. So what does it <em>not</em>
          prevent, and why?
      a: >-
          <b>Hit these points:</b> <em>write skew</em> — two txns read an overlapping condition and each
          write a <em>different</em> row &rarr; SI's first-committer-wins only catches same-row
          write-write conflicts, so neither write is blocked &rarr; both commit and together break an
          invariant no serial order could produce &rarr; name a concrete instance: last unit of
          inventory, double-booked room, both on-call doctors going off shift &rarr; the one-liner:
          "SI's only hole is write skew — read a shared condition, write different rows."
    - level: 'core'
      q: 'What is serializable snapshot isolation, in one breath?'
      a: >-
          <b>Hit these points:</b> SSI runs every txn on ordinary MVCC snapshots — reads stay
          non-blocking, exactly like SI &rarr; on top of that it <em>additionally</em> tracks read/write
          dependencies between concurrent txns at runtime &rarr; when it spots the pattern that can cause
          write skew (the dangerous structure of rw-antidependencies), it aborts one txn rather than let
          the anomaly commit &rarr; so it delivers true serializability with SI's read performance,
          paying in aborts instead of read locks &rarr; "SI plus a runtime detector that aborts the txn
          that would break serializability."
    - level: 'core'
      q: "What is an rw-antidependency, and why is it the only edge SI can't prevent?"
      a: >-
          <b>Hit these points:</b> an rw-antidependency is "T1 reads a version of x, then T2 writes a
          <em>newer</em> version of x" — T1 read stale, so in any serial order T1 must come
          <em>before</em> T2 &rarr; SI already prevents ww (first-committer-wins) and cross-txn wr (you
          only read committed snapshot versions) &rarr; but a write that lands after your frozen snapshot
          is invisible to you, yet it still orders you first — SI has no way to notice it &rarr; hence the
          only cycles SI can produce are made of rw-edges &rarr; that's why SSI watches rw-edges
          specifically and nothing else.
    - level: 'core'
      q: 'Define serializability precisely enough that you could build a detector from the definition.'
      a: >-
          <b>Hit these points:</b> model each committed txn as a node &rarr; draw a directed edge whenever
          one txn must precede another in any equivalent serial order (wr = read another's write, ww =
          write-write order, rw = read-then-overwritten) &rarr; a history is serializable <em>iff</em> the
          resulting dependency graph is acyclic &rarr; a cycle means each txn is both before and after the
          others, so no serial order exists &rarr; the detector's only job: don't let a cycle commit
          &rarr; this is the conflict-serializability / dependency-graph framing the whole SSI argument
          rests on.
    - level: 'senior'
      q: "State Fekete's theorem and explain why it makes SSI cheap."
      a: >-
          <b>Hit these points:</b> every non-serializable SI history contains <em>two consecutive
          rw-edges</em> meeting at a pivot txn — an inbound rw from T-in and an outbound rw to T-out, with
          the pivot concurrent with both &rarr; that local two-edge shape (the "dangerous structure") is
          <em>necessary</em> for any SI anomaly &rarr; so the engine never has to build the full graph or
          confirm a cycle actually closed — it just watches each txn for one inbound + one outbound rw
          flag &rarr; break any one of the three edges and the cycle can't form &rarr; that's the whole
          reason SSI is cheap: detect a local pattern, not a global cycle.
    - level: 'senior'
      q: 'Why does SSI abort transactions that were actually serializable — and is that a bug?'
      a: >-
          <b>Hit these points:</b> the dangerous structure is <em>necessary</em> for an anomaly but
          <em>not sufficient</em> — the same two-edge shape also appears in histories where the cycle
          never closes and the result is serializable &rarr; aborting on the shape alone is deliberately
          <strong>conservative</strong>: it catches every real anomaly plus some innocents (false
          positives) &rarr; that's the price of detecting a local pattern instead of proving a global
          cycle, which would be far more expensive &rarr; so it's a designed trade-off, not a bug &rarr;
          the false-positive rate climbs with contention and with coarse predicate-lock granularity, which
          is exactly what you tune.
    - level: 'senior'
      q: >-
          How does PostgreSQL detect an rw-edge at runtime without making reads block, and what must the
          app do?
      a: >-
          <b>Hit these points:</b> SIRead predicate locks — non-blocking <em>records</em> of what each
          serializable read touched (tuple &rarr; page &rarr; relation granularity, plus index-range
          locks to catch phantoms) &rarr; they take no real lock, so they never block a writer; they just
          leave a trace &rarr; a write landing on a SIRead-locked datum is a detected rw-conflict, flagged
          on both txns involved &rarr; when one txn ends up with both an inbound and an outbound rw flag
          (the pivot), the engine aborts it with SQLSTATE <code>40001</code> &rarr; the app contract:
          <strong>retry on 40001</strong>, made safe by idempotent transactions &rarr; note SIRead locks
          outlive the reader's commit, because a later writer can still close the cycle.
    - level: 'senior'
      q: 'A serializable workload is aborting too often under load. Walk me through your tuning levers.'
      a: >-
          <b>Hit these points:</b> keep txns short and reads selective — this cuts both false positives
          and predicate-lock pressure, and it's the highest-leverage fix &rarr; raise
          <code>max_pred_locks_per_transaction</code> to delay summarization, because once SIRead locks
          summarize up to coarse (page/relation) granularity, false positives spike &rarr; move long
          reporting reads to <code>READ ONLY DEFERRABLE</code> for a free safe snapshot that can't be
          aborted &rarr; watch the oldest open txn — like VACUUM, it sets the cost &rarr; if contention is
          genuinely high, measure the crossover where 2PL's blocking tax beats SSI's abort tax, rather
          than assuming SSI always wins.
    - level: 'staff'
      q: "Compare SI, SSI, and 2PL on what they guarantee and how they fail — and when you'd pick each."
      a: >-
          <b>Hit these points:</b> <b>SI</b> — consistent snapshot, first-committer-wins; fast
          non-blocking reads but leaks write skew; pick it when no cross-row invariant is at risk or you
          can materialize the conflict &rarr; <b>SSI</b> — SI plus rw-dependency tracking; true
          serializability with non-blocking reads, fails by <em>aborting</em> (40001) and consumes
          predicate-lock memory; pick it for invariant-critical workloads at low-to-moderate contention
          with a retry loop &rarr; <b>2PL</b> — locks held to commit; serializable by blocking, fails by
          <em>waiting</em> and deadlocks; pick it (or explicit <code>FOR UPDATE</code>) at high contention
          where abort-and-retry would thrash &rarr; synthesis: the choice is "fail by aborting (optimistic,
          SSI) vs fail by blocking (pessimistic, 2PL)," decided by contention; measure the crossover, don't
          assume.
    - level: 'staff'
      q: 'Your fleet is sharded and each Postgres node runs SERIALIZABLE. Is the system serializable?'
      a: >-
          <b>Hit these points:</b> no — PostgreSQL SSI guarantees serializability <em>within one
          server</em> only &rarr; it tracks rw-dependencies through that node's own SIRead locks; it has
          no visibility into reads/writes on another shard, so a cycle that spans shards is invisible to
          every node &rarr; the guarantee does not extend across shards or logical-replication boundaries
          for free &rarr; distributed serializability is a separate, harder problem requiring a global
          commit order (consensus / a distributed SSI, as in CockroachDB/Spanner-style systems) &rarr; the
          staff move: state the scope of the guarantee explicitly, and don't let "each node is
          serializable" be mistaken for "the system is serializable."
    - level: 'staff'
      q: >-
          A team wants to default the whole OLTP service to SERIALIZABLE to "never think about anomalies
          again." Make the principal call.
      a: >-
          <b>Hit these points:</b> don't decide by slogan — characterize the workload: contention level,
          txn duration, which txns actually carry cross-row invariants &rarr; upside: Serializable removes
          the whole anomaly class, and on SSI reads stay non-blocking, so at low contention the cost is
          modest &rarr; but the cost is real and workload-dependent: abort rate (and false positives)
          rises with contention, predicate-lock memory grows, and <em>every</em> serializable txn now
          needs a correct 40001 retry loop or you've shipped a latent failure &rarr; and the guarantee is
          single-node, so it gives nothing across shards &rarr; the principal call: apply Serializable to
          the specific txns whose invariants need it, keep the rest at the lowest sufficient level, mandate
          retry + idempotency, and gate the rollout on measured abort rate with a kill criterion &rarr;
          make it a measured decision, not a global flag flipped on faith.
    - level: 'design'
      q: >-
          Choose a concurrency-control design for a hospital on-call scheduler that must always keep at
          least one doctor on call.
      a: >-
          <b>A strong answer covers:</b> name the hazard — this is the canonical write-skew case: two
          doctors each read "another is still on call" and each go off shift, dropping the count to zero
          &rarr; clarify scale: low write volume, a hard invariant, correctness dominates throughput
          &rarr; option A, the cheap fix: materialize the conflict into a single lockable row (an on-call
          <em>count</em> or a guard row) and take <code>SELECT … FOR UPDATE</code> before going off shift,
          so the two txns serialize on one row &rarr; option B: run the go-off-shift txn at SERIALIZABLE
          so SSI's rw-tracking aborts the loser, with a 40001 retry loop &rarr; given low contention
          either works; prefer SSI if there are many such invariants and you don't want to hand-place
          locks, prefer the materialized lock if it's one hot invariant &rarr; reserve fixed effort for
          idempotent retries &rarr; trade-off: explicit lock (predictable, blocks, must remember to take
          it) vs SSI (declarative, non-blocking reads, aborts under contention) &rarr; note the guarantee
          is single-node, so a sharded scheduler needs a global ordering.
    - level: 'design'
      q: >-
          Design isolation for a service that runs short invariant-critical writes and long analytical
          reads against the same database, choosing among SI, SSI, and 2PL.
      a: >-
          <b>A strong answer covers:</b> partition by workload — the two paths have opposite needs and
          shouldn't share one global level &rarr; the short invariant-critical writes are where write skew
          bites, so run those at SERIALIZABLE (SSI) with a 40001 retry loop, or pin the conflict with
          <code>FOR UPDATE</code> if contention is high enough that aborts would thrash &rarr; the long
          analytical reads must not be aborted and must not pin cleanup: run them as
          <code>READ ONLY DEFERRABLE</code> to get a safe snapshot with zero SSI overhead, or route them
          to a read replica &rarr; keep write txns short and reads selective to bound predicate-lock memory
          and false positives &rarr; watch the crossover: if write contention is genuinely high, 2PL's
          blocking may beat SSI's abort churn — measure it &rarr; observability: track abort rate,
          predicate-lock usage, and the oldest open txn &rarr; trade-offs: SSI (non-blocking reads, aborts)
          vs 2PL (no aborts, blocks + deadlocks); safe-snapshot reads (free, must be read-only) vs replica
          reads (isolated, but lag); and the reminder that all of it is single-node.

        <h2>Serializability is a graph property</h2>
    </div>

    <table>
        <tr>
            <th>Edge</th>
            <th>Meaning</th>
            <th>Orders</th>
            <th>Under SI?</th>
        </tr>
        <tr>
            <td>wr</td>
            <td>T1 writes x; T2 reads that version</td>
            <td>T1 → T2</td>
            <td>Never crosses concurrent txns — you read only committed snapshot versions</td>
        </tr>
        <tr>
            <td>ww</td>
            <td>T1 writes x; T2 overwrites it</td>
            <td>T1 → T2</td>
            <td>Blocked — first-committer-wins stops two concurrent writers of x</td>
        </tr>
        <tr>
            <td>rw</td>
            <td>
                T1 reads x; T2 writes a <em>later</em> x
            </td>
            <td>T1 → T2</td>
            <td>
                <strong>The only edge SI can't prevent</strong> — your snapshot is frozen, later
                writes are invisible yet still order you first
            </td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Master theorem</b> A multiversion history is serializable <em>exactly when</em> its
            dependency graph has no cycle. (Adya 1999; Bernstein/Hadzilacos/Goodman.)
            <sup><a href="#s6" id="r6">6</a></sup>
        </div>
        <div class="fact">
            <b>Why a cycle is fatal</b> T1→T2→…→T1 asserts each txn is both before and after the
            others — impossible to lay on a line.
        </div>
        <div class="fact blue">
            <b>The narrowing</b> SI already prevents ww and cross-txn wr cycles, so the{' '}
            <em>only</em> way an SI history breaks is a cycle made of <code>rw</code>-edges.
        </div>
        <div class="fact blue">
            <b>Why it matters</b> The whole job of a serializable engine reduces to one rule:{' '}
            <em>don't let a cycle commit.</em>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "You must build and topologically sort the whole graph"</div>
        <div class="t">
            ✓ You only need to prevent <em>rw-cycles</em> — and §3 shrinks even that
        </div>
    </div>
    <div class="rule">
        <span>🔗 Memory rule:</span> Serializable ⟺ acyclic dependency graph; under SI the only
        cycles left are made of rw-antidependencies.
    </div>
    <MemoryCheck items={frontmatter.mc.slice(3, 6)} />
</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>The dangerous structure — two rw-edges and a pivot</h2>
    </div>
    <p class="tagline">
        The result SSI is built on, and the reason it can be cheap. Sharper than "watch for cycles."
    </p>
    <figure>
        <img
            src="diagrams/ssi-03-dangerous-structure.png"
            alt="Every snapshot-isolation anomaly contains this exact shape. In any non-serializable SI history, some transaction T-pivot has BOTH an incoming rw-edge (T-in to T-pivot) and an outgoing rw-edge (T-pivot to T-out) — two rw-antidependencies in a row, meeting at the pivot. Both conflicting transactions are concurrent with the pivot, and T-out commits first."
        />
        <figcaption>
            <strong>Every SI anomaly contains this exact shape.</strong> Some{' '}
            <strong>T-pivot</strong> has BOTH an inbound <code>rw</code>-edge (from T-in) and an
            outbound one (to T-out) — two rw-antidependencies <em>in a row</em>. Both conflicting
            txns are concurrent with the pivot; T-out commits first (Fekete's refinement).
        </figcaption>
    </figure>
    <Fig3 />
    <blockquote style="margin:1rem 0;padding:.5rem 0 .5rem 1rem;border-left:3px solid var(--rule);font-family:ui-sans-serif,system-ui,sans-serif;font-size:.86rem;color:#3a372f;font-style:italic">
        <strong>Fekete's theorem (2005):</strong> in any non-serializable SI execution, the cycle
        contains two <em>consecutive</em> rw-edges — a T-pivot with an inbound rw-edge from a T-in
        and an outbound rw-edge to a T-out, both concurrent with the pivot.
        <sup><a href="#s2" id="r2b">2</a></sup>
    </blockquote>
    <div class="facts">
        <div class="fact">
            <b>Necessary…</b> Every real anomaly contains the dangerous structure. Catch the pattern
            → catch every write skew.
        </div>
        <div class="fact">
            <b>…not sufficient</b> The pattern also appears in <em>perfectly serializable</em>{' '}
            histories (the cycle never closes, or the third edge runs the other way).
        </div>
        <div class="fact blue">
            <b>So a detector is conservative</b> Aborting on the pattern catches every anomaly{' '}
            <em>and</em>
            some innocents — by design.
        </div>
        <div class="fact blue">
            <b>The payoff</b> Cheap-and-conservative beats exact-and-expensive: no full graph, no
            cycle confirmation. That's what makes §4 possible.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Seeing the dangerous structure means a cycle definitely exists"</div>
        <div class="t">✓ It's necessary, not sufficient — the would-be cycle may never close</div>
    </div>
    <div class="rule">
        <span>⚠️ Memory rule:</span> Dangerous structure = one pivot with an inbound AND outbound
        rw-edge; necessary for every anomaly, sufficient for none.
    </div>
    <MemoryCheck items={frontmatter.mc.slice(6, 9)} />
</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>How PostgreSQL catches it — SIReads &amp; a conservative abort</h2>
    </div>
    <p class="tagline">
        Detect rw-antidependencies <em>as they form</em>, without ever making a read block.
    </p>
    <figure>
        <img
            src="diagrams/ssi-04-ssi-detection.png"
            alt="SSI detection at runtime. Every read under SERIALIZABLE leaves a non-blocking SIRead lock recording what was read, escalating from tuples to pages to relations under pressure. A later write landing on SIRead-locked data flags an rw-conflict and sets an in/out flag. When one transaction has BOTH an inbound and outbound flag — the dangerous structure — the engine aborts a transaction (preferring the pivot) with SQLSTATE 40001. SIRead locks survive the reader's commit."
        />
        <figcaption>
            <strong>SSI detection at runtime.</strong> Every read leaves a non-blocking{' '}
            <strong>SIRead "lock"</strong> recording what was touched. A later write landing on
            SIRead-locked data is a detected rw-conflict and sets an in/out flag. Both flags on one
            txn → the engine aborts (preferring the pivot) with <strong>SQLSTATE 40001</strong>.
            SIRead locks survive the reader's commit — a conflicting writer can still arrive.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Piece</th>
            <th>What it does</th>
        </tr>
        <tr>
            <td>SIRead locks</td>
            <td>
                Predicate locks taken on what a serializable read touched. They{' '}
                <strong>never block</strong>a writer — they only leave a record "this txn read
                this." A write later falling on one = a detected rw-edge.
            </td>
        </tr>
        <tr>
            <td>Granularity</td>
            <td>
                Tuple → page → relation as a txn reads more; index-range locks cover{' '}
                <em>predicates</em>, which is how SSI catches phantoms (rows that don't exist yet
                but would match). Coarser = less memory, more conflicts.
            </td>
        </tr>
        <tr>
            <td>Abort rule</td>
            <td>
                Each txn carries an inbound-conflict flag and an outbound-conflict flag. When
                <strong>both</strong> set on one txn (the pivot), roll one back — preferring the
                pivot itself, unless it already committed.
            </td>
        </tr>
        <tr>
            <td>Locks outlive reader</td>
            <td>
                A SIRead lock is retained <em>after</em> the reading txn commits, because a writer
                creating the dangerous structure can show up later. This is why long, read-heavy
                txns inflate the footprint.
            </td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>The contract</b> On <code>40001</code> (
            <code>could not serialize access due to read/write dependencies</code>),{' '}
            <strong>retry the transaction.</strong> Non-negotiable.
        </div>
        <div class="fact">
            <b>Why retry is safe</b> Idempotency makes the retry harmless — see the Idempotency Keys
            deep dive. An app that doesn't retry is broken, not unlucky.
        </div>
        <div class="fact blue">
            <b>Conservative, again</b> Fires on the <em>structure</em>, not a confirmed cycle, so
            every genuine write skew is caught — plus a few innocents.
        </div>
        <div class="fact blue">
            <b>Citation</b> Cahill et al. (SIGMOD 2008) gave the algorithm; Ports &amp; Grittner
            (VLDB 2012) shipped it in PostgreSQL.
            <sup><a href="#s1" id="r1b">1</a></sup>
            <sup><a href="#s3" id="r3b">3</a></sup>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "SIRead locks block writers like real locks"</div>
        <div class="t">
            ✓ They never block — they only record reads so a later write reveals an rw-edge
        </div>
    </div>
    <div class="rule">
        <span>🔍 Memory rule:</span> Reads leave non-blocking SIRead marks; a write on a marked
        datum is an rw-edge; both flags on one txn → abort with 40001 → retry.
    </div>
    <MemoryCheck items={frontmatter.mc.slice(9, 13)} />
</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>The price, the tuning, and the escape hatch</h2>
    </div>
    <p class="tagline">
        SSI buys serializability at ~SI throughput. "Roughly" and "conservative" each cost something
        — senior use means knowing the bill.
    </p>
    <figure>
        <img
            src="diagrams/ssi-05-tradeoff.png"
            alt="The trade-off and the fast path. Versus strict 2PL (blocks readers, deadlocks) and plain SI (fast but admits write skew), SSI keeps reads non-blocking and stays correct, paying instead in false-positive aborts and predicate-lock memory. The escape hatch: a READ ONLY transaction proven to a safe snapshot can neither suffer nor cause an anomaly, so PostgreSQL runs it as plain REPEATABLE READ with zero SSI overhead."
        />
        <figcaption>
            <strong>The trade-off, and the fast path.</strong> Vs strict 2PL (blocks readers,
            deadlocks) and plain SI (fast, admits write skew), SSI keeps reads non-blocking and
            stays correct — paying in false-positive aborts and predicate-lock memory. A{' '}
            <code>READ ONLY</code> txn proven to a "safe snapshot" runs as plain REPEATABLE READ,
            zero SSI overhead.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th></th>
            <th>Strict 2PL</th>
            <th>Plain SI</th>
            <th>SSI</th>
        </tr>
        <tr>
            <td>Correct (serializable)</td>
            <td>Yes</td>
            <td>No — write skew</td>
            <td>Yes</td>
        </tr>
        <tr>
            <td>Reads block writers</td>
            <td>Yes</td>
            <td>No</td>
            <td>No</td>
        </tr>
        <tr>
            <td>Failure mode</td>
            <td>Deadlocks, latency</td>
            <td>Silent anomalies</td>
            <td>False-positive aborts</td>
        </tr>
        <tr>
            <td>Stance</td>
            <td>Pessimistic (block up front)</td>
            <td>—</td>
            <td>Optimistic (abort at the end)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>False positives</b> Rise with contention and with <em>coarse</em> predicate locks —
            as SIReads summarize tuple→page→relation, unrelated rows collide and innocents abort.
        </div>
        <div class="fact">
            <b>Predicate-lock memory</b> Bounded by <code>max_pred_locks_per_transaction</code> ×
            connections. Past it, Postgres <em>summarizes</em> to coarser granularity (more false
            positives) rather than failing.
        </div>
        <div class="fact blue">
            <b>The oldest txn sets the cost</b> One long serializable txn holds its SIRead locks the
            whole time — same way a long txn starves VACUUM in MVCC.
        </div>
        <div class="fact blue">
            <b>Safe-snapshot escape hatch</b> A read-only txn that can't be part of any dangerous
            structure drops to REPEATABLE READ. <code>READ ONLY DEFERRABLE</code> waits for that
            snapshot on purpose.
            <sup><a href="#s3" id="r3c">3</a></sup>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Each node serializable ⇒ the sharded fleet is serializable"</div>
        <div class="t">
            ✓ Postgres SSI is <em>single-node</em>; distributed serializability is a harder problem
        </div>
    </div>
    <div class="rule">
        <span>💸 Memory rule:</span> 2PL pays up front in blocking; SSI pays at the end in
        conservative aborts — for read-mostly, low-conflict OLTP, optimism wins.
    </div>
    <MemoryCheck items={frontmatter.mc.slice(13, 17)} />
</section>

<h2 class="sec" id="cheat">
    ★ Cheat sheet — term → engineering
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> Write skew = read a shared condition, write different rows ·
        Serializable = acyclic dependency graph · rw-antidependency =
        read-then-someone-writes-later, the only edge SI can't stop · Dangerous structure = one
        pivot, two rw-edges (necessary, not sufficient) · SIRead lock = a non-blocking read receipt
        · 40001 = retry · Safe snapshot = abort-proof read-only.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Term</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>Snapshot isolation</td>
            <td>
                Read a frozen consistent snapshot; first-committer-wins on same-row writes —{' '}
                <em>not</em> serializable
            </td>
        </tr>
        <tr>
            <td>Write skew</td>
            <td>
                Read overlapping condition → each writes a different row → both commit → invariant
                broken
            </td>
        </tr>
        <tr>
            <td>Dependency graph</td>
            <td>Nodes = txns, edges = must-precede (wr / ww / rw); serializable ⟺ acyclic</td>
        </tr>
        <tr>
            <td>rw-antidependency</td>
            <td>T1 reads old x, T2 writes newer x; T1 must come first — SI can't prevent it</td>
        </tr>
        <tr>
            <td>Dangerous structure</td>
            <td>A pivot with both an inbound and outbound rw-edge; necessary, not sufficient</td>
        </tr>
        <tr>
            <td>SIRead lock</td>
            <td>Non-blocking predicate lock recording a read; outlives the reader's commit</td>
        </tr>
        <tr>
            <td>SQLSTATE 40001</td>
            <td>"could not serialize access" → the app must retry the transaction</td>
        </tr>
        <tr>
            <td>READ ONLY DEFERRABLE</td>
            <td>Wait for a safe snapshot, then run as REPEATABLE READ with zero SSI cost</td>
        </tr>
    </table>
    <h3>Three rules for the principal chair</h3>
    <p style="margin-bottom:0">
        ① SERIALIZABLE is the cheap-and-conservative bet: it <strong>will</strong> abort under
        contention — so <strong>always retry on 40001</strong>. ② Keep txns <strong>short</strong>{' '}
        and reads
        <strong>selective</strong> to cut false positives and lock memory; push long reads to
        <code>READ ONLY DEFERRABLE</code>. ③ The guarantee is <strong>single-node</strong> — don't
        assume a sharded fleet is serializable because each node is.
    </p>
</div>
