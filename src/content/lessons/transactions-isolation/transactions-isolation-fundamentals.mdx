# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What does ACID actually promise — and what does "atomic" <em>not</em> mean?'
      a: >-
          <b>Hit these points:</b> A = abortable — undo a partial txn on failure, <em>not</em> anything
          about concurrency &rarr; C = your application invariant, mostly your code's job, the database
          only enforces what you declare &rarr; I = concurrent txns behave <em>as if</em> run in some
          serial order &rarr; D = committed data survives planned faults (crash, power loss) &rarr;
          bonus: A and D ride the same WAL, and "ACID-compliant" says nothing about the default
          isolation level you actually run at.
    - level: 'core'
      q: 'Name the standard isolation levels from weakest to strongest, and what each adds.'
      a: >-
          <b>Hit these points:</b> <b>Read Uncommitted</b> — may see another txn's uncommitted writes
          (dirty reads); almost no engine actually offers true dirty reads as a default &rarr; <b>Read
          Committed</b> — only committed data, but a fresh read each statement, so values can change
          mid-txn &rarr; <b>Repeatable Read</b> — a stable per-txn view; in practice this is snapshot
          isolation in Postgres &rarr; <b>Serializable</b> — the result equals <em>some</em> serial
          order; forbids all anomalies &rarr; the ladder is "which anomalies are still allowed," and
          the name on the tin lies — reason from the matrix, not the label.
    - level: 'core'
      q: 'List the read/write anomalies and give a one-line definition of each.'
      a: >-
          <b>Hit these points:</b> <b>dirty read</b> — read data another txn hasn't committed &rarr;
          <b>non-repeatable read (read skew)</b> — same row read twice in one txn returns two values
          &rarr; <b>phantom</b> — a range/predicate query returns a different set because rows were
          inserted/deleted &rarr; <b>lost update</b> — two read-modify-write cycles on the same row,
          one overwrites the other &rarr; <b>write skew</b> — two txns read an overlapping condition
          and each write a <em>different</em> row, breaking an invariant &rarr; an anomaly is any
          result no serial order could produce.
    - level: 'core'
      q: 'Explain how an MVCC engine serves a consistent read without taking locks.'
      a: >-
          <b>Hit these points:</b> every write appends a <em>new</em> versioned tuple tagged with its
          creating txn id rather than overwriting in place &rarr; a reader carries a snapshot and walks
          the version chain, returning the newest version committed before its snapshot &rarr;
          therefore readers never block writers and writers never block readers &rarr; the cost is
          dead versions that need VACUUM/GC, and a long-running txn pins the cleanup horizon open
          &rarr; bloat &rarr; "consistent read = read your snapshot, ignore everything newer."
    - level: 'senior'
      q: 'Two services both set "Repeatable Read." Do they give the same guarantees?'
      a: >-
          <b>Hit these points:</b> no — the name is not the guarantee &rarr; Postgres RR is snapshot
          isolation: it leaks write skew and has no gap locks &rarr; MySQL InnoDB RR is snapshot-ish
          but adds next-key (gap) locks that block many phantoms RR doesn't require &rarr; Oracle's
          "Serializable" is actually SI and still admits write skew &rarr; the senior move: reason from
          the level&times;anomaly matrix for <em>your</em> engine and version, never from the ANSI
          label, and verify with a concurrency test before you rely on it.
    - level: 'senior'
      q: 'A counter is losing increments under load. Walk me through diagnosis and the fix ladder.'
      a: >-
          <b>Hit these points:</b> classic <em>lost update</em> — read-modify-write in app code where
          both txns read the same stale value and write back, so one increment vanishes &rarr; confirm
          by checking whether the write is computed in the app from a prior SELECT rather than in the
          database &rarr; fixes cheapest-first: atomic <code>SET n = n + 1</code> in one statement
          &rarr; <code>SELECT … FOR UPDATE</code> to serialize the read-modify-write &rarr;
          compare-and-set with a version column plus a retry loop &rarr; note Postgres SI auto-detects
          lost updates and aborts a loser, but MySQL InnoDB RR does not, so the engine changes which
          fix you need.
    - level: 'senior'
      q: "Why doesn't Snapshot Isolation prevent write skew, and how do you actually fix it?"
      a: >-
          <b>Hit these points:</b> both txns read their pre-write snapshots and each write a
          <em>different</em> row, so first-committer-wins has no same-row conflict to catch &rarr; both
          commit and break an invariant (last on-call doctor, last unit of inventory, double-booked
          room) that no serial order would allow &rarr; fixes: raise to Serializable — Postgres SSI
          tracks read/write dependencies and aborts a loser at commit, so budget for retries on
          SQLSTATE 40001 &rarr; or materialize the conflict into a single lockable row (a guard row /
          summary row) and take <code>FOR UPDATE</code> on it &rarr; "lock the row you can name;
          serialize the conflict you can't."
    - level: 'senior'
      q: "Locking (2PL) vs MVCC for isolation — what's the real trade-off?"
      a: >-
          <b>Hit these points:</b> strict 2PL takes shared/exclusive locks and holds them to commit,
          so readers and writers block each other and you get serializability but contention and
          deadlocks &rarr; MVCC keeps multiple versions so reads run off a snapshot and never block
          writes, giving great read concurrency but only snapshot isolation by default (write skew
          leaks) &rarr; pure MVCC needs an extra layer (SSI, or explicit locks) to reach serializable
          &rarr; cost shifts from blocking (2PL) to version cleanup + abort/retry (MVCC) &rarr; most
          engines are MVCC for reads with locks on writes; the crossover is contention — measure where
          2PL's blocking tax beats MVCC's abort tax.
    - level: 'staff'
      q: '"Just set everything to Serializable and stop worrying about anomalies." Argue for, then against.'
      a: >-
          <b>For:</b> Serializable is the only level that removes the whole anomaly class, so you stop
          reasoning case-by-case about write skew and phantoms; on Postgres SSI reads stay
          non-blocking, so the baseline cost is modest at low contention. <b>Against:</b> under
          contention SSI aborts climb (false positives included), and <em>every</em> serializable txn
          now needs a correct retry loop on 40001 — without it you've shipped a latent 500; the
          guarantee is single-node only, so a sharded fleet isn't serializable just because each node
          is; latency and abort rate become workload-dependent. <b>Synthesis:</b> default to the
          lowest level that provably forbids the anomalies your invariants need; reserve Serializable
          for the txns that truly require it, and only after the retry loop and observability are in
          place.
    - level: 'staff'
      q: 'You inherit a service on Read Committed with occasional "impossible" data states. How do you find and classify the bug?'
      a: >-
          <b>Hit these points:</b> don't guess the level — reproduce and characterize the corruption
          first: is it a single row diverging (lost update), a cross-row invariant broken (write
          skew), or a count/range that shifted (phantom)? &rarr; map the symptom to the anomaly using
          the level&times;anomaly matrix for the actual engine/version, not the ANSI name &rarr;
          confirm with a deterministic two-session interleaving that reproduces it, so the fix is
          verified, not hoped &rarr; choose the narrowest fix: atomic statement or <code>FOR UPDATE</code>
          for lost update, materialized lock row or Serializable for write skew, predicate/index-range
          locks or Serializable for phantoms &rarr; add the missing observability — log isolation level
          per txn and alert on retry/abort spikes &rarr; the staff move is turning "occasional
          impossible state" into a named, reproduced anomaly with a regression test, not bumping the
          global level and hoping.
    - level: 'staff'
      q: 'An invariant must hold across two microservices with separate databases. How do you reason about isolation here?'
      a: >-
          <b>Hit these points:</b> first reframe — isolation levels are a <em>single-database</em>
          tool; there is no shared serializable order across two stores &rarr; options: a distributed
          txn (2PC) buys real atomicity but blocks on coordinator crash and couples availability,
          acceptable only for short steps on a shared XA stack or inside one engine with a replicated
          coordinator &rarr; the usual answer is a <em>saga</em>: a local txn per service plus a
          compensating action each, ordered so the irreversible pivot is last &rarr; make it safe with
          a transactional outbox (publish the event atomically with the local commit) and idempotency
          keys for at-least-once delivery &rarr; accept that you trade ACID isolation for eventual
          consistency with a visible <code>PENDING</code> state &rarr; the staff call is naming what
          guarantee you actually need (atomic vs eventually-consistent) before reaching for 2PC.
    - level: 'design'
      q: 'Design the transaction &amp; isolation strategy for a ticket-booking system where each seat may be sold once.'
      a: >-
          <b>A strong answer covers:</b> the invariant is "one seat &rarr; at most one paid booking," a
          classic write-skew / oversell risk &rarr; clarify scale: bursty contention on hot events,
          mostly short txns &rarr; cheapest correct design is to make the conflict a same-row write: a
          <code>seat</code> row with a status, taken with <code>SELECT … FOR UPDATE</code> (or a unique
          constraint on <code>(seat_id)</code> in a bookings table) so the second writer blocks or
          fails &rarr; if you'd rather not lock, run the reserve-then-pay txn at Serializable and retry
          on 40001 &rarr; reserve a short hold with a TTL so abandoned carts free the seat &rarr; keep
          the reserve txn tiny to cut contention and abort rate &rarr; name the trade-off: explicit
          lock (predictable, blocks) vs Serializable (non-blocking reads, aborts under contention)
          &rarr; observability: alert on oversell-attempt rate and abort spikes; cross-service payment
          uses a saga + idempotency, not a distributed txn.
    - level: 'design'
      q: 'Choose isolation levels for a system with a high-volume write path and a separate analytics reporting path on the same database.'
      a: >-
          <b>A strong answer covers:</b> partition by workload — the two paths have opposite needs
          &rarr; the OLTP write path wants short txns at the lowest level that protects its invariants
          (often Read Committed with targeted <code>FOR UPDATE</code> or Serializable only on the few
          invariant-critical txns) to minimize contention and abort rate &rarr; the analytics path
          wants a stable consistent view but must not pin the cleanup horizon: on Postgres use
          <code>READ ONLY DEFERRABLE</code> for a safe snapshot with zero SSI cost, and ideally route
          it to a read replica so long scans don't bloat the primary &rarr; call out the cross-path
          hazard: a long-running report's snapshot holds back VACUUM/purge DB-wide &rarr; bloat and
          freeze lag, so cap it with statement/idle-in-txn timeouts &rarr; observability: monitor
          oldest <code>xact_start</code> / backend_xmin and dead-tuple growth &rarr; trade-off: same-DB
          simplicity vs replica isolation, and freshness (replica lag) vs primary protection.

            default level is almost always weaker than that.
        </div>
        <div class="fact blue">
            <b>Durability</b> Committed data survives the faults you planned for (WAL + fsync, then
            replication). Never absolute — Jepsen proves it.
        </div>
    </div>

        <img
            src="diagrams/02-lost-update.png"
            alt="The canonical lost update. Two transactions both read counter=10, both add 1, both write 11 — the result is 11 instead of 12, and one increment is silently lost."
        />
        <figcaption>
            The canonical lost update: both read 10, both write 11. Result 11, not 12 — one
            increment vanishes. Look closely; this is the one you'll ship by accident.
        </figcaption>
    </figure>
    <div class="myth">
        <div class="m">✗ "If two txns write the same table at once, that's an anomaly"</div>
        <div class="t">
            ✓ An anomaly is a result <em>no serial order</em> could have produced — that clause is
            the entire game
        </div>
    </div>
    <div class="rule">
        <span>🐛 Memory rule:</span> Dirty read = saw uncommitted · Non-repeatable = row changed
        under me · Phantom = search result changed · Lost update = same-row clobber · Write skew =
        disjoint writes break one invariant.
    </div>
    <MemoryCheck items={mc2} />
</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>Isolation levels — the menu, and why the names lie</h2>
    </div>
    <p class="tagline">
        A level is a contract: <em>which anomalies will the DB prevent for me?</em> Read the matrix,
        never the name.
    </p>
    <figure>
        <img
            src="diagrams/03-isolation-matrix.png"
            alt="The isolation-level by anomaly matrix. Each cell shows whether a level prevents an anomaly; the names real databases use map onto these rows in surprising ways."
        />
        <figcaption>
            The level × anomaly matrix. Read it bottom-up: every step adds protection. Snapshot
            Isolation is the awkward guest — blocks phantoms, still leaks write skew, so it sits
            below true Serializable.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Level</th>
            <th>Dirty read</th>
            <th>Non-repeat.</th>
            <th>Phantom</th>
            <th>Write skew</th>
        </tr>
        <tr>
            <td>Read Uncommitted</td>
            <td>possible</td>
            <td>possible</td>
            <td>possible</td>
            <td>possible</td>
        </tr>
        <tr>
            <td>Read Committed</td>
            <td>prevented</td>
            <td>possible</td>
            <td>possible</td>
            <td>possible</td>
        </tr>
        <tr>
            <td>Snapshot (SI)</td>
            <td>prevented</td>
            <td>prevented</td>
            <td>prevented*</td>
            <td>
                <b>possible</b>
            </td>
        </tr>
        <tr>
            <td>Serializable</td>
            <td>prevented</td>
            <td>prevented</td>
            <td>prevented</td>
            <td>prevented</td>
        </tr>
    </table>
    <p class="tagline">
        ANSI SQL-92 names only three phenomena (dirty/non-repeatable/phantom).{' '}
        <b>Lost update and write skew aren't in the standard at all</b> — which is why SI doesn't
        fit any ANSI row cleanly.
    </p>
    <div class="facts">
        <div class="fact blue">
            <b>Postgres "Repeatable Read"</b> is actually Snapshot Isolation — stronger than ANSI RR
            (blocks phantoms) but still leaks write skew.
        </div>
        <div class="fact blue">
            <b>Oracle "Serializable"</b> is actually Snapshot Isolation — so it{' '}
            <em>allows write skew</em>. Not serializable in the textbook sense.
        </div>
        <div class="fact">
            <b>MySQL InnoDB "Repeatable Read"</b> is the default; blocks many phantoms via next-key
            (gap) locks — a different mechanism, different profile.
        </div>
        <div class="fact">
            <b>A level is a ceiling, not a floor</b> Nothing stops an engine preventing{' '}
            <em>more</em> than the level requires. Always check the engine, not the word.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Two DBs both set to 'Repeatable Read' give the same guarantees"</div>
        <div class="t">
            ✓ Same name, different real guarantees — a bug impossible on one is live on the other
        </div>
    </div>
    <div class="rule">
        <span>📖 Memory rule:</span> The standard defines levels by anomalies prevented; vendors
        slap those names on whatever their engine does. Reason from the matrix, never the name.
    </div>
    <MemoryCheck items={mc3} />
</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>Snapshot Isolation &amp; MVCC — how Read Committed / SI actually work</h2>
    </div>
    <p class="tagline">
        Real engines don't lock readers. They keep multiple versions of each row and let every
        reader see a frozen, consistent past.
    </p>
    <figure>
        <img
            src="diagrams/04-mvcc-snapshot.png"
            alt="Two snapshots, one row, no blocking. The version chain for row x and how Ta and Tb each read a different committed version while writers keep appending."
        />
        <figcaption>
            MVCC: every write <em>appends</em> a new version tagged with its txn id; readers walk
            the chain to the newest version committed before their snapshot. That's why readers
            never block writers.
        </figcaption>
    </figure>
    <Fig4 />
    <div class="facts">
        <div class="fact">
            <b>Read Committed</b> No dirty reads, no dirty writes; a fresh snapshot per statement;
            write locks held to commit. The default in Postgres, Oracle, SQL Server.
        </div>
        <div class="fact">
            <b>Snapshot Isolation</b> One snapshot per transaction; reads never block. Write
            conflicts resolved at commit by <em>first-committer-wins</em>; the loser aborts.
        </div>
        <div class="fact blue">
            <b>MVCC rules</b> Every UPDATE writes a new tuple (Postgres tags <code>xmin</code>/
            <code>xmax</code>); versions form a chain; visibility = was it committed before my
            snapshot?
        </div>
        <div class="fact blue">
            <b>The cost is garbage</b> Dead versions must be reclaimed (VACUUM / GC). A long-running
            txn pins old versions open → table bloat. The #1 MVCC incident.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "A long read-only transaction is harmless"</div>
        <div class="t">
            ✓ Its open snapshot pins old versions, stalls VACUUM, and bloats tables by gigabytes
        </div>
    </div>
    <div class="rule">
        <span>📸 Memory rule:</span> MVCC = append a version per write, read the newest one
        committed before your snapshot. RC snapshots per statement; SI per transaction; neither
        blocks reads.
    </div>
    <MemoryCheck items={mc4} />
</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Lost updates &amp; write skew — the subtle invariant-breakers</h2>
    </div>
    <p class="tagline">
        These survive a green test suite and a casual BEGIN/COMMIT, then corrupt prod under load.
        The fixes are structural, not "be careful".
    </p>
    <figure>
        <img
            src="diagrams/05-write-skew.png"
            alt="Write skew breaks an invariant that no single transaction violated. Two doctors, T1 and T2, each read the same on-call count (both see 2), each decide it is safe, each write only their own row to off-call; the combined result is zero doctors on call."
        />
        <figcaption>
            Write skew: two on-call doctors each read "count = 2, safe to leave", each write{' '}
            <em>their own</em> row. No lock conflict — different rows. Final state: zero on call.
            Each txn alone did nothing wrong.
        </figcaption>
    </figure>
    <h4 class="sans" style="color:#7a1f12">
        Lost update — three fixes, in this order
    </h4>
    <table>
        <tr>
            <th>Fix</th>
            <th>What it is</th>
            <th>Reach for it when</th>
        </tr>
        <tr>
            <td>1 · Atomic write</td>
            <td>
                <code>UPDATE c SET n = n + 1</code> — computed in-engine under its own lock
            </td>
            <td>new value is a pure fn of the old; cheapest &amp; safest</td>
        </tr>
        <tr>
            <td>2 · Explicit lock</td>
            <td>
                <code>SELECT … FOR UPDATE</code> — pessimistic row lock to commit
            </td>
            <td>you must read→compute→write in app code</td>
        </tr>
        <tr>
            <td>3 · Compare-and-set</td>
            <td>
                <code>… WHERE version = :v</code>; retry if 0 rows — optimistic
            </td>
            <td>scale; no lock across user think-time</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Lost update</b> Two read-modify-writes to the <em>same</em> object; one silently
            overwrites the other. Counters, balances, JSON merges.
        </div>
        <div class="fact">
            <b>Write skew</b> Two reads of overlapping rows, each passes a check, each writes a{' '}
            <em>different</em> row → joint invariant broken. SI does NOT prevent it.
        </div>
        <div class="fact blue">
            <b>Why SI misses it</b> Both read pre-write snapshots; write sets are disjoint, so
            first-committer-wins has nothing to conflict on.
        </div>
        <div class="fact blue">
            <b>Materialize the conflict</b> Add a lock row per shift/room/seat; force{' '}
            <code>FOR UPDATE</code> on it so disjoint writers collide. Ugly — use only without SSI.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Snapshot isolation will protect my on-call invariant"</div>
        <div class="t">
            ✓ Write skew slips through SI; use Serializable (SSI) or a materialized lock row
        </div>
    </div>
    <div class="rule">
        <span>⚖️ Memory rule:</span> Lock the row you can name (atomic write / FOR UPDATE / CAS for
        lost update); serialize the conflict you can't name (SSI / materialized row for write skew).
    </div>
    <MemoryCheck items={mc5} />
</section>

{/* ================= LEVEL 6 ================= */}

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Serializability — serial · 2PL · SSI</h2>
    </div>
    <p class="tagline">
        Instead of plugging anomalies one by one, demand that <em>none</em> can ever happen. Three
        ways to deliver it, three currencies to pay in.
    </p>
    <figure>
        <img
            src="diagrams/06-serializability-strategies.png"
            alt="A 2PL lock count rises in the growing phase and falls in the shrinking phase, forming a triangle. The dashed line marks the moment of first release, after which no new lock may ever be taken."
        />
        <figcaption>
            2PL's lock count rises (growing phase, acquire only) then falls (shrinking phase,
            release only). The first release ends growth — no new lock after it. That discipline is
            what produces serializability.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Strategy</th>
            <th>Style</th>
            <th>You pay in</th>
            <th>Wins when</th>
        </tr>
        <tr>
            <td>Serial execution</td>
            <td>no concurrency (1 thread)</td>
            <td>throughput ceiling</td>
            <td>data fits RAM; txns tiny, single-partition (VoltDB, Redis)</td>
        </tr>
        <tr>
            <td>2PL</td>
            <td>pessimistic locking</td>
            <td>lock waits, deadlocks, bad tail latency</td>
            <td>contention high &amp; aborts intolerable</td>
        </tr>
        <tr>
            <td>SSI</td>
            <td>optimistic, abort on conflict</td>
            <td>aborts + retries under contention</td>
            <td>contention low–moderate; reads must not block</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>"Serializable" formally</b> The result equals <em>some</em> serial order — not a
            specific one, not wall-clock order. Constrains the <em>outcome</em>, not the physical
            execution.
        </div>
        <div class="fact">
            <b>2PL adds predicate locks</b> Row locks can't lock a row that doesn't exist yet — so
            index-range / predicate locks stop phantom inserts. (Eswaran et al. 1976.)
        </div>
        <div class="fact blue">
            <b>SSI</b> Start from SI (no read locks), track read-write dependencies, abort one txn
            at commit if a dangerous structure forms. Postgres ships it as SERIALIZABLE. (Cahill et
            al. 2008.)
            <sup>
                <a href="#s3" id="r3">
                    3
                </a>
            </sup>
        </div>
        <div class="fact blue">
            <b>2PL ≠ 2PC</b> Two-phase <em>locking</em> = concurrency control on one node. Two-phase{' '}
            <em>commit</em> = atomic commit across nodes (Level 7). Same number, unrelated.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Serializable means transactions can't overlap in time"</div>
        <div class="t">
            ✓ They still overlap physically — only the <em>result</em> must match some serial order
        </div>
    </div>
    <div class="rule">
        <span>🔒 Memory rule:</span> Serial = forbid concurrency · 2PL = block (pessimistic,
        deadlocks) · SSI = let it run, abort losers (optimistic, retries). Read your workload to
        choose.
    </div>
    <MemoryCheck items={mc6} />
</section>

{/* ================= LEVEL 7 ================= */}

<section class="lvl" id="l7">
    <div class="lvltop">
        <span class="num">7</span>
        <h2>Distributed transactions &amp; 2PC — why two-phase commit blocks</h2>
    </div>
    <p class="tagline">
        Now the data is on two machines. Atomicity is no longer one log write. You need a tiny
        consensus protocol — and it inherits consensus's worst property.
    </p>
    <figure>
        <img
            src="diagrams/07-two-phase-commit.png"
            alt="2PC commits cleanly when nobody fails, but the coordinator crashing between phases freezes the participants in doubt. Top: prepare/vote then commit succeeds; bottom: the coordinator dies after both participants prepared, leaving them holding locks with no safe move."
        />
        <figcaption>
            2PC: prepare/vote, then commit/abort, driven by a coordinator. The coordinator's COMMIT
            log write is the single irrevocable moment. If it crashes after participants vote YES,
            they're stuck in doubt holding locks.
        </figcaption>
    </figure>
    <Fig5 />
    <div class="facts">
        <div class="fact">
            <b>Phase 1 — prepare/vote</b> Coordinator sends PREPARE; each participant does
            everything short of commit, writes a <em>prepared</em> record, replies YES → and
            surrenders its right to abort.
        </div>
        <div class="fact">
            <b>Phase 2 — commit/abort</b> All YES → coordinator writes COMMIT (the point of no
            return) → tells everyone. Any NO/timeout → ABORT.
        </div>
        <div class="fact blue">
            <b>The fatal flaw: it blocks</b> Coordinator crash after the vote leaves prepared
            participants in doubt, holding locks, with no safe unilateral move. A single point of
            failure that freezes data.
        </div>
        <div class="fact blue">
            <b>3PC doesn't save you</b> Non-blocking only under a synchronous network — which
            doesn't exist. You can't tell a crashed coordinator from a slow one. Essentially never
            used.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Add a third phase (3PC) and it's non-blocking and safe"</div>
        <div class="t">
            ✓ 3PC trades blocking for inconsistency under partition — worse on a real async network
        </div>
    </div>
    <div class="rule">
        <span>⛓️ Memory rule:</span> 2PC = vote then act, via a coordinator. The COMMIT log write is
        the point of no return. Coordinator crash post-vote = in-doubt participants block on their
        locks.
    </div>
    <MemoryCheck items={mc7} />
</section>

{/* ================= LEVEL 8 ================= */}

<section class="lvl" id="l8">
    <div class="lvltop">
        <span class="num">8</span>
        <h2>Sagas, outbox &amp; idempotency — eventual consistency for workflows</h2>
    </div>
    <p class="tagline">
        Checkout spans orders, payments, inventory — three DBs, three teams. No single BEGIN/COMMIT
        wraps them. So you trade isolation for availability.
    </p>
    <figure>
        <img
            src="diagrams/08-saga-compensation.png"
            alt="A three-step saga unwinding on failure. The forward path commits each local transaction in order; when inventory fails, the saga runs compensating transactions backward, semantically undoing rather than rolling back."
        />
        <figcaption>
            A saga = local transactions T₁…Tₙ, each with a compensating Cᵢ. Fail at step j → run
            Cⱼ…C₁ backward. A compensation is <em>not</em> a rollback: T₂ already committed, so you
            issue a new offsetting transaction (refund, not delete).
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Choose 2PC when</th>
            <th>Choose a saga when</th>
        </tr>
        <tr>
            <td>All participants share an XA stack</td>
            <td>Independent services / vendors</td>
        </tr>
        <tr>
            <td>Steps are short; locks held briefly</td>
            <td>The operation is long-running</td>
        </tr>
        <tr>
            <td>You need true isolation between ops</td>
            <td>You can tolerate visible intermediate states</td>
        </tr>
        <tr>
            <td>Operations have no natural undo</td>
            <td>Each step has a clean semantic undo</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Saga</b> Each Tᵢ is a real short local ACID txn; no locks across steps. Not
            serializable — others can see a charged-but-unfulfilled order. (Garcia-Molina &amp;
            Salem 1987.)
            <sup>
                <a href="#s4" id="r4">
                    4
                </a>
            </sup>
        </div>
        <div class="fact">
            <b>Transactional outbox</b> In the <em>same</em> local txn that updates business tables,
            INSERT the event into an <code>outbox</code> row. One commit, fully atomic. A relay
            publishes later.
        </div>
        <div class="fact blue">
            <b>Idempotency key</b> The relay gives at-least-once → duplicates are inevitable. Stamp
            each step <code>{'{sagaId}:{step}'}</code>; dedup on the consumer so retries don't
            double-charge.
        </div>
        <div class="fact blue">
            <b>The three are a set</b> Outbox (never lost) + idempotency (never double-applied) +
            compensations (forward or clean unwind). Drop one and you have a correctness hole.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "A compensating transaction is just a delayed rollback"</div>
        <div class="t">
            ✓ The committed row is real; you can't un-send the email — you send an apology (a new
            offsetting txn)
        </div>
    </div>
    <div class="rule">
        <span>🔄 Memory rule:</span> Sagas swap global locks for eventual consistency: local txn +
        compensation + outbox (at-least-once) + idempotency key (dedup). Model a PENDING state;
        tolerate the window.
    </div>
    <MemoryCheck items={mc8} />
</section>
