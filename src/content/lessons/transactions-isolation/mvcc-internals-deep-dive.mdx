# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What does multi-version storage mean — what happens to old data when a row is updated?'
      a: >-
          <b>Hit these points:</b> an UPDATE never overwrites in place — it creates a <em>new
          version</em> of the row and keeps the old one around &rarr; in Postgres the new tuple is
          inserted into the heap with <code>xmin</code> = the current xid and the old tuple's
          <code>xmax</code> is stamped; the old version lives until VACUUM removes it &rarr; in InnoDB
          the clustered row is changed in place but the prior image is written to the undo log,
          reachable via <code>DB_ROLL_PTR</code> &rarr; so a logical row is really a <em>chain</em> of
          physical versions &rarr; "no overwrite" is the core idea — both engines retain the prior
          version so concurrent readers can still see it.
    - level: 'core'
      q: 'What exactly is a snapshot, and what is in it?'
      a: >-
          <b>Hit these points:</b> a snapshot is the set of information a txn uses to decide which
          versions it may see &rarr; in Postgres it's <code>xmin</code> (oldest still-running xid),
          <code>xmax</code> (next xid not yet assigned), and <code>xip[]</code> (the list of
          in-progress xids) &rarr; together they define "committed as of this instant" &rarr; Read
          Committed takes a fresh snapshot per statement; Repeatable Read / SI takes one snapshot at
          the first read and reuses it for the whole txn &rarr; the snapshot is the txn's frozen view
          of the commit timeline — it never sees writes that commit after it.
    - level: 'core'
      q: 'State the visibility rule: how does a reader decide a given version is visible?'
      a: >-
          <b>Hit these points:</b> for each tuple the reader checks two gates against its snapshot
          &rarr; the creating xid (<code>xmin</code>) must be committed <em>and</em> in the past (not
          in <code>xip[]</code>, below <code>xmax</code>) &rarr; the deleting xid (<code>xmax</code>)
          must be absent, aborted, or still in the future relative to the snapshot &rarr; if both
          hold, that version is the one this reader sees; otherwise it walks to the next version in
          the chain &rarr; commit status comes from the commit log (clog) &rarr; "visible =
          created-by-a-committed-past-txn AND not-yet-deleted-as-of-my-snapshot."
    - level: 'core'
      q: 'What is version GC — VACUUM in Postgres, purge in InnoDB — and why is it needed?'
      a: >-
          <b>Hit these points:</b> because updates and deletes leave old versions behind, something
          must reclaim the ones no live snapshot can ever see again &rarr; Postgres VACUUM marks dead
          tuples reusable (and updates the visibility map / freezes old xids); plain VACUUM frees
          space for reuse but <code>VACUUM FULL</code> is what actually shrinks the file &rarr;
          InnoDB's purge thread reclaims undo records and delete-marked rows once no read view needs
          them &rarr; the cleanup ceiling is the oldest live transaction's horizon: nothing that died
          after it can be removed &rarr; without GC the table bloats and reads slow down.
    - level: 'senior'
      q: 'Walk me through exactly what happens on disk when you <code>UPDATE</code> one row in Postgres vs InnoDB.'
      a: >-
          <b>Hit these points:</b> Postgres inserts a <em>new</em> tuple (<code>xmin</code> = current
          xid), stamps the old tuple's <code>xmax</code>, and writes new index entries unless HOT
          applies; the old version stays in the heap for VACUUM &rarr; InnoDB modifies the
          clustered-index row <em>in place</em> but first writes an undo record reachable via
          <code>DB_ROLL_PTR</code>, and old readers walk the undo chain backward &rarr; key contrast:
          Postgres keeps history <em>in the heap</em> (cheap update, expensive cleanup, index churn),
          InnoDB keeps history <em>in undo</em> (in-place update, cheaper indexes, purge pressure)
          &rarr; in neither case is the prior version destroyed at write time.
    - level: 'senior'
      q: 'A table is 5&times; the size of its live row count and seq scans are slow. Diagnose it.'
      a: >-
          <b>Hit these points:</b> bloat from dead tuples VACUUM can't reclaim &rarr; find the cause:
          an old or <code>idle in transaction</code> connection, or a long analytics query, pinning
          the <code>xmin</code> horizon so nothing that died after it can be removed — and the effect
          is DB-wide, not just that table &rarr; check <code>pg_stat_activity</code>
          (<code>xact_start</code>, <code>backend_xmin</code>) and <code>n_dead_tup</code> &rarr; fix
          the offending txn, don't just run VACUUM &rarr; remember plain VACUUM frees space for reuse
          but won't shrink the file — only <code>VACUUM FULL</code> does, and it takes an exclusive
          lock.
    - level: 'senior'
      q: 'Why is updating an <em>indexed</em> column more expensive under MVCC — in both engines?'
      a: >-
          <b>Hit these points:</b> a new version generally means new or changed index entries &rarr;
          Postgres writes new index tuples unless HOT applies (HOT needs the update to stay on the
          same page and touch no indexed column) &rarr; index bloat; and because visibility lives in
          the heap, index-only scans depend on the visibility map &rarr; InnoDB delete-marks the old
          secondary-index entry and inserts a new one, and secondary-index reads often drop to
          clustered + undo to find the visible version &rarr; net effect in both: write amplification
          on the write path plus extra read cost — which is why "just add an index" isn't free under
          heavy updates.
    - level: 'senior'
      q: 'Connect a long-running analytics query to table bloat, and give the operational fix.'
      a: >-
          <b>Hit these points:</b> the query's snapshot pins the <code>xmin</code> horizon, so VACUUM
          can't reclaim anything that died <em>after</em> the query started — bloat grows for the
          whole duration of that one read &rarr; it also holds back freezing, so it's a wraparound
          risk too &rarr; the read path is fine (snapshots are cheap); the damage is to cleanup &rarr;
          levers: <code>idle_in_transaction_session_timeout</code> and statement timeouts to bound txn
          age, route heavy analytics to a read replica, and alert on the oldest <code>xact_start</code>
          / <code>backend_xmin</code> (and InnoDB history list length) &rarr; the rule: cleanup is
          only as current as your oldest open transaction.
    - level: 'staff'
      q: '"The database suddenly stopped accepting writes." What''s your first hypothesis and how does it tie to MVCC?'
      a: >-
          <b>Hit these points:</b> first hypothesis: transaction-ID <em>wraparound</em> — Postgres
          went read-only to avoid silent data loss &rarr; mechanism: xids are finite and compared
          modulo-2^31, so old versions must be <em>frozen</em> ("visible to all, ignore xmin") before
          the oldest unfrozen xid ages past the safety threshold &rarr; root cause is always
          "freezing fell behind": autovacuum disabled or starved, or blocked by a long-running txn
          pinning the horizon &rarr; recovery: let anti-wraparound autovacuum run (or VACUUM in
          single-user mode), then kill the long txn and fix monitoring &rarr; prevention: never
          disable autovacuum, alert on max xid age and oldest open txn &rarr; cite the Sentry /
          Mailchimp postmortems — every one is "autovacuum couldn't keep up."
    - level: 'staff'
      q: 'Postgres heap-history vs InnoDB undo-history — how does that one design choice ripple through operations?'
      a: >-
          <b>Hit these points:</b> the storage location of old versions sets the whole operational
          profile &rarr; Postgres keeps history in the heap: cheap in-place-ish writes but cleanup
          (VACUUM) competes for I/O, index entries churn, and a pinned horizon bloats the live table
          &rarr; you monitor <code>n_dead_tup</code>, autovacuum lag, <code>backend_xmin</code>, and
          freeze age, and you never disable autovacuum &rarr; InnoDB keeps history in undo: cheaper
          indexes and in-place clustered updates, but a long read view grows the undo/history list and
          purge falls behind &rarr; you monitor history list length and purge lag, and keep undo
          bounded &rarr; synthesis: different janitor, same root cause — a long transaction starving
          cleanup &rarr; the staff move is operating each engine to its specific failure mode rather
          than treating MVCC as one thing.
    - level: 'staff'
      q: 'How is PostgreSQL <code>SERIALIZABLE</code> built on top of MVCC, and what does it add — and cost?'
      a: >-
          <b>Hit these points:</b> SSI runs txns on ordinary MVCC snapshots with <em>no</em> extra
          locks on reads, then additionally tracks read/write dependencies &rarr; when it detects the
          dangerous structure (rw-antidependencies forming a potential cycle) that causes write skew,
          it aborts one txn with SQLSTATE 40001 &rarr; so MVCC is the substrate and SSI is an
          optimistic safety layer over it, not a replacement &rarr; cost: it's conservative (aborts
          some serializable histories), abort rate climbs with contention, and it consumes
          predicate-lock (SIRead) memory &rarr; operational contract: the app must retry on 40001 and
          keep txns short and reads selective &rarr; the guarantee is single-node only &rarr; the
          synthesis: SSI buys serializability without read locks by paying in aborts and lock memory
          instead of blocking.
    - level: 'design'
      q: 'Design the version-storage and GC strategy for an append-mostly audit-log table that is rarely updated but read by long analytics scans.'
      a: >-
          <b>A strong answer covers:</b> the workload is insert-heavy with few updates, so version
          churn is low — the real risk is long read scans, not write amplification &rarr; because
          updates are rare, dead-tuple bloat from updates is minimal; the threat is a long analytics
          snapshot pinning the cleanup horizon DB-wide and holding back freezing &rarr; defenses: run
          those scans on a read replica or under <code>READ ONLY DEFERRABLE</code> for a safe snapshot
          with no SSI cost, and cap them with statement timeouts &rarr; aggressively freeze old,
          immutable rows (they never change) so wraparound is a non-issue and they're skipped by
          VACUUM &rarr; partition by time so old partitions become read-only/freeze-once and cleanup
          stays local &rarr; observability: alert on oldest <code>xact_start</code> /
          <code>backend_xmin</code> and max xid age &rarr; trade-off: replica isolation (fresh-enough,
          isolates bloat) vs same-DB simplicity (one long scan can bloat the primary).
    - level: 'design'
      q: "You're choosing between a Postgres-style (heap history) and an InnoDB-style (undo history) engine for a write-heavy, high-contention OLTP service. Walk the decision."
      a: >-
          <b>A strong answer covers:</b> clarify the workload first — heavy updates on hot rows, short
          txns, many indexes &rarr; heap-history (Postgres-style): updates are cheap but every non-HOT
          update churns index entries and leaves dead tuples, so under heavy writes you pay in VACUUM
          I/O, bloat, and a wraparound/freeze obligation; you must tune autovacuum hard and never let
          a long txn pin the horizon &rarr; undo-history (InnoDB-style): in-place clustered updates
          and cheaper secondary indexes suit write-heavy workloads, but long read views grow the
          undo/history list and reads via secondary indexes pay a clustered+undo hop &rarr; for
          write-heavy high-contention OLTP the undo model often operates more smoothly,
          <em>provided</em> you keep read views short so purge keeps up &rarr; name the decisive
          trade-off: VACUUM/bloat discipline vs purge/history-list discipline &rarr; whichever you
          pick, the binding constraint is the same — bound transaction duration, because the oldest
          open txn sets the cleanup cost; instrument it before deciding.

        <h2>Two ways to store versions — heap vs undo</h2>
    </div>
    <p class="tagline">
        Same isolation guarantee, opposite on-disk layout. Everything below follows from this fork.
    </p>

    <figure>
        <img
            src="diagrams/dd-01-mvcc-storage.png"
            alt="Two MVCC storage models. PostgreSQL keeps every row version in the heap (old versions stay until VACUUM removes them). InnoDB updates the row in place and keeps the old version in an undo log, reconstructing past versions on demand."
        />
        <figcaption>
            <strong>Two MVCC storage models.</strong> PostgreSQL keeps every row version in the heap
            (old versions stay until VACUUM removes them). InnoDB updates the row in place and keeps
            the old version in an undo log, reconstructing past versions on demand.
        </figcaption>
    </figure>

    <div class="facts">
        <div class="fact">
            <b>Postgres — append to heap</b> An <code>UPDATE</code> does <em>not</em> overwrite: it
            inserts a brand-new tuple (<code>xmin</code>=current xid) and stamps the old tuple's{' '}
            <code>xmax</code>. Both versions sit in the heap. <code>DELETE</code> just sets{' '}
            <code>xmax</code>.
        </div>
        <div class="fact blue">
            <b>InnoDB — update in place + undo</b> The clustered row is modified in place; the prior
            version is written as an <em>undo record</em> reached via <code>DB_ROLL_PTR</code>. The
            table stays the latest; history lives in the undo log.
        </div>
        <div class="fact">
            <b>Postgres tuple header</b> <code>xmin</code> = xid that created it; <code>xmax</code> =
            xid that deleted/superseded it (0 = live). That's it — two ids per version.
        </div>
        <div class="fact blue">
            <b>InnoDB hidden fields</b> <code>DB_TRX_ID</code> = last txn to touch the row;{' '}
            <code>DB_ROLL_PTR</code> = pointer into the undo log. Old reads walk the undo chain
            backward.
        </div>
    </div>

    <div class="rule">
        <span>🗄️ Memory rule:</span> Postgres keeps history <span>in the table</span> (fast
        latest-row reads, but it bloats and needs VACUUM); InnoDB keeps history <span>in undo</span>{' '}
        (compact table, but old reads pay to walk undo and long readers grow the log).
    </div>

    <div class="myth">
        <div class="m">✗ "An UPDATE overwrites the row in place"</div>
        <div class="t">
            ✓ Postgres inserts a new tuple; InnoDB writes undo first — the old version survives either
            way
        </div>
    </div>

        </div>
        <div class="fact blue">
            <b>Repeatable Read / SI</b> One snapshot taken at first read, reused all txn → one
            consistent view.
        </div>
        <div class="fact">
            <b>InnoDB read view</b> Same logic: low-water id, high-water id, active set — applied{' '}
            <em>while walking undo</em> until <code>DB_TRX_ID</code> is visible.
        </div>
        <div class="fact blue">
            <b>Cost of the check</b> Per tuple, a commit-log (CLOG) lookup to learn "did that xid
            commit or abort?"
        </div>
    </div>

    <div class="rule">
        <span>👁️ Memory rule:</span> Visible ⇔ <span>xmin committed in the past</span> AND{' '}
        <span>xmax not committed in the past</span> — two header fields plus one commit-log lookup.
    </div>

    <div class="myth">
        <div class="m">✗ "A snapshot copies the rows it will read"</div>
        <div class="t">✓ It stores xmin/xmax/xip[ ] and judges each version on the fly</div>
    </div>

    <MemoryCheck items={mc2} />

</section>

{/* ================= LEVEL 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>VACUUM, purge &amp; the long-transaction bloat trap</h2>
    </div>
    <p class="tagline">
        Dead versions don't vanish on their own. One forgotten transaction can bloat a whole database
        — silently.
    </p>

    <figure>
        <img
            src="diagrams/dd-03-vacuum-bloat.png"
            alt="The long-transaction bloat trap. VACUUM can only remove versions older than the oldest running transaction's snapshot (the xmin horizon). One long-running or idle-in-transaction connection pins that horizon, so dead tuples pile up and the table bloats — unbounded, with no error."
        />
        <figcaption>
            <strong>The long-transaction bloat trap.</strong> VACUUM can only remove versions older
            than the oldest running transaction's snapshot — the <code>xmin</code> horizon. One
            idle-in-transaction connection pins that horizon, so dead tuples pile up unbounded, with
            no error.
        </figcaption>
    </figure>

    <div class="facts">
        <div class="fact">
            <b>When is a version removable?</b> Once invisible to <em>every</em> snapshot that could
            ever run again — its <code>xmax</code> committed before the oldest still-running txn.
        </div>
        <div class="fact blue">
            <b>The xmin horizon</b> The <code>xmin</code> of the oldest live txn. VACUUM cannot touch
            anything that died <em>after</em> it — across the whole DB.
        </div>
        <div class="fact">
            <b>VACUUM vs VACUUM FULL</b> Plain VACUUM (usually autovacuum) frees space <em>for reuse
            in the table</em>; the file doesn't shrink. Only <code>VACUUM FULL</code> rewrites &amp;
            shrinks, under an exclusive lock.
        </div>
        <div class="fact blue">
            <b>InnoDB equivalent</b> The <strong>purge</strong> thread reclaims undo + delete-marked
            rows; a long read view shows up as a growing <strong>history list length</strong>.
        </div>
    </div>

    <div class="rule">
        <span>🧹 Memory rule:</span> Your cleanup is only ever as current as your{' '}
        <span>oldest open transaction</span> — one <code>idle in transaction</code> connection bloats
        the entire database, with no error.
    </div>

    <div class="myth">
        <div class="m">✗ "Plain VACUUM returns disk to the OS"</div>
        <div class="t">✓ It marks space reusable inside the table; only VACUUM FULL shrinks the file</div>
    </div>

    <p class="tagline">
        <b>Defenses (operational, not magical):</b> cap age with{' '}
        <code>idle_in_transaction_session_timeout</code>; watch <code>pg_stat_activity</code> for
        ancient <code>xact_start</code> / <code>backend_xmin</code>; alert on <code>n_dead_tup</code>.
        InnoDB: alert on history list length.
    </p>

    <MemoryCheck items={mc3} />

</section>

{/* ================= LEVEL 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>The transaction-ID wraparound time bomb</h2>
    </div>
    <p class="tagline">
        PostgreSQL-specific, famous, and has taken down real companies. The fix is freezing.
    </p>

    <Fig3 />

    <div class="facts">
        <div class="fact">
            <b>The bomb</b> 32-bit xids (~4.2B), compared circularly. After ~2B newer xids, an old{' '}
            <code>xmin</code> looks future → the row vanishes. Silent, catastrophic data loss.
        </div>
        <div class="fact blue">
            <b>The defense</b> <strong>Freezing</strong>: VACUUM marks old tuples "committed &amp;
            visible to everyone — ignore <code>xmin</code>." A frozen tuple is wrap-immune.
        </div>
        <div class="fact">
            <b>Escalation</b> As the oldest unfrozen xid ages (<code>autovacuum_freeze_max_age</code>,
            default 200M), Postgres runs aggressive anti-wraparound autovacuums.
        </div>
        <div class="fact blue">
            <b>The hard stop</b> If freezing falls too far behind, Postgres <strong>refuses new xids
            and goes read-only</strong> rather than risk corruption — "DB stopped accepting writes."
        </div>
    </div>

    <div class="rule">
        <span>⏰ Memory rule:</span> Wraparound is prevented by <span>freezing</span> — marking old
        tuples visible-to-all so their soon-to-wrap xmin is ignored. Fall behind and the database goes
        read-only.
    </div>

    <div class="myth">
        <div class="m">✗ "Wraparound is fixed by resetting the counter to zero"</div>
        <div class="t">✓ It's prevented by freezing old tuples; nothing resets the global xid counter</div>
    </div>

    <p class="tagline">
        The same long transaction that bloats (§3) <em>also</em> holds back freezing — nudging you
        toward wraparound. The postmortems (Sentry, Mailchimp, others) all read the same:{' '}
        <em>autovacuum couldn't keep up.</em>
        <sup><a href="#s5" id="r5">5</a></sup>
    </p>

    <MemoryCheck items={mc4} />

</section>

{/* ================= LEVEL 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>Loose ends — indexes, SSI &amp; which model wins</h2>
    </div>
    <p class="tagline">Three things to know before you close the book.</p>

    <Fig4 />

    <div class="facts">
        <div class="fact">
            <b>Indexes carry no version</b> A PG index points at a heap tuple; visibility lives in the
            heap, so index-only scans consult the <strong>visibility map</strong>. InnoDB secondary
            entries are delete-marked on update and often force a drop to the clustered index + undo.
        </div>
        <div class="fact blue">
            <b>Indexed-column updates are pricey</b> Under MVCC, updating an indexed column writes new
            index entries in both engines — extra cost, extra bloat.
        </div>
        <div class="fact">
            <b>SSI is built on MVCC</b> PostgreSQL <code>SERIALIZABLE</code> (SSI) runs on ordinary
            snapshots and <em>additionally</em> tracks read/write dependencies, aborting one txn on the
            dangerous-structure pattern that causes write skew.
            <sup><a href="#s4" id="r4">4</a></sup>
        </div>
        <div class="fact blue">
            <b>Pick your discipline</b> Knowing your engine tells you what you owe it: VACUUM/freeze
            monitoring for Postgres, undo/history-list + purge monitoring for InnoDB.
        </div>
    </div>

    <div class="rule">
        <span>🧩 Memory rule:</span> MVCC is the substrate; <span>SSI is the safety layer over
        it</span>. Neither storage model wins — they trade, and a long transaction starves cleanup in
        both.
    </div>

    <div class="myth">
        <div class="m">✗ "Serializable replaces MVCC with locks"</div>
        <div class="t">
            ✓ Postgres SSI runs on MVCC snapshots + dependency tracking — same substrate, extra checks
        </div>
    </div>

    <MemoryCheck items={mc5} />

</section>

<h2 class="sec" id="cheat">
    ★ Cheat sheet — internal → what it means
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> Row = chain of versions · UPDATE = new version, never
        overwrite · Postgres = history in the heap · InnoDB = history in undo · Snapshot =
        xmin/xmax/xip[] judging each version · Visible = xmin past+committed AND xmax not-past · xmin
        horizon = oldest open txn = cleanup limit · VACUUM/purge = the janitor · Freezing = wraparound
        vaccine · SSI = safety layer over MVCC.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Term</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>
                <code>xmin</code> / <code>xmax</code>
            </td>
            <td>Postgres tuple header: creator xid / deleter xid (0 = live)</td>
        </tr>
        <tr>
            <td>
                <code>DB_TRX_ID</code> / <code>DB_ROLL_PTR</code>
            </td>
            <td>InnoDB hidden fields: last writer / pointer into undo</td>
        </tr>
        <tr>
            <td>Snapshot</td>
            <td>xmin (oldest running), xmax (next unassigned), xip[] (in progress)</td>
        </tr>
        <tr>
            <td>xmin horizon</td>
            <td>xmin of the oldest live txn — VACUUM's cleanup ceiling</td>
        </tr>
        <tr>
            <td>Bloat</td>
            <td>Dead tuples a pinned horizon won't let VACUUM reclaim</td>
        </tr>
        <tr>
            <td>Freezing</td>
            <td>Flag "visible to all; ignore xmin" → immune to wraparound</td>
        </tr>
        <tr>
            <td>History list length</td>
            <td>InnoDB's bloat signal: undo a long read view can't release</td>
        </tr>
        <tr>
            <td>Purge thread</td>
            <td>InnoDB's VACUUM: reclaims undo + delete-marked rows</td>
        </tr>
        <tr>
            <td>SSI</td>
            <td>Serializable on MVCC + read/write dependency tracking</td>
        </tr>
    </table>
    <h3>Three rules for the on-call chair</h3>
    <p style="margin-bottom:0">
        ① Cleanup is only as current as your <strong>oldest open transaction</strong> — hunt it in{' '}
        <code>pg_stat_activity</code> first. ② Plain VACUUM frees-for-reuse; <strong>only{' '}
        <code>VACUUM FULL</code> shrinks</strong> the file (exclusive lock). ③ Never disable
        autovacuum — <strong>"DB went read-only" is wraparound</strong>, and it's always "autovacuum
        couldn't keep up."
