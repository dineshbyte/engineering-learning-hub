# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Walk me through the design method you run on any blank-page prompt.'
      a: >-
          <b>Hit these points:</b> clarify functional + the four non-functionals (scale, latency,
          availability, consistency) &rarr; estimate QPS, storage, bandwidth &rarr; API for the hot
          operation &rarr; data model for the hot query &rarr; draw the high-level boxes &rarr;
          deep-dive the one bottleneck and name its trade-off &rarr; the system changes but the method
          does not; skipping a step (usually estimation) is the #1 failure mode &rarr; "I clarify
          before estimating because the read:write ratio falls out of scope."
    - level: 'core'
      q: '100M DAU at 10 reads/day each — size the read QPS and storage out loud.'
      a: >-
          <b>Hit these points:</b> 100M &times; 10 = 1B reads/day &rarr; &divide; 86,400 s &asymp; 12K
          avg QPS &rarr; &times;2&ndash;3 for peak &asymp; 35K peak QPS (numbers illustrative) &rarr; if
          those were 1 KB writes: 1B &times; 1 KB = 1 TB/day, &times;3 replicas &asymp; 1 PB/year &rarr;
          conclusion each number drives: 35K QPS is past one box &rarr; cache + replicas; 1 PB forces a
          sharded fleet &rarr; estimate reads and writes separately because the ratio picks the
          architecture.
    - level: 'core'
      q: 'What goes in the API and data model for a URL shortener, and why a KV store?'
      a: >-
          <b>Hit these points:</b> two endpoints &mdash; <code>POST /urls {longUrl}</code> &rarr; short
          code, and <code>GET /{code}</code> &rarr; 301/302 redirect &rarr; data model is pure
          key-value: <code>code (PK) &rarr; longUrl, createdAt</code> &rarr; access is single-key point
          lookups, no joins or range scans &rarr; so a KV engine beats a relational schema you'd never
          query relationally &rarr; the mapping is immutable, so cache invalidation is nearly free and
          TTLs can be long.
    - level: 'core'
      q: 'Name the five reusable levers and what each one buys you.'
      a: >-
          <b>Hit these points:</b> <b>cache</b> &rarr; faster reads + DB load shed (trade: staleness)
          &rarr; <b>replicate</b> &rarr; survive node loss + scale reads (trade: replication lag / write
          cost) &rarr; <b>partition / shard</b> &rarr; more storage + write throughput past one box
          (trade: cross-shard queries, rebalancing) &rarr; <b>queue</b> &rarr; absorb spikes + decouple
          producers from consumers (trade: async, eventual) &rarr; <b>tune consistency</b> &rarr; lower
          latency / higher availability by accepting staleness (trade: read-your-writes is no longer
          free) &rarr; every design is "which lever, on which box, at what cost."
    - level: 'senior'
      q: 'A teammate wants to size the database first and check the math later. Push back.'
      a: >-
          <b>Hit these points:</b> the estimate IS the design constraint, not a sanity check bolted on
          after &rarr; the QPS and storage arithmetic is what decides single box vs replicas vs a
          sharded fleet &rarr; design decisions differ by ~1000&times; (one machine vs a thousand), so
          being off by 2&times; never flips the answer &mdash; that's why two minutes of
          back-of-envelope is enough &rarr; without it you've committed to a shape arbitrarily and may
          discover at build time that storage alone forces sharding you didn't plan for &rarr; estimate
          reads and writes separately: 100:1 read-heavy screams cache + replicas, write-heavy screams
          LSM-trees and a log-first ingest path.
    - level: 'senior'
      q: "A rate limiter shares a Redis counter across N gateway replicas. What's the trap and the fix?"
      a: >-
          <b>Hit these points:</b> the lost-update race &mdash; naive <code>GET</code> &rarr; +1 &rarr;
          <code>SET</code> interleaves across replicas and silently drops increments, so the effective
          limit drifts above the target &rarr; fix: atomic <code>INCR</code>, which is a single
          uninterruptible step &rarr; for multi-step token-bucket logic (refill + decrement + TTL) wrap
          it in a Lua <code>EVAL</code> so it runs as one atomic unit &rarr; don't use per-replica local
          counters: N replicas fragment the count to an effective N&times;L &rarr; decide fail-open vs
          fail-closed when Redis is unreachable (usually fail-open: availability over precision, with a
          local fallback bucket).
    - level: 'senior'
      q: 'Your cache fronts the DB at a 95% hit ratio. Which two failure modes will page you?'
      a: >-
          <b>Hit these points:</b> first quantify the win &mdash; 100K rps at 95% hit leaves the DB
          serving ~5K rps, a 20&times; reduction &rarr; (1) <b>cache stampede</b>: a hot key expires and
          every concurrent request misses at once, stampeding the DB; fix with request coalescing
          (single-flight) + jittered/staggered TTLs + probabilistic early refresh &rarr; (2)
          <b>invalidation</b>: a cache is derived data with no idea when its source changed; TTL is
          blunt, explicit invalidation races, CDC from the DB change stream is the robust answer &rarr;
          bonus: a cold-cluster restart is a self-inflicted stampede across every key at once.
    - level: 'senior'
      q: 'When does fan-out-on-write break for a news feed, and how do you resolve it?'
      a: >-
          <b>Hit these points:</b> push (fan-out on write) is the default because reads dominate
          (~5:1), so a feed read becomes a cheap precomputed cache lookup &rarr; it breaks on the
          celebrity: one post by a millions-follower account = millions of feed writes, a fan-out
          explosion that floods the write path and is a hot, skewed partition &rarr; resolution is the
          hybrid: push for normal accounts, pull (merge their recent posts at read time) for accounts
          above a follower threshold &rarr; store post IDs not bodies (a viral post is stored once,
          referenced N times) in a capped Redis sorted set &rarr; run fan-out as an async, idempotent
          outbox/log job so a spike degrades freshness, not availability.
    - level: 'staff'
      q: 'Make the case that the estimate, not the feature list, should drive the architecture — then steelman the opposite.'
      a: >-
          <b>For:</b> decisions span ~1000&times; (one box vs a fleet) and fall directly out of the
          QPS/storage/bandwidth arithmetic; features tell you what to build, numbers tell you what shape
          survives; clarifying then estimating makes the rest of the design near-inevitable rather than
          improvised. <b>Steelman against:</b> early-stage products with unknown load shouldn't shard
          from day one &mdash; over-architecting for hypothetical scale is real cost and operational
          drag, and premature partitioning can be harder to undo than to add. <b>Synthesis:</b> let the
          estimate set the <em>shape</em> (stateless app tier, clean partition key chosen even if you
          run one shard), but defer the <em>cost</em> (actual fleet size) until load is real; design so
          scaling is "add replicas / split shards," never "re-architect."
    - level: 'staff'
      q: "You're handed a system that's slow at p99 but fine at the mean. How do you reason about it as the senior in the room?"
      a: >-
          <b>Hit these points:</b> the SLO is the tail, not the average &mdash; p99 can be 10&times; the
          mean under load, and at high fan-out a 1-in-100 slow call hits nearly every request, so a
          "fine average" hides a broken experience &rarr; attribute the tail before tuning: is it GC
          pauses, a cold cache, lock/queue contention, a hot/skewed partition, or a slow downstream
          dependency? &rarr; instrument with percentile histograms and per-dependency latency, not just
          means &rarr; common levers map to causes: stampede &rarr; coalescing + jittered TTLs; hot key
          &rarr; better partition key / replication of the hot item; tail amplification from fan-out
          &rarr; hedged requests or reducing fan-out width &rarr; name the trade-off of each fix and
          verify with a before/after percentile, not a vibe.
    - level: 'staff'
      q: 'An engineer proposes adding a cache to "make everything faster." Make the principal-level call.'
      a: >-
          <b>Hit these points:</b> a cache is not free speed &mdash; it's a bet that you can tolerate
          staleness in exchange for not touching the DB, and it adds the two hardest operational
          problems in the book (invalidation and stampede) &rarr; first ask whether the workload is
          actually read-heavy and re-readable; caching a write-heavy or low-hit-ratio path adds
          complexity for little load shed &rarr; decide the consistency need: is a 200 ms-stale read
          OK? If not, you've committed to write-through or a quorum path, which has its own cost &rarr;
          choose the strategy deliberately (cache-aside workhorse vs write-through vs write-back, where
          write-back can lose acknowledged data on crash) &rarr; require a hit-ratio target and an
          invalidation story (TTL vs CDC) up front &rarr; the principal move: cache the specific hot
          read path with a measured hit-ratio and a defined staleness budget, not "everything."
    - level: 'design'
      q: 'Design a news feed for 300M users with a sub-200 ms feed-read budget.'
      a: >-
          <b>A strong answer covers:</b> clarify &mdash; home timeline of followees' posts, ranked or
          reverse-chron, read-your-own-writes for the author, eventual for others &rarr; estimate
          &mdash; reads dominate (~5:1), so the read path is the SLO; size feed-read QPS and fan-out
          write amplification &rarr; data model &mdash; feed = per-user list of
          <code>(postId, authorId, createdAt)</code> in a capped Redis sorted set; posts stored once in
          a KV/blob store &rarr; core decision: aggregate at write time (push) or read time (pull)
          &rarr; default push so reads are a cheap precomputed lookup; the celebrity breaks pure push
          (fan-out explosion / hot partition), so go hybrid: pull for high-follower accounts, merged in
          at read time &rarr; run fan-out as an async idempotent outbox/log job so spikes degrade
          freshness not availability &rarr; cache the hot feeds; CDN media &rarr; failure modes:
          fan-out lag, hot partitions, the inactive-user wasted-work problem &rarr; trade-off: push pays
          at write time and wastes work on inactive users; pull pays at read time; the hybrid pays pull
          only where push would explode.
    - level: 'design'
      q: 'Design a distributed key-value store that stays writeable under node and network failure.'
      a: >-
          <b>A strong answer covers:</b> clarify &mdash; durability, target availability, and the
          consistency bar (do we accept eventual + conflict resolution?) &rarr; placement: a
          consistent-hashing ring with virtual nodes so a membership change moves only ~K/N keys and
          load stays even &rarr; replication: each key to the next N=3 distinct physical nodes clockwise
          (the preference list), skipping vnodes of the same host &rarr; leaderless, tunable quorums
          R + W &gt; N for strong-ish reads; W=1 for always-writeable (Dynamo's shopping cart) at the
          cost of more reconciliation &rarr; concurrent writes &rarr; vector clocks (or version vectors)
          return siblings to merge, not blind last-write-wins &rarr; per-node storage: an LSM-tree for
          write-heavy load &rarr; anti-entropy: Merkle-tree comparison + read-repair + gossip-based
          membership and failure detection &rarr; failure modes: hinted handoff for a briefly-down
          node, permanent-loss re-replication, and the partition behavior &rarr; trade-off: this is an
          AP design (CAP) &mdash; it chooses availability and accepts temporary inconsistency you
          resolve at read time; if the use case needs strong consistency, you'd move to
          single-leader-per-shard with synchronous replication instead.

        <div class="t">✓ Start by understanding what + at what scale, then commit to a shape</div>
    </div>
    <div class="rule">
        <span>🧭 Memory rule:</span> Clarify → Estimate → API + data → Draw → Deep-dive. The
        estimate, not the features, decides whether you shard from day one.
    </div>

<section class="lvl" id="l2">
    <div class="lvltop">
        <span class="num">2</span>
        <h2>Back-of-the-envelope estimation — the bridge to architecture</h2>
    </div>
    <p class="tagline">
        Two minutes of arithmetic decides "one box" vs "a thousand." Correct to within 10× is
        enough.
    </p>
    <Fig2 />
    <div class="rule">
        <span>🧮 The QPS formula:</span> avg QPS = (DAU × actions/day) ÷ 86,400, then peak ≈ avg ×
        2–3. There are 86,400 seconds in a day — that's the denominator of every estimate.
    </div>
    <figure>
        <img
            src="diagrams/02-estimation.png"
            alt="Back-of-the-envelope reference card. Left: the latency ladder from L1 cache to cross-region round-trip with bars that grow by orders of magnitude. Right: a worked QPS-and-storage estimate."
        />
        <figcaption>
            Worked example: 100M DAU × 10 reads = 1B/day ÷ 86,400 ≈ 12K avg QPS, ×3 ≈ 35K peak. 1B
            writes/day × 1 KB × 3 replicas ≈ 1 PB/year → unambiguously a sharded fleet.
        </figcaption>
    </figure>
    <div class="facts">
        <div class="fact">
            <b>The anchor number</b> A record is ~1 KB. A tweet, an order, a Kafka event — call it 1
            KB unless you know better. Billions of records → terabytes by inspection.
        </div>
        <div class="fact">
            <b>One multiply per resource</b> QPS, storage, bandwidth, memory each get one
            multiplication. Whichever blows past a single machine is your deep-dive.
        </div>
        <div class="fact blue">
            <b>Estimate reads + writes separately</b> 100:1 read-heavy screams cache + replicas;
            write-heavy screams LSM-trees + a log-first ingest path.
        </div>
        <div class="fact blue">
            <b>Tail, not average, is the SLO</b> p99 can be 10× the mean under load; at 100-way
            fan-out a 1-in-100 slow call hits nearly every request.
            <sup><a href="#s3" id="r3b">3</a></sup>
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "I'll size the database, then sanity-check the math later"</div>
        <div class="t">
            ✓ The estimate IS the design constraint — it makes the rest inevitable, not arbitrary
        </div>
    </div>

</section>

        <span class="num">4</span>
        <h2>Rate limiter — atomicity on the hot path</h2>
    </div>
    <p class="tagline">
        The smallest system in the book, and the one that forces an atomic counter on every single
        request.
    </p>
    <figure>
        <img
            src="diagrams/04-rate-limiter.png"
            alt="Left: the token-bucket algorithm. Right: a shared atomic counter in Redis. Token bucket allows a request when a token is present and returns 429 when the bucket is empty; the distributed counter shows many app servers issuing a single atomic INCR against one Redis key."
        />
        <figcaption>
            Token bucket allows a request when a token is present, returns 429 when empty. Many
            gateway replicas issue one atomic <code>INCR</code> against a single Redis key so the
            limit stays global.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Algorithm</th>
            <th>Memory / client</th>
            <th>Accuracy</th>
            <th>Notes</th>
        </tr>
        <tr>
            <td>Fixed window</td>
            <td>1 integer</td>
            <td>Low (2L at edges)</td>
            <td>Cheapest</td>
        </tr>
        <tr>
            <td>Sliding-window counter</td>
            <td>2 integers</td>
            <td>High (~0.003% err)</td>
            <td>Best general default</td>
        </tr>
        <tr>
            <td>Sliding-window log</td>
            <td>O(L) timestamps</td>
            <td>Exact</td>
            <td>Costly at high L</td>
        </tr>
        <tr>
            <td>Token bucket</td>
            <td>2 numbers</td>
            <td>High, allows bursts</td>
            <td>Production default</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>The lost-update trap</b> Naive <code>GET</code>→+1→<code>SET</code> lets two replicas
            overwrite each other. Redis <code>INCR</code> is atomic by construction → no lost
            update.
        </div>
        <div class="fact">
            <b>Multi-step? Use a Lua script</b> Refill + decrement + TTL wrapped in{' '}
            <code>EVAL</code> runs as one atomic unit — a transaction collapsed to one instruction.
        </div>
        <div class="fact blue">
            <b>Where it lives</b> Standard answer: the API gateway — one chokepoint sees all traffic
            before fan-out. Add per-service limits for expensive endpoints.
        </div>
        <div class="fact blue">
            <b>Fail-open vs fail-closed</b> If shared Redis is unreachable, usually fail-open
            (availability over precision) with a per-replica local fallback.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Local in-memory counters are fine across replicas"</div>
        <div class="t">
            ✓ N replicas fragment the count → effective limit becomes N × L; you need one shared
            counter
        </div>
    </div>
    <div class="rule">
        <span>🪣 Memory rule:</span> Token bucket allows bursts up to B at average rate r;
        correctness across replicas needs an atomic shared counter (Redis INCR / a Lua script).
    </div>

        time or read time.

        />
        <figcaption>
            Push (left): copy a post into every follower's precomputed feed → cheap reads, expensive
            writes. Pull (right): merge followees' posts at read time → cheap writes, expensive
            reads.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th></th>
            <th>Fan-out on write (push)</th>
            <th>Fan-out on read (pull)</th>
        </tr>
        <tr>
            <td>Cost of a post</td>
            <td>high — write to every follower</td>
            <td>low — one insert</td>
        </tr>
        <tr>
            <td>Cost of a feed read</td>
            <td>low — read precomputed list</td>
            <td>high — merge at read time</td>
        </tr>
        <tr>
            <td>Wasted work</td>
            <td>feeds for inactive users</td>

        </div>
        <div class="fact">

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Distributed cache — trade freshness for latency &amp; load</h2>
    </div>
    <p class="tagline">
        Every cache decision is a bet on how stale a copy you can tolerate in exchange for not
        touching the DB.
    </p>
    <figure>
        <img
            src="diagrams/06-distributed-cache.png"
            alt="Cache-aside read flow and the consistent-hashing ring. The left panel numbers the miss path that populates the cache; the right shows keys owned by the next node clockwise and how adding a node remaps only one arc."
        />
        <figcaption>
            Cache-aside (left): on a miss, read the DB, populate, return. Consistent hashing
            (right): a key is owned by the next node clockwise; adding a node remaps only one arc
            (~K/N keys), not all of them.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Strategy</th>
            <th>Consistency</th>
            <th>Write latency</th>
            <th>Failure risk</th>
        </tr>
        <tr>
            <td>Cache-aside</td>
            <td>Eventual; stale until invalidated</td>
            <td>DB only</td>
            <td>Stale reads on a race</td>
        </tr>
        <tr>
            <td>Write-through</td>
            <td>Strong (cache tracks DB)</td>
            <td>Cache + DB</td>
            <td>Wasted writes</td>
        </tr>
        <tr>
            <td>Write-back</td>
            <td>Weak; cache leads DB</td>
            <td>Cache only</td>
            <td>Data loss on crash</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>The load win</b> 100K rps at 95% hit ratio leaves the DB serving 5K rps — a 20×
            reduction in the hardest thing to scale.
        </div>
        <div class="fact">
            <b>Why mod-N fails</b> Change N (a deploy, a death) and almost every key remaps → the
            whole cache cold-starts and hammers the DB. Consistent hashing + virtual nodes fix it.
        </div>
        <div class="fact blue">
            <b>Cache stampede</b> A hot key expires → every concurrent request misses at once. Fix
            with request coalescing (single-flight) + jittered/staggered TTLs.
        </div>
        <div class="fact blue">
            <b>Invalidation is the hard 20%</b> A cache is derived data with no idea when its source
            changed. TTL is blunt; explicit invalidation is fragile; CDC from the DB's change stream
            is the robust answer.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Write-back is just a faster write-through"</div>
        <div class="t">
            ✓ Write-back acks from cache memory → a crash in the flush window loses acknowledged
            data
        </div>
    </div>
    <div class="rule">
        <span>🗄️ Memory rule:</span> Cache-aside is the workhorse; consistent hashing keeps reshards
        cheap; the two things that page you are stampede and invalidation.
    </div>

</section>

    <div class="lvltop">
        <span class="num">7</span>
        <h2>Message queue — the partitioned, replicated log</h2>
    </div>
    <p class="tagline">
        Decouple producers from consumers. The core choice: a log that doesn't delete on read.
    </p>
    <figure>
        <img
            src="diagrams/07-message-queue.png"
            alt="A Kafka-like message queue: producers append to a partitioned topic, each partition replicated leader-to-followers, consumed by a consumer group tracking offsets. Read it as Part 4's log plus Part 1's replication."
        />
        <figcaption>
            Producers append to a partitioned topic; each partition is replicated leader→followers;
            a consumer group tracks offsets. Estimate: 1M msg/s × 1 KB × 3 days × RF 3 ≈ 780 TB → a
            disk-resident system.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Setting</th>
            <th>Durability</th>
            <th>Latency</th>
            <th>Survives</th>
        </tr>
        <tr>
            <td>
                <code>acks=0</code>
            </td>
            <td>none</td>
            <td>lowest</td>
            <td>nothing</td>
        </tr>
        <tr>
            <td>
                <code>acks=1</code> (leader)
            </td>
            <td>leader's disk</td>
            <td>low</td>
            <td>follower loss</td>
        </tr>
        <tr>
            <td>
                <code>acks=all</code> (full ISR)
            </td>
            <td>whole ISR</td>
            <td>higher</td>
            <td>leader loss (2 of 3)</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Log &gt; delete-on-read</b> Reading doesn't delete — the consumer remembers its
            offset. Buys replayability, multiple independent consumers, cheap append-only
            durability.
        </div>
        <div class="fact">
            <b>Partition = parallelism + ordering</b> Order holds only within a partition; route by{' '}
            <code>hash(key) mod N</code> so all events for one key stay ordered. More partitions =
            more throughput, weaker global order.
        </div>
        <div class="fact blue">
            <b>acks=all is R+W&gt;N in disguise</b> Leader + ISR; an acked write survives losing any
            2 of 3 brokers. A new leader is elected from the ISR on failure.
        </div>
        <div class="fact blue">
            <b>Backpressure = lag</b> A slow consumer just lags (tail offset − consumer offset); the
            buffer is retention itself. Monitor consumer lag — the #1 operational signal.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "acks=all always means 3 copies"</div>
        <div class="t">
            ✓ A silently-shrunk ISR gives acks=1 durability; set <code>min.insync.replicas=2</code>{' '}
            to reject under-replicated writes
        </div>
    </div>
    <div class="rule">
        <span>📨 Memory rule:</span> A partitioned append-only log + ISR replication; offset-commit
        timing picks at-most/at-least-once, and exactly-once = delivery + idempotent processing.
    </div>

            A key hashes to a ring position and is replicated on the next N=3 distinct nodes
            clockwise (its preference list). A W=2 write and an R=2 read overlap because R+W&gt;N.
            <sup><a href="#s4" id="r4">4</a></sup>
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Setting</th>
            <th>R</th>
            <th>W</th>
            <th>Behaviour</th>
        </tr>
        <tr>
            <td>Balanced</td>
            <td>2</td>
            <td>2</td>
            <td>One node down tolerated; strong-ish reads</td>
        </tr>
        <tr>
            <td>Fast write</td>
            <td>1</td>
            <td>3</td>
            <td>Cheap reads, every write hits all replicas</td>
        </tr>
        <tr>
            <td>Always writeable</td>
            <td>3</td>
            <td>1</td>
            <td>
                <code>put</code> survives 2 dead nodes; reads reconcile
            </td>
        </tr>

        <div class="fact">
            <b>R + W &gt; N</b> The W nodes that took the latest write and the R nodes you read must
            share ≥1 node → the read sees the most recent acknowledged write.
        </div>
        <div class="fact">
            <b>Virtual nodes</b> Each machine claims many small ring positions → load smooths across
            heterogeneous boxes; a dead node's keys spread across many successors, not one.
        </div>
        <div class="fact blue">
            <b>Concurrent writes → vector clocks</b> Track causality, not wall-clock. If neither
            clock descends from the other, the writes are concurrent — return both siblings, let the
            app merge (the cart unions items).
        </div>
        <div class="fact blue">
            <b>LSM per node + gossip</b> Write-heavy → LSM-tree (sequential appends), not a B-tree.
            Merkle-tree anti-entropy repairs drift; gossip spreads membership in O(log N) with no
            coordinator.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "Last-write-wins by timestamp is safe"</div>
        <div class="t">
            ✓ Clocks drift (no global clock) → LWW silently drops an update; vector clocks detect
            the conflict
        </div>
    </div>
    <div class="rule">
        <span>💍 Memory rule:</span> The ring places keys + nodes; the preference list replicates;
        R+W&gt;N tunes consistency; vector clocks catch conflicts; LSM persists each node.
    </div>

        <h2>Putting it together — the universal checklist &amp; five levers</h2>

            The checklist is a <em>gate</em>, not a menu: Clarify → Estimate → API → Data model →
            High-level design → Deep-dive the bottleneck → State the trade-offs. The five-lever
            palette runs down the right.
        </figcaption>
    </figure>
    <table>
        <tr>
            <th>Lever</th>
            <th>What it buys</th>
            <th>Reach for it when…</th>
        </tr>
        <tr>
            <td>Cache</td>
            <td>Read latency, fewer DB hits</td>
            <td>read path is hot</td>
        </tr>
        <tr>
            <td>Replicate</td>
            <td>Read throughput, availability</td>
            <td>survive a node loss / scale reads</td>
        </tr>
        <tr>
            <td>Partition / shard</td>
            <td>Write throughput, storage past one node</td>
            <td>writes or data exceed one box</td>
        </tr>
        <tr>
            <td>Queue (async)</td>
            <td>Decouple, absorb spikes</td>
            <td>survive a traffic spike</td>
        </tr>
        <tr>
            <td>Tune consistency</td>
            <td>Trade freshness for speed/availability</td>
            <td>staleness is tolerable</td>
        </tr>
    </table>
    <div class="facts">
        <div class="fact">
            <b>Senior signal</b> Not picking the "right" answer — naming the alternative you
            rejected and the condition that would flip the call ("…but if this were account
            balances, I'd want sync replication").
        </div>
        <div class="fact">
            <b>Immutable data = free caching</b> Pastebin's pastes never change, so the hard half of
            caching (invalidation) simply disappears. Spot immutability and exploit it.
        </div>
        <div class="fact blue">
            <b>Most "scale" is read-scale</b> Cache + read replicas solve a huge fraction of
            prompts. Write-scale (sharding, outbox, Kafka) is the harder, rarer case.
        </div>
        <div class="fact blue">
            <b>The single-box baseline</b> Before sharding, ask whether one big NVMe node + a read
            replica would do. Many "web-scale" problems fit on one box.
        </div>
    </div>
    <div class="myth">
        <div class="m">✗ "The pipeline delivers each event exactly once"</div>
        <div class="t">
            ✓ The wire gives at-least-once; effectively-once = at-least-once + an idempotency key
        </div>
