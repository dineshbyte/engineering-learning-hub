# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is a Bloom filter, in one sentence, and what does it store?'
      a: >-
          <b>Hit these points:</b> a space-efficient probabilistic set-membership structure &rarr; it
          stores <em>no keys</em>, just one <code>m</code>-bit array &rarr; <code>add(x)</code> sets
          the <code>k</code> bits that <code>x</code> hashes to; <code>contains(x)</code> reads those
          <code>k</code> bits &rarr; the whole point is the asymmetry: it answers "definitely not
          here" or "probably here," never the reverse &rarr; you trade exactness for a tiny,
          controllable footprint.
    - level: 'core'
      q: 'Which error can a standard Bloom filter make — and which can it never make?'
      a: >-
          <b>Hit these points:</b> it can return a <b>false positive</b> — "probably present" for
          something never added &rarr; it can <em>never</em> return a false negative — a "not present"
          is certain &rarr; reason: a "no" requires at least one checked bit to be 0, which only
          happens if the item was never inserted &rarr; "all k bits set" can be a coincidence from
          <em>other</em> items, so "present" is only a probability &rarr; this is why a "no" is always
          safe to act on.
    - level: 'core'
      q: 'Walk through what add and contains actually do at the bit level.'
      a: >-
          <b>Hit these points:</b> <code>add(x)</code>: run <code>k</code> independent hashes of
          <code>x</code>, take each mod <code>m</code>, set those <code>k</code> bits to 1 (some may
          already be 1) &rarr; <code>contains(x)</code>: hash the same way, read those <code>k</code>
          bits &rarr; if <em>any</em> is 0 &rarr; "definitely absent"; if <em>all</em> are 1 &rarr;
          "probably present" &rarr; no comparison against stored keys ever happens — there are none
          &rarr; cost is <code>k</code> hashes and <code>k</code> bit-probes, independent of how many
          items are in the set.
    - level: 'core'
      q: "Why can't a standard Bloom filter support delete?"
      a: >-
          <b>Hit these points:</b> bits are <em>shared</em> across items &rarr; clearing a bit to
          "remove" <code>x</code> might un-set a bit that a still-present key <code>y</code> also
          relies on &rarr; that would introduce false negatives, which breaks the one guarantee the
          structure offers &rarr; so deletion is simply unsupported in the standard form &rarr; if you
          need deletes, reach for a counting Bloom filter (bits &rarr; small counters) or a cuckoo
          filter, or periodically rebuild from the live set.
    - level: 'senior'
      q: 'Where would you put a Bloom filter in a read-heavy database, and what does it buy you?'
      a: >-
          <b>Hit these points:</b> put a small per-segment filter in front of the on-disk lookup — in
          an LSM store like RocksDB or Cassandra, one per SSTable &rarr; on a read, check the filter
          first: a "not present" lets you <em>skip that disk/SSD read entirely</em> &rarr; because
          there are no false negatives, you never skip a key that's actually there &rarr; it cuts read
          amplification for absent keys, at the cost of a few bits of RAM per key and the occasional
          wasted lookup on a false positive &rarr; size for ~1% FP (~10 bits/key, illustrative) and
          rebuild/partition if it saturates as segments grow.
    - level: 'senior'
      q: '"Probably present" is useless — why bother if you still have to verify?'
      a: >-
          <b>Hit these points:</b> the value isn't in the "yes," it's in the <em>certain "no"</em>
          &rarr; in workloads dominated by absent-key lookups, the filter eliminates the expensive
          path for most queries with a cheap RAM check &rarr; only the small fraction that come back
          "maybe" pay the real lookup, and even there the FP rate is bounded by design &rarr; so you
          swap "every query hits disk/DB" for "only true hits plus a few-percent false positives hit
          disk/DB" &rarr; the win scales with how often the answer is genuinely "not here."
    - level: 'senior'
      q: "What is the optimal hash count, and what's the catch with using too many hashes?"
      a: >-
          <b>Hit these points:</b> optimal <code>k = (m/n)&middot;ln 2</code>, which minimizes the FP
          rate for a given array size &rarr; at that <code>k</code> the FP rate is about
          <code>(0.5)<sup>k</sup></code> &rarr; intuition: too few hashes means too few independent
          bits checked; too many hashes fills the array faster, so each query trips over more set bits
          &rarr; there's a sweet spot — past it, adding hashes <em>raises</em> the FP rate <em>and</em>
          costs more CPU per op &rarr; also: real hashes must be well-mixed and effectively
          independent, or you fall short of the theoretical rate (double-hashing is the common cheap
          trick).
    - level: 'senior'
      q: 'The textbook FP formula — what is it, and why call it an approximation, not a law?'
      a: >-
          <b>Hit these points:</b> FP rate &asymp; <code>(1&minus;e<sup>&minus;kn/m</sup>)<sup>k</sup></code>
          &rarr; it assumes bit-set events are independent, which isn't strictly true — bits set by
          different items are correlated &rarr; so it's an excellent estimate, slightly optimistic,
          not an exact identity (NIST has a more precise analysis) &rarr; saying this out loud signals
          you understand the model's assumptions, not just the formula &rarr; practically it's
          accurate enough to size with, but don't treat the last decimal as gospel, and always add
          headroom for hash quality and growth.
    - level: 'staff'
      q: 'Make the case for a Bloom filter over just caching, then steelman the opposite.'
      a: >-
          <b>For:</b> a Bloom filter answers "is it absent?" in a few bits/key with a certain "no," so
          it kills the expensive path for misses at a fraction of a cache's memory; it doesn't store
          values, so it stays tiny even for billions of keys. <b>Steelman against:</b> if your
          workload is dominated by <em>hits</em>, the filter says "maybe" almost every time and buys
          nothing; a cache that actually returns values would serve those hits. <b>Synthesis:</b>
          Bloom for negative-lookup-heavy workloads (LSM reads, dedup, "have I seen this?"); a real
          cache when repeated hits dominate — they're complementary, often layered (filter to skip
          misses, cache to serve hits).
    - level: 'staff'
      q: >-
          A teammate proposes a Bloom filter as the source of truth for "has this user already claimed
          the coupon?" Where does this break?
      a: >-
          <b>Hit these points:</b> a false positive here means a legitimate user is wrongly told
          "already claimed" — a correctness/UX defect, not a perf nuisance &rarr; the filter is only
          safe as a <em>negative</em> filter: "no" &rarr; certainly not claimed, fast path; "maybe"
          &rarr; must verify against the authoritative store &rarr; never let "maybe" become the final
          answer for anything user-visible or money-touching &rarr; also flag: no deletes (can't
          un-claim), and saturation drives the FP rate up silently over time &rarr; the principal
          move: filter in front of an exact store, sized so the verify path is cheap, with a rebuild
          plan — not the filter <em>as</em> the store.
    - level: 'staff'
      q: 'When would you NOT use a Bloom filter at all?'
      a: >-
          <b>Hit these points:</b> when false positives are unacceptable and you can't afford a verify
          step — use an exact set/index &rarr; when you must <em>enumerate</em> or return the items —
          the filter stores no keys, so it can't list anything &rarr; when the set is tiny (a few
          thousand) — a plain hash set is smaller, exact, and supports deletes, so there's no win
          &rarr; when the dominant query is a <em>hit</em> rather than a miss — "maybe" every time buys
          nothing &rarr; the staff framing: a Bloom filter earns its keep only when negative answers
          are common and a cheap, certain "no" removes real work.
    - level: 'design'
      q: 'Design a "have we already crawled this URL?" check for a web crawler seeing billions of URLs.'
      a: >-
          <b>A strong answer covers:</b> the workload is miss-heavy early and the cost of re-crawling
          is real, so a Bloom filter as a <em>cheap negative filter</em> fits &rarr; "no" &rarr;
          definitely unseen, schedule the crawl; "maybe" &rarr; check the authoritative
          seen-set/store before crawling &rarr; size from a tolerable FP rate (a false positive just
          skips a URL — acceptable if rare; numbers illustrative) and from the expected <code>n</code>
          &rarr; <code>n</code> grows unboundedly, so use a <em>scalable</em> Bloom (chained layers) or
          shard filters by URL hash so the FP rate stays capped &rarr; no deletes needed (URLs don't
          un-crawl), so standard/scalable beats counting &rarr; instrument bits-set fraction and FP
          rate to trigger a new layer before saturation &rarr; name the trade-off: a false positive
          silently drops a page vs the memory of an exact set of billions of URLs.
    - level: 'design'
      q: 'Design a "weak / breached password" rejection check at signup using a Bloom filter.'
      a: >-
          <b>A strong answer covers:</b> goal — reject any password in a huge known-breached list
          without shipping or querying the full list per request &rarr; load the breached set into a
          Bloom filter: "maybe present" &rarr; reject (ask for another password); "definitely absent"
          &rarr; allow &rarr; here the asymmetry flips in your favor: a false positive just makes a
          user pick a different password (mildly annoying, safe); a false <em>negative</em> would let
          a breached password through — and Bloom has none &rarr; size for a low FP rate so you rarely
          over-reject (illustrative), from the known <code>n</code> of the breach corpus &rarr; the
          set is static and rebuilt on corpus updates, so no deletes and a simple rebuild pipeline
          &rarr; ship the filter to the edge/client for latency, since it stores no actual passwords
          &rarr; name the trade-off: occasional over-rejection (false positive) vs the privacy/size
          cost of querying or shipping the raw list.

# —— design-round rubric (rendered by <Interview> above the design questions) ——
rubricIntro: >-
    <b>Design-round framework</b> — drive any "use a Bloom filter here" prompt through these, out
    loud:
rubric:
    - 'Clarify the workload: how often is the answer "absent"? Hits vs misses ratio decides if a filter helps at all.'
    - 'Pick the guarantee you need: a certain "no" (Bloom fits) vs an exact yes/no (needs a real index).'
    - 'Size it: choose target FP <code>p</code>, count <code>n</code>, derive <code>m = &minus;(n&middot;ln p)/(ln 2)<sup>2</sup></code> and <code>k = log&#8322;(1/p)</code>.'
    - 'Place it: in front of the expensive/authoritative path; "maybe" always falls through to verify.'
    - 'Plan for growth: rebuild/shard/scalable variant before saturation pushes the FP rate up.'
    - 'Handle deletes/churn: counting or cuckoo variant, or periodic rebuild from the live set.'
    - 'Observe it: track measured FP rate, bits-set fraction, and verify-path hit rate, not just "it''s there."'

## Rehearse out loud

<Rehearse
    promptHtml={`Interviewer: <strong>"Where would you put a Bloom filter in a read-heavy database, and what does it buy you?"</strong> ~60 seconds, from memory &#8212; then reveal and compare.`}

        Put a small per-segment Bloom filter in front of the on-disk lookup &#8212; in an LSM-tree

        wasted lookup on a false positive. I'd size it for ~1% FP (about 10 bits/key) and revisit if
        the filter saturates as segments grow &#8212; at which point I'd rebuild or partition rather
        than let the FP rate climb.
    </p>
