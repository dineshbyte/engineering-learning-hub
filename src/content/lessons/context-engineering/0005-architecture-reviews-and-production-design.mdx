interview:
    - level: 'core'
      q: 'Name each product&apos;s context strategy: Cursor, Claude Code, ChatGPT, GitHub Copilot.'
      a: >-
          <b>Hit these points:</b> <b>Cursor</b> = pre-indexed embeddings of the repo + semantic
          retrieval (plus open files) &rarr; <b>Claude Code</b> = agentic read-on-demand via tools
          (grep / glob / read), follows the dependency graph live, no persistent index &rarr;
          <b>ChatGPT</b> = conversation history + optional uploads / connectors / memory &rarr;
          <b>GitHub Copilot</b> = open files + nearby code, with newer repo/workspace indexing for
          chat &rarr; the frame: each is a <em>context strategy with a model attached</em>, and the
          strategy — not the model — is what you&apos;re naming.
    - level: 'core'
      q: 'What is the pre-indexed &harr; read-on-demand spectrum, and where do those four products sit on it?'
      a: >-
          <b>Hit these points:</b> the axis is <em>when</em> retrieval happens — precompute an index
          ahead of time vs search the live source per task &rarr; <b>pre-indexed (left)</b>: Cursor
          builds embeddings up front so retrieval is one fast lookup &rarr; <b>read-on-demand
          (right)</b>: Claude Code searches the live tree each task, always fresh &rarr; Copilot is
          local-window-plus-index (left-of-centre), ChatGPT is history/connectors which is
          "none-to-shallow" grounding &rarr; placing a product on the axis is the reviewer&apos;s first
          move because position predicts the failure mode.
    - level: 'core'
      q: 'Match each production agent to the retrieval its data type demands: billing, support, engineering, research.'
      a: >-
          <b>Hit these points:</b> <b>billing</b> = exact/transactional &rarr; deterministic
          structured query (SQL keyed by <code>account_id</code>), never embeddings &rarr;
          <b>support</b> = fuzzy natural-language KB + customer history &rarr; hybrid + rerank, cite
          sources &rarr; <b>engineering</b> = structured code (imports, symbols, tests) &rarr; hybrid +
          structural retrieval or read-on-demand, run tests as a grounding signal &rarr; <b>research</b>
          = volatile, multi-source &rarr; fan-out web/doc retrieval, compress intermediate findings,
          keep provenance per claim &rarr; the rule underneath: match the retrieval mechanism to the
          data type.
    - level: 'core'
      q: 'Define agent, runtime, and RAG as you&apos;d use them in a production-design review.'
      a: >-
          <b>Hit these points:</b> an <b>agent</b> is an LLM that uses tools to act, not just generate
          text &rarr; the <b>runtime</b> is the orchestration layer around it — it wires tools,
          retrieval, memory, and the loop that feeds context in and reads results back &rarr; <b>RAG</b>
          is the pattern retrieve &rarr; assemble &rarr; LLM &rarr; answer, so the answer is grounded in
          fetched facts rather than the model&apos;s parametric memory &rarr; in a review these let you
          separate "what does the model see" (RAG/retrieve) from "what can it do" (agent/tools) from
          "who assembles it" (runtime).
    - level: 'senior'
      q: 'Cursor and Claude Code use opposite strategies. Name each one&apos;s signature failure mode and why position on the spectrum causes it.'
      a: >-
          <b>Hit these points:</b> <b>Cursor (pre-indexed)</b> fails by retrieving a topically-similar
          but <em>wrong</em> file — embeddings match meaning, not exact identifiers — and by serving a
          stale index after code changes &rarr; that&apos;s the structural tax of precompute: the index
          drifts from reality the moment the repo moves &rarr; <b>Claude Code (read-on-demand)</b>
          fails by missing a file it never thought to grep, and pays more tool round-trips &rarr;
          latency &amp; token cost &rarr; that&apos;s the tax of searching live: freshness is bought
          with hops and a dependence on a good search plan &rarr; neither is "better"; the review names
          <em>which</em> failure your use-case can&apos;t tolerate.
    - level: 'senior'
      q: 'Why is semantic embedding search the wrong retrieval for a billing agent, and what does the right design cost you?'
      a: >-
          <b>Hit these points:</b> embeddings return the <em>most-similar</em> value, not the
          <em>correct</em> one — for a charge that means a confidently-wrong amount with no audit trail
          &rarr; financial data is exact, transactional, and disputable, so it needs a deterministic
          structured query keyed by <code>account_id</code>/invoice id &rarr; the design rule: the
          agent may quote only values it pulled, never synthesize a number &rarr; the cost you accept is
          less "magic" — you can&apos;t fuzzily match a vaguely-worded question to a row, so you invest
          in intent parsing and exact keys &rarr; and you must log the assembled context per call, which
          billing needs anyway for audit and dispute resolution.
    - level: 'senior'
      q: 'A research agent fans out across many sources. Which failure modes does that shape invite, and how do you defend each?'
      a: >-
          <b>Hit these points:</b> <em>conflicting</em> — sources disagree and the model silently picks
          one &rarr; dedupe, prefer authoritative sources, surface the conflict rather than hide it
          &rarr; <em>outdated</em> — fetch live for volatile facts, don&apos;t lean on a stale cache
          &rarr; <em>excessive</em> — fan-out blows the budget and buries signal ("lost in the middle")
          &rarr; compress intermediate findings before assembly &rarr; cost: fan-out multiplies calls,
          so cap breadth and cache repeated lookups &rarr; reliability: adversarial verification of
          claims &rarr; observability: source provenance per claim so any sentence is traceable to
          where it came from.
    - level: 'senior'
      q: '"We&apos;ll add observability later — ship the agent first." Push back like a reviewer.'
      a: >-
          <b>Hit these points:</b> without logging the assembled context per call, a wrong answer is
          undebuggable — there&apos;s no stack trace pointing at "wrong document retrieved" &rarr; you
          can&apos;t tell a retrieval bug (wrong context in) from a reasoning bug (right context, bad
          answer), so you can&apos;t decide whether to fix retrieval or the model &rarr; for billing
          it&apos;s not optional: audit and dispute resolution legally require the trail &rarr;
          observability is a design property of the retrieve/assemble stages, not a follow-up sprint
          &rarr; minimum bar: log what was retrieved and assembled per call, plus retrieval hit-rate,
          before launch.
    - level: 'staff'
      q: 'You&apos;re reviewing an unfamiliar vendor "AI assistant" with 20 minutes. What&apos;s your line of questioning, and what would make you walk away?'
      a: >-
          <b>Hit these points:</b> first question is always "what&apos;s the context strategy —
          pre-indexed embeddings, read-on-demand tools, or just history?" — that one answer predicts
          strengths, cost shape, and failure mode &rarr; pre-indexed &rarr; probe index
          freshness/invalidation and exact-identifier recall &rarr; read-on-demand &rarr; probe
          round-trip latency and search-coverage gaps &rarr; history/none &rarr; ask what grounds it in
          <em>our</em> live systems &rarr; then: is retrieval hybrid + reranked to cover the embedding
          blind spot, and is retrieval <em>evaluated</em> (recall@k) not just answers? &rarr; walk away
          if there&apos;s no observability of assembled context and no answer on freshness — that&apos;s
          an undebuggable, stale-by-default system &rarr; frame throughout: I&apos;m comparing retrieval
          architectures, not models.
    - level: 'staff'
      q: 'Build vs buy: a VP asks whether to adopt a vendor assistant or build our own retrieval. Make the principal-level call.'
      a: >-
          <b>Hit these points:</b> decide by where the differentiation lives — buy when the data is
          generic and the vendor&apos;s context strategy already fits the use-case; build when our edge
          <em>is</em> the retrieval over proprietary data the vendor can&apos;t reach &rarr; the model
          is a commodity any competitor can rent; the retrieval pipeline over our live systems is the
          moat &rarr; the pivot question: does off-the-shelf grounding actually reach our systems, or
          does it hallucinate without connectors? &rarr; sequence it: instrument the failing cases
          first — if the gap is retrieval coverage, no vendor model upgrade closes it &rarr; hybrid
          stance is common: buy the model/runtime, build the retrieval and evaluation that compound
          across every feature.
    - level: 'staff'
      q: 'Across billing, support, engineering, and research agents, where does retrieval investment actually pay off — and where would you NOT over-invest?'
      a: >-
          <b>Hit these points:</b> investment compounds where wrong answers carry the highest
          business/compliance risk and where the data is yours — billing (audit/dispute exposure) and
          engineering (proprietary code) reward deep, exact retrieval &rarr; support rewards hybrid +
          rerank plus caching hot articles, because volume and deflection make small accuracy gains pay
          back fast &rarr; research rewards verification and provenance more than index sophistication,
          since freshness beats precompute on volatile sources &rarr; where <em>not</em> to
          over-invest: building a bespoke embedding stack for generic data a vendor already serves, or
          chasing a model upgrade (a flat tax on every call) to paper over a retrieval gap &rarr; the
          staff move: instrument, attribute failures by class, and put the build effort where it
          compounds across features rather than where it&apos;s merely fashionable.
    - level: 'design'
      q: 'Design the context system for a billing assistant that quotes exact charges and also answers billing-policy questions.'
      a: >-
          <b>A strong answer covers:</b> spot two data types in one product &rarr; <b>charges</b> are
          exact/transactional/auditable &rarr; deterministic SQL keyed by <code>account_id</code>/invoice
          id, never embeddings, because a semantic match returns the most-similar amount — a
          confidently-wrong number &rarr; <b>policy</b> is fuzzy natural-language text &rarr; hybrid
          retrieval + rerank over the help-center KB, with citations &rarr; one agent, two retrieval
          mechanisms matched to two data types &rarr; reliability rule: figures are quoted from a pulled
          record, never synthesized &rarr; guardrails: PII redaction and access control at the retrieve
          stage, injection defense on KB content &rarr; observability: log every assembled context +
          tool call for audit and dispute resolution &rarr; name the failure modes the naive "vector DB
          for everything" design invites: wrong (similar-not-correct amounts) and outdated (stale
          figures) &rarr; close with freshness: invalidate on write since balances and policies change.
    - level: 'design'
      q: 'Design the context system for a research agent that fans out across many sources and must produce a cited answer.'
      a: >-
          <b>A strong answer covers:</b> the data is volatile and spread across many fresh sources, so
          fetch live rather than rely on a precomputed index &rarr; orchestration: a planner decomposes
          the question, fans out retrieval across web/docs/APIs, each sub-step returning candidates
          &rarr; recall wide then rerank per sub-question; dedupe near-identical sources &rarr;
          <b>compress</b> intermediate findings into a running summary so the budget survives the
          fan-out, but keep a citation handle on every retained fact &rarr; reliability: adversarial
          verification — cross-check claims against independent sources and surface conflicts instead of
          silently picking one &rarr; provenance per claim so the final answer is traceable
          sentence-by-sentence &rarr; cost is the dominant constraint — cap breadth, cache repeated
          lookups, and stop when marginal sources stop changing the answer &rarr; observability: log the
          source set, what each sub-step retrieved, and which claims survived verification &rarr; name
          the failure modes this shape invites: conflicting, outdated, excessive — and the defense for
          each.

    <div class="facts">
        <div class="fact"><b>Pre-index (Cursor)</b> Builds embeddings ahead of time so retrieval is one fast lookup. The bet: precompute relevance. The tax: an index drifts from reality the moment code changes.</div>
        <div class="fact"><b>Read-on-demand (Claude Code)</b> No precompute; it searches the live tree per task. The bet: always-fresh, auditable reads. The tax: round-trips cost tokens and latency, and a bad search plan misses files.</div>
        <div class="fact blue"><b>History + connectors (ChatGPT)</b> Strong generalist whose grounding in your systems is only as good as the connectors you wire up. With nothing retrieved, it fills gaps confidently.</div>
        <div class="fact blue"><b>Local window (Copilot)</b> Optimised for instant inline completion from what's near the cursor; whole-repo constraints far from the cursor can be silently violated.</div>
    </div>

        <thead><tr><th>Agent</th><th>Retrieval (matched to data)</th><th>Scalability</th><th>Cost</th><th>Reliability</th><th>Observability</th></tr></thead>
        <tbody>
            <tr><td><b>Billing</b></td><td><em>Deterministic</em> structured query — SQL keyed by <code>account_id</code>, never fuzzy embeddings</td><td>Cheap point lookups</td><td>Low tokens</td><td>No hallucinated numbers; never invent amounts</td><td>Log every assembled context + tool call for audit</td></tr>
            <tr><td><b>Support</b></td><td>Hybrid over help-center KB + customer history (memory); rerank</td><td>KB grows; cache hot articles</td><td>Moderate</td><td>KB freshness; PII handling / compliance</td><td>Retrieval hit-rate; deflection/escalation; cite sources to user</td></tr>
            <tr><td><b>Engineering</b></td><td>Codebase retrieval (hybrid + structural) and/or read-on-demand; tests as a grounding signal</td><td>Large repos; token-heavy</td><td>Token cost dominates</td><td>Ground in real code; run tests</td><td>Log which files / symbols loaded per task</td></tr>
            <tr><td><b>Research</b></td><td>Multi-source web/doc retrieval, multi-step; compress intermediate findings; keep citations</td><td>Many calls / sources</td><td>High (fan-out)</td><td>Adversarial verification; dedupe conflicting sources</td><td>Source provenance per claim</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact"><b>Billing — exact</b> Financial data must be exact and auditable. A SQL point-lookup returns the true number; a semantic search returns the <em>most-similar</em> number — which is a bug. Match mechanism to data type.</div>
        <div class="fact"><b>Support — fuzzy</b> Natural-language KB articles are the home turf of hybrid retrieval + rerank, with customer history as memory. Cache hot articles; cite sources back to the user.</div>
        <div class="fact blue"><b>Engineering — structural</b> Code has structure (imports, symbols, tests). Hybrid + structural retrieval or read-on-demand grounds answers in real code; tests are a verification signal the agent can run.</div>
        <div class="fact blue"><b>Research — volatile</b> Truth is spread across many fresh sources. Fan out, verify adversarially, compress intermediate findings to fit the budget, and keep provenance per claim.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Just use vector search for everything — it's the AI way."</div>
        <div class="t">&#10003; Vector search returns the <em>most similar</em>, not the <em>correct</em>. For exact/transactional data (an invoice total) you need a deterministic structured query, not semantic similarity.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "We'll add observability later — ship the agent first."</div>
        <div class="t">&#10003; Without logging the assembled context per call, a wrong answer is undebuggable. For billing it's also non-negotiable for audit. Observability is part of the design, not a follow-up.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> <span>Design the retrieval to the data:</span> exact &amp; financial &rarr; structured query; fuzzy &amp; textual &rarr; hybrid + rerank; volatile &rarr; fetch live; long-running &rarr; <span>compress + remember</span>.</div>

    <details class="mc"><summary>Memory check</summary><ul><li>Why is embedding search wrong for a billing agent? <span class="ans">&rarr; it returns the most-similar value, not the exact one; financial data needs deterministic, auditable structured queries</span></li><li>What's the dominant cost driver for an engineering agent? <span class="ans">&rarr; tokens — large repos mean token-heavy context; cost scales with what you load per task</span></li><li>What does a research agent need that a billing agent doesn't? <span class="ans">&rarr; adversarial verification, dedupe of conflicting sources, and source provenance per claim across a fan-out</span></li></ul></details>

    <details class="iq">
        <summary>Design the retrieval layer for a billing-support agent that must quote exact charges. Walk the trade-offs.</summary>
        <div class="a">
            <b>Hit these points:</b> charges are exact, transactional, auditable &rarr; retrieve via deterministic SQL keyed by <code>account_id</code>/invoice id, <em>not</em> embeddings &rarr; embeddings would return the most-similar amount, i.e. a confidently-wrong number &rarr; reliability rule: the agent may quote only values it pulled, never synthesize amounts &rarr; log every assembled context + tool call for audit and dispute resolution &rarr; if the user also asks policy questions, that's a <em>different</em> data type &rarr; layer hybrid KB retrieval for the textual part &rarr; one agent, two retrieval mechanisms matched to two data types.
        </div>
    </details>

<h3 id="d1">1 · One-page Context Engineering cheat sheet</h3>
<div class="facts">
    <div class="fact">
        <b>The window</b> A fixed token budget shared by prompt, tools, history, retrieved context,
        and the answer. A bigger window raises the ceiling; it doesn't fix what you put in it.
    </div>
    <div class="fact">
        <b>Context = the new query</b> Retrieval/assembly plays SQL's role: it decides what the
        model ever sees.
    </div>
    <div class="fact">
        <b>Selection &rarr; retrieve</b> Lexical (BM25) · semantic (embeddings) · <em>hybrid</em>;
        recall first.
    </div>
    <div class="fact">
        <b>Rerank</b> Precision second: re-order the recalled set so the best lands near the top.
    </div>
    <div class="fact blue">
        <b>Chunking</b> Split on meaning, add overlap; add per-chunk context so a chunk stands
        alone.
    </div>
    <div class="fact blue">
        <b>RAG</b> retrieve &rarr; assemble &rarr; LLM &rarr; answer; the answer is grounded in the
        fetched facts.
    </div>
    <div class="fact blue">
        <b>Memory vs KB</b> Memory = per-user/session state you carry; KB = shared corpus you
        retrieve from.
    </div>
    <div class="fact blue">
        <b>Compression</b> For long runs: summarize/prune intermediate context to stay in budget.
    </div>
</div>

<h3 id="d2">2 · Context Engineering mental models</h3>
<div class="facts">
    <div class="fact">
        <b>Constant vs variable</b> Context is the variable; the model is the constant. Change the
        context, change the answer.
    </div>
    <div class="fact">
        <b>The new query</b> Context retrieval is the new database query — same role SQL played.
    </div>
    <div class="fact">
        <b>Recall, then precision</b> Cast wide first; rerank to surface the best. Two stages, two
        jobs.
    </div>
    <div class="fact">
        <b>Match retrieval to data</b> Exact &rarr; structured; fuzzy &rarr; hybrid; volatile &rarr;
        live; long &rarr; compress.
    </div>
    <div class="fact blue">
        <b>Most AI bugs are context bugs</b> Suspect the input before blaming the model.
    </div>
    <div class="fact blue">
        <b>The window is RAM</b> A budget you allocate, not a pipe to your data.
    </div>
</div>

<h3 id="d5">5 · Interview questions (Core &rarr; Staff)</h3>
<details class="iq">
    <summary>
        Core — What is a context window and why can't an LLM just read the whole codebase?
    </summary>
    <div class="a">
        <b>Hit these points:</b> the window is a fixed token budget — the model's only channel
        &rarr; real repos are orders of magnitude larger &rarr; bigger windows cost more, run
        slower, and recall the middle poorly &rarr; so you must <em>select</em> a relevant subset
        &rarr; the job is "which slice," not "read it all."
    </div>
</details>
<details class="iq">
    <summary>Core — Lexical vs semantic vs hybrid retrieval — when each?</summary>
    <div class="a">
        <b>Hit these points:</b> lexical (BM25) nails exact tokens/identifiers but misses synonyms
        &rarr; semantic (embeddings) matches meaning but misses exact strings and can drift &rarr;
        hybrid runs both and fuses (e.g. reciprocal rank fusion) &rarr; rerank then sharpens
        precision &rarr; default to hybrid + rerank in production.
    </div>
</details>
<details class="iq">
    <summary>Mid — Walk me through a RAG pipeline and where each failure mode enters.</summary>
    <div class="a">
        <b>Hit these points:</b> ingest &rarr; chunk &rarr; index &rarr; retrieve &rarr; rerank
        &rarr; assemble &rarr; LLM &rarr; log &rarr; <em>missing</em> enters at retrieve (recall
        gap), <em>wrong</em> at rerank, <em>outdated</em> at index freshness, <em>conflicting</em>{' '}
        at assemble, <em>excessive</em> at assemble/budget &rarr; each stage gets its own metric.
    </div>
</details>
<details class="iq">
    <summary>Senior — How do you evaluate retrieval quality, not just answer quality?</summary>
    <div class="a">
        <b>Hit these points:</b> build a labeled set of query&rarr;gold-doc pairs &rarr; measure
        recall@k and rerank precision separately from end-answer accuracy &rarr; log assembled
        context per call so failures are attributable &rarr; a wrong answer with the right context
        is a reasoning bug; with the wrong context it's a retrieval bug &rarr; you can't fix what
        you can't separate.
    </div>
</details>
<details class="iq">
    <summary>Senior — Why is matching retrieval to data type a senior decision?</summary>
    <div class="a">
        <b>Hit these points:</b> exact/transactional (a balance) &rarr; deterministic structured
        query; semantic search would return the most-similar, i.e. wrong &rarr; fuzzy text &rarr;
        hybrid + rerank &rarr; volatile &rarr; fetch live, don't cache stale &rarr; long-running
        &rarr; compress + remember &rarr; the mistake juniors make is one-size-fits-all vector
        search.
    </div>
</details>
<details class="iq">
    <summary>
        Staff — Review this design: an AI agent answers finance questions over a vector DB of
        reports. Pushback?
    </summary>
    <div class="a">
        <b>Hit these points:</b> numbers must be exact and auditable — vector search returns
        similar, not correct &rarr; risk: confidently-wrong amounts, no audit trail &rarr; redesign:
        structured query for the figures, vector/hybrid only for narrative text &rarr; add a rule
        that figures are quoted, never synthesized &rarr; log assembled context per call &rarr; add
        freshness/invalidation since reports update &rarr; name the failure modes this design
        currently invites (wrong, outdated).
    </div>
</details>
<details class="iq">
    <summary>
        Staff — Same model, two products, one feels far smarter. Explain to a skeptical interviewer.
    </summary>
    <div class="a">
        <b>Hit these points:</b> the model is the constant; context is the variable &rarr; the
        smarter product retrieves and assembles better context per call &rarr; "smart vs dumb" is
        mostly an artifact of what got loaded &rarr; therefore investment goes to retrieval,
        evaluation, and assembly — not chasing model versions &rarr; verify by logging context for
        the failing cases.
    </div>
</details>

<h3 id="d6">6 · VP Engineering review questions (leadership framing)</h3>
<details class="iq">
    <summary>
        VP — Build vs buy: when do we build our own retrieval vs adopt a vendor assistant?
    </summary>
    <div class="a">
        <b>Hit these points:</b> buy when the data is generic and the vendor's context strategy fits
        our use-case &rarr; build when our differentiation <em>is</em> the retrieval over
        proprietary data &rarr; the model is a commodity; the retrieval pipeline is the moat &rarr;
        decision pivots on whether off-the-shelf grounding reaches our live systems.
    </div>
</details>
<details class="iq">
    <summary>VP — Where are the cost levers in an AI feature?</summary>
    <div class="a">
        <b>Hit these points:</b> cost scales with tokens sent &rarr; tighter selection + rerank cuts
        wasted context &rarr; compress long sessions; trim history &rarr; cache hot retrievals
        &rarr; right context on a cheaper model often beats wrong context on the flagship — a
        structural saving, not a discount.
    </div>
</details>
<details class="iq">
    <summary>VP — Where does investment in retrieval actually pay off?</summary>
    <div class="a">
        <b>Hit these points:</b> retrieval fixes compound across every feature on the stack &rarr; a
        model upgrade is a flat tax on every call &rarr; better retrieval lifts accuracy, lowers
        cost, and reduces hallucination simultaneously &rarr; prioritize where wrong answers carry
        the highest business/compliance risk.
    </div>
</details>
<details class="iq">
    <summary>VP — How do we measure retrieval quality as a leadership metric?</summary>
    <div class="a">
        <b>Hit these points:</b> recall@k and rerank precision on a labeled set &rarr; retrieval
        hit-rate in production &rarr; deflection/escalation for support, audit-pass for billing
        &rarr; track these <em>separately</em> from end-answer accuracy so we know whether to invest
        in retrieval or the model.
    </div>
</details>
<details class="iq">
    <summary>VP — What are the risk and compliance exposures?</summary>
    <div class="a">
        <b>Hit these points:</b> PII in retrieved context &rarr; redaction + access control at the
        retrieve stage &rarr; prompt injection via poisoned/retrieved content (OWASP LLM Top 10)
        &rarr; guardrails before the model &rarr; auditability: log assembled context per call
        &rarr; for finance, deterministic retrieval + no-synthesis rule.
    </div>
</details>
<details class="iq">
    <summary>VP — When do we upgrade the model vs invest in retrieval?</summary>
    <div class="a">
        <b>Hit these points:</b> instrument first: was the right context retrieved for the failing
        cases? &rarr; if no &rarr; fix retrieval (cheaper, compounds) &rarr; if yes and reasoning is
        the bottleneck (multi-step synthesis) &rarr; pay for model capability &rarr; never upgrade
        the model to paper over a retrieval gap.
    </div>
</details>
