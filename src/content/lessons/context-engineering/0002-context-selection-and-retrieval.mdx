# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Name the three stages of the selection pipeline, in order, and what each one does.'
      a: >-
          <b>Hit these points:</b> <b>candidate generation</b> &rarr; cast a wide net to pull anything
          that might be relevant &rarr; <b>ranking</b> &rarr; score those candidates so the best ones
          rise &rarr; <b>selection / packing</b> &rarr; fit the top of the list into the token budget.
          The whole point is choosing the smallest set that still contains the answer when you can't
          send everything.
    - level: 'core'
      q: 'What does lexical / BM25 search actually match on, and what is it good at?'
      a: >-
          <b>Hit these points:</b> it scores documents by overlap of the <b>exact terms</b>, weighted
          by term frequency and rarity (TF-IDF / BM25) &rarr; meaning plays no part, only the literal
          strings &rarr; strong on identifiers, error codes, function names, exact IDs &rarr; fast,
          cheap, interpretable, no model required — you can read why it matched.
    - level: 'core'
      q: 'What does semantic / embedding search match on, and why use it?'
      a: >-
          <b>Hit these points:</b> it encodes query and chunks into vectors and ranks by similarity
          (cosine over an ANN index) &rarr; matches on <b>meaning</b>, so synonyms and paraphrase still
          find the right passage even with zero shared words &rarr; exists because users rarely use the
          document's exact wording &rarr; robust to phrasing, weak on rare exact tokens.
    - level: 'core'
      q: 'What is hybrid search, and what does fusion (e.g. RRF) do in it?'
      a: >-
          <b>Hit these points:</b> run lexical and semantic <b>together</b>, then merge their two result
          lists into one ranking &rarr; <b>fusion</b> (Reciprocal Rank Fusion or a weighted blend)
          combines the per-engine ranks into a single ordered list &rarr; the reason: the two engines
          fail on opposite cases, so one usually catches what the other drops &rarr; gives you
          exact-match and meaning-match coverage at once.
    - level: 'senior'
      q: "Users say semantic search returns &ldquo;close but wrong&rdquo; results and sometimes whiffs on exact error codes. What's your fix and why?"
      a: >-
          <b>Hit these points:</b> name the root cause — embeddings score topical similarity, so they
          return plausible-but-wrong and wash out rare exact tokens like error codes &rarr; add
          <b>hybrid</b>: run BM25 alongside the vector search and fuse (RRF) so exact strings are caught
          by lexical and meaning by semantic &rarr; add a <b>cross-encoder re-rank</b> over the fused
          top-N to push the genuinely best result up (precision) &rarr; frame the split: hybrid fixes
          recall, re-rank fixes precision &rarr; verify with retrieval hit-rate on a labelled set, not
          end-answer vibes; tune fusion weights against that set.
    - level: 'senior'
      q: 'Your code assistant keeps missing the actual buggy file. Where in the pipeline do you look first, and how do you prove it?'
      a: >-
          <b>Hit these points:</b> instrument per stage, don't guess &rarr; first check
          <b>candidate-generation recall</b>: was the buggy file ever in the candidate set? if no,
          ranking and packing are irrelevant &rarr; if recall is the gap, add the missing signal —
          usually structural (imports / call sites) or lexical for an exact symbol the embedding washed
          out &rarr; if recall is fine but it ranked low, fix ranking / add a re-rank &rarr; if it
          ranked high but got dropped, it's a packing/budget problem &rarr; track retrieval hit-rate as
          a first-class metric, not just end-answer quality.
    - level: 'senior'
      q: 'Distinguish ranking from re-ranking. When is a cross-encoder worth its cost, and when is it a waste?'
      a: >-
          <b>Hit these points:</b> <b>ranking</b> = first stage, cheap score (BM25 / cosine) over many
          candidates, optimizing recall — &ldquo;don't miss it&rdquo; &rarr; <b>re-ranking</b> = second
          stage, expensive cross-encoder reading query + document jointly over the top-N, optimizing
          precision — &ldquo;best at the top&rdquo; &rarr; worth it when the final top-k reaching the
          window must be high-precision and first-stage ordering is noisy &rarr; a waste if recall is
          the real gap — it can't recover a missed doc &rarr; bound cost by running it on top-N only and
          caching; prove the precision lift on a labelled set before keeping it.
    - level: 'senior'
      q: 'Someone proposes &ldquo;just embed everything and drop BM25 — embeddings are strictly better.&rdquo; Push back.'
      a: >-
          <b>Hit these points:</b> embeddings are <b>weak on exact tokens</b> — error codes, IDs,
          function names get washed into topical neighbours &rarr; for a query that <em>is</em> the
          literal string, meaning is irrelevant and lexical is exactly right &rarr; the two engines have
          opposite blind spots, so dropping BM25 trades one failure mode for another &rarr; hybrid keeps
          BM25 precisely for those cases; the small extra infra is cheaper than silently missing every
          ID lookup &rarr; decide with hit-rate on a query mix that includes exact-token queries, not a
          vibe.
    - level: 'staff'
      q: "Design a retrieval strategy when you don't yet know the query mix — some lookups are exact IDs, some are vague conceptual questions, and the corpus keeps changing."
      a: >-
          <b>Hit these points:</b> start from <b>recall first</b>: a missed doc can't be ranked or
          packed, so over-invest stage 1 &rarr; default to <b>hybrid</b> (BM25 + vector, fused) so
          exact-token and conceptual queries are both covered without knowing the split up front &rarr;
          add a <b>cross-encoder re-rank</b> on the fused top-N for precision where it matters &rarr;
          layer signals the domain needs — structural for code, recency/version for changing docs,
          authority for canonical sources &rarr; keep re-indexing fresh as the corpus changes &rarr;
          instrument hit-rate by query type, learn the real mix, then tune fusion weights and decide
          where re-rank earns its latency.
    - level: 'staff'
      q: '&ldquo;Relevant&rdquo; means different things per system. Contrast a billing platform, a code repo, and a docs portal — and say how that changes your retrieval.'
      a: >-
          <b>Hit these points:</b> relevance is set by the question's intent, not one universal score
          &rarr; <b>billing</b>: the exact account / ledger rows — lexical/ID match + authority; a
          &ldquo;similar&rdquo; account is actively harmful &rarr; <b>repo</b>: the right symbols plus
          their dependencies — semantic concept + structural call sites and imports &rarr; <b>docs</b>:
          the right article at the right version — semantic gated by recency/version + authority &rarr;
          so you pick and weight signals to the domain's notion of relevance, rather than shipping one
          generic retriever everywhere.
    - level: 'staff'
      q: 'You can raise recall (wider net) or precision (aggressive re-rank + tighter packing), but not both for free. How do you reason about the trade-off?'
      a: >-
          <b>Hit these points:</b> the two failures cost differently — a recall miss means the answer
          never reaches the model (unrecoverable), a precision miss means noise the model must see past
          &rarr; so buy recall cheaply and wide in stage 1, then buy precision narrowly in stage 2 where
          it's affordable &rarr; watch the packing tension: more candidates raise recall but bury the
          right lines in the &ldquo;lost in the middle&rdquo; zone and cost more &rarr; tie the dial to
          the domain's cost of a miss (billing error vs a fuzzy docs answer) &rarr; measure both
          hit-rate and top-k precision on a labelled set and move the knob deliberately, not by feel.
    - level: 'design'
      q: 'Design the retrieval layer for a documentation Q&amp;A bot over versioned, frequently-updated docs.'
      a: >-
          <b>A strong answer covers:</b> the query mix is mostly conceptual &ldquo;how do I…&rdquo;
          questions, so semantic carries weight — but keep BM25 for exact API names and error strings
          &rarr; gate on <b>recency / version</b> so an old article doesn't beat the current one, and on
          <b>authority</b> for canonical pages &rarr; hybrid stage 1, fuse with RRF, re-rank the top-N
          for precision since users read only the first answer &rarr; pack the best article(s) within
          budget; prefer the right section over dumping whole pages &rarr; re-index on every doc change
          so freshness holds &rarr; call the failure modes: plausible-but-wrong from semantic,
          vocabulary mismatch from lexical &rarr; measure hit-rate on labelled questions, not end-answer
          vibes.
    - level: 'design'
      q: 'Design retrieval for an internal-search feature across code, tickets, and wiki docs in one box.'
      a: >-
          <b>A strong answer covers:</b> the corpus is heterogeneous, so &ldquo;relevant&rdquo; differs
          per source — code wants structural signals (imports, call sites) plus lexical for exact
          symbols; tickets/wiki lean semantic &rarr; run <b>hybrid</b> per source and fuse, or fuse
          across sources, so exact-token and conceptual queries both land &rarr; recall first: a missed
          file can't be ranked or packed &rarr; re-rank the fused top-N for precision; weight by recency
          where it matters (latest ticket, newest doc) and authority for canonical sources &rarr; pack
          within budget — signatures/docstrings for code, snippets for docs &rarr; keep indexes fresh as
          all three sources change &rarr; instrument hit-rate by source and query type, then tune fusion
          weights and decide where re-rank pays for its latency.

    <h3>Search (lexical / keyword / BM25)</h3>

    <div class="facts">
        <div class="fact">
            <b>Is</b> Exact-term matching: score documents by how well their words match the query
            words, weighted by term frequency and rarity (TF-IDF / BM25).
            <sup><a href="#s3" id="r3">3</a></sup>
        </div>
        <div class="fact">
            <b>Why it exists</b> Some queries <em>are</em> the exact tokens — an error code, a function
            name, an account ID. For those, meaning is irrelevant; the literal string is the answer.
        </div>
        <div class="fact blue">
            <b>Advantages</b> Precise on exact terms &amp; identifiers, fast, cheap, interpretable,
            needs no model — you can read why it matched.
        </div>
        <div class="fact blue">
            <b>Failure mode</b> No synonyms or semantics: if the user phrases it differently than the
            doc (vocabulary mismatch), it returns a <em>total miss</em>.
        </div>
    </div>

    <h3>Semantic search (embeddings / vector)</h3>

    <div class="facts">
        <div class="fact">
            <b>Is</b> Encode query and chunks into vectors; rank by cosine similarity using an
            approximate-nearest-neighbour index (e.g. HNSW).
            <sup><a href="#s4" id="r4">4</a></sup>
        </div>
        <div class="fact">
            <b>Why it exists</b> Users rarely use the doc's exact words. Embeddings match on{' '}
            <em>meaning</em>, so a paraphrase still finds the right passage.
        </div>
        <div class="fact blue">
            <b>Advantages</b> Captures meaning, synonyms, paraphrase; robust to wording; finds the
            right idea even with zero shared words.
        </div>
        <div class="fact blue">
            <b>Failure mode</b> Returns &ldquo;topically similar but wrong&rdquo;; weak on rare exact
            tokens — it can't reliably match an exact error string or ID.
        </div>
    </div>

    <h3>Hybrid search</h3>

    <div class="facts">
        <div class="fact">
            <b>Is</b> Run lexical and semantic together, then fuse the two result lists into one
            ranking — commonly Reciprocal Rank Fusion or a weighted blend.
            <sup><a href="#s5" id="r5">5</a></sup>
        </div>
        <div class="fact">
            <b>Why it exists</b> The two engines fail on opposite cases: lexical whiffs on paraphrase,
            semantic whiffs on exact tokens. Run both and one usually catches what the other drops.
        </div>
        <div class="fact blue">
            <b>Advantages</b> Covers both exact and meaning, so fewer right documents slip through.
            Anthropic's Contextual Retrieval pairs embeddings with BM25 for exactly this reason.
            <sup><a href="#s2" id="r2b">2</a></sup>
        </div>
        <div class="fact blue">
            <b>Failure mode</b> More infra and complexity, fusion weights to tune; still bounded by the
            recall of both first-stage retrievers.
        </div>
    </div>

    <h3>Ranking vs re-ranking — two different jobs</h3>

    <Fig4 />

    <div class="facts">
        <div class="fact">
            <b>Ranking — is</b> Order candidates by a cheap score (BM25, cosine). First stage. Purpose:
            optimize <em>recall</em> so the right document is somewhere in the list.
        </div>
        <div class="fact">
            <b>Re-ranking — is</b> A second, expensive, accurate pass (cross-encoder) that re-scores
            the top-N jointly with the query. Purpose: optimize <em>precision</em> at the top.
        </div>
        <div class="fact blue">
            <b>Re-rank advantage</b> Large precision lift on the final top-k the model actually sees —
            the items most likely to be packed into the window.
        </div>
        <div class="fact blue">
            <b>Re-rank failure</b> Latency and cost; runs on top-N only; <em>cannot recover</em> a
            document the first stage already missed.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; &ldquo;Embeddings replaced keyword search.&rdquo;</div>
        <div class="t">
            &#10003; Embeddings are weak on exact tokens (error codes, IDs, function names). Hybrid
            keeps BM25 precisely for those cases.
        </div>
    </div>
    <div class="myth">
        <div class="m">&#10007; &ldquo;A great re-ranker fixes bad retrieval.&rdquo;</div>
        <div class="t">
            &#10003; Re-ranking only reorders what stage 1 returned. If recall missed the doc, no
            re-ranker can surface it — fix recall first.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span>{' '}
        <span>First-stage retrieval buys recall; re-ranking buys precision.</span> Lexical finds
        exact, semantic finds meaning, <span>hybrid finds both</span>.
    </div>
