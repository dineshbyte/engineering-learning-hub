# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is an embedding, and what does "near" mean in embedding space?'
      a: >-
          <b>Hit these points:</b> an embedding is a dense vector of meaning — text mapped to a point
          in a learned high-dimensional space &rarr; the model places semantically similar text close
          together, so "near" is a distance/angle, not keyword overlap &rarr; "log the user out" and
          "revoke the session" share no words yet land close &rarr; that's why semantic search catches
          paraphrase where lexical match misses &rarr; the unit you embed is the chunk, so chunk
          granularity decides vector quality.
    - level: 'core'
      q: 'Name the three similarity metrics and say what cosine actually measures.'
      a: >-
          <b>Hit these points:</b> cosine, dot product, Euclidean &rarr; cosine scores the <em>angle</em>
          between two vectors — direction, not magnitude — so a long doc and a short one on the same
          topic still score close &rarr; normalize vectors to unit length and cosine and dot product
          agree, and Euclidean ranks the same way &rarr; the metric must match how the model was
          trained &rarr; the failure mode is mixing a metric the model wasn't trained for and quietly
          getting worse rankings.
    - level: 'core'
      q: 'What does ANN stand for, and why approximate instead of exact?'
      a: >-
          <b>Hit these points:</b> approximate nearest neighbour &rarr; exact k-NN compares the query
          to every vector — O(n) per query, too slow at millions of vectors &rarr; ANN gives up a few
          percent recall to make search sub-linear, orders of magnitude faster &rarr; recall = fraction
          of the true top-k the index actually returns &rarr; the bargain is fine for retrieval: 95%
          recall at ~1ms beats 100% at 500ms, and a reranker cleans up the top.
    - level: 'core'
      q: 'What are the five stages of an ingestion pipeline?'
      a: >-
          <b>Hit these points:</b> load &rarr; parse &rarr; chunk &rarr; embed &rarr; upsert &rarr;
          embed is the stage that bills you per token &rarr; the key idea is it's a pipeline that runs
          forever, not a one-time script — it re-runs incrementally as data changes &rarr; the
          canonical failure is treating it as "index once at launch," which lets the store drift out of
          sync with the source.
    - level: 'senior'
      q: 'A teammate wants to bump the embedding model "for a quick quality win." What do you flag?'
      a: >-
          <b>Hit these points:</b> it's not a config flip — old and new vectors live in different
          spaces and are incomparable, so every chunk must be re-embedded and the index rebuilt &rarr;
          that's a corpus-wide embed-token bill plus pipeline time, and a cutover plan (dual-write /
          shadow index) so search isn't down mid-migration &rarr; quantify it: tokens-per-chunk
          &times; chunk count &times; price &rarr; validate the win on an eval set first — "newest"
          isn't always better on <em>your</em> domain &rarr; cheaper alternatives: Matryoshka
          truncation or a domain model if the real goal is quality-per-dollar.
    - level: 'senior'
      q: 'Pick an index family for 50M vectors with tenant filtering and a p99 latency budget. Walk the trade-offs.'
      a: >-
          <b>Hit these points:</b> you can't max recall, latency, and memory at once — name the
          triangle and pick a corner &rarr; HNSW buys recall and low latency by spending memory; IVF
          tunes speed via n-probe; PQ buys memory back by spending recall, often IVF+PQ at this scale
          &rarr; tenant filtering must be a <em>pre-filter</em> that scopes the search to the allowed
          set — right result count and a security/isolation boundary — not a post-filter that can
          return too few &rarr; size RAM against vector count &times; dims, and validate recall against
          an exact baseline before shipping.
    - level: 'senior'
      q: 'Vector search is fast but misses obviously relevant chunks for some queries. Where do you look?'
      a: >-
          <b>Hit these points:</b> "fast but misses" smells like low recall from ANN settings — raise
          ef-search / n-probe and re-measure recall against an exact baseline &rarr; check whether PQ
          compression is too aggressive (lossy codes drop near-neighbours) and whether the index needs
          a rebuild after heavy upserts &rarr; confirm metadata filtering isn't post-filtering away
          good hits or scoping to the wrong tenant/date &rarr; verify the query is embedded with the
          <em>same</em> model and normalization as the corpus &rarr; if recall is genuinely good but
          ranking is wrong, that's a reranker problem, not the index.
    - level: 'senior'
      q: 'Your RAG cost-per-query is 3&times; the budget. Name the levers, in order.'
      a: >-
          <b>Hit these points:</b> first attribute it — break cost into LLM input, LLM output, query
          embed, ANN search, rerank; input tokens usually dominate &rarr; cut retrieved chunks to
          fewer-but-better (helps accuracy too) and trim the prompt &rarr; rerank only the top-n, not
          every candidate &rarr; cache the stable prefix (system prompt, tool defs) so you stop
          re-paying for it &rarr; drop to a cheaper / smaller-dim embedding model if quality holds on an
          eval set &rarr; batch corpus embedding instead of real-time &rarr; only then consider a
          smaller generation model — and measure each change against an eval set so you don't trade
          dollars for accuracy.
    - level: 'staff'
      q: 'Design an embedding + index + cost strategy for a 200M-chunk corpus on a fixed monthly budget.'
      a: >-
          <b>Hit these points:</b> start from the budget and work backwards — model the dominant cost
          line (LLM input tokens at serve time, plus one-off corpus embedding) and set targets per line
          &rarr; choose model + dims by quality-per-dollar on an eval set, lean on Matryoshka so you
          can dial dims down without re-embedding &rarr; index family scoped to scale: IVF+PQ for
          memory at 200M, pre-filter for tenancy, tune n-probe to hit a recall floor at a latency
          ceiling &rarr; serve-time levers: fewer-better chunks, rerank top-n, cache the stable prefix
          &rarr; ingestion is incremental (CDC) with batched embedding, not real-time &rarr; close the
          loop with recall and cost-per-query as monitored SLOs so a regression is visible, not a
          surprise invoice.
    - level: 'staff'
      q: 'When is a vector index the wrong tool, and what would you build instead?'
      a: >-
          <b>Hit these points:</b> name the cases where it's the wrong default — small corpus
          (brute-force or pgvector in existing Postgres is simpler and fast enough), data that's
          exact-match or highly structured (a relational/keyword index beats meaning-space), and
          volatile facts like price or stock that are stale minutes after embedding &rarr; for volatile
          fields, fetch live via a structured tool at query time or TTL them, don't bake them into the
          index &rarr; the staff judgement is matching retrieval mechanism to data shape and freshness,
          and not adding vector-DB infrastructure before scale, latency, or filtering actually demand
          it.
    - level: 'staff'
      q: 'A stale index keeps serving a deleted doc. Frame this as a systemic failure and fix it at the root.'
      a: >-
          <b>Hit these points:</b> name it — a stale index is the outdated-context failure: deleted
          docs and old facts served fluently and confidently &rarr; root cause is treating ingestion as
          a one-time script with no invalidation, not a bad query &rarr; fix the system: incremental
          upserts / CDC for edits, hard deletes for removed chunks, invalidation triggered on source
          change &rarr; split by volatility — stable prose ingested, volatile fields fetched live &rarr;
          model changes go through a full re-embed behind a shadow index, then cutover &rarr; make
          staleness observable: monitor oldest-chunk age as an SLO so it's caught by a dashboard, not a
          user.
    - level: 'design'
      q: 'Design the vector index + ingestion pipeline for a fast-changing corpus (news + product catalog, edits every minute).'
      a: >-
          <b>A strong answer covers:</b> clarifies scale, edit rate, and latency/freshness SLOs first
          &rarr; splits by volatility — article prose is embedded and incrementally ingested via CDC
          (re-embed only changed docs, delete removed ones); price/stock/availability are <em>not</em>
          embedded but fetched live at query time or TTL'd, because a stale index is the
          outdated-context failure &rarr; index family chosen for scale and recall needs (HNSW for high
          recall if memory allows, IVF+PQ if it doesn't), pre-filter for category/tenant scoping &rarr;
          embedding model + dims justified on an eval set, with Matryoshka headroom &rarr; invalidation
          wired to source changes so a single edit re-chunks/re-embeds just that doc &rarr; model
          upgrades run behind a shadow index then cut over &rarr; staleness monitored as an SLO
          (oldest-chunk age) so drift is visible.
    - level: 'design'
      q: 'Design a cost-bounded retrieval stack: serve relevant context under a hard per-query dollar/latency ceiling.'
      a: >-
          <b>A strong answer covers:</b> starts from the ceiling and attributes cost per line — LLM
          input usually dominates, then output, query embed, ANN search, rerank &rarr; primary lever is
          fewer-but-better chunks (cheaper <em>and</em> more accurate) plus a trimmed prompt &rarr;
          rerank only the top-n; cache the stable prefix so the system prompt and tool defs aren't
          re-paid every call &rarr; embedding cost controlled with a cheaper/smaller-dim or
          Matryoshka-truncated model, validated on an eval set &rarr; latency budget met by tuning
          recall vs latency (ef-search / n-probe) and the right index family, not by buying hardware
          &rarr; corpus embedding batched, not real-time &rarr; cost-per-query and recall tracked as
          SLOs with alerts, so a regression shows on a dashboard before it shows on the invoice &rarr;
          every cut measured against an eval set so dollars aren't traded for accuracy.

<section class="lvl" id="m1">
    <div class="lvltop"><span class="num">1</span><h2>Embeddings Deep-Dive</h2></div>
    <p class="tagline">An embedding turns text into a point in <b>meaning-space</b>. Texts that mean similar things land close together; that closeness is what semantic search measures.<sup><a href="#s3" id="r2">3</a></sup></p>

    <h3>What an embedding is: meaning becomes geometry</h3>
    <Fig2 />

    <h3>The four knobs you actually choose</h3>
    <table>
        <thead><tr><th>Knob</th><th>What it sets</th><th>Effect</th><th>Trade-off</th></tr></thead>
        <tbody>
            <tr><td><b>Model</b></td><td>General vs domain-specific vs multilingual</td><td>Decides what "similar" means for your data</td><td>Quality vs speed vs price; domain models win on jargon</td></tr>
            <tr><td><b>Dimensions</b></td><td>Vector length (e.g. 384 · 768 · 1536 · 3072)</td><td>More dims capture more nuance</td><td>Bigger = more storage, RAM, and compute per query</td></tr>
            <tr><td><b>Metric</b></td><td>Cosine · dot product · Euclidean</td><td>How "closeness" is scored</td><td>Cosine ignores magnitude; must match how the model was trained</td></tr>
            <tr><td><b>What you embed</b></td><td>The chunk (granularity from Lesson 3)</td><td>Defines the unit of retrieval</td><td>Too big &rarr; averaged, diluted vector; too small &rarr; lost context</td></tr>
        </tbody>
    </table>

    <h3>Dimensions &amp; the similarity metric</h3>
    <div class="facts">
        <div class="fact"><b>Is</b> Dimensions are the length of the vector. A 3072-dim embedding has more axes to encode nuance than a 384-dim one — at 8&times; the storage and compute. <b>Matryoshka</b> embeddings are trained so you can truncate to fewer dims and keep most of the quality, trading dims for cost on demand.</div>
        <div class="fact"><b>Why it exists</b> Cosine similarity scores the <em>angle</em> between vectors — direction, not magnitude — so a long document and a short one about the same topic still score as close. Normalize vectors to unit length and cosine and dot product agree; Euclidean then ranks the same way.</div>
        <div class="fact blue"><b>Like (world)</b> A map: latitude/longitude are 2 dimensions placing every town. Add altitude and you separate towns that look adjacent on a flat map. More dimensions = more ways for two points to be "different."</div>
        <div class="fact blue"><b>Like (code)</b> A feature vector for a recommender. You compare users by the angle between their preference vectors (cosine), not by how many items each rated (magnitude). Same math, same normalize-first habit.</div>
    </div>

    <h3>Chunk granularity decides embedding quality</h3>
    <div class="myth">
        <div class="m">&#10007; "Embed the whole document — one vector per file is simplest."</div>
        <div class="t">&#10003; A vector is an <em>average</em> of its text's meaning. Embed a 5,000-word file and you get a blurry centroid that matches nothing precisely; chunk first (Lesson 3) so each vector means one thing.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Switching to a better embedding model is a config change."</div>
        <div class="t">&#10003; It's a migration. Old and new vectors live in <em>different spaces</em> and can't be compared — you must re-embed the entire corpus and rebuild the index.</div>
    </div>

    <h3>Re-embedding is a real migration</h3>
    <table>
        <thead><tr><th>Change you make</th><th>Blast radius</th><th>Cost driver</th></tr></thead>
        <tbody>
            <tr><td>Tune the prompt / retrieval k</td><td>None — query-time only</td><td>Free; no re-index</td></tr>
            <tr><td>Truncate Matryoshka dims</td><td>Re-index vectors (cheap re-store)</td><td>Storage + rebuild, no re-embed</td></tr>
            <tr><td>Change the embedding model</td><td>Re-embed <em>every</em> chunk + rebuild index</td><td>Embed tokens for the whole corpus</td></tr>
            <tr><td>Re-chunk the corpus</td><td>Re-embed + re-index everything</td><td>Embed tokens + pipeline run</td></tr>
        </tbody>
    </table>

    <div class="rule"><span>&#128229; Memory rule:</span> The <span>embedding model and the chunk are the lens</span>. Change the lens and you <span>re-index the world</span> — so choose both before you fill the store.</div>

<section class="lvl" id="m2">
    <div class="lvltop"><span class="num">2</span><h2>Vector Index Internals (ANN)</h2></div>
    <p class="tagline">Exact nearest-neighbour search compares the query to <b>every</b> vector — O(n) per query. At millions of vectors that's too slow, so we approximate: trade a little recall for a huge speed-up.<sup><a href="#s3" id="r4">3</a></sup></p>

    <h3>Why approximate? Brute force doesn't scale</h3>
    <Fig3 />

    <h3>The three index families</h3>
    <table>
        <thead><tr><th>Index</th><th>How it works</th><th>Strength</th><th>Cost</th></tr></thead>
        <tbody>
            <tr><td><b>HNSW</b> (graph)</td><td>Navigable small-world graph; greedily hop toward neighbours</td><td>Fast, high recall</td><td>Memory-heavy; slower to build</td></tr>
            <tr><td><b>IVF</b> (cluster)</td><td>Partition space into cells; probe only the nearest few</td><td>Tunable speed via #probes</td><td>Recall dips if a neighbour is in an un-probed cell</td></tr>
            <tr><td><b>PQ</b> (compress)</td><td>Product quantization: compress vectors to codes</td><td>Big memory savings</td><td>Lossy &rarr; lower recall; often paired as <b>IVF+PQ</b></td></tr>
            <tr><td><b>Flat</b> (brute force)</td><td>Compare against all vectors exactly</td><td>Perfect recall, zero build</td><td>O(n) per query — only for small corpora</td></tr>
        </tbody>
    </table>

    <h3>The trade-off triangle &amp; metadata filtering</h3>
    <Fig4 />

    <div class="facts">
        <div class="fact"><b>Is</b> ANN is a family of data structures (graphs, partitions, compressed codes) that find <em>probably</em> the nearest vectors without scanning all of them. The control knobs (ef-search, n-probe) directly trade recall against latency.</div>
        <div class="fact"><b>Why it exists</b> Vector similarity at scale is a high-dimensional search problem with no cheap exact index. ANN accepts a tiny, measurable recall loss to make queries sub-linear — the only way to serve millions of vectors in milliseconds.</div>
        <div class="fact blue"><b>Like (world)</b> Finding the nearest coffee shop: you don't measure the distance to every shop in the city (exact). You look in your neighbourhood and the next one over (probe a region) — almost always right, far faster.</div>
        <div class="fact blue"><b>Like (code)</b> A database index: a B-tree doesn't scan every row, it narrows the search. ANN is the same idea for "nearest by meaning" instead of "equals by key" — and like indexes, it has build and memory cost.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Production AI always needs a dedicated vector database."</div>
        <div class="t">&#10003; For a small corpus, brute-force or <code>pgvector</code> in your existing Postgres is fine and far simpler. Reach for a vector DB when scale, latency, or filtering demand it — not by default.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Post-filtering metadata is equivalent to pre-filtering."</div>
        <div class="t">&#10003; Post-filter retrieves k, then drops non-matches — you can end up with too few results. Pre-filter scopes the search to the allowed set first; it's also the tenant-isolation hook.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> <span>ANN buys speed by approximating</span> — tune <span>recall vs latency vs memory</span>, and don't reach for a vector DB before you actually need one.</div>

    <MemoryCheck items={mcM2} />

    <details class="iq">
        <summary>Your vector search is fast but misses obviously relevant chunks for some queries. Where do you look?</summary>
        <div class="a">
            <b>Hit these points:</b> "fast but misses" smells like low recall from the ANN settings — raise ef-search / n-probe and re-measure recall against an exact baseline &rarr; check if PQ compression is too aggressive (lossy codes drop near-neighbours) and whether the index needs a rebuild after heavy upserts &rarr; verify metadata filtering isn't post-filtering away good hits, or scoping to the wrong tenant/date &rarr; confirm the query is embedded with the <em>same</em> model and normalization as the corpus &rarr; if recall is genuinely good and ranking is wrong, that's a reranker problem (Lesson 2), not the index.
        </div>
    </details>

<section class="lvl" id="m3">
    <div class="lvltop"><span class="num">3</span><h2>Indexing Pipelines, Freshness &amp; Cost</h2></div>
    <p class="tagline">Indexing is a <b>data pipeline that runs forever</b>, not a one-time script. Documents change, models change, and a stale index quietly serves yesterday's answer.<sup><a href="#s2" id="r6">2</a></sup></p>

    <h3>The ingestion pipeline</h3>
    <Fig5 />

    <h3>Freshness: a stale index is the "outdated context" failure</h3>
    <table>
        <thead><tr><th>Freshness need</th><th>Mechanism</th><th>Why</th></tr></thead>
        <tbody>
            <tr><td>Edited / deleted docs</td><td>Incremental upsert · change-data-capture</td><td>Only re-embed what changed; delete removed chunks</td></tr>
            <tr><td>New embedding model</td><td>Full re-embed + rebuild (Module 1)</td><td>Spaces are incompatible — no partial path</td></tr>
            <tr><td>Volatile facts (prices, stock)</td><td>TTLs · prefer live structured fetch</td><td>Don't index data that's wrong minutes later</td></tr>
            <tr><td>Stale chunk lingering</td><td>Index invalidation on source change</td><td>A stale index <em>is</em> Lesson 4's outdated-context bug</td></tr>
        </tbody>
    </table>
    <div class="myth">
        <div class="m">&#10007; "Index it once at launch and you're done."</div>
        <div class="t">&#10003; Sources change constantly. Without incremental updates and invalidation, the store serves deleted docs and old prices — confident, fluent, wrong.</div>
    </div>

    <h3>Where the tokens and dollars go</h3>
    <table>
        <thead><tr><th>Cost line</th><th>Roughly equals</th><th>Lever to pull</th></tr></thead>
        <tbody>
            <tr><td><b>LLM generation</b></td><td>(input + output tokens) &times; price</td><td>Select fewer-but-better chunks; trim the prompt</td></tr>
            <tr><td><b>Query embedding</b></td><td>query tokens &times; embed price</td><td>Cheaper / smaller embedding model</td></tr>
            <tr><td><b>ANN search</b></td><td>index compute + latency</td><td>Tune recall vs latency; right index family</td></tr>
            <tr><td><b>Rerank</b></td><td>top-n candidates &times; rerank price</td><td>Rerank only the top-n, not all candidates</td></tr>
            <tr><td><b>Corpus embedding</b></td><td>corpus tokens &times; embed price (one-off / on re-index)</td><td>Batch (not real-time); avoid needless re-embeds</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact"><b>Is</b> Per-call cost &asymp; (input + output tokens) &times; model price, plus the embedding and ANN-search latency the retrieval step adds before the model ever runs. Tokens are the bill; latency is the user-facing tax.</div>
        <div class="fact"><b>Why it exists</b> Most cost hides in the input side — stuffing the window with marginal chunks pays for tokens that don't improve the answer. Liu et al. show more isn't better; fewer-but-relevant beats more.<sup><a href="#s4" id="r7">4</a></sup></div>
        <div class="fact blue"><b>Like (world)</b> A restaurant bill: the entrée (LLM tokens) dominates, but the sides (embed, rerank) add up. You cut the bill by ordering less of what you won't eat, not by switching restaurants.</div>
        <div class="fact blue"><b>Like (code)</b> An N+1 query: the fix isn't a faster DB, it's fetching less, smarter. Prompt caching (Lesson 9) is the equivalent of memoizing the stable prefix so you stop paying for it every call.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Stuff more context in — tokens are cheap and it can't hurt."</div>
        <div class="t">&#10003; Tokens are the bill, and noise lowers accuracy (Lesson 1). Select less but better: cheaper, faster, <em>and</em> more accurate at once.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> <span>Indexing is a pipeline, not a script</span> — a stale index = wrong answers; <span>tokens are the bill</span>, so select less but better.</div>

    <MemoryCheck items={mcM3} />

    <details class="iq">
        <summary>Your RAG feature's cost-per-query is 3&times; the budget. Walk through how you'd bring it down.</summary>
        <div class="a">
            <b>Hit these points:</b> first attribute the cost — break it into LLM input, LLM output, query embed, ANN search, rerank; the input tokens usually dominate &rarr; cut retrieved chunks to fewer-but-better (helps accuracy too) and trim the prompt; rerank only the top-n &rarr; cache the stable prefix with prompt caching (Lesson 9) so you stop re-paying for system prompt and tool defs &rarr; drop to a cheaper embedding model or truncated Matryoshka dims if quality holds on an eval set &rarr; batch corpus embedding instead of real-time &rarr; only then consider a smaller generation model. Measure each change against an eval set so you don't trade dollars for accuracy.
        </div>
    </details>

    <details class="iq">
        <summary>How do you keep an index fresh for a docs site that updates daily, with some live-changing fields?</summary>
        <div class="a">
            <b>Hit these points:</b> split by volatility — stable prose goes through incremental ingestion (CDC / re-embed only changed pages, delete removed ones) &rarr; fast-changing fields (price, availability) shouldn't be embedded at all; fetch them live via a structured tool at query time, or TTL them &rarr; wire invalidation so a source edit triggers re-chunk/re-embed of just that doc &rarr; on an embedding-model change, run a full re-embed behind a shadow index and cut over &rarr; monitor staleness (oldest-chunk age) as an SLO, because a stale index is the silent outdated-context failure from Lesson 4.
        </div>
    </details>
