# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What are the three discovery steps a tool runs before it assembles a codebase-context window?'
      a: >-
          <b>Hit these points:</b> <b>file discovery</b> — which files are even relevant, by path/glob/tree
          &rarr; <b>symbol discovery</b> — the definitions and references of the functions/classes in play
          (an LSP/ctags-style lookup) &rarr; <b>dependency traversal</b> — what those symbols import and
          call, the surrounding graph &rarr; then <b>assembly</b> packs the slice within the token budget,
          preferring signatures over full bodies &rarr; the goal throughout: surface the smallest
          sufficient slice, the way a human navigates an unfamiliar repo.
    - level: 'core'
      q: "Contrast Cursor's and Claude Code's strategy for understanding a large repo."
      a: >-
          <b>Hit these points:</b> Cursor <b>pre-indexes</b> — at setup it chunks + embeds files and stores
          vectors; at query time it does semantic retrieval over that index &rarr; Claude Code is
          <b>read-on-demand</b> — no persistent embedding index; it fetches live via tools (grep/glob/read)
          and follows imports like a human &rarr; the trade-off is freshness vs round-trips: pre-index is
          fast recall but a snapshot that drifts; read-on-demand is always current but pays latency and
          tokens per hit &rarr; same end goal: the right symbols within the budget.
    - level: 'core'
      q: 'What is chunking, and what makes one chunk "good"?'
      a: >-
          <b>Hit these points:</b> chunking is the rule for splitting a document into retrievable units —
          you can't embed or retrieve a whole file, it's too big and too coarse &rarr; a good chunk is
          <em>complete</em> (it answers on its own) and <em>retrievable</em> (its embedding matches the
          questions it should serve) &rarr; one chunk = one retrievable unit, ideally one coherent idea
          &rarr; how you cut decides what you can ever retrieve, so the chunk grain is the retrieval grain.
    - level: 'core'
      q: 'Name the five stages of the RAG pipeline in order, and say what RAG is.'
      a: >-
          <b>Hit these points:</b> <b>Question &rarr; Retrieval &rarr; Context assembly &rarr; LLM &rarr;
          Answer</b> &rarr; RAG retrieves relevant external chunks at query time and packs them into the
          window so the model answers from <em>them</em> instead of its frozen weights &rarr; that's what
          lets it cite a source, stay current, and reach your private data &rarr; it's the productized form
          of "context is the new database query" — the LLM only ever sees what Retrieval surfaced.
    - level: 'senior'
      q: "Your embedding-indexed assistant keeps citing a function deleted last week. What's wrong and how do you fix it?"
      a: >-
          <b>Hit these points:</b> a pre-built embedding index is a snapshot — if it isn't reindexed on
          change it serves stale vectors pointing at code that no longer exists &rarr; root cause is
          index/source skew, not the model &rarr; fix with incremental reindexing on commit/save plus a
          freshness check or TTL on the index &rarr; pair embeddings with a lexical/grep pass so exact
          current identifiers are verified against the live tree &rarr; or route latency-tolerant queries
          to read-on-demand so they always hit fresh files.
    - level: 'senior'
      q: 'Walk through chunk-size trade-offs, and explain how zero-overlap fixed-size splitting silently loses facts.'
      a: >-
          <b>Hit these points:</b> too big = diluted/averaged embedding, imprecise hits, wasted tokens; too
          small = lost surrounding context and many scattered fragments &rarr; fixed-size character
          splitting cuts mid-function/mid-sentence, so chunks are incoherent &rarr; with zero overlap a fact
          that straddles a boundary ("retry limit is set" | "to 5/min") lives whole in neither chunk, so no
          chunk is a complete answer and retrieval misses it — context fragmentation &rarr; fixes:
          structure-aware splitting on function/heading bounds, modest overlap so cross-boundary facts
          survive, and contextual chunking that prepends the section heading.
    - level: 'senior'
      q: 'A RAG bot answers most questions but always whiffs on "what is the rate limit?" though the docs state it. Where do you look first?'
      a: >-
          <b>Hit these points:</b> suspect context fragmentation — the fact likely straddles a chunk
          boundary, so neither chunk is a complete retrievable answer &rarr; inspect the actual chunks
          around that sentence, not the model output &rarr; check whether the right chunk even appears in
          top-k for that query — a retrieval check, not an answer check &rarr; fixes: add overlap so the
          fact survives whole, switch to structure-aware splitting so the sentence isn't cut, apply
          contextual chunking so the chunk carries its heading &rarr; the model is innocent; the bug is
          upstream in chunking/retrieval.
    - level: 'senior'
      q: 'An embedding-only RAG returns the right document for paraphrased questions but misses when users type an exact error code. Why, and what do you add?'
      a: >-
          <b>Hit these points:</b> embeddings capture semantic similarity, so they shine on paraphrase but
          can fail to match a rare exact token — an error code or identifier sits in a sparse region where
          nearest-neighbor recall is weak &rarr; that's a recall miss, not a model failure &rarr; add
          lexical retrieval (BM25) so exact strings hit, and run it as a <em>hybrid</em> with embeddings
          &rarr; rerank the merged candidates so the best chunk rises above the cut &rarr; verify with a
          retrieval eval that includes exact-match queries, not just semantic ones — otherwise the gap stays
          invisible in answer-only vibes.
    - level: 'staff'
      q: 'At scale, how do you decide between a pre-built embedding index and read-on-demand — and when do you combine them?'
      a: >-
          <b>Hit these points:</b> read-on-demand when freshness and exact-identifier precision dominate and
          the repo changes constantly — nothing to maintain or go stale, but you pay latency + tokens per
          round-trip &rarr; pre-index when fast semantic recall across a huge corpus matters and queries are
          fuzzy ("find the code that does X" before you know the name), accepting upfront/reindex cost and
          staleness risk &rarr; the senior move is to combine: embeddings for semantic recall, lexical/live
          read to verify exact current code, hybrid (embeddings + BM25), reindex incrementally on change
          &rarr; the choice is a staleness-vs-latency trade-off, not a winner.
    - level: 'staff'
      q: 'A team says "our RAG is bad, the model hallucinates — we need a smarter model." Diagnose this as the senior in the room.'
      a: >-
          <b>Hit these points:</b> reframe — most "the LLM is dumb" bugs are retrieval bugs: garbage in,
          fluent garbage out &rarr; first instrument what was actually retrieved per failing query; teams
          grade the answer and never the recall &rarr; check the usual suspects: fixed-size chunking
          destroyed the fact, embedding-only retrieval missed exact terms, no reranking so the right doc
          sits below top-k (or lost in the middle), or a stale index / conflicting passages &rarr; fixes map
          to chunking + retrieval: structure-aware + contextual chunking, hybrid + rerank, reindex on change
          &rarr; gate on a retrieval-quality eval (hit-rate / recall@k), not end-answer vibes &rarr; only
          after retrieval is verified good is model capability the lever worth paying for.
    - level: 'staff'
      q: 'Make the case that retrieval quality matters more than model quality for a code assistant — then steelman the opposite.'
      a: >-
          <b>For:</b> the model only reasons over what it's given, so a stale index or a fragmented chunk
          yields a wrong answer at any model size; the model is a shared commodity while retrieval is your
          design, and retrieval fixes compound across features and cost far less than model upgrades.
          <b>Steelman against:</b> below a capability floor a weak model can't synthesize even a perfectly
          retrieved slice — multi-file refactors, long dependency reasoning — so there the model is the
          binding constraint. <b>Synthesis:</b> fix retrieval first; it's the common cause and the cheapest
          lever, and you can't even tell whether reasoning is the bottleneck until retrieval is verified
          good. Pay for model capability once recall@k is high and reasoning is provably the limit.
    - level: 'design'
      q: 'Design a RAG system for a help-center support bot over a changing knowledge base.'
      a: >-
          <b>A strong answer covers:</b> ingest the help center, chunk structure-aware on headings/sections
          with overlap, and apply contextual chunking so each chunk carries its article/section context
          &rarr; embed + index, and add lexical (BM25) so exact product names and error codes match &rarr;
          per question: retrieve hybrid candidates, rerank the top-N, assemble into the window with the
          question, and have the LLM answer <em>with citations</em> to the source chunk &rarr; keep the
          index fresh — reindex changed articles incrementally, TTL stale ones &rarr; name the three break
          points: chunking destroyed the fact, retrieval missed (no hybrid/rerank, buried below top-k or
          lost in the middle), stale/conflicting index &rarr; close the loop with a retrieval-hit-rate eval,
          not just end-answer quality.
    - level: 'design'
      q: 'Design the code-context layer for a coding agent over a 2M-token monorepo with a 200k window.'
      a: >-
          <b>A strong answer covers:</b> you can surface ~10% of the repo at most, so retrieval is the
          product, not the window size &rarr; index symbols + a dependency/call graph, not just flat text
          chunks &rarr; on a query, gather candidates by exact symbol match (lexical) <em>and</em> semantic
          similarity, then walk imports/call-sites for structural context &rarr; rerank, then pack within
          budget: signatures and docstrings over full bodies, the files named in the query first, and
          reserve fixed budget for history + the answer &rarr; keep it fresh — incremental re-embed/ctags on
          change, or read-on-demand for always-fresh precision on exact identifiers &rarr; instrument what
          was retrieved so you can measure hit-rate &rarr; name the trade-off explicitly: pre-indexed
          embeddings (fast, can go stale) vs read-on-demand (fresh, more round-trips).

                <td>Zero overlap</td>
                <td>Boundary-spanning facts fragment and become unretrievable.</td>
                <td>Modest overlap so cross-boundary facts survive.</td>
            </tr>
            <tr>
                <td>Ignoring document structure</td>
                <td>A chunk lacks the heading/section it belongs to; ambiguous alone.</td>
                <td>
                    <b>Contextual chunking:</b> prepend a short doc/section summary to each chunk so it
                    self-describes.
                </td>
            </tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact">
            <b>Is</b> Chunking is the rule for splitting a document into retrievable units. A good chunk is
            <em>complete</em> (answers on its own) and <em>retrievable</em> (its embedding matches the
            questions it should serve).
        </div>
        <div class="fact">
            <b>Why it exists</b> Files exceed embedding input limits, and retrieval needs granularity — you
            want the relevant passage back, not a whole file. Chunking is the unit of both storage and
            recall.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> Index cards from a book. Cut them on chapter/section lines and each card
            stands alone; cut every 500 letters and you get cards that start mid-sentence and answer nothing.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> Choosing a primary key / row grain in a table: too coarse and a query returns
            bloated rows; too fine and you must join many rows to answer. Chunk grain is the retrieval grain.
        </div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "Just split every N characters — simple and good enough."</div>
        <div class="t">
            &#10003; Fixed-size splits cut mid-thought and fragment facts. Split on meaning (function/heading
            bounds) and add overlap so each chunk is complete and retrievable.
        </div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Smaller chunks are always more precise."</div>
        <div class="t">
            &#10003; Too small strips the surrounding context that made the passage meaningful — and
            multiplies fragments. Precision needs the right grain plus enough context, not minimum size.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span> <span>Chunk on meaning, not character count.</span> A chunk
        should be <span>retrievable and complete on its own.</span>
    </div>

    <MemoryCheck items={mc2} />

    <details class="iq">
        <summary>
            A RAG bot answers most questions but always whiffs on "what is the rate limit?" even though the
            docs clearly state it. Where do you look first?
        </summary>
        <div class="a">
            <b>Hit these points:</b> suspect context fragmentation — the fact ("limit is set to 5/min")
            likely straddles a chunk boundary, so neither chunk is a complete, retrievable answer &rarr;
            inspect the actual chunks around that sentence, not the model output &rarr; fixes: add overlap so
            the fact survives whole; switch to structure-aware splitting so the sentence isn't cut; apply
            contextual chunking so the chunk carries its section heading &rarr; verify by checking whether
            the right chunk now appears in top-k for that query — a retrieval check, not an answer check.
        </div>
    </details>

</section>

<section class="lvl" id="m3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>RAG — From First Principles</h2>
    </div>
    <p class="tagline">
        <b>Retrieval-Augmented Generation</b> is Lesson 1's idea, shipped: take a question, <em>retrieve</em>
        the relevant chunks, <em>assemble</em> them into the window, and let the LLM answer — grounded in
        <em>your</em> data instead of its frozen weights.
    </p>

    <h3>The pipeline</h3>

    <Fig5 />

    <h3>Why RAG exists vs the alternatives</h3>
    <table>
        <thead>
            <tr>
                <th>Need</th>
                <th>Stuff it all in the window</th>
                <th>Fine-tune the model</th>
                <th>RAG</th>
            </tr>
        </thead>
        <tbody>
            <tr>
                <td>Private / current data</td>
                <td>Doesn't fit; truncated</td>
                <td>Stale at next data change</td>
                <td>&#10003; fresh, pulled per query</td>
            </tr>
            <tr>
                <td>Citations / provenance</td>
                <td>Buried in noise</td>
                <td>&#10007; weights can't cite</td>
                <td>&#10003; returns the source chunk</td>
            </tr>
            <tr>
                <td>Cost to update</td>
                <td>Pay per token every call</td>
                <td>Retrain run each time</td>
                <td>&#10003; reindex changed docs only</td>
            </tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact">
            <b>Is</b> RAG is the pattern of retrieving relevant external text at query time and putting it in
            the context window so the model answers from it — the productized form of "context is the new
            database query."
        </div>
        <div class="fact">
            <b>Why it exists</b> To ground the model in private, current, authoritative data; cut
            hallucination; enable citations; and do it cheaper and fresher than fine-tuning — without
            stuffing everything into a fixed window.
        </div>
        <div class="fact blue">
            <b>Like (world)</b> An open-book exam. The student (LLM) is fixed, but pulling the right page
            before answering changes the grade. RAG is the "go fetch the right page" step.
        </div>
        <div class="fact blue">
            <b>Like (code)</b> A read-through cache in front of a database: on each request you fetch the few
            relevant records and hand them to the handler. Retrieval is that fetch; the LLM is the handler.
        </div>
    </div>

    <h3>Why so many RAG systems are poor</h3>
    <table>
        <thead>
            <tr>
                <th>The naive default</th>
                <th>What it costs</th>
                <th>The fix (maps to Module 2)</th>
            </tr>
        </thead>
        <tbody>
            <tr>
                <td>Fixed-size chunking</td>
                <td>Fragmented, incoherent chunks</td>
                <td>Structure-aware + contextual chunking</td>
            </tr>
            <tr>
                <td>Embedding-only retrieval</td>
                <td>Misses exact keywords/IDs</td>
                <td>Hybrid: embeddings + BM25 lexical</td>
            </tr>
            <tr>
                <td>Top-k, no reranking</td>
                <td>Right doc buried below the cut</td>
                <td>Rerank so the best chunk rises to top</td>
            </tr>
            <tr>
                <td>No retrieval evaluation</td>
                <td>You measure the answer, never the recall</td>
                <td>Eval retrieval hit-rate, not just vibes</td>
            </tr>
        </tbody>
    </table>
    <div class="pull">
        Read the middle column. Every failure is upstream of the model: bad chunks, blind retrieval, no
        rerank, no measurement. Teams grade the <em>answer</em> and never check whether the right document
        was even retrieved &mdash; so they "fix the LLM" and nothing improves.
    </div>

    <div class="myth">
        <div class="m">&#10007; "RAG gave a wrong answer, so we need a smarter model."</div>
        <div class="t">
            &#10003; Usually retrieval missed: the model got nothing or the wrong passage and answered
            confidently anyway. Garbage in &rarr; fluent garbage out. Fix recall before model.
        </div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Embeddings + top-k is a complete RAG system."</div>
        <div class="t">
            &#10003; That's the naive baseline. Without hybrid search and reranking, the relevant doc often
            sits below the cut (and "lost in the middle" even when included) — so it never reaches the model
            usefully.
        </div>
    </div>

    <div class="rule">
        <span>&#128229; Memory rule:</span> <span>RAG is only as good as its retrieval.</span> Most
        <span>"the LLM is dumb" bugs are retrieval bugs.</span>
    </div>

    <MemoryCheck items={mc3} />

    <details class="iq">
        <summary>
            Sketch a docs-Q&amp;A or support assistant as RAG, then name the three places it most often
            breaks in production.
        </summary>
        <div class="a">
            <b>Hit these points:</b> pipeline — chunk the help center / docs, embed + index, then per
            question retrieve top chunks, assemble into the window with the question, LLM answers with
            citations &rarr; break 1: chunking destroyed the fact (fragmentation / fixed-size cuts) &rarr;
            break 2: retrieval missed — embedding-only with no hybrid/rerank, so the right doc is buried
            below top-k or lost in the middle &rarr; break 3: stale index — the doc changed but wasn't
            reindexed, or conflicting passages confuse the model &rarr; in all three the model is innocent;
            the fix is upstream, and you must <em>measure retrieval hit-rate</em>, not just answer quality.
        </div>
    </details>
