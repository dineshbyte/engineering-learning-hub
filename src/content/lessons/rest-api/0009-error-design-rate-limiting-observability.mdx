# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is the current standard for HTTP API error bodies?'
      a: >-
          <p><strong>RFC 9457, "Problem Details for HTTP APIs"</strong> — media type
          <code>application/problem+json</code>. It defines a single, consistent shape for error
          responses across an API. Naming the RFC number (not just "problem+json") is the senior tell;
          it <strong>supersedes RFC 7807</strong>, so cite the right one in the room.</p>
    - level: 'core'
      q: '<code>400</code> vs <code>422</code> — when each?'
      a: >-
          <p><code>400 Bad Request</code> for a <strong>syntactically malformed</strong> request —
          broken JSON, missing required structure, can't even parse it. <code>422 Unprocessable
          Content</code> for a request that <strong>parsed fine but is semantically invalid</strong> —
          e.g. valid JSON whose <code>email</code> field fails validation. Parsed-but-wrong is the
          <code>422</code> signal.</p>
    - level: 'core'
      q: "What's the HTTP contract for a throttled request?"
      a: >-
          <p><code>429 Too Many Requests</code> with a <code>Retry-After</code> header telling the
          client when to come back. Ideally also expose <code>RateLimit-Limit</code> /
          <code>RateLimit-Remaining</code> / <code>RateLimit-Reset</code> on <em>every</em> response so
          well-behaved clients self-throttle <em>before</em> they hit the wall.</p>
    - level: 'core'
      q: 'What are the RED metrics?'
      a: >-
          <p><strong>R</strong>ate (requests/sec), <strong>E</strong>rrors (failed requests),
          <strong>D</strong>uration (latency). Tracked per endpoint, they're the minimal dashboard that
          tells you whether a service is healthy — and the first thing to pull up when something's
          wrong.</p>
    - level: 'senior'
      q: 'Why must clients branch on error codes, not error prose?'
      a: >-
          <p>Prose is for humans and changes freely; codes are part of your contract and must stay
          stable. The moment you reword a string and a partner's <code>if</code>-statement breaks,
          you've taught them to never trust your contract again. So: <strong>stable codes, human
          prose</strong> — and document the codes as an error catalog they can code against.</p>
    - level: 'senior'
      q: 'Why does consistent status mapping matter beyond aesthetics?'
      a: >-
          <p>Because the whole HTTP ecosystem acts on the status: caches, retry logic, circuit
          breakers, monitoring, and clients all decide behavior from it. Map a transient overload to
          <code>400</code> and clients won't retry; map a validation error to <code>500</code> and you
          page yourself for client mistakes and inflate your error budget. The status class (4xx =
          "you," 5xx = "me") drives <em>who retries</em> and <em>who gets paged</em>.</p>
    - level: 'senior'
      q: "Token bucket — how it works and why it's the common default?"
      a: >-
          <p>Tokens refill at a steady rate; each request spends one; the bucket size caps how many can
          accumulate. It <strong>allows bursts up to the bucket size, then settles to the steady refill
          rate</strong>. It's the common choice because real traffic is spiky-but-bounded — token
          bucket tolerates a legitimate spike while still enforcing a sustained average.</p>
    - level: 'senior'
      q: 'Why report latency as p95/p99 instead of the mean?'
      a: >-
          <p>Because the mean hides the tail. A p50 of 50ms can sit alongside a p99 of 4s — and that
          p99 is exactly the slow experience losing you the partner. Averages get dragged around by
          outliers and conceal them at once. Percentiles describe what real users actually feel; judge
          and alert on <strong>p95/p99</strong>, never the mean.</p>
    - level: 'staff'
      q: 'How do you enforce a global rate limit across many distributed API nodes?'
      a: >-
          <p>The naive per-node counter lets the real limit balloon by N nodes. Options, in tension: a
          <strong>shared store</strong> (Redis with atomic INCR/Lua, or a token-bucket script) gives an
          accurate global count but adds a hop and a dependency on every request; <strong>local
          counters with a per-node share</strong> are fast but approximate and unfair under skew;
          <strong>sampled/sync'd local buckets</strong> trade a little accuracy for latency. Most real
          systems do limiting at the <strong>gateway/edge</strong> with a shared store, accepting
          eventual-consistency slop near the boundary.</p>
    - level: 'staff'
      q: 'How do rate limiting, retries, idempotency, and circuit breakers fit together as one story?'
      a: >-
          <p>They're a closed loop. <strong>Rate limits/outages</strong> trigger <strong>retries</strong>;
          retries need <strong>backoff + jitter</strong> so they don't stampede; retries on writes need
          <strong>idempotency keys</strong> so they don't duplicate; a persistently failing dependency
          needs a <strong>circuit breaker</strong> so you stop retrying into the void and <strong>degrade
          gracefully</strong>; and <strong>observability</strong> (correlation ids, RED, p99, traces) is
          how you see the whole loop and tune it. Naming the loop, not just one piece, is the staff
          signal.</p>
    - level: 'staff'
      q: 'Tracing every request is too expensive at scale. How do you sample without going blind?'
      a: >-
          <p>Mix strategies. <strong>Head-based sampling</strong> (decide at ingress, e.g. keep 1%) is
          cheap but may drop the rare failure you needed. <strong>Tail-based sampling</strong> buffers
          spans and keeps traces that are <em>interesting</em> — errors, high latency — at the cost of
          more infrastructure. Keep <strong>100% of error/slow traces</strong> and sample the boring
          success path; keep full-fidelity <em>metrics</em> always so aggregates stay accurate even when
          individual traces are sampled out.</p>
    - level: 'design'
      q: >-
          Design the error contract, abuse protection, and observability for a public write-heavy REST
          API a partner says is "impossible to integrate against and keeps falling over."
      a: >-
          <p><b>A strong answer covers:</b> diagnose the two complaints separately — "impossible to
          integrate" is an error-contract/DX problem, "keeps falling over" is an abuse/resilience
          problem &rarr; <strong>error contract</strong>: adopt one <strong>RFC 9457
          <code>problem+json</code></strong> shape across every endpoint, add a <strong>stable
          machine-readable code</strong> as an extension member (clients branch on the code, never the
          prose), emit <strong>field-level validation errors</strong>, map status correctly
          (<code>400</code> malformed vs <code>422</code> validation, <code>409</code> conflict,
          <code>401</code>/<code>403</code>), leak no stack traces/SQL, and publish an <strong>error
          catalog</strong> enforced by contract tests in CI &rarr; <strong>abuse protection</strong>:
          <strong>token bucket per API key</strong> (absorbs legitimate bursts, enforces a sustained
          average) with separate burst vs sustained quotas tied to billing tiers, return
          <code>429</code> + <code>Retry-After</code> and expose <code>RateLimit-*</code> headers so
          good clients self-throttle; enforce at the gateway with a shared store (Redis atomic
          INCR/Lua) accepting eventual-consistency slop; add load shedding (<code>503</code> +
          <code>Retry-After</code>) before you fall over &rarr; <strong>safe recovery</strong>: require
          <strong>idempotency keys</strong> on writes so retried <code>POST</code>s don't double-charge,
          advise <strong>exponential backoff + jitter</strong> (and vary <code>Retry-After</code>
          server-side) to avoid a thundering herd, and circuit-break failing downstreams &rarr;
          <strong>observability</strong>: structured logs keyed by a propagated correlation id (W3C
          <code>traceparent</code>), RED metrics per endpoint, OpenTelemetry traces, alert on p95/p99
          and SLO error-budget burn (not the mean), and split dashboards by status class so partner 4xx
          doesn't pollute server 5xx signals &rarr; name the trade-off: a shared-store global limiter is
          accurate but adds a hop/dependency vs fast-but-approximate local counters, and tail-based
          trace sampling keeps the failures you need at the cost of more infrastructure.</p>
    - level: 'design'
      q: >-
          Design the end-to-end resilience and observability story for a payments-style API where
          retries must never double-charge and outages must degrade gracefully.
      a: >-
          <p><b>A strong answer covers:</b> treat resilience as a closed loop, not isolated features
          &rarr; <strong>idempotency</strong> is the backbone: every write carries an
          <code>Idempotency-Key</code>; the server stores key &rarr; result on first success and returns
          the stored response on replay, with a lock/"in progress" record for the concurrent in-flight
          retry and a sensible key-expiry window — this makes a non-idempotent charge safe to retry
          &rarr; <strong>retry discipline</strong>: clients use exponential backoff + jitter so a
          recovering service isn't stampeded; the server signals retryability honestly —
          <code>429</code> + <code>Retry-After</code> for throttling, <code>503</code> +
          <code>Retry-After</code> for temporary overload, and never dresses a client error as a
          retryable <code>5xx</code> &rarr; <strong>circuit breakers</strong> around the payment
          processor and other dependencies: open on a failure threshold to fail fast, half-open to test
          recovery, so a struggling downstream gets room instead of being retried to death &rarr;
          <strong>graceful degradation</strong>: load-shed excess early, serve stale-but-cached
          non-critical data, disable optional features, and prioritize the core charge path; decide in
          advance what's droppable &rarr; <strong>error contract</strong>: RFC 9457
          <code>problem+json</code> with stable codes (e.g. <code>card_declined</code>) clients can
          branch on, correct status mapping, zero leakage of internal detail &rarr;
          <strong>observability</strong>: correlation id propagated across every hop, structured logs
          that <em>never</em> log PANs/secrets/PII (redact at the boundary), RED metrics, OpenTelemetry
          traces to see where time/failures land, and paging tied to SLO error-budget burn against the
          user-facing SLI rather than raw error counts &rarr; name the trade-off: idempotency adds
          storage + a write-path lookup and forces you to define exactly-once windows, but it's the only
          honest way to make money-moving retries safe.</p>

## Error design: one machine-readable contract

<p>
    The fastest way to make an API "impossible to integrate against" is inconsistent errors — a bare
    <code>500</code> here, a <code>200</code> with <code>{'{"ok":false}'}</code> there, a different
    shape per endpoint. The senior move is{' '}
    <strong>one consistent, machine-readable error contract across the entire API</strong>. The
    standard is <strong>RFC 9457, "Problem Details for HTTP APIs"</strong> — media type{' '}
    <code>application/problem+json</code> — which supersedes RFC 7807.
    <sup><a href="#s1" id="r1">1</a></sup>
</p>

<Fig1 />

{/* 2 · STATUS CODE ROUTING */}

<p>
    Pick the <strong>correct status code</strong> — this ties back to Lesson 2 — and never tunnel
    errors through <code>200</code>; the status line is part of your contract and breaks every
    intermediary that reads it.
    <sup><a href="#s3" id="r3">3</a></sup>
</p>

    <div class="t">
        ✓ Truth: clients branch on a stable <code>code</code>; the prose is for humans and can be
        reworded freely.

<p class="pull">
    <strong>Clients branch on your error codes, not your error prose</strong> — so make the codes
    stable and the prose human. The moment you reword a string and a partner's <code>if</code>
    -statement breaks, you've taught them to never trust your contract again.
</p>

## Rate limiting: protect availability and cost

<p>
    Without limits, one client — buggy retry loop, scraper, or attacker — can exhaust your capacity
    or your cloud bill. This is OWASP <strong>API4:2023 Unrestricted Resource Consumption</strong>,
    a top-ten risk for a reason.
    <sup><a href="#s2" id="r2">2</a></sup>{' '}
    The contract for a throttled request: <code>429 Too Many Requests</code> with{' '}
    <code>Retry-After</code>, plus <code>RateLimit-*</code> headers on <em>every</em> response so
    well-behaved clients self-throttle <em>before</em> they hit the wall.
</p>

<Fig3 />

<div class="facts">
    <div class="fact">
        <b>Fixed window</b> Count per calendar window. Trivial, but allows a <em>boundary burst</em>{' '}
        — up to 2× the limit straddling the window edge.
    </div>
    <div class="fact">
        <b>Sliding window</b> Rolling time window (weighted log/counter). Smooths the boundary burst
        at modest extra state cost.
    </div>
    <div class="fact blue">
        <b>Token bucket</b> Tokens refill at a steady rate; a request spends one; the cap bounds the
        burst. The common choice — tolerates spiky-but-bounded traffic.
    </div>
    <div class="fact blue">
        <b>Leaky bucket</b> Requests queue and drain at a fixed rate. Smooths <em>output</em> to a
        constant rate; protects a downstream that hates spikes.
    </div>
</div>

<p>
    Limit per <strong>principal</strong> — API key, account, or user — so one tenant can't starve
    the rest; fall back to <strong>per-IP</strong> only for unauthenticated traffic. And separate
    <strong>burst</strong> from <strong>sustained</strong> quotas (e.g. 100/sec burst, 10k/hour
    sustained) so a legitimate spike isn't punished like sustained abuse.
</p>

{/* 4 · OBSERVABILITY TRIAD */}

## Observability: see what actually broke

<p>
    The three pillars are <strong>logs</strong> (structured/JSON, not free-text),
    <strong>metrics</strong>, and <strong>traces</strong>. The connective tissue is a
    <strong>correlation / request id</strong> — W3C <code>traceparent</code> or{' '}
    <code>X-Request-Id</code> — propagated across every service so one request is traceable end to
    end. Without it, a distributed bug is a needle in N haystacks.
</p>

<Fig4 />

<p>
    Track <strong>RED metrics</strong> per endpoint — Rate, Errors, Duration — and measure duration
    as
    <strong>latency percentiles</strong> (p50/p95/p99), never averages: an average of 50ms hides the
    p99 of 4s that's losing you the partner.
</p>

<div class="rule">
    <span>🛡️ Memory rule:</span> one <span>problem+json</span> contract (stable code, human prose,
    no leaks) · throttle per principal with <span>token bucket → 429 + Retry-After</span> · see it
    via logs + RED metrics + traces stitched by one <span>request id</span> · keep retries safe with
    idempotency keys.
</div>

<p>
    Tie it back to retries: rate limits and outages <em>cause</em> client retries, and retries on
    writes cause duplicates — which is exactly why <strong>idempotency keys</strong> (Lesson 3) are
    the safety net under all of this. Observability tells you it's falling over; idempotency keeps
    the recovery from corrupting state.
</p>
