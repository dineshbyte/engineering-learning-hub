# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'Define a <em>safe</em> and an <em>idempotent</em> HTTP method.'
      a: >-
          <p><strong>Safe</strong> = read-only <em>intent</em>, no intended state change
          (<code>GET</code>, <code>HEAD</code>, <code>OPTIONS</code>). <strong>Idempotent</strong> = N
          identical requests have the same effect on server state as one (<code>GET</code>,
          <code>HEAD</code>, <code>OPTIONS</code>, <code>PUT</code>, <code>DELETE</code>). The key word
          is <em>effect on state</em>, not a byte-identical response.</p>
    - level: 'core'
      q: 'Is <code>POST</code> idempotent? Why does it matter?'
      a: >-
          <p>No. A retried <code>POST</code> can create a duplicate (a second charge or order). It
          matters because dropped connections and retries are normal in distributed systems —
          non-idempotency is exactly why <code>POST</code> creation needs an
          <code>Idempotency-Key</code> to be retry-safe.</p>
    - level: 'core'
      q: '<code>200</code> vs <code>201</code> vs <code>204</code>, and what must a <code>201</code> include?'
      a: >-
          <p><code>200 OK</code>: success with a body. <code>201 Created</code>: a new resource was
          created and <strong>must carry a <code>Location</code></strong> header to it. <code>204 No
          Content</code>: success with no body (common for <code>DELETE</code> and for
          <code>PUT</code>/<code>PATCH</code> that don't echo the resource).</p>
    - level: 'core'
      q: "<code>401</code> vs <code>403</code> — what's the difference?"
      a: >-
          <p><code>401 Unauthorized</code> = <strong>authentication</strong> failed ("we don't know
          who you are" — credentials missing/invalid/expired; re-authenticate). <code>403
          Forbidden</code> = <strong>authorization</strong> failed ("we know who you are, you're not
          allowed" — re-authenticating won't help).</p>
    - level: 'senior'
      q: 'Explain "safe &sub; idempotent," and why "safe means secure" is false.'
      a: >-
          <p>Every safe method is idempotent (reading changes nothing, so repeating it changes
          nothing); the reverse fails — <code>PUT</code>/<code>DELETE</code> are idempotent but not
          safe. "Safe" is about read-only <em>intent</em>, nothing about authn/authz/TLS: a
          <code>GET</code> can still expose sensitive data and still needs authorization and transport
          security. Keep safe / idempotent / cacheable as three separate axes.</p>
    - level: 'senior'
      q: 'Is <code>PATCH</code> idempotent, and how do JSON Merge Patch vs JSON Patch relate?'
      a: >-
          <p>Not guaranteed — it depends on the patch document. A "set field to X" patch is
          idempotent; "increment by 1" / "add to array" is not. <strong>JSON Merge Patch (RFC
          7386)</strong> sets values (<code>null</code> = remove), so it tends to be idempotent;
          <strong>JSON Patch (RFC 6902)</strong> is an op sequence where <code>add</code> isn't. Make
          <code>PATCH</code> idempotent when you can, and know why a given patch isn't.</p>
    - level: 'senior'
      q: 'Why prefer <code>307</code>/<code>308</code> over <code>302</code>/<code>301</code>, and why is <code>304</code> a feature?'
      a: >-
          <p><code>307</code>/<code>308</code> <strong>preserve the method and body</strong>; with
          <code>301</code>/<code>302</code> clients historically rewrote a <code>POST</code> into a
          <code>GET</code>, dropping the body. <code>304 Not Modified</code> is the payoff of
          conditional GET: the client revalidates with
          <code>If-None-Match</code>/<code>If-Modified-Since</code> and, if nothing changed, gets
          <code>304</code> with no body and reuses its cache — bandwidth saved, not a failure.</p>
    - level: 'senior'
      q: 'When should a forbidden resource return <code>404</code> instead of <code>403</code>?'
      a: >-
          <p>When the very <em>existence</em> of the resource is sensitive. A <code>403</code> confirms
          "this exists, you just can't see it," which leaks information (resource enumeration — e.g.
          private repo names). <code>404</code> hides existence entirely. Trade-off: <code>404</code>
          is harder to debug for legitimate users, so reserve it for genuinely sensitive resources,
          not as a blanket policy.</p>
    - level: 'staff'
      q: 'A teammate puts "mark as read" behind <code>GET /messages/42?read=true</code>. What breaks?'
      a: >-
          <p>It makes a state-changing operation safe-in-name only, and the ecosystem assumes
          <code>GET</code> is safe: prefetchers, crawlers, link-preview bots, and proxies fire it and
          silently mutate state, and caches may serve stale results. Intended state change belongs on
          a non-safe verb — <code>POST</code>, or <code>PATCH</code>/<code>PUT</code> for an idempotent
          update. The classic casualty was an admin panel where a crawler followed every "delete"
          link.</p>
    - level: 'staff'
      q: "A downstream dependency is timing out under load. Walk the status codes and behavior you'd return."
      a: >-
          <p>If a circuit breaker is open / you're shedding load, return <code>503</code> +
          <code>Retry-After</code> so clients back off. If you actually called upstream and it timed
          out, <code>504</code> is honest; if it returned garbage, <code>502</code>. Don't let
          upstream's timeout surface as a <code>500</code> for a valid client request, and never
          convert it into a <code>4xx</code> — the client did nothing wrong. Combine with idempotency
          keys so the inevitable retries don't double-act.</p>
    - level: 'staff'
      q: 'Status code as the contract, or body error codes — how do you reconcile them?'
      a: >-
          <p>Use both at the right layer. The HTTP status is the <strong>coarse, machine-actionable
          verdict</strong> that infrastructure (caches, LBs, retry logic, alerting) acts on — get it
          right first. The body carries a <strong>stable, fine-grained error code</strong> plus
          human/field detail for the application to branch on (e.g. <code>"code":
          "card_declined"</code> alongside a <code>402</code>/<code>422</code>). Status for the
          network, body codes for the application — not one instead of the other.</p>
    - level: 'design'
      q: 'Design the request/response contract for <code>POST /payments</code> where the connection can drop mid-flight. Walk the methods and status codes.'
      a: >-
          <p><b>A strong answer covers:</b> <code>POST</code> is correct because the server assigns the
          payment id &rarr; sync path returns <code>201 Created</code> + <code>Location</code>; async
          path returns <code>202 Accepted</code> + a status URL to poll, never <code>200</code> for
          unfinished work &rarr; the dropped connection is the hard part: <code>POST</code> isn't
          idempotent, so a client-generated <code>Idempotency-Key</code> lets a retry replay the
          original result instead of double-charging &rarr; map the failure modes to honest codes:
          <code>422</code> for validation, <code>409</code> for a state conflict,
          <code>402</code>/body error code for a declined card,
          <code>503</code>+<code>Retry-After</code> when shedding load, <code>504</code> if upstream
          timed out &rarr; never tunnel an error through <code>200</code>, never return
          <code>5xx</code> for bad client input &rarr; name the contract split: status code for the
          network/infra, stable body error code for the application.</p>
    - level: 'design'
      q: 'Design the update + concurrency story for a resource many clients edit at once (PUT vs PATCH, conditional requests).'
      a: >-
          <p><b>A strong answer covers:</b> choose <code>PUT</code> (full replace, idempotent by
          construction) vs <code>PATCH</code> (partial edit) by resource size and whether "set to
          null" must differ from "leave untouched" &rarr; for <code>PATCH</code>, pick the format
          deliberately — JSON Merge Patch (value-setting, tends to be idempotent) vs JSON Patch (op
          sequence, <code>add</code> isn't idempotent) &rarr; guard lost updates with optimistic
          concurrency: hand out an <code>ETag</code> on reads, require <code>If-Match</code> on writes,
          return <code>412 Precondition Failed</code> on a stale version &rarr; distinguish
          <code>412</code> ("your version is stale") from <code>409</code> ("this action conflicts with
          current state") &rarr; make retries safe: idempotent verbs retry freely, and a
          turned-into-explicit conflict is retryable after a re-read &rarr; name the trade-off:
          <code>PUT</code> is simpler and retry-safe but bandwidth-heavy; <code>PATCH</code> saves
          bytes but adds format and idempotency nuance.</p>

<div class="facts">
    <div class="fact">
        <b>201 vs 202</b> 201 means done and created (with <code>Location</code>); 202 means
        accepted for async processing — <em>not done yet</em>, hand back a status URL to poll.
    </div>
    <div class="fact">
        <b>304 is a feature</b> Not Modified powers conditional GET / revalidation. It's a cache
        win, not an error.
    </div>
    <div class="fact blue">
        <b>405 vs 409 vs 410</b> 405 Method Not Allowed adds an <code>Allow</code> header; 409 for a
        state conflict; 410 Gone for permanently removed.
    </div>
    <div class="fact blue">
        <b>429 + Retry-After</b> Too Many Requests should tell the client when to come back — same
        courtesy as 503.
    </div>
</div>

<div class="rule">
    <span>🎯 Memory rule:</span> Three axes (safe ⊂ idempotent, cacheable apart); PUT vs POST = who
    owns the URI; pick the <em>exact</em> code — 422 not 400 for valid-but-invalid, 401 vs 403 for
    authn vs authz, 404 to hide existence — and never tunnel errors through a 200.
</div>
