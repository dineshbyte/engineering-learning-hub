# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What does it mean for an operation to be idempotent?'
      a: >-
          <p>Making the same request <em>N</em> times has the <strong>same effect on server
          state</strong> as making it once. The key word is <em>state</em>, not response: it's about
          what's left behind, not what comes back.</p>
    - level: 'core'
      q: 'How do you make a non-idempotent <code>POST</code> safe to retry?'
      a: >-
          <p>Bolt on the <strong>Idempotency-Key pattern</strong>: the client sends a unique key per
          logical operation in an <code>Idempotency-Key</code> header; the server stores key →
          outcome on first execution and <em>replays</em> the stored response on any retry carrying
          the same key. Stripe is the canonical implementation.</p>
    - level: 'core'
      q: 'Who generates the idempotency key, and why?'
      a: >-
          <p>The <strong>client</strong> — typically a <code>UUID v4</code>, one key per <em>logical
          operation</em>, reused across every retry of that intent. Only the client knows that "this
          retry" and "the first attempt" are the same intent; the server can't infer it from two
          byte-identical requests.</p>
    - level: 'core'
      q: 'Should you apply idempotency keys to every endpoint?'
      a: >-
          <p>No — only to <strong>unsafe, non-idempotent</strong> operations: payments, order
          creation, message sends, transfers. Reads and naturally idempotent writes
          (<code>GET</code>, <code>PUT</code>, <code>DELETE</code>) don't need the machinery; adding
          it everywhere is wasted storage and complexity.</p>
    - level: 'senior'
      q: 'Is "idempotent" the same as "returns the same response"?'
      a: >-
          <p>No — a classic trap. Idempotency is about <strong>effect on state</strong>, not the
          response body or status. A repeated <code>DELETE</code> is idempotent even though the first
          returns <code>204</code> and the second may return <code>404</code> — the resource is
          absent either way. Likewise "no side effects" is wrong: a <code>GET</code> may still log or
          update analytics.</p>
    - level: 'senior'
      q: 'Two requests arrive concurrently with the same key — what happens?'
      a: >-
          <p>They must <strong>not both execute</strong>. Guard the key with a <strong>unique
          constraint</strong> or a lock: the first wins and writes; the loser either <em>waits</em>
          for the in-flight result, or gets a <code>409 Conflict</code> ("operation in progress").
          This concurrency case — not the header — is where the interview actually lives.</p>
    - level: 'senior'
      q: 'Why fold the key-write into the same transaction as the effect?'
      a: >-
          <p>So the effect and the record-of-the-effect commit <strong>atomically</strong>. If
          they're separate, a crash between "did the work" and "saved the key" leaves you having
          executed without a record — the next retry re-executes and duplicates. One transaction
          makes "executed" and "remembered" inseparable.</p>
    - level: 'senior'
      q: 'Does idempotency give you exactly-once delivery?'
      a: >-
          <p>No — say this out loud, it's the differentiator. <strong>Exactly-once <em>delivery</em>
          is impossible</strong> in a distributed system. Idempotency gives you exactly-once
          <strong>effect</strong>: the network may deliver five times, but the operation takes effect
          once. The equation: exactly-once effect = at-least-once delivery + idempotent processing
          (dedup on the receiver).</p>
    - level: 'staff'
      q: 'Should the idempotency layer live in the gateway or the service?'
      a: >-
          <p>A gateway/middleware layer is clean for caching the stored response and short-circuiting
          retries cheaply, but it can't atomically fold the key-write into the <em>business</em>
          transaction. The strongest guarantee comes from writing the key in the <strong>same
          transaction as the effect</strong>, which forces it into the service/data layer. Common
          compromise: gateway handles fast-path replay and the concurrency gate, the service owns the
          transactional write of key+effect. Name the trade-off rather than asserting one is
          "correct."</p>
    - level: 'staff'
      q: 'The process crashes after the effect commits but before responding. What does the next retry see?'
      a: >-
          <p>If key+effect were written in <strong>one transaction</strong>, the retry hits a stored
          key and replays the original response — correct. If they were separate and only the effect
          committed, the retry sees a <em>miss</em> and re-executes — duplicate. This is exactly why
          atomic key+effect persistence is the load-bearing detail, not an optimization.</p>
    - level: 'staff'
      q: 'What goes wrong if you cache the response but the original operation actually failed?'
      a: >-
          <p>You'd replay a failure forever, or — worse — cache a <code>5xx</code> and turn a
          transient error into a permanent one. Best practice: only persist the key for
          <strong>terminal outcomes</strong>; for in-flight or transient failures, leave the key open
          (or store a "pending" state) so a legitimate retry can complete the operation rather than
          being told it already failed.</p>
    - level: 'design'
      q: 'Design <code>POST /orders</code> so a flaky client can retry without creating duplicate orders.'
      a: >-
          <p><b>A strong answer covers:</b> the client mints an <code>Idempotency-Key</code> (UUID
          v4) per logical order and sends it on every retry &rarr; the server keeps a durable
          key&rarr;response store (status + body) with a TTL (~24h) and a request fingerprint so a key
          can't be reused for a different order &rarr; the first request executes inside a transaction
          that also writes the key under a unique constraint, making "executed" and "remembered"
          atomic &rarr; a concurrent/retried request with the same key replays the stored response,
          waits for the in-flight one, or gets a <code>409</code> — it must never re-execute &rarr;
          persist the key only for terminal outcomes so a transient failure stays retryable &rarr;
          scope keys per endpoint + per tenant &rarr; frame it precisely: this delivers exactly-once
          <em>effect</em>, not exactly-once delivery; pair it client-side with bounded retries,
          exponential backoff + jitter, and honor <code>Retry-After</code>.</p>
    - level: 'design'
      q: 'Design a message/event consumer that must not double-process a payment when the broker delivers at-least-once.'
      a: >-
          <p><b>A strong answer covers:</b> accept that exactly-once delivery is impossible (Two
          Generals) — the design goal is exactly-once <em>effect</em> = at-least-once delivery +
          idempotent processing &rarr; deduplicate on a stable event/operation id carried by the
          message, recording processed ids with a TTL and skipping any already handled &rarr; fold the
          dedup write and the business effect into one transaction (or use transactional offset
          commits) so a crash between "did the work" and "recorded it" can't duplicate &rarr;
          recognize the boundary: in-broker exactly-once stops the moment the effect crosses to an
          external system (a charge, an email), where you're back to at-least-once + an idempotency
          key &rarr; protect the downstream with backoff + jitter and a circuit breaker to prevent a
          thundering herd &rarr; only mark terminal outcomes as processed so a transient failure can
          be retried.</p>

## What "idempotent" actually means

<p>
    Idempotent = repeating the request leaves the <strong>same server state</strong> as doing it
    once. Not "returns the same body" — <em>leaves the same state</em>. HTTP's method semantics
    <sup><a href="#s2" id="r2">2</a></sup>
    bake this in:
</p>

<Fig2 />

<div class="myth">
    <div class="m">✗ Myth: "idempotent means it returns the same response."</div>
    <div class="t">
        ✓ Truth: it leaves the same server <em>state</em> — the response may differ.
    </div>
</div>

## The Idempotency-Key pattern

<p>
    For genuinely non-idempotent operations — create, charge, transfer — you bolt idempotency on.
    Stripe is the canonical exemplar.
    <sup><a href="#s1" id="r1">1</a></sup>
</p>

<Fig3 />

<div class="steps">
    <div class="step">
        <b>1 · Client mints the key</b> One UUID per <em>logical</em> operation, sent as an
        <code>Idempotency-Key</code> header; reused on every retry.
    </div>
    <div class="step">
        <b>2 · Server stores key → response</b> On first sight, execute, then persist the key with
        the status, body, and a request fingerprint.
    </div>
    <div class="step">
        <b>3 · Retry replays</b> A second request with the same key returns the stored response — it
        never re-executes.
    </div>
    <div class="step">
        <b>4 · Mismatch is rejected</b> Same key + a different body errors out, so a key can't be
        smuggled onto another operation.
    </div>
</div>

<p class="pull">
    <strong>The key belongs to the client, not the server.</strong> Only the client knows that "this
    retry" and "that first attempt" are the same intent.
</p>

## Implement it like a senior

<p>The header is the easy part. The interview lives in these four decisions:</p>

<div class="facts">
    <div class="fact">
        <b>Where to store</b> A durable store, or Redis with a TTL. It must outlive one process and
        survive the retry window.
    </div>
    <div class="fact">
        <b>What to persist</b> Status code + response body + a fingerprint (hash) of the request, to
        replay exactly and detect mismatch.
    </div>
    <div class="fact blue">
        <b>Concurrency — the real test</b> Two in-flight requests, one key: a unique constraint or
        lock lets one win and write; the other waits or gets <code>409</code>.
    </div>
    <div class="fact blue">
        <b>Scope &amp; TTL</b> Scope keys per endpoint and per account; retain long enough to cover
        real retry windows (Stripe ≈ 24h), short enough to bound storage.
    </div>
</div>

## Idempotency ≠ exactly-once

<Fig4 />

<div class="rule">
    <span>♻️ Memory rule:</span> Idempotency makes duplicate delivery harmless — the client owns the
    key, the server stores key→response and replays it; you get exactly-once <em>effect</em>, never
    exactly-once delivery.
</div>
