# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What are the two distinct halves of HTTP caching?'
      a: >-
          <p><strong>Freshness</strong> lets a cache reuse a stored response with <em>no network call
          at all</em>. <strong>Validation</strong> lets a cache cheaply confirm a stored response is
          still good via a round-trip that carries no body. Conflating the two is the tell of a shallow
          answer — you design with both: aggressive freshness for the common case, validation as the
          cheap fallback when freshness lapses.</p>
    - level: 'core'
      q: "What's the difference between <code>public</code> and <code>private</code>?"
      a: >-
          <p><code>public</code> may be stored by <em>any</em> cache, including shared ones (CDN/proxy).
          <code>private</code> restricts storage to the end user's own browser — never a shared cache.
          <code>private</code> is the guard for per-user payloads.</p>
    - level: 'core'
      q: 'Walk through a conditional <code>GET</code> with <code>If-None-Match</code>. What status comes back if nothing changed?'
      a: >-
          <p>The client sends the stored <code>ETag</code> in <code>If-None-Match</code>. If the
          representation still matches, the server returns <code>304 Not Modified</code> <strong>with
          no body</strong> — the client reuses its cached copy. Only if it changed does the server
          return <code>200</code> with the full body.</p>
    - level: 'core'
      q: 'What status code signals a failed precondition, and what does the client do?'
      a: >-
          <p><code>412 Precondition Failed</code>. The client should re-<code>GET</code> the resource
          (and its new ETag), merge or re-apply its change against the current state, and retry the
          write with the updated <code>If-Match</code>.</p>
    - level: 'senior'
      q: "<code>no-store</code> vs <code>no-cache</code> — what's the difference? (Most candidates trip here.)"
      a: >-
          <p><code>no-store</code> means <strong>never write it to any cache</strong> — the directive
          for genuinely sensitive data. <code>no-cache</code> is confusingly named: it <em>does</em>
          store the response, but must <strong>revalidate with the origin before every reuse</strong>
          (expecting a <code>304</code>). So <code>no-cache</code> still saves bandwidth on unchanged
          bodies; <code>no-store</code> saves nothing because nothing is kept.</p>
    - level: 'senior'
      q: 'How does <code>If-Match</code> prevent lost updates, end to end?'
      a: >-
          <p>The client <code>GET</code>s the resource and its <code>ETag</code>, then sends
          <code>If-Match: "&lt;etag&gt;"</code> on the <code>PUT</code>/<code>PATCH</code>. The server
          applies the write only if the current ETag still matches. If the resource changed underneath,
          the ETag no longer matches and the server returns <code>412 Precondition Failed</code> — the
          write is rejected and the client must <strong>re-fetch and reconcile</strong> rather than
          clobber. This is <strong>optimistic concurrency control</strong>.</p>
    - level: 'senior'
      q: 'Strong vs weak ETags (<code>"abc"</code> vs <code>W/"abc"</code>) — what''s the distinction and when do you use each?'
      a: >-
          <p>A <strong>strong</strong> validator means <em>byte-for-byte identical</em>. A
          <strong>weak</strong> one (<code>W/</code> prefix) means <em>semantically equivalent</em> —
          the payload may differ cosmetically (whitespace, a regenerated timestamp) but is "the same"
          for the user. Strong is required for <code>Range</code> requests and is what you want for
          concurrency control; weak is fine for cache revalidation of responses that vary only
          cosmetically.</p>
    - level: 'senior'
      q: 'What is the cache key by default, and what does <code>Vary</code> change?'
      a: >-
          <p>By default a cache keys on the <strong>request URL</strong> (plus method).
          <code>Vary</code> adds named request headers to that key so different header values get
          different cache entries — e.g. <code>Vary: Accept</code> partitions JSON from XML,
          <code>Vary: Accept-Encoding</code> separates gzip from brotli. Get it wrong and the cache
          serves the wrong representation — or worse, the wrong user's data.</p>
    - level: 'staff'
      q: "How would you generate ETags, and what's the cost trade-off?"
      a: >-
          <p>Two families: <strong>content hashes</strong> (hash the serialized body) — exact but you
          pay CPU to render and hash on every request, even ones that end in a <code>304</code>; or
          <strong>version metadata</strong> (a row's <code>updated_at</code> or a monotonic version
          column) — cheap to read, strong if the version is reliable, and computable without rendering
          the full body. At scale, prefer a stored version/sequence number so you can answer a
          conditional request without materializing the representation. Watch out for hashing that
          differs across nodes (key ordering, serializer version) — it'll break revalidation and tank
          your hit rate.</p>
    - level: 'staff'
      q: "Same ETag drives caching and concurrency — articulate why that's elegant, and any tension."
      a: >-
          <p>One value, two jobs: <em>read</em> it back as <code>If-None-Match</code> and you get a
          <code>304</code> cache validation; <em>send</em> it back as <code>If-Match</code> on a write
          and you get optimistic locking. Elegant because the server maintains a single version notion.
          The tension: concurrency wants a <strong>strong, exact</strong> validator (any change must
          fail the write), while cache revalidation tolerates a <strong>weak</strong> one — so if you
          serve weak ETags for cosmetic-variance caching, don't reuse them for <code>If-Match</code>,
          or you'll accept writes against a "semantically equal" but actually-different revision.</p>
    - level: 'staff'
      q: "A write happens at the origin but a CDN still serves the old copy. What's going on and how do you fix it?"
      a: >-
          <p>The edge entry is still within its <code>s-maxage</code> freshness window, so the CDN
          serves it without asking the origin — by design. Fixes: shorten <code>s-maxage</code> and
          lean on revalidation; issue an explicit <strong>purge/ban</strong> to the CDN on write
          (eventual, per-POP propagation, so allow a moment); or use <code>stale-while-revalidate</code>
          so the next request triggers a background refresh. Pick based on how tight your staleness SLA
          is — instant correctness means active purge plus a short TTL as a safety net.</p>
    - level: 'design'
      q: 'Design the caching &amp; conditional-request strategy for a read-heavy public product catalog API fronted by a CDN.'
      a: >-
          <p><b>A strong answer covers:</b> classify resources first — catalog/product pages are
          public-stable, price/inventory are public-volatile, the cart/account is per-user &rarr; for
          public-stable, set <code>public</code> with a generous <code>s-maxage</code> at the CDN and a
          shorter <code>max-age</code> in the browser so you can purge the edge while clients return
          soon, and emit <strong>strong ETags</strong> so lapsed entries revalidate with a cheap
          <code>304</code> rather than re-shipping the body &rarr; add <code>stale-while-revalidate</code>
          so users never wait on origin during a refresh &rarr; for volatile price/inventory, short TTL
          or validation-only plus an <strong>active purge on write</strong> where instant correctness
          matters &rarr; per-user data: <code>private, no-store</code>, kept off the shared cache &rarr;
          set <code>Vary: Accept, Accept-Encoding</code> precisely and never over-vary (high-cardinality
          headers shred hit rate) &rarr; name the trade-off: long edge TTL maximizes offload but risks
          stale-after-write unless you can purge; ETag validation never serves wrong data but costs a
          round-trip &rarr; measure cache hit rate and origin offload as the success metric, and guard
          against a thundering herd on simultaneous expiry with jittered TTLs or
          <code>stale-while-revalidate</code>.</p>
    - level: 'design'
      q: 'Design optimistic-concurrency control for a collaborative document API where many clients edit the same resources.'
      a: >-
          <p><b>A strong answer covers:</b> the hazard is the <strong>lost update</strong> — concurrent
          read-modify-write over a stateless protocol &rarr; on <code>GET</code>, return the document
          with a <strong>strong, exact ETag</strong> for that revision (a stored version/sequence
          column, not a body hash, so you can answer without materializing the representation) &rarr;
          require every <code>PUT</code>/<code>PATCH</code> to carry <code>If-Match: &lt;etag&gt;</code>;
          if the current version no longer matches, reject with <code>412 Precondition Failed</code> so
          the client must re-fetch and reconcile rather than clobber &rarr; enforce the check
          <strong>atomically at the store</strong> — <code>UPDATE … WHERE id=? AND version=?</code> and
          check rows-affected, or a transaction — because the HTTP layer only communicates the
          precondition, it doesn't enforce it &rarr; offer race-free create via
          <code>If-None-Match: *</code> &rarr; reuse the same version as an <code>If-None-Match</code>
          validator for <code>304</code> caching (one value, two jobs), but do <strong>not</strong>
          reuse a <em>weak</em> ETag for <code>If-Match</code> or you'll accept a write against a merely
          "semantically equal" revision &rarr; name the trade-off: optimistic is lock-free and cheap on
          the happy path but thrashes under frequent conflicts, where a short-lived <em>scoped</em> lock
          with a lease/TTL is the fallback &rarr; surface conflicts to the UI for merge, and log
          <code>412</code> rates as a signal that contention is too high.</p>

## Freshness — the <code>Cache-Control</code> directives map

<p>
    The origin declares a response's lifetime; while it's fresh, caches serve it directly. The
    directives you must be fluent in:
    <sup><a href="#s1">1</a></sup>
</p>

<Fig2 />

<div class="myth">
    <div class="m">
        ✗ Myth: "<code>no-cache</code> means don't cache it."
    </div>
    <div class="t">
        ✓ Truth: <code>no-cache</code> stores it but revalidates before every reuse;
        <code>no-store</code> is the one that never writes to any cache.
    </div>
</div>

## Validation — the <code>If-None-Match</code> → <code>304</code> round-trip

<p>
    When freshness lapses, a cache <em>validates</em> rather than re-downloads. The client echoes
    the stored <code>ETag</code> in <code>If-None-Match</code>; if it still matches, the server
    replies
    <code>304 Not Modified</code> with no body.
    <sup><a href="#s2" id="r2">2</a></sup>
</p>

<Fig3 />

<div class="facts">
    <div class="fact">
        <b>ETag + If-None-Match</b> An opaque version tag for the representation.
        <sup><a href="#s3" id="r3">3</a></sup>{' '}
        Echoed in <code>If-None-Match</code>; still matching → <code>304</code> with no body.
    </div>
    <div class="fact">
        <b>Last-Modified + If-Modified-Since</b> Timestamp-based, weaker (1-second granularity,
        clock issues). The fallback when you can't compute an ETag cheaply.
    </div>
    <div class="fact blue">
        <b>Strong validator — "abc"</b> Byte-for-byte identical. Required for <code>Range</code>
        requests and what you want for concurrency control.
    </div>
    <div class="fact blue">
        <b>Weak validator — W/"abc"</b> Semantically equivalent. Fine for cache revalidation of
        cosmetically-varying responses.
    </div>
</div>

## The fresh → stale → revalidate lifecycle

<Fig4 />

<p class="pull">
    <strong>
        An ETag is both a cache validator and a concurrency token — same header, two superpowers.
    </strong>{' '}
    Read it, and you confirm freshness with a <code>304</code>. Send it back on a write, and you
    enforce optimistic locking. One value, two roles.
</p>

## Conditional writes — optimistic concurrency

<p>
    This is the senior highlight and the part most candidates miss. The{' '}
    <strong>lost-update problem</strong>: two clients <code>GET</code> the same resource, both edit,
    both <code>PUT</code> — the second silently overwrites the first. The HTTP-native fix is{' '}
    <strong>optimistic concurrency control</strong> using the same ETag.
    <sup><a href="#s2">2</a></sup>
</p>

<Fig5 />

<div class="facts">
    <div class="fact">
        <b>The happy path</b> Client sends <code>If-Match: "etag"</code> on <code>PUT</code>/
        <code>PATCH</code>; resource unchanged → write applies, costs nothing extra.
    </div>
    <div class="fact">
        <b>The conflict</b> Resource changed → <code>412 Precondition Failed</code>. The client
        re-fetches and reconciles rather than clobbers.
    </div>
    <div class="fact blue">
        <b>Why not pessimistic locking</b> Holding a lock across stateless requests violates
        statelessness, ties up resources, and breaks when a client vanishes mid-edit.
    </div>
    <div class="fact blue">
        <b>Why optimistic wins</b> Assumes conflicts are rare, detects them at write time, costs
        nothing on the happy path — a far better fit for HTTP.
    </div>
</div>

## Caching layers &amp; the <code>Vary</code> trap

<p>
    Caches live in two places, mapping straight onto REST's <em>cacheable</em> and{' '}
    <em>layered system</em> constraints: <strong>private</strong> caches (the user's browser) and
    <strong>shared</strong> caches (CDN, reverse proxy, API gateway). A shared cache is one node
    many users hit — which is exactly why the cache <em>key</em> matters.
</p>

<Fig6 />

<div class="rule">
    <span>♻️ Memory rule:</span> Freshness serves free, validation revalidates cheap (
    <code>304</code>, no body); the <em>same ETag</em> that proves freshness also enforces
    optimistic locking via
    <code>If-Match</code> → <code>412</code>; on shared caches, mark authed responses
    <code>private, no-store</code>.
</div>
