# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is a breaking change?'
      a: >-
          <p>Any change that can cause a <strong>deployed client to fail</strong> without the client
          changing its own code. The line that decides versioning: if a change can break an
          integration silently, it's breaking; if it can't, it's safe to ship in place.</p>
    - level: 'core'
      q: 'What is a "tolerant reader"?'
      a: >-
          <p>A client that <strong>ignores fields and values it doesn't recognize</strong> instead of
          failing on them. It's the client half of additive evolution: the server only adds, the
          client only tolerates, and you can ship new fields and enum values without breaking
          anyone.</p>
    - level: 'core'
      q: 'What are the three places to put a version?'
      a: >-
          <p>(1) <strong>URI path</strong> — <code>/v1/users</code>; (2) <strong>custom header</strong>
          — e.g. <code>Accept-Version: 2</code> or <code>API-Version</code>; (3) <strong>media-type
          negotiation</strong> — <code>Accept: application/vnd.company.v2+json</code>. None is free;
          each buys clarity in one dimension and pays for it in another.</p>
    - level: 'core'
      q: 'What does the <code>Sunset</code> header do?'
      a: >-
          <p>Defined by <strong>RFC 8594</strong>, it carries the <strong>date an endpoint or version
          will be retired</strong>, in every response on the old version — so even silent clients that
          never read a changelog carry the warning in-band.</p>
    - level: 'senior'
      q: '"How do you version a public API?" What''s the senior reframe?'
      a: >-
          <p>The question is really <em>"how do I evolve the API without breaking the clients I can't
          redeploy?"</em> A new version is a <strong>last resort, not a first move</strong>. Most
          changes can be made compatibly — additive change plus tolerant readers — with zero version
          bump. You version only when a change is genuinely impossible to make compatibly. Leading
          with <code>/v1</code> vs. headers is the mid-level reflex; reframing to <em>evolution</em>
          is the senior signal.</p>
    - level: 'senior'
      q: 'Why is "just cut a new version" the wrong default?'
      a: >-
          <p>The cost of a version isn't the URL — it's the <strong>permanent support burden</strong>:
          a migration imposed on every integrator, a fragmented client base, and a combinatorial
          testing matrix across parallel surfaces you now maintain forever. Mint one per change and
          you're running five APIs in perpetuity. Versions are expensive; you spend one only when
          compatible evolution is genuinely impossible.</p>
    - level: 'senior'
      q: 'Why is per-account pinning (Stripe) such a strong pattern?'
      a: >-
          <p>It converts <em>"we changed the API"</em> from a client-breaking event into a
          <strong>client-controlled</strong> one. Existing integrations stay frozen on the version
          they were built against; nothing reshapes underneath them. New users get the latest;
          upgrades are opt-in and overridable per request. It's the cleanest way to ship breaking
          changes without a flag day for every integrator.</p>
    - level: 'senior'
      q: 'How do you decide <em>when</em> it''s safe to actually remove the old version?'
      a: >-
          <p><strong>Monitor real traffic, don't guess on a calendar.</strong> Instrument usage per
          version (and ideally per consumer), watch it drain, and reach out directly to the stragglers
          still on the old surface. Sunset only when usage has genuinely gone — a date in the
          <code>Sunset</code> header is a target, not a guillotine. Pulling a version with live
          traffic on it because "the date arrived" is how you cause an outage you own.</p>
    - level: 'staff'
      q: 'A field is technically optional but every client has always sent it. You want to drop it from responses. Breaking?'
      a: >-
          <p>Treat it as breaking in practice. The <em>contract</em> says optional, but the <em>de
          facto</em> contract is what clients actually depend on (Hyrum's Law: with enough consumers,
          every observable behavior is depended on by someone). Seniority is measuring real usage
          before deciding, not arguing from the spec. Check traffic; if anyone reads it, removing it
          breaks them.</p>
    - level: 'staff'
      q: 'How do you coordinate a migration across clients you don''t control?'
      a: >-
          <p>You can't force them, so you make migration cheap and the status visible. Instrument
          <strong>per-consumer usage</strong> of the old version. Publish a precise migration guide
          and, ideally, automated diffs or a compatibility shim. Reach the long tail individually —
          the top consumers by volume are a handful of conversations. Use in-band
          <code>Deprecation</code>/<code>Sunset</code> headers so even silent clients carry the
          signal. Then sunset by <em>drained traffic</em>, segmenting stragglers rather than pulling
          on a date. The work is communication and measurement, not code.</p>
    - level: 'staff'
      q: 'How do you keep the support cost of multiple live versions from compounding?'
      a: >-
          <p>Hold a hard line on the number of concurrent majors (often N and N−1 only), so the
          testing matrix and bug-fix surface stay bounded. Where possible, implement old versions as a
          <strong>translation layer in front of a single current core</strong> — adapters that reshape
          requests/responses rather than forked codepaths — so business logic isn't duplicated per
          version. And keep the bar for cutting a version high: every version you don't mint is
          support cost you never pay. The cheapest version is still no new version.</p>
    - level: 'design'
      q: 'Design a versioning &amp; deprecation strategy for a public payments API with thousands of integrators.'
      a: >-
          <p><b>A strong answer covers:</b> leading with evolution, not versioning — additive-first so
          most changes ship in place with zero migration for clients you can't redeploy. For genuinely
          breaking changes: <strong>pin a version per account</strong> (Stripe's model) so a server
          change never silently reshapes a live integration; new accounts get the latest, upgrades are
          explicit and per-request overridable. Choose placement deliberately (URI path for
          visibility/routing, or a dated header à la GitHub) and name the trade-off. Run the full
          deprecation lifecycle: parallel old/new, in-band <code>Deprecation</code> +
          <code>Sunset</code> (RFC 8594) headers plus out-of-band changelog/email, a generous
          months-not-days window scaled to blast radius. Instrument <strong>per-consumer usage</strong>,
          retire only on drained traffic, and segment stragglers (reach the high-volume few directly)
          rather than pulling on a date. Bound concurrent majors (N and N−1) and implement old versions
          as a translation layer over one core to cap the cost, testing matrix, and operational
          complexity of running N live surfaces. For payments specifically, weigh data-correctness and
          security: sometimes the contract itself is the bug and a forced breaking change is the only
          honest fix — done with pinning, hard comms, and migration help.</p>
    - level: 'design'
      q: 'You must make a breaking change to a widely-used response shape — design the rollout so no client breaks.'
      a: >-
          <p><b>A strong answer covers:</b> first exhausting the additive path — add the new
          field/shape alongside the old, keep the old populated, document it deprecated, and rely on
          tolerant readers to ignore what they don't use, so it's a zero-version outcome. If the change
          is truly incompatible (renamed/retyped/dropped field, changed meaning): cut a new
          <strong>major</strong>, <strong>pin existing accounts</strong> to their current version so
          nothing reshapes underneath them, and run both surfaces in parallel behind a translation
          layer over a single core. Signal in-band with <code>Deprecation</code> + <code>Sunset</code>
          (RFC 8594) headers so even silent clients carry the warning, and out-of-band with a migration
          guide and direct outreach to top consumers. Instrument per-version (ideally per-consumer)
          traffic, give a generous window, and <strong>retire only once usage has drained</strong> —
          segmenting the long tail rather than enforcing a calendar date. The discipline throughout:
          never break a deployed client silently; make migration cheap and the deprecation status
          visible in both the code path and to the humans who own it.</p>

# —— design-round rubric (rendered by <Interview> above the design questions) ——
rubricIntro: >-
    <b>Designing a REST versioning &amp; evolution strategy — a framework:</b>
rubric:
    - >-
          <strong>Classify the change.</strong> Breaking vs non-breaking is the line that decides
          everything. Apply additive-change discipline first: add optional fields/endpoints, never
          remove, rename, retype, or repurpose — most "version" requests dissolve here with zero
          migration.
    - >-
          <strong>Decide where the version lives</strong> only if a bump is unavoidable: URI path
          (visible, routes/caches by URL, but couples version to identity), custom header (clean URLs,
          but invisible and needs <code>Vary</code>), or media-type negotiation (most RESTful, but
          poor tooling support). State the pick <em>and</em> its trade-off.
    - >-
          <strong>Define the compatibility contract.</strong> Spell out tolerant-reader expectations
          on both ends — server append-only, clients ignore unknowns and open-enum — so additive
          evolution actually holds in practice, including in generated SDKs.
    - >-
          <strong>Pin to insulate clients.</strong> Per-account (Stripe) or dated-header (GitHub)
          pinning converts "we changed the API" from a client-breaking event into a client-controlled
          one; new breaking surfaces never reshape existing integrations silently.
    - >-
          <strong>Run a deprecation lifecycle.</strong> Announce → run old + new in parallel → emit
          in-band <code>Deprecation</code> + <code>Sunset</code> (RFC 8594) headers → publish a
          migration guide with a generous window → reach humans out-of-band.
    - >-
          <strong>Migrate via dual-running / translation layer.</strong> Implement old versions as
          adapters in front of one current core rather than forked codepaths, so business logic isn't
          duplicated per version.
    - >-
          <strong>Observe version usage, then retire on drained traffic</strong> — per-version
          (ideally per-consumer) instrumentation tells you when it's safe to sunset; never pull on a
          calendar guess. Bound concurrent majors (often N and N−1) to cap the testing matrix,
          support, and cost of N live versions.

{/* THE LINE: breaking vs non-breaking */}

<h2 class="sec">The line that decides everything</h2>

<p>
    Every evolution question collapses to one classification: is the change <strong>breaking</strong>
    or <strong>non-breaking</strong>? Get this line right and the rest is mechanics.
</p>

<Fig2 />

<div class="myth">
    <div class="m">✗ Myth: "any change to the response shape needs a new version."</div>
    <div class="t">
        ✓ Truth: pure additions are non-breaking — ship them in place, no version, if your readers
        are tolerant.
    </div>
</div>

<h2 class="sec">Where the version lives: three strategies</h2>

<p>
    Once a change is genuinely breaking, you must place the version somewhere. There's no free
    option — each buys clarity in one dimension and pays for it in another.
</p>

<Fig3 />

{/* HOW EXEMPLARS DO IT */}

<h2 class="sec">How the exemplars actually do it</h2>

<p>
    Two reference points worth quoting by name, because they encode the senior instinct — <em>never
    break a deployed client silently.</em>
</p>

<div class="facts">
    <div class="fact">
        <b>Stripe · pin per account</b> Your version is fixed when you first integrate; you upgrade
        explicitly, and a single request can override with a header. A server-side change never
        silently reshapes an existing integration.
        <sup><a href="#s1" id="r1">1</a></sup>
    </div>
    <div class="fact blue">
        <b>GitHub · date in a header</b> A date-based version sent in <code>X-GitHub-Api-Version</code>
        — the header strategy paired with calendar versioning (<code>2024-10-01</code>).
        <sup><a href="#s2" id="r2">2</a></sup>
    </div>
    <div class="fact">
        <b>Date vs semver = labeling</b> <code>2024-10-01</code> vs <code>v2</code> is a labeling
        choice, not a strategy. Both expose only <strong>major</strong> versions; minors/patches are
        the compatible changes you ship in place.
    </div>
    <div class="fact blue">
        <b>Pinning is the real move</b> Per-account pinning converts "we changed the API" from a
        client-breaking event into a client-<em>controlled</em> one. That's the senior signal.
    </div>
</div>

{/* DEPRECATION LIFECYCLE */}

<h2 class="sec">Deprecation is a lifecycle, not a delete</h2>

<p>
    The version is the easy half. Retiring the old one without burning integrators is where
    seniority shows.
</p>

<Fig4 />

<div class="steps">
    <div class="step">
        <b>1 · Announce &amp; overlap</b> Publish the change, then run old and new <em>in parallel</em>
        for a long, generous window — months for a public API, not days.
    </div>
    <div class="step">
        <b>2 · Signal in-band</b> Emit <code>Deprecation</code> and <code>Sunset</code> (RFC 8594)
        with the retirement date, so even silent clients carry the warning.
    </div>
    <div class="step">
        <b>3 · Communicate out-of-band</b> Changelog, email, docs — reach the humans, not just the
        code.
    </div>
    <div class="step">
        <b>4 · Monitor, then remove</b> Watch real traffic on the old version; sunset only once usage
        has drained, not on a calendar guess.
    </div>
</div>

<p class="pull">
    <strong>Pinning is the move that scales.</strong> It turns "we changed the API" from a
    client-breaking event into a client-controlled one — additive-first, then major, parallel,
    signaled, monitored, retired.
</p>
