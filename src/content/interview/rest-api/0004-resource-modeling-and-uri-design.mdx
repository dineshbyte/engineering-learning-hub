# —— interview Q&A (verbatim from <div data-interview>) ——
interview:
    - level: 'core'
      q: 'Why is <code>POST /getOrder</code> an anti-pattern?'
      a: >-
          <p>It encodes the <em>verb in the path</em>, which is RPC wearing a REST costume. In resource-oriented design the URI names a <strong>noun</strong> — the thing — and the HTTP method <em>is</em> the verb. The read should be <code>GET /orders/42</code>: the resource is <code>/orders/42</code>, the action is <code>GET</code>.</p>
    - level: 'core'
      q: 'Should collection names be singular or plural?'
      a: >-
          <p>Plural, consistently: <code>/orders</code>, <code>/customers</code>, <code>/products</code>. A collection holds many items, so the plural reads naturally — <code>/orders</code> is the set, <code>/orders/42</code> is one member of it. Mixing singular and plural across the surface is a consistency smell.</p>
    - level: 'core'
      q: 'What is the difference between a collection URI and an item URI?'
      a: >-
          <p>A <strong>collection URI</strong> (<code>/orders</code>) addresses the set: <code>GET</code> lists it, <code>POST</code> adds to it (server assigns the id). An <strong>item URI</strong> (<code>/orders/42</code>) addresses one member: <code>GET</code> reads it, <code>PUT</code>/<code>PATCH</code> updates it, <code>DELETE</code> removes it. The method's meaning shifts with which level you're at.</p>
    - level: 'core'
      q: "What's the right way to create a new order?"
      a: >-
          <p><code>POST /orders</code> with the order body. The server assigns the id and returns <code>201 Created</code> with a <code>Location</code> header pointing at <code>/orders/{new-id}</code>. You don't <code>POST /createOrder</code> and you don't let the client pick the id on a collection <code>POST</code>.</p>
    - level: 'core'
      q: "Why don't actions like <em>cancel</em> or <em>publish</em> map cleanly to CRUD?"
      a: >-
          <p>Because they're genuine verbs that don't reduce to create/read/update/delete. "Cancel an order" isn't naturally a <code>PUT</code> of a new representation — it's a state transition. CRUD covers the easy 80%; modeling the remaining verbs is where the interview gets interesting.</p>
    - level: 'core'
      q: 'What does nesting like <code>/customers/1/orders</code> communicate?'
      a: >-
          <p>Containment — "the orders belonging to customer 1." That's a legitimate, readable use of nesting for a real ownership relationship. The mistake isn't nesting one level; it's going deep.</p>
    - level: 'core'
      q: 'Why prefer opaque identifiers over sequential integers in URIs?'
      a: >-
          <p>Sequential ints <strong>leak business volume</strong> (reading order <code>#100002</code> tells an attacker your throughput) and invite <strong>enumeration</strong> — walking <code>/orders/1</code>, <code>/orders/2</code>… UUIDs/ULIDs are opaque and non-guessable, so neither attack works from the id alone.</p>
    - level: 'core'
      q: 'What are slugs for, and what should they <em>not</em> be?'
      a: >-
          <p>Slugs (<code>/articles/how-to-design-apis</code>) are for <strong>humans and SEO</strong> — readable, memorable URLs. They should <strong>not</strong> be your stable primary key: titles change, slugs collide, and they're mutable. Use an opaque id as the canonical key and treat the slug as a lookup alias (often redirecting to the id-based canonical URL).</p>
    - level: 'core'
      q: 'What are the boring-on-purpose URI formatting conventions?'
      a: >-
          <p>Lowercase; hyphen-separated words (<code>/purchase-orders</code>, not <code>/purchaseOrders</code> or <code>/purchase_orders</code>); no trailing slash; no file extensions (<code>.json</code>) — use the <code>Accept</code> header for format. Boring and predictable beats clever.</p>
    - level: 'core'
      q: "What belongs in query parameters and what doesn't?"
      a: >-
          <p>Query params carry <strong>filtering, sorting, and pagination</strong> — <code>?status=open&amp;sort=-created_at&amp;page=2</code>. They must <strong>not</strong> carry identity: the resource's identifier belongs in the <em>path</em> (<code>/orders/42</code>, not <code>/orders?id=42</code>). Path = which resource; query = how to view the collection.</p>
    - level: 'core'
      q: 'Why is resource modeling usually the first artifact in a design interview?'
      a: >-
          <p>Because everything else hangs off it — status codes, auth, pagination, caching all assume a resource model. Clean nouns and defensible nesting are one of the earliest senior signals an interviewer reads; get the frame right and the rest of the design falls out cleanly. Get it wrong and every downstream answer inherits the mess.</p>
    - level: 'senior'
      q: 'Why does the noun/verb discipline matter beyond aesthetics?'
      a: >-
          <p>Because it's what keeps the contract <strong>uniform and cacheable</strong>. Once verbs leak into paths, every endpoint becomes a bespoke procedure and the HTTP machinery — caching, idempotency, conditional requests, status semantics — stops applying. <code>GET /orders/42</code> is cacheable and safe by definition; <code>POST /getOrder</code> tells intermediaries nothing. Naming the resource correctly is the load-bearing decision the rest of the design hangs off.</p>
    - level: 'senior'
      q: 'A teammate wants <code>GET /searchOrders</code>. What do you propose instead?'
      a: >-
          <p>Search is reading a collection with criteria, so <code>GET /orders?status=open&amp;customer_id=7</code>. The collection is the noun; query params carry the filter. <code>searchOrders</code> reintroduces a verb in the path. The one legitimate exception is a search with a body too large/complex for a query string, where teams use <code>POST /orders/search</code> (or <code>:search</code>) — but acknowledge that's a pragmatic concession that costs you GET cacheability.</p>
    - level: 'senior'
      q: 'What are the three principled ways to model a non-CRUD action?'
      a: >-
          <ul>
                <li><strong>Sub-resource / state collection:</strong> treat the outcome as a thing — <code>POST /orders/42/cancellations</code>, or a status you read back.</li>
                <li><strong>Custom method (Google AIP):</strong> colon-suffixed verb on the resource — <code>POST /orders/42:cancel</code>.</li>
                <li><strong>Reify the action as its own resource:</strong> promote the verb to a noun — <code>POST /refunds</code> with <code>{order_id}</code>.</li>
              </ul><p>The senior signal is naming the trade-off, not just picking one.</p>
    - level: 'senior'
      q: 'When do you reify an action as its own top-level resource?'
      a: >-
          <p>When the action has its <strong>own durable lifecycle, state, and history</strong>. A refund is processed → settled → reversible; it's queried, listed, audited, and reported on independently. That earns it standalone existence as <code>/refunds</code> sitting alongside <code>/payments</code>. Decide by lifecycle, not by grammar.</p>
    - level: 'senior'
      q: 'When is a custom method (<code>POST /orders/42:cancel</code>) the right call?'
      a: >-
          <p>When the action is a <strong>momentary transition on an existing resource</strong> that has no life of its own, and inventing a noun ("a cancellation") would create something nobody queries. The custom method is honest that it's a non-CRUD verb, keeps the resource in the path, and is explicitly sanctioned by Google AIP for verbs that don't fit standard methods. It's cleaner than forcing a fake resource.</p>
    - level: 'senior'
      q: 'Sub-resource vs custom method for the same action — how do you choose?'
      a: >-
          <p>If you want the action to leave an <strong>auditable, listable record</strong>, model it as a sub-resource/state collection (<code>POST /orders/42/cancellations</code>) — now you can <code>GET</code> the history. If it's a pure transition you'll never enumerate, a custom method (<code>:cancel</code>) is leaner and avoids a collection that only ever holds zero or one rows. Both keep the parent resource in the path; they differ on whether the action deserves to be a queryable record.</p>
    - level: 'senior'
      q: >-
          What status code does an async action like <code>:cancel</code> return if it can't complete synchronously?
      a: >-
          <p><code>202 Accepted</code> — the request is acknowledged but processing isn't done. Return a representation (or <code>Location</code>) for a status/operation resource the client can poll (<code>GET /operations/{id}</code> or the order's own status). Returning <code>200</code> immediately would lie about completion.</p>
    - level: 'senior'
      q: 'Why is deep nesting like <code>/customers/1/orders/42/items/9</code> fragile?'
      a: >-
          <p>It hard-codes the entire hierarchy into the URL. The day a relationship changes — items move under a fulfillment, an order spans customers, you add line-item history — every client URL breaks. Deep paths couple your contract to today's org chart. They also bloat with redundant ids you don't need to locate the leaf.</p>
    - level: 'senior'
      q: "What's the shallow alternative to deep nesting, and what do you keep?"
      a: >-
          <p>Address resources flatly by identity — <code>/orders/42</code>, <code>/items/9</code> — and express relationships with <strong>filtering</strong>: <code>/orders?customer_id=1</code> instead of <code>/customers/1/orders</code>. You get the same query power without coupling the URL to the hierarchy. Keep nesting (≈ one level) only for genuine ownership where the child has no identity of its own.</p>
    - level: 'senior'
      q: 'Link, embed, or separate endpoint — what are the three ways to handle related data?'
      a: >-
          <ul>
                <li><strong>Link:</strong> return a reference/URL (hypermedia) the client follows when it wants the related resource.</li>
                <li><strong>Embed/expand:</strong> inline the related data on demand — <code>GET /orders/42?expand=customer</code> or <code>?fields=</code>.</li>
                <li><strong>Separate endpoint:</strong> the client fetches <code>/customers/7</code> itself.</li>
              </ul><p>The choice is a round-trips-vs-payload-size trade-off.</p>
    - level: 'senior'
      q: 'Over-embedding vs the N+1 problem — describe the tension.'
      a: >-
          <p>Always embedding everything makes <em>every</em> payload huge — clients download relations they didn't ask for, caching gets coarser, and the response couples to many tables. Never embedding makes clients <strong>chatty</strong>: list 50 orders, then fire 50 follow-up requests for each customer (the N+1 round-trip). The middle path is opt-in expansion (<code>?expand=</code> / field selection) so the client pays only for what it needs.</p>
    - level: 'senior'
      q: 'Where should a many-to-many relationship be modeled — under which resource?'
      a: >-
          <p>Neither parent "owns" it, so deep nesting under one side is arbitrary. Either expose the relationship from both sides as filtered sub-collections (<code>/students/1/courses</code> and <code>/courses/9/students</code>) backed by the same join, or — if the link itself carries data (enrollment date, grade) — <strong>reify the join</strong> as its own resource: <code>/enrollments?student_id=1</code>. Reify when the relationship has attributes; otherwise filtered views suffice.</p>
    - level: 'senior'
      q: 'What is a BOLA / IDOR vulnerability and how does id choice relate to it?'
      a: >-
          <p>Broken Object-Level Authorization (a.k.a. IDOR): a user swaps the id in <code>/orders/42</code> for <code>/orders/43</code> and reads someone else's data because the server checks <em>authentication</em> but not <em>ownership</em>. Opaque ids make resources <em>harder to guess</em>, but they are <strong>not the fix</strong> — they're defense in depth. The real fix is an authorization check on every object access ("does this caller own this order?"). Calling opaque ids a substitute for authz is a junior mistake.</p>
    - level: 'senior'
      q: 'UUID vs ULID — when does the difference matter?'
      a: >-
          <p>Both are opaque and non-guessable. ULIDs (and UUIDv7) are <strong>time-sortable</strong> — they encode a timestamp prefix, so they're roughly monotonic, which keeps database B-tree index inserts sequential and avoids the write-amplification of random UUIDv4. Pick ULID/UUIDv7 when insert performance and natural ordering matter; classic random UUIDv4 when you specifically want no embedded time signal.</p>
    - level: 'senior'
      q: 'Should the URI expose your internal database primary key?'
      a: >-
          <p>Avoid it. Exposing the raw DB key couples your <em>public contract</em> to a storage detail — you can't reshard, migrate to a different store, or merge tables without breaking URLs, and an auto-increment key leaks volume. Mint a separate external identifier (opaque, stable) that you map to the internal key. The public id is part of your API contract; the internal key isn't.</p>
    - level: 'senior'
      q: 'When is a singleton resource the right shape, and how do you spot the mistake?'
      a: >-
          <p>When there's <strong>exactly one</strong> of a thing in a context — <code>/users/me/settings</code>, <code>/account/preferences</code>. A singleton has <em>no id and no plural</em>; you <code>GET</code>/<code>PUT</code>/<code>PATCH</code> it directly. The tell is modeling it as a collection (<code>/users/me/settings/1</code>) — reaching for the plural-collection pattern reflexively instead of fitting it to the domain.</p>
    - level: 'senior'
      q: 'What does <code>/users/me</code> express, and why is it useful?'
      a: >-
          <p>It's an <strong>alias for the authenticated principal</strong> — "whoever this token belongs to" — so the client doesn't need to know or send its own id. It avoids leaking other users' ids into client code, sidesteps a class of IDOR mistakes (you can't swap <code>me</code> for someone else), and reads cleanly: <code>/users/me/orders</code>. The server resolves <code>me</code> to the real id from the credentials.</p>
    - level: 'senior'
      q: 'Why does consistency across the surface beat a locally clever endpoint?'
      a: >-
          <p>Because clients learn your API by <strong>pattern</strong>: once they know <code>/orders</code>, <code>/orders/42</code>, <code>?status=</code>, they predict the rest. One odd-but-clever endpoint forces every consumer to special-case it, breaks codegen/SDK assumptions, and costs documentation and support forever. The marginal cleverness almost never pays back the cognitive tax. Predictability is a feature.</p>
    - level: 'senior'
      q: 'How should the path represent filtering by multiple values or ranges?'
      a: >-
          <p>Keep it in the query string with a consistent grammar: repeated keys or CSV for sets (<code>?status=open&amp;status=pending</code> or <code>?status=open,pending</code>), and explicit suffixes/operators for ranges (<code>?created_after=…&amp;created_before=…</code> or a documented <code>created_at[gte]=</code>). Pick one convention and apply it everywhere. The path still names the collection; all of this is query, never identity.</p>
    - level: 'senior'
      q: "Walk through deriving a resource model from a domain — what's your method?"
      a: >-
          <p>(1) List the <strong>nouns</strong> the domain holds (order, cart, customer, payment, refund) — those are candidate resources. (2) Decide which are top-level collections vs owned sub-resources by asking "does this have identity on its own?" (3) Map the verbs to HTTP methods; for non-CRUD verbs, decide sub-resource / custom method / reify by lifecycle. (4) Choose relationships: nest one level for true ownership, filter for the rest. (5) Pick opaque ids and consistent conventions. Narrate the trade-offs as you go — the reasoning is what's scored.</p>
    - level: 'senior'
      q: 'How do you handle pluralization edge cases (uncountable nouns, irregulars)?'
      a: >-
          <p>Default to the natural English plural and stay consistent. For uncountable/mass nouns (<code>/audit-info</code>, <code>/news</code>, <code>/weather</code>) pick a form and document it rather than inventing <code>/informations</code>. For irregulars use the correct plural (<code>/people</code>, not <code>/persons</code>, unless your domain says otherwise). The rule that actually matters: be <strong>uniform</strong> — clients should be able to guess the plural from the singular without surprises.</p>
    - level: 'senior'
      q: 'How does URI design interact with HTTP caching?'
      a: >-
          <p>The URI <em>is</em> the cache key, so design affects cacheability directly. Stable, canonical, identity-in-path URIs cache cleanly; <code>GET /orders/42</code> is cacheable with ETags. Smuggling verbs into paths or forcing reads through <code>POST</code> kills caching. Query-param explosion fragments the cache (every filter combo is a new key), and dynamic <code>?expand=</code> shapes do too. One canonical URI per resource (avoid duplicate paths for the same thing) keeps caches and conditional requests effective.</p>
    - level: 'senior'
      q: 'How does resource modeling interact with versioning?'
      a: >-
          <p>Good modeling reduces the need to version: stable nouns, identity-in-path, and opaque ids let you add fields, relations, and optional expansions without breaking clients. Versioning bites when the <em>shape</em> or <em>relationships</em> change — renaming resources, re-homing sub-resources, changing id schemes. Those are exactly the deep-nesting / leaked-internal-id decisions that age badly. So the modeling discipline (shallow nesting, filtering over containment, external ids) is partly a versioning-cost-avoidance strategy. (Versioning mechanics are Lesson 5.)</p>
    - level: 'senior'
      q: 'Interviewer pushes back: "Deep nesting is more RESTful and self-documenting." Your reply?'
      a: >-
          <p>Nesting documents <em>a</em> relationship but hard-codes <em>one</em> hierarchy into the contract. It reads nicely until the relationship changes or the resource participates in several relationships, at which point the URL lies or breaks. REST cares that resources are identified and addressable — a flat <code>/items/9</code> plus <code>?order_id=42</code> is no less RESTful and far more evolvable. I keep one level for true ownership and filter for everything else; self-documentation comes from consistency and hypermedia/links, not from a long path.</p>
    - level: 'staff'
      q: 'Articulate the pure-REST vs pragmatist tension on custom methods.'
      a: >-
          <p>Purists argue <em>everything</em> should be a resource: a custom method like <code>:cancel</code> smuggles a verb back into the URI and breaks the uniform interface, so they'd reify state into resources (a cancellation record, a status sub-resource). Pragmatists accept a colon-method for a genuinely non-CRUD verb because inventing nouns nobody queries is its own cost. The staff answer doesn't pick a religion — it picks by lifecycle: durable state and history → reify; momentary transition → sub-resource or custom method. Say the trade-off out loud; that's the signal.</p>
    - level: 'staff'
      q: 'Should a state transition be a <code>PATCH</code> on a status field instead of a custom method?'
      a: >-
          <p>You <em>can</em> model "cancel" as <code>PATCH /orders/42 {"status":"cancelled"}</code>, and it's defensible when the transition is genuinely just a field change. The risks: it hides business rules (cancelling may trigger refunds, restocking, notifications) behind an innocuous field write, lets clients set arbitrary illegal states, and makes authorization/validation per-transition harder. A custom method or sub-resource names the operation explicitly, so you can authorize and validate <em>that transition</em> and run its side effects. Prefer the explicit form when the transition carries behavior, not just data.</p>
    - level: 'staff'
      q: 'What are the downsides of a flexible <code>?expand=</code> / field-selection mechanism?'
      a: >-
          <p>It pushes you toward GraphQL-shaped problems without GraphQL's tooling: every expansion combination is a distinct query and cache key, so <strong>caching fragments</strong>; arbitrary expansion enables expensive N+1 / fan-out queries you must bound; authorization must be enforced <em>per expanded field</em>, not just on the root; and response shape becomes dynamic, complicating client typing and contract tests. Mitigate with an allowlist of expandable relations, depth/complexity limits, and per-field authz. At some point if clients need fully arbitrary shapes, that's the signal to consider GraphQL.</p>
    - level: 'staff'
      q: 'A mobile team complains your API is too chatty. How do you respond at the design level?'
      a: >-
          <p>Diagnose first: is it N+1 over a list (add opt-in <code>expand</code> / embedded relations), too many round-trips for one screen (offer a purpose-built composite/aggregate endpoint or a BFF that fans out server-side), or pagination thrash (bigger pages, cursor paging)? Resist the urge to deep-nest everything just to save calls — that re-couples URLs. Options: bounded expansion, a read-optimized aggregate resource, batch endpoints, or a separate BFF layer. Name the caching and coupling cost of each rather than reflexively embedding.</p>
    - level: 'staff'
      q: 'Client-generated vs server-generated ids — what changes in the design?'
      a: >-
          <p>Server-generated: <code>POST /orders</code>, server mints the id, returns <code>201</code> + <code>Location</code>. Simple, but the create isn't naturally idempotent — a retry can double-create, so you bolt on an <code>Idempotency-Key</code>. Client-generated (e.g. client picks a UUID): the create becomes a natural <strong>idempotent upsert</strong> via <code>PUT /orders/{client-uuid}</code> — a retry targets the same URI and is safe by construction, and offline/mobile clients can mint ids without a round-trip. The cost is trusting clients with the id namespace (collision/forgery handling). Choose by whether you need offline id minting and free idempotency.</p>
    - level: 'staff'
      q: 'Whiteboard prompt: model an e-commerce checkout (browse cart, place order, cancel, refund).'
      a: >-
          <p>Cart is <code>/carts/{id}</code> with line items at <code>/carts/{id}/items/{id}</code> — one level, real containment. Place order: <code>POST /orders</code>, server assigns id, returns <code>201</code> + <code>Location</code>, built from the cart; require an <code>Idempotency-Key</code> so a retry doesn't double-charge. Cancel is a transition on an existing order, so <code>POST /orders/{id}/cancellations</code> or AIP-style <code>POST /orders/{id}:cancel</code> — no top-level noun. Refund has its own lifecycle (processed/settled/reversible) so <strong>reify</strong> it: <code>POST /refunds</code> with <code>{order_id}</code>, beside <code>/payments</code>. Ids opaque (UUID/ULID); list a customer's orders via <code>/orders?customer_id=</code> rather than deep nesting. That hits nouns-not-verbs, principled action modeling with a stated trade-off, shallow nesting + filtering, and security-aware ids.</p>
    - level: 'staff'
      q: 'What''s your test for "does this action deserve to be its own resource?"'
      a: >-
          <p>Ask: does it have a <strong>lifecycle, state, and history you'd query independently</strong>? If yes — you'd list it, report on it, transition it through states, audit it — reify (a refund, a shipment, a dispute). If it's a one-shot transition with no afterlife you'd ever enumerate, don't invent a noun nobody reads; use a custom method or status sub-resource. The litmus test is "would I ever <code>GET</code> a collection of these?"</p>
    - level: 'staff'
      q: 'How does URI shape interact with object-level authorization?'
      a: >-
          <p>Identity-in-path makes the authorization target explicit: every <code>/orders/{id}</code> access needs an ownership check (BOLA defense), and consistent URI structure lets you enforce that with uniform middleware rather than ad-hoc checks. Aliases like <code>/users/me</code> remove a whole class of IDOR by not letting clients name other principals. Nesting can <em>imply</em> scope (<code>/customers/1/orders</code>) but you must still verify the caller may act as customer 1 — the path is a claim, not proof. Flat URIs + per-object authz is usually cleaner than relying on nesting for security.</p>
    - level: 'staff'
      q: 'When would you accept verbs in paths or otherwise break the conventions?'
      a: >-
          <p>Pragmatically: explicitly-sanctioned custom methods (<code>:cancel</code>) for non-CRUD verbs; a <code>POST .../search</code> when query criteria are too large/sensitive for the URL (trading GET cacheability knowingly); RPC-style or gRPC for chatty internal service-to-service traffic where REST's overhead doesn't pay. The staff move is that these are <strong>deliberate, named exceptions with a stated cost</strong>, isolated and consistent, not accidental drift. Purity isn't the goal — defensible, evolvable design is.</p>
    - level: 'design'
      q: 'Model an e-commerce checkout (cart, place order, cancel, refund) on a whiteboard.'
      a: >-
          <p><b>A strong answer covers:</b> Cart is <code>/carts/{id}</code> with line items at <code>/carts/{id}/items/{id}</code> — one level, real containment. Place order: <code>POST /orders</code>, server assigns id, returns <code>201</code> + <code>Location</code>, built from the cart; require an <code>Idempotency-Key</code> so a retry doesn't double-charge. Cancel is a transition on an existing order, so <code>POST /orders/{id}/cancellations</code> or AIP-style <code>POST /orders/{id}:cancel</code> — no top-level noun. Refund has its own lifecycle (processed/settled/reversible) so <strong>reify</strong> it: <code>POST /refunds</code> with <code>{order_id}</code>, beside <code>/payments</code>. Ids opaque (UUID/ULID); list a customer's orders via <code>/orders?customer_id=</code> rather than deep nesting. That hits nouns-not-verbs, principled action modeling with a stated trade-off, shallow nesting + filtering, and security-aware ids.</p>

      q: 'Design the resource model for a multi-tenant project-management API with users, projects, tasks, comments, and memberships.'
      a: >-
          <p><b>A strong answer covers:</b> Top-level collections by identity: <code>/projects</code>, <code>/tasks</code>, <code>/users</code> — each has its own life. Scope tenancy implicitly from the authenticated principal (resolve the tenant from the token), not via a <code>/tenants/{id}/…</code> prefix clients can tamper with. Tasks belong to a project, but keep nesting shallow: <code>/projects/{id}/tasks</code> for the owned list, and address a single task flatly by identity as <code>/tasks/{id}</code> so it doesn't break if it's re-homed; filter cross-cuts with <code>/tasks?assignee=…&amp;status=open</code>. Comments are owned by their parent and have no independent life, so <code>/tasks/{id}/comments</code> (one level). Membership is a many-to-many between users and projects that <em>carries data</em> (role, joined-at), so <strong>reify the join</strong>: <code>/memberships</code> (or <code>/projects/{id}/members</code> as a filtered view) rather than deep-nesting under one side. Ids opaque (UUID/ULID); enforce per-object authorization on every access (BOLA) <em>and</em> tenant isolation so user A can't read project B in another tenant — the path is a claim, not proof. Use <code>/users/me</code> for the caller. Conventions consistent throughout; <code>?expand=</code> bounded by an allowlist if clients need related data inline.</p>

rubricIntro: >-
    <b>A framework for whiteboarding a REST resource model — narrate each step out loud:</b>
rubric:
    - >-
        <strong>Clarify the domain &amp; list candidate nouns.</strong> Name the things the domain holds (order, cart, customer, payment, refund). Those are your candidate resources before any URL is drawn.
    - >-
        <strong>Top-level collections vs owned sub-resources, by identity.</strong> Ask "does this have identity on its own?" If yes it's a top-level collection; if it only exists inside a parent, it's an owned sub-resource (one level).
    - >-
        <strong>Map verbs to HTTP methods; place non-CRUD verbs deliberately.</strong> CRUD → standard methods. For genuine verbs (cancel, publish, refund) choose sub-resource, custom method (<code>:verb</code>), or reify-as-resource — driven by lifecycle, not grammar.
    - >-
        <strong>Relationships: one-level nesting for ownership, filtering otherwise.</strong> Nest <code>/customers/1/orders</code> only for true containment; express the rest with <code>/orders?customer_id=1</code>. Reify many-to-many joins that carry data.
    - >-
        <strong>Identifiers: opaque external ids + per-object authz.</strong> Use UUID/ULID, never the raw DB key. Opaque ids are defense in depth, not the authz fix — every object access still needs an ownership (BOLA) check.
    - >-
        <strong>Conventions &amp; consistency.</strong> Lowercase, hyphenated, plural collections, no extensions; query carries filter/sort/page, never identity. Predictability across the surface beats local cleverness.
    - >-
        <strong>Evolvability / versioning cost.</strong> Stable nouns, identity-in-path, and external ids let you add fields and expansions without breaking clients — name where each choice would force a version bump later.
