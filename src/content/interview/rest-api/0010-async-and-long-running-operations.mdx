# —— interview Q&A (verbatim from <div data-interview>) ——
interview:
    - level: 'core'
      q: 'When should an API handle work asynchronously instead of in the request?'
      a: >-
          <p>The moment the work outlives the request: <strong>long jobs</strong> (data exports, video transcode, batch recompute), calls into <strong>slow external dependencies</strong> you don't control, and anything that risks blowing a gateway or load-balancer timeout. If you can't reliably finish inside a few seconds, don't hold the connection — return immediately and let the client track the work.</p>
    - level: 'core'
      q: "What's wrong with just holding the connection open for a few minutes?"
      a: >-
          <p>It's a resource leak and a reliability trap. Open connections tie up server/proxy resources, and intermediaries (LBs, gateways, proxies) impose their own timeouts that will kill it anyway. Worse, a dropped socket loses the client's <em>only handle</em> on work that's still running — they can't tell if it succeeded, failed, or is mid-flight.</p>
    - level: 'core'
      q: >-
          What does <code>202 Accepted</code> mean?
      a: >-
          <p>"I've <strong>accepted</strong> the request for processing, but it <em>isn't done</em> and may not even have started." It's deliberately non-committal about the outcome — unlike <code>201 Created</code>, nothing is guaranteed to exist yet. The response points the client at where to check progress.</p>
    - level: 'core'
      q: 'Walk through the async request–reply (202 + polling) pattern.'
      a: >-
          <p>(1) Client <code>POST</code>s the work. (2) Server accepts it, kicks off processing, returns <code>202 Accepted</code> with a <code>Location</code> header to a status/operation resource like <code>/operations/{id}</code>. (3) Client <strong>polls</strong> that resource with <code>GET</code>, which returns state (<code>pending</code>/<code>running</code>). (4) When finished, the status shows <code>done</code>/<code>succeeded</code> plus a link to the actual <strong>result resource</strong>.</p>
    - level: 'core'
      q: >-
          What does the <code>Location</code> header point to in a 202 response?
      a: >-
          <p>To the <strong>status/operation resource</strong> the client should poll (e.g. <code>/jobs/{id}</code> or <code>/operations/{id}</code>), <em>not</em> to the final result. Contrast with <code>201 Created</code>, where <code>Location</code> points at the newly created resource itself.</p>
    - level: 'core'
      q: 'What is a webhook, and how does it differ from polling?'
      a: >-
          <p>A webhook inverts the direction: instead of the client polling, the client <strong>registers a callback URL</strong> and the server <code>POST</code>s an event to it when something happens. It's push-based — efficient and near-real-time — but it makes the client a server and hands it a fistful of distributed-systems problems (duplicates, ordering, auth).</p>
    - level: 'core'
      q: 'What delivery guarantee do webhooks give, and what follows from it?'
      a: >-
          <p>Delivery is <strong>at-least-once</strong>: failed deliveries are retried, so the same event can arrive more than once. The direct consequence: the <strong>consumer must be idempotent</strong> and <strong>dedupe on a stable event id</strong>. Assuming exactly-once delivery is the classic bug — it's how you bill a customer twice.</p>
    - level: 'core'
      q: 'Anyone can POST to a public callback URL. How do you trust the payload?'
      a: >-
          <p><strong>Sign the payload.</strong> The sender computes an <strong>HMAC</strong> over the body using a shared signing secret and puts it in a header (e.g. a Stripe-style <code>Stripe-Signature</code> header). The receiver recomputes the HMAC with its copy of the secret and rejects the request unless they match — proving the event genuinely came from the sender and wasn't tampered with.</p>
    - level: 'core'
      q: 'Polling vs webhooks — the one-line trade-off each.'
      a: >-
          <p><strong>Polling (202 + status):</strong> dead simple, no client endpoint needed, works through firewalls — but wasteful and laggy (you learn on the next tick). <strong>Webhooks:</strong> efficient, near-real-time push — but the client must expose a public endpoint and handle every delivery semantic (duplicates, ordering, signing, retries).</p>
    - level: 'senior'
      q: "What's the core mental shift when moving from sync to async design?"
      a: >-
          <p><strong>Turn the work into a resource.</strong> Instead of returning the result, you create a first-class <em>operation</em>/<em>job</em> resource the client can address, query, and recover against. The request returns fast with a handle to that resource; the result becomes a separate resource the operation links to once it's done.</p>
    - level: 'senior'
      q: 'Give concrete examples of work that belongs behind an async API.'
      a: >-
          <p>Large CSV/PDF <strong>exports</strong>, <strong>media transcoding</strong>, <strong>bulk/batch</strong> imports and recomputations, ML inference/training jobs, report generation, and anything fanning out to a <strong>slow third party</strong> (payment settlement, KYC, shipping carriers). Rule of thumb: if completion time is unbounded or measured in tens of seconds-plus, make it async.</p>
    - level: 'senior'
      q: >-
          Describe Google's long-running <code>Operation</code> model.
      a: >-
          <p>A uniform envelope (Google AIP-151) carrying <code>name</code>, a <code>done</code> boolean, and — once done — <strong>either</strong> an <code>error</code> <strong>or</strong> a <code>response</code> (never both). Clients poll <code>GET /operations/{id}</code> until <code>done:true</code>, then read the result or error from the same shape. The win is a consistent operation surface across every long job in the API rather than a bespoke status format per endpoint.</p>
    - level: 'senior'
      q: 'How should you control how often clients poll?'
      a: >-
          <p>Use <code>Retry-After</code> on the status response so the server dictates a sane interval and clients don't hammer you; pair it with client-side <strong>backoff</strong> (and jitter) toward a ceiling. The server stays in control of its own load, and clients converge on a polite poll cadence instead of tight-looping.</p>
    - level: 'senior'
      q: 'Why keep the operation queryable after it has completed?'
      a: >-
          <p>So a client that <strong>crashed or lost the network mid-poll can recover</strong> — it re-queries the operation id and learns the outcome and result link. If you delete the operation the instant it finishes, you create a race where the client never learns it succeeded. Retain completed operations for a sensible TTL.</p>
    - level: 'senior'
      q: 'Should the result live in the operation resource or a separate resource? Why?'
      a: >-
          <p>A <strong>separate result resource</strong> the operation links to. It keeps the operation envelope small and uniform, lets the result have its own lifecycle (expiry, auth, content type — e.g. a signed, expiring download URL), and avoids stuffing large payloads into a polling response that clients fetch repeatedly.</p>
    - level: 'senior'
      q: "Webhooks aren't ordered. How do you build a correct consumer anyway?"
      a: >-
          <p>Carry a <strong>timestamp and/or sequence number</strong> in each event and never assume the last event you received is the latest write. Reconcile against current state (e.g. "ignore an update older than what I've already applied"), or fetch authoritative state from a status resource. Design so out-of-order arrival is merely <em>handled</em>, not <em>fatal</em>.</p>
    - level: 'senior'
      q: 'Why should a webhook receiver ack fast and process later?'
      a: >-
          <p>Return a quick <code>2xx</code> the instant you've durably <em>captured</em> the event, then do heavy work asynchronously. Doing the work before acking ties up the sender's delivery attempt, risks its timeout (→ spurious retries and duplicates), and couples your processing latency to their retry policy. Ack = "I have it safely," not "I've finished acting on it."</p>
    - level: 'senior'
      q: >-
          What happens when the receiver returns a non-<code>2xx</code> (or times out)?
      a: >-
          <p>The sender treats it as a failed delivery and <strong>retries with backoff</strong> (typically exponential + jitter, over hours). After enough give-ups it routes the event to a <strong>dead-letter</strong> for inspection/manual replay, and usually surfaces the failing endpoint to the developer. This is exactly why at-least-once duplicates happen — a slow receiver that <em>did</em> process but failed to ack gets the event again.</p>
    - level: 'senior'
      q: 'Explain dead-lettering for webhooks.'
      a: >-
          <p>When deliveries keep failing past the retry budget, the event is parked in a <strong>dead-letter queue/store</strong> instead of being dropped or retried forever. It preserves the event for diagnosis and lets you <strong>replay</strong> it once the endpoint is fixed. It bounds wasted retries while guaranteeing you don't silently lose events.</p>
    - level: 'senior'
      q: 'How do you dedupe reliably on the consumer side?'
      a: >-
          <p>Persist processed <strong>event ids</strong> (with a TTL matching the sender's max retry window) and check-then-process atomically — e.g. an upsert/unique-constraint on event id, or an idempotency table — so a duplicate is a no-op. Make the <em>effect</em> idempotent too (use the event id as the key for any side effect), so even a race that processes twice converges to one outcome.</p>
    - level: 'senior'
      q: "Why HMAC over the raw body specifically, and what's the common pitfall?"
      a: >-
          <p>HMAC ties authenticity <em>and</em> integrity to a shared secret cheaply. The classic pitfall: verifying against a re-serialized/parsed body instead of the <strong>exact raw bytes</strong> received — any reformatting breaks the signature. Capture the raw payload before JSON parsing, and use a <strong>constant-time compare</strong> to avoid timing attacks.</p>
    - level: 'senior'
      q: 'A valid signed request can be captured and resent. How do you stop replay?'
      a: >-
          <p>Include a <strong>timestamp in the signed payload</strong> and <strong>reject stale ones</strong> (outside a small tolerance window, e.g. a few minutes). A replayed event then fails even though its signature is genuine, because it's too old. Combine with consumer-side dedupe on event id so a replay inside the window is also a no-op.</p>
    - level: 'senior'
      q: 'How do you rotate a webhook signing secret without dropping events?'
      a: >-
          <p>Support <strong>multiple active secrets</strong> during an overlap window: the sender can sign with the new secret (or send signatures for both), and the receiver accepts a payload if it validates against <em>any</em> currently-valid secret. Roll forward, confirm traffic verifies on the new one, then retire the old. Never hard-cut a single secret — in-flight/retried events would fail verification.</p>
    - level: 'senior'
      q: 'Where do SSE, WebSockets, and long-polling fit?'
      a: >-
          <p>They're the <strong>streaming</strong> shape — for low-latency, continuous updates where a per-event <code>POST</code> is too coarse. <strong>SSE</strong>: simple server→client stream over HTTP, auto-reconnect. <strong>WebSockets</strong>: full duplex, for interactive/bidirectional. <strong>Long-poll</strong>: a fallback that mimics push by holding a request open. The cost is a <strong>held connection</strong> (statefulness, scaling, reconnection logic).</p>
    - level: 'senior'
      q: >-
          A client is behind a corporate firewall / can't run a public server. Which pattern?
      a: >-
          <p><strong>Polling</strong>, because webhooks require the client to expose an inbound, internet-reachable endpoint — often impossible behind NAT/firewalls or for browser/mobile clients. Polling only needs outbound calls, so it works everywhere. This constraint, not just latency, frequently decides the pattern.</p>
    - level: 'senior'
      q: >-
          Why do mature APIs often offer <em>both</em> a status endpoint and webhooks?
      a: >-
          <p>It's belt-and-suspenders, not indecision. The <strong>webhook</strong> delivers the happy-path, near-real-time push; the <strong>pollable status resource</strong> is the fallback for <em>missed deliveries</em>, crash recovery, and clients that can't receive webhooks. Together they give push performance with poll reliability — the consumer can always reconcile authoritative state.</p>
    - level: 'senior'
      q: 'Design an API for a large data export that may take minutes, then notifies the client.'
      a: >-
          <p>Don't block. <code>POST /exports</code> → <code>202 Accepted</code> + <code>Location</code> to <code>/exports/{id}</code> (a status resource). Client <strong>polls</strong> (<code>pending</code>→<code>running</code>→<code>succeeded</code>), honoring <code>Retry-After</code>; on done it links to a <strong>result resource</strong> — a signed, expiring download URL. <strong>Also</strong> offer a completion webhook: a <strong>signed (HMAC)</strong> event <code>POST</code>. Because delivery is at-least-once, the consumer dedupes on <strong>event id</strong> and stays idempotent, doesn't rely on ordering, and uses a timestamp to reject replays. Receiver acks fast <code>2xx</code> and works async; non-<code>2xx</code> → retries with backoff → dead-letter. Protect the result URL with auth + expiry. Offering both push and poll, plus securing the result, is the senior signal.</p>
    - level: 'senior'
      q: 'Walk me through making a webhook consumer robust.'
      a: >-
          <ul><li><strong>Verify the signature</strong> (HMAC over raw body, constant-time compare) and reject if stale (timestamp window) — auth + anti-replay.</li><li><strong>Capture durably, ack fast</strong> with <code>2xx</code>, then process async.</li><li><strong>Dedupe on event id</strong> and make side effects idempotent — at-least-once is the default.</li><li><strong>Don't trust order</strong>: reconcile using timestamps/sequence or re-fetch state.</li><li><strong>Handle failure</strong>: let the sender retry; have your own retry + dead-letter for downstream errors after the ack.</li></ul>
    - level: 'senior'
      q: 'Two events for the same object arrive out of order. What goes wrong, and how do you handle it?'
      a: >-
          <p>The naive failure: you apply the <em>older</em> event last and clobber newer state (e.g. "active" then "cancelled" arrives reordered → you end up "active"). Fix by making each event carry a <strong>monotonic version/timestamp</strong> and applying it conditionally — ignore an event older than the state you've already recorded — or by treating the webhook as a trigger to <strong>re-fetch authoritative state</strong> rather than blindly applying the payload.</p>
    - level: 'senior'
      q: 'The same payment-succeeded webhook is delivered twice. How do you avoid double-fulfilment?'
      a: >-
          <p>Use the <strong>event id</strong> (or a business key like the payment/charge id) as an idempotency key for fulfilment: check-then-act atomically (unique constraint / conditional write) so the second delivery is a no-op. This is Lesson 3 cashing in — at-least-once delivery is fine precisely <em>because</em> the effect is idempotent.</p>
    - level: 'staff'
      q: 'A sync endpoint occasionally takes 90 seconds. Do you just bump the timeout?'
      a: >-
          <p>No — raising timeouts pushes the failure surface around without fixing it. Long-held connections still break under deploys, LB recycling, and client networks, and they hurt throughput and tail latency for everyone. The staff move is to convert the slow path to async (<code>202</code> + status resource), keep the fast path sync, and possibly split the endpoint by expected cost so the common case stays snappy.</p>
    - level: 'staff'
      q: 'How do idempotency keys interact with the 202 submission?'
      a: >-
          <p>The submitting <code>POST</code> is not idempotent, so a retried submission could spawn a duplicate job. Accept an <code>Idempotency-Key</code> (Lesson 3): on a repeat, return the <em>same</em> <code>202</code> and <code>Location</code> for the already-created operation instead of starting a second one. That makes "submit a long job" safely retryable — the client gets one operation no matter how many times the request is replayed.</p>
    - level: 'staff'
      q: 'How would you let a client cancel a long-running operation?'
      a: >-
          <p>Model cancellation as an action on the operation — e.g. <code>POST /operations/{id}:cancel</code> (or <code>DELETE</code>) — and treat it as a <strong>request</strong>, not a guarantee: set the state to <code>cancelling</code>, signal the worker, and let it reach a terminal <code>cancelled</code> when it actually stops. Make cancel idempotent and define behavior for work that already finished (it's a no-op / terminal-state response). The honest framing is cooperative cancellation, not a hard kill.</p>
    - level: 'staff'
      q: >-
          "Just give me exactly-once delivery." How do you respond?
      a: >-
          <p>Exactly-once <em>delivery</em> is effectively unachievable over an unreliable network (the two-generals problem). What you can get is exactly-once <strong>effect</strong>: at-least-once delivery plus an idempotent consumer keyed on the event id. The senior framing is "I want exactly-once <em>effect</em>, not exactly-once delivery" — and then you build dedupe rather than chase an impossible guarantee.</p>
    - level: 'staff'
      q: 'Beyond signatures, what else hardens a webhook endpoint?'
      a: >-
          <p><strong>HTTPS only</strong> (confidentiality + integrity in transit); optional <strong>source-IP allowlisting</strong> if the sender publishes stable ranges (defense-in-depth, not a substitute for signing); rate limiting / payload-size caps to blunt abuse; and treating the endpoint as untrusted input — validate, dedupe, and never let an unsigned/invalid request trigger side effects. Signatures prove <em>who</em>; transport + IP + limits reduce the attack surface.</p>
    - level: 'staff'
      q: 'How does pattern choice change as the event rate scales 100×?'
      a: >-
          <p>Polling cost scales with <em>clients × poll frequency</em> regardless of activity, so at high fan-out it wastes enormous request volume on "nothing changed" — push (webhooks/streaming) wins. But webhooks then concentrate <strong>delivery, retry, and dead-letter</strong> load on you, demanding a durable queue, batching/coalescing of events, and per-endpoint backpressure so one slow consumer can't stall the fleet. The trade-off flips from client simplicity toward sender-side delivery infrastructure.</p>
    - level: 'staff'
      q: "A customer's webhook endpoint has been down for two hours. What should your platform do?"
      a: >-
          <p><strong>Retry with exponential backoff + jitter</strong> over a bounded window (hours), then <strong>dead-letter</strong> undelivered events and surface the failing endpoint to the customer (dashboard/alert). Critically, give them a <strong>recovery path that doesn't depend on the webhook</strong>: a list/status API to fetch events or current state for the gap, plus the ability to <strong>replay</strong> dead-lettered events. Don't retry forever (it amplifies load and duplicates); don't drop silently (they lose data). Belt-and-suspenders: the pollable resource is the safety net.</p>
    - level: 'staff'
      q: 'How do you keep one slow or failing consumer from degrading webhook delivery for everyone?'
      a: >-
          <p><strong>Isolate per-consumer:</strong> independent delivery queues, per-endpoint concurrency limits and circuit breakers so a timing-out endpoint backs off without starving others; bound and parallelize retries; and dead-letter aggressively rather than letting a black-hole endpoint consume the retry budget. The principle is <strong>bulkheading</strong> — failure of one tenant's endpoint must not become a shared-fate outage of the delivery pipeline.</p>
    - level: 'design'
      q: "Design an API for a large data export that can take several minutes, then notifies the client when it's ready."
      a: >-
          <p><b>A strong answer covers:</b> don't block &mdash; <code>POST /exports</code> &rarr; <code>202 Accepted</code> + <code>Location</code> to <code>/exports/{id}</code>, a status resource &rarr; client <strong>polls</strong> (<code>pending</code>&rarr;<code>running</code>&rarr;<code>succeeded</code>) honoring <code>Retry-After</code>; on done it links to a <strong>result resource</strong> &mdash; a signed, expiring download URL &rarr; <strong>also</strong> offer a completion webhook: an HMAC-signed event <code>POST</code> &rarr; because delivery is at-least-once, the consumer dedupes on <strong>event id</strong> and stays idempotent, doesn't rely on ordering, and uses a timestamp to reject replays &rarr; receiver acks fast <code>2xx</code> and works async; non-<code>2xx</code> &rarr; retries with backoff &rarr; dead-letter &rarr; protect the result URL with auth + expiry &rarr; keep the operation queryable after completion so a client that crashed mid-poll can recover &rarr; name the trade-off: offering both push and poll buys push performance with poll reliability.</p>
    - level: 'design'
      q: 'Design a webhook delivery platform: how do you guarantee customers eventually get every event without one bad endpoint taking the system down?'
      a: >-
          <p><b>A strong answer covers:</b> persist every event durably first, then deliver from a queue &mdash; never couple production of the event to its delivery &rarr; delivery is <strong>at-least-once</strong>: sign each payload (HMAC over raw bytes, constant-time compare) and include a timestamp so consumers can verify authenticity and reject replays &rarr; expect non-<code>2xx</code>/timeouts &rarr; retry with exponential backoff + jitter over a bounded window (hours), then <strong>dead-letter</strong> and surface the failing endpoint to the customer &rarr; give a webhook-independent recovery path: a list/status API to fetch the gap and the ability to <strong>replay</strong> dead-lettered events (belt-and-suspenders with a pollable resource) &rarr; isolate per consumer &mdash; independent queues, per-endpoint concurrency limits and circuit breakers (bulkheading) so one slow/black-hole endpoint can't starve the fleet &rarr; coalesce/batch at high fan-out and apply per-endpoint back-pressure &rarr; observability: track delivery success, retry counts, and lag per endpoint &rarr; name the trade-off: exactly-once <em>delivery</em> is unachievable over an unreliable network, so you provide at-least-once delivery + an idempotent consumer keyed on event id (exactly-once <em>effect</em>).</p>

    <b>Async / long-running design framework</b> &mdash; drive any "design a long-running operation" prompt through these, out loud:
rubric:
    - 'Clarify: expected duration, completion SLA, who the client is (can it receive a callback?), idempotency &amp; consistency needs.'
    - >-
        Make the work a resource: <code>POST</code> &rarr; <code>202 Accepted</code> + <code>Location</code> to a status/operation resource; keep it queryable after completion (TTL) for crash recovery.
    - >-
        Model the operation as a state machine: <code>queued &rarr; running &rarr; succeeded | failed | cancelled</code>; <code>done=true</code> resolves to exactly one of result or error; link the result as its own resource.
    - >-
        Choose the notification axis: polling (<code>Retry-After</code> + client backoff, works behind firewalls) vs signed webhooks (push, near-real-time) &mdash; mature APIs offer both, poll as the fallback.
    - >-
        Make submission safe: <code>Idempotency-Key</code> so a retried submit returns the same operation, not a duplicate job.
    - >-
        Webhook delivery semantics: at-least-once &amp; unordered &rarr; consumer dedupes on event id, stays idempotent, carries timestamp/sequence; HMAC-sign + reject stale (replay window); ack fast <code>2xx</code> then work async.
    - >-
        Reliability &amp; scale: retries with backoff + jitter &rarr; dead-letter; per-consumer isolation/bulkheading; observability on delivery, retry, and lag.
