# —— interview Q&A (verbatim from <div data-interview>) ——
interview:
    - level: 'core'
      q: >-
          What is a "design X's API" round actually testing?
      a: >-
          <p>Not whether you produce <em>the</em> answer — there isn't one. It grades <strong>structure</strong> (did you clarify before designing?), <strong>breadth</strong> (writes got idempotency, reads got pagination, you named object-level auth), and <strong>judgment</strong> (you named a trade-off and what changes at 100×). It's a 45-minute integration exam over every prior topic at once, not a quiz on one.</p>
    - level: 'core'
      q: >-
          What should you do <em>first</em> in a design round, before any endpoint?
      a: >-
          <p>Clarify requirements and constraints — clients, scale, read/write ratio, consistency needs, auth model, SLAs. <strong>Diagnose, then design.</strong> The resource model falls out of the questions you ask; the endpoints fall out of the resource model. Drawing URIs before clarifying is the classic junior tell.</p>
    - level: 'core'
      q: >-
          What's the winning arc of a design answer in one phrase?
      a: >-
          <p><strong>Diagnose, design, defend.</strong> Clarify constraints first, walk the resource model and endpoints with a reason attached to each choice, then close with a trade-off and what changes at scale. A candidate who narrates a clear checklist and the <em>why</em> beats one who free-associates brilliant fragments.</p>
    - level: 'core'
      q: >-
          For any "design X" prompt, what two things must every <em>list</em> endpoint have?
      a: >-
          <p><strong>Pagination</strong> (cursor-based at scale, with a bounded/hard-capped page size) and a <strong>filter/sort contract</strong>. An unbounded list is a latency and memory landmine. Reaching for these on every collection without being prompted is a fast breadth signal — the mirror of reaching for idempotency on every unsafe write.</p>
    - level: 'core'
      q: >-
          Which write should return <code>202</code> instead of <code>201</code>?
      a: >-
          <p>A write whose work hasn't finished yet. <code>201 Created</code> means the resource exists now; <strong><code>202 Accepted</code></strong> means "I've taken the request and will process it asynchronously" — return it for async capture, job submission, fan-out, or anything long-running, along with a status resource the client can poll.</p>
    - level: 'core'
      q: >-
          Which HTTP methods are already idempotent, so they don't need an idempotency key?
      a: >-
          <p><code>GET</code>, <code>HEAD</code>, <code>PUT</code>, and <code>DELETE</code> are idempotent by HTTP semantics — repeating them has the same effect as doing them once. <code>POST</code> is the exception: it's not idempotent, so unsafe <code>POST</code>s (create, charge) are the ones that need an <code>Idempotency-Key</code>.</p>
    - level: 'core'
      q: >-
          Why prefer a structured error format (RFC 9457) over a bare string?
      a: >-
          <p>So clients can react programmatically and humans can debug. RFC 9457 problem+json carries a stable machine-readable <code>code</code>/<code>type</code>, a human <code>title</code>/<code>detail</code>, and a per-request <code>trace_id</code> that ties the response to your logs/traces. A bare string forces clients to brittle string-matching and gives you nothing to grep on when it breaks.</p>
    - level: 'core'
      q: >-
          What makes an API horizontally scalable in the first place?
      a: >-
          <p><strong>Statelessness</strong> — each request carries everything the server needs, so any node can serve any request. No sticky sessions, so you scale by adding boxes behind a load balancer and a node dying loses nothing. That property is the foundation everything else (caching, replicas, sharding) builds on.</p>
    - level: 'core'
      q: >-
          "Why is object-level authorization so important?"
      a: >-
          <p>Because authentication only proves <em>who</em> you are, not <em>what</em> you're allowed to touch. Without a per-object ownership check, a logged-in user A can pass user B's id in the path (<code>/orders/{B}</code>) and read B's data — that's <strong>BOLA</strong>, the #1 real-world API vulnerability. Every read/write must verify the caller owns <em>this</em> object, not just that they're authenticated.</p>
    - level: 'senior'
      q: >-
          What's the single biggest mistake candidates make in this round?
      a: >-
          <p>Starting to type endpoints. <strong>Don't design — diagnose, then design.</strong> The first five minutes are requirements and constraints. The endpoints fall out of the resource model; the resource model falls out of the questions you asked. Leading with verbs before you know the clients, scale, and consistency needs signals junior.</p>
    - level: 'senior'
      q: >-
          Name the clarifying questions you ask before drawing anything.
      a: >-
          <ul>       <li><strong>Clients:</strong> first-party only, or untrusted third parties (changes auth, rate limits, versioning rigor)?</li>       <li><strong>Scale &amp; read/write ratio:</strong> read-heavy (cache, replicas) vs write-heavy (contention, idempotency)?</li>       <li><strong>Consistency:</strong> which paths must be strongly consistent vs eventually consistent?</li>       <li><strong>Auth model</strong> and tenancy: who owns what; multi-tenant isolation?</li>       <li><strong>SLAs / latency budget</strong> and the one operation that absolutely cannot fail (e.g. never double-charge).</li>     </ul><p>Pin these before any URI.</p>
    - level: 'senior'
      q: >-
          Walk the 11-step framework from memory.
      a: >-
          <p>(1) Clarify requirements/constraints; (2) resource model &amp; URIs; (3) endpoints — methods + status codes; (4) idempotency on unsafe writes; (5) collections — pagination/filter/sort; (6) caching &amp; concurrency (ETag/If-Match); (7) versioning &amp; evolution; (8) authN/authZ + object-level (BOLA); (9) errors, rate limiting, observability; (10) async/webhooks where needed; (11) trade-offs &amp; "what changes at 100×." Name the step, make the one decision that matters, signal you know the rest is there.</p>
    - level: 'senior'
      q: >-
          How do you drive a whiteboard round — manage time and structure?
      a: >-
          <p>Narrate the framework as a spine so the interviewer can follow. Box the first ~5 min for clarify + resource model (the part that earns the most). Don't boil the ocean: name each step, make the headline decision, move on — <em>"I'd also add ETags here, but let me keep moving and come back if you want depth."</em> Leave the last few minutes for trade-offs at scale. Re-state assumptions when they twist a constraint, and think out loud so they grade your reasoning, not just your output.</p>
    - level: 'senior'
      q: >-
          How do you communicate a trade-off so it reads as "senior"?
      a: >-
          <p>State the option, <strong>pick a side</strong>, attach the reason, and name when you'd decide differently: <em>"That's a deliberate trade-off because X; at 100× I'd revisit it because Y."</em> Defaulting silently (offset pagination, no idempotency) reads junior; choosing and defending reads senior. Other tells: <em>"it's at-least-once, so the consumer must be idempotent,"</em> <em>"I care about p99, not the mean,"</em> <em>"exactly-once effect, not exactly-once delivery."</em></p>
    - level: 'senior'
      q: >-
          Design the API for a URL shortener (create, redirect, analytics).
      a: >-
          <ul>       <li><strong>Clarify:</strong> read-dominated (redirects ≫ creates), public reads but authed creates, custom-alias support, latency-critical redirect path.</li>       <li><strong>Resources/writes:</strong> <code>POST /links</code> {url, custom_alias?} → <code>201</code>; make it <strong>idempotent</strong> via <code>Idempotency-Key</code> or by hashing the long URL so retries don't mint duplicate codes.</li>       <li><strong>Redirect:</strong> <code>GET /{code}</code> → <strong>301</strong> for permanent/SEO vs <strong>302</strong> when you must count every hit or may re-point. 301 is cached hard so analytics suffer — a deliberate trade-off.</li>       <li><strong>Analytics:</strong> <code>GET /links/{code}/stats</code> with cursor pagination; counts are eventually consistent, aggregated async off a click stream, never on the hot redirect path.</li>       <li><strong>Protect &amp; auth:</strong> rate-limit creates per key (<code>429</code>), object-level ownership on stats/delete, edge-cache redirects.</li>       <li><strong>At 100×:</strong> precompute codes from a key-gen service to avoid write contention; the redirect becomes a cache lookup, not a DB hit.</li>     </ul>
    - level: 'senior'
      q: >-
          Design the API for a notifications service (multi-channel send, preferences, delivery status).
      a: >-
          <ul>       <li><strong>Clarify:</strong> fan-out across channels (push/email/SMS), each provider is flaky and async, users have preferences/quiet hours, dedup matters.</li>       <li><strong>Resources:</strong> <code>Notification</code>, <code>Template</code>, <code>Preference</code>, <code>DeliveryAttempt</code>. <code>POST /notifications</code> → <code>202 Accepted</code> + a status resource (sending is inherently async).</li>       <li><strong>Idempotency:</strong> <code>Idempotency-Key</code> so a retried send doesn't notify the user twice — the whole point of the service is to not spam.</li>       <li><strong>Status:</strong> <code>GET /notifications/{id}</code> shows per-channel delivery state; or push results via signed at-least-once webhooks.</li>       <li><strong>Lists/auth:</strong> <code>GET /notifications?status=&amp;limit=</code> cursor-paginated; object-level checks; per-tenant rate caps to protect providers and budget.</li>       <li><strong>At 100×:</strong> durable queue per channel with retry/back-off + DLQ for poison messages; respect provider rate limits as a back-pressure source.</li>     </ul>
    - level: 'senior'
      q: >-
          Design a rate-limited public API (developer-facing, plans, quotas).
      a: >-
          <ul>       <li><strong>Clarify:</strong> untrusted third-party developers, tiered plans, abuse is expected, contract stability is a product promise.</li>       <li><strong>Auth:</strong> API keys or OAuth2 client-credentials with scopes; key identifies the plan/quota bucket.</li>       <li><strong>Limiting:</strong> <strong>token-bucket per key</strong>, returning <code>429</code> with <code>RateLimit-Limit/Remaining/Reset</code> and <code>Retry-After</code>; document the algorithm so clients can back off correctly.</li>       <li><strong>Errors/observability:</strong> RFC 9457 problem+json with a stable machine <code>code</code> and a per-request <code>trace_id</code>; publish an OpenAPI spec and a changelog.</li>       <li><strong>Versioning:</strong> additive-first; breaking changes get a version bump + <code>Sunset</code> header and a long deprecation window — you can't redeploy their clients.</li>       <li><strong>At 100×:</strong> distributed limit counters (Redis), per-endpoint cost weighting, edge caching, and a separate burst tier so one abuser can't starve neighbors.</li>     </ul>
    - level: 'senior'
      q: >-
          Design the API for a job/task queue service (submit work, poll status, get result).
      a: >-
          <ul>       <li><strong>Clarify:</strong> work is long-running and may fail/retry; clients want submit-and-forget plus a way to track and fetch results.</li>       <li><strong>Resources:</strong> <code>Job</code> with a <code>status</code> (queued/running/succeeded/failed). <code>POST /jobs</code> → <strong><code>202 Accepted</code></strong> + a status resource (this is the canonical async pattern, not <code>201</code>).</li>       <li><strong>Idempotency:</strong> <code>Idempotency-Key</code> on submit so a retry doesn't enqueue the work twice.</li>       <li><strong>Status/result:</strong> <code>GET /jobs/{id}</code> to poll, with the result inline or a link when done; or push completion via a signed webhook. Support <code>POST /jobs/{id}/cancel</code>.</li>       <li><strong>Errors/auth:</strong> failed jobs carry a structured error; object-level checks so callers only see their jobs.</li>       <li><strong>At 100×:</strong> durable queue with retry/back-off + DLQ, priority lanes, and back-pressure (<code>429</code>) when the queue is saturated.</li>     </ul>
    - level: 'senior'
      q: >-
          REST vs GraphQL vs gRPC — how do you pick for a given need?
      a: >-
          <p><strong>REST</strong>: resource-oriented, HTTP-cache-friendly, broad reach — default for public/edge-facing APIs. <strong>GraphQL</strong>: client shapes the query (kills over-/under-fetching), one endpoint, great for many varied clients (e.g. mobile + web) — but HTTP caching is hard, and you manage query cost / N+1. <strong>gRPC</strong>: typed protobuf contracts, low latency, streaming — best for internal service-to-service, weaker browser/edge story. Rule of thumb: gRPC between services, REST at the edge, GraphQL when client-shaped fetching is the dominant pain.</p>
    - level: 'senior'
      q: >-
          Sync vs async — how do you decide per endpoint?
      a: >-
          <p>Go async (<code>202</code> + status resource / webhook) when the work is long-running, depends on a flaky downstream, or fans out — anything that would blow your latency budget or pin a request thread. Stay sync (<code>201</code>/<code>200</code>) when the operation is fast and the caller genuinely needs the result inline. The trade-off: async adds a status/polling or webhook surface and forces idempotent consumers, but protects p99 and survives downstream failures.</p>
    - level: 'senior'
      q: >-
          Cursor vs offset pagination — defend your default.
      a: >-
          <p>Default to <strong>cursor (keyset)</strong> at any real scale: deep pages stay cheap (no <code>OFFSET N</code> scan) and stable under concurrent inserts/deletes — offset skips or repeats rows when the set shifts. Offset is fine only for small, slow-changing, jump-to-page-N UIs. Always cap page size. The senior tell is naming <em>stability under writes</em>, not just performance.</p>
    - level: 'senior'
      q: >-
          Versioning: URI path vs header vs date-based — which and why?
      a: >-
          <p>There's no purist winner — pick for the audience. <strong>Path</strong> (<code>/v1</code>) is explicit, cache- and router-friendly, easiest for third parties — most common. <strong>Header/media-type</strong> is "purer" but invisible and easy to get wrong. <strong>Date-pinned per account</strong> (Stripe model) enables fine-grained non-breaking evolution. A strong answer: coarse path boundary + additive-first evolution, bumping the version only on a true breaking change, with a <code>Sunset</code>/deprecation window.</p>
    - level: 'senior'
      q: >-
          Should every endpoint support idempotency keys?
      a: >-
          <p>No — only <strong>non-idempotent unsafe writes</strong> need them: <code>POST</code>s that create resources or move money/inventory. <code>GET</code>/<code>PUT</code>/<code>DELETE</code> are already idempotent by HTTP semantics, so a key adds nothing. Blanketing every endpoint adds a dedup store and latency for no benefit. The judgment is identifying which writes are dangerous on retry and protecting exactly those.</p>
    - level: 'senior'
      q: >-
          "Tell me about an API you designed and a decision you'd redo."
      a: >-
          <p>Structure it: context + constraints → the decision → the consequence → what you learned. Pick a <em>real</em> trade-off you got wrong (e.g. chose offset pagination and deep pages melted under load; or skipped idempotency keys and a retry storm double-charged). Show you now know <em>why</em> it was wrong and what you'd do instead. Owning a concrete mistake with a clear lesson reads far more senior than a flawless fairy tale.</p>
    - level: 'senior'
      q: >-
          "What's the biggest API mistake you've seen or made?"
      a: >-
          <p>Good candidates: <strong>BOLA</strong> — trusting an id in the path without an object-level ownership check, letting user A read user B's data (the #1 real-world API vuln); a non-idempotent money <code>POST</code> with no key; a "small" breaking change shipped without a version bump that broke every third-party client. Name the failure, the blast radius, and the systemic fix (object-level authZ middleware, idempotency by default on writes, contract tests in CI) — not just the one-off patch.</p>
    - level: 'senior'
      q: >-
          "How do you decide a resource model?"
      a: >-
          <p>Start from the domain nouns and their relationships/ownership, not the screens or the verbs. Pick plural collections, opaque ids, and <strong>shallow nesting</strong> (deep <code>/a/b/c/d</code> hierarchies couple you to one access path). Model state transitions as either status fields or action sub-resources. Pressure-test it: can every required operation map cleanly to a method on a resource? If a key operation only fits as an RPC verb, that's a signal to reconsider the nouns — or to deliberately allow one action endpoint.</p>
    - level: 'senior'
      q: >-
          "How do you ensure API quality across a whole team?"
      a: >-
          <p>Make quality systematic, not heroic: a written <strong>style guide</strong> (naming, errors, pagination, versioning conventions) so APIs look like one team built them; <strong>linting</strong> on the OpenAPI spec (e.g. Spectral) in CI to enforce it automatically; <strong>contract tests</strong> so changes can't silently break consumers; <strong>design review</strong> for new resources before code; and a <strong>single source-of-truth OpenAPI spec</strong> that drives docs and client codegen. The point is to move correctness left and make the right thing the default.</p>
    - level: 'senior'
      q: >-
          "A stakeholder demands a breaking change next sprint. How do you handle it?"
      a: >-
          <p>Don't just say no — quantify and redirect. First check if it can be done <strong>additively</strong> (new field/endpoint, old path untouched) — usually it can, removing the conflict. If it's genuinely breaking, surface the cost: who's on the old contract (usage data), the deprecation window required, and the migration work. Offer a versioned path with a <code>Sunset</code> timeline so the stakeholder gets the capability now without breaking existing clients. Frame it as protecting <em>their</em> customers, and bring options, not a blanket refusal.</p>
    - level: 'senior'
      q: >-
          "How do you balance shipping speed against API design rigor?"
      a: >-
          <p>Make the high-cost-to-change decisions carefully and defer the rest. The <strong>public contract</strong> (URIs, resource shapes, error format, auth, versioning) is expensive to undo, so get it right up front. Internal details and additive features can iterate fast behind that stable contract. Where you're unsure, ship behind a version/feature flag or mark it experimental so you keep the freedom to change it. The senior judgment is knowing which decisions are one-way doors.</p>
    - level: 'staff'
      q: >-
          The interviewer twists a constraint mid-answer ("now there are 10M third-party clients"). How do you respond?
      a: >-
          <p>Don't restart. Re-anchor to the framework and say what <em>changes</em>: untrusted third parties tighten authZ (object-level checks, scoped tokens), force strict versioning + deprecation discipline, demand per-tenant rate limits and abuse controls, and push you toward HATEOAS/discoverability so you can evolve URLs without redeploying their code. Naming the deltas — rather than re-drawing — is the signal that you understand which decisions are constraint-sensitive.</p>
    - level: 'staff'
      q: >-
          Design the API for a hotel / room booking system (search, hold, book, cancel).
      a: >-
          <ul>       <li><strong>Clarify:</strong> inventory is scarce and contended — double-booking is the cardinal sin; consistency over availability on the book path.</li>       <li><strong>Hold as a first-class resource:</strong> <code>POST /holds</code> reserves a room with a short <strong>TTL</strong> (e.g. 10 min) → <code>201</code> + expiry, decrementing inventory atomically. Separating hold from book is the senior move.</li>       <li><strong>Book:</strong> <code>POST /bookings</code> {hold_id} → <code>201</code>, <strong>idempotent</strong> via <code>Idempotency-Key</code> so a retried payment doesn't book twice; <code>409</code> if the hold expired.</li>       <li><strong>Concurrency:</strong> optimistic concurrency (<code>If-Match</code>/version) on the inventory row so two simultaneous holds can't both win; loser gets <code>412</code>/<code>409</code>.</li>       <li><strong>Search/cancel:</strong> <code>GET /rooms?dates=&amp;guests=&amp;limit=</code> cursor-paginated, cacheable; <code>POST /bookings/{id}/cancel</code> enforces a refund-by-time policy and releases inventory.</li>       <li><strong>At 100×:</strong> sweep stale holds via a queue, partition inventory by property/date, guard thundering herds on hot dates.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for a file storage service like Dropbox (upload large files, share, list, versions).
      a: >-
          <ul>       <li><strong>Clarify:</strong> files can be GBs, uploads must survive flaky networks, sharing needs fine-grained access; bytes go to object storage, metadata to the API.</li>       <li><strong>Large uploads:</strong> resumable/multipart — <code>POST /uploads</code> opens a session → <code>202</code> + session URL; client PUTs chunks; <code>POST /uploads/{id}/complete</code>. Decouples bytes from metadata and lets retries resume.</li>       <li><strong>Content addressing:</strong> identify blobs by content hash so identical files dedup and uploads are naturally <strong>idempotent</strong>; <code>If-Match</code>/ETag on metadata mutations.</li>       <li><strong>Sharing:</strong> <strong>signed URLs</strong> (time-boxed, scoped) for direct transfer to object storage, plus permission resources; object-level checks throughout (BOLA).</li>       <li><strong>List/versions:</strong> <code>GET /folders/{id}/files?limit=&amp;cursor=</code> cursor-paginated; <strong>versions as a sub-resource</strong> <code>GET /files/{id}/versions</code> with restore.</li>       <li><strong>At 100×:</strong> push transfer to CDN/object store via signed URLs (the API never proxies bytes), shard metadata, fan out share events via webhooks.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for a payments / charges service (charge, refund, list).
      a: >-
          <ul>       <li><strong>Clarify:</strong> untrusted merchant callers, write-heavy money path / read-heavy history, charges strongly consistent &amp; exactly-once from the caller's view, server-to-server auth, "never double-charge."</li>       <li><strong>Resources:</strong> <code>Charge</code>, <code>Refund</code> (sub-resource of a charge), <code>Customer</code>, <code>PaymentMethod</code>. Money as integer minor units + currency, never floats.</li>       <li><strong>Endpoints:</strong> <code>POST /v1/charges</code> → <code>201</code> sync / <code>202</code> async capture; <code>POST /v1/charges/{id}/refunds</code> → <code>201</code> (<code>422</code> if over captured total, <code>409</code> if fully refunded); <code>GET /v1/charges?customer=&amp;limit=</code>.</li>       <li><strong>Idempotency:</strong> headline of the answer — <code>Idempotency-Key</code> on every charge/refund, key→result stored 24h; a replay returns the original response.</li>       <li><strong>Auth + object-level:</strong> OAuth2 client-credentials, scopes like <code>charges:write</code>; verify this account <em>owns</em> the charge or return <code>404</code> (don't leak existence). Errors as RFC 9457; per-account <code>429</code> with limit headers.</li>       <li><strong>Async + at 100×:</strong> async capture → <code>202</code> + <code>pending</code>, poll or HMAC-signed at-least-once webhooks (consumer dedups). Shard idempotency/charge stores by account, watch hot merchants, replica for list reads, queue + DLQ behind webhooks.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for a chat / messaging service (send, fetch history, real-time delivery).
      a: >-
          <ul>       <li><strong>Clarify:</strong> real-time delivery, ordering and dedup matter, fan-out to many recipients, presence/read receipts, mobile reconnects.</li>       <li><strong>Resources:</strong> <code>Conversation</code>, <code>Message</code> (sub-resource), <code>Membership</code>. <code>POST /conversations/{id}/messages</code> → <code>201</code>.</li>       <li><strong>Idempotency:</strong> client-generated message id (or <code>Idempotency-Key</code>) so a resend on flaky mobile networks doesn't duplicate the message — critical here.</li>       <li><strong>History:</strong> <code>GET /conversations/{id}/messages?before=&amp;limit=</code> — cursor pagination keyed on a monotonic message id/timestamp; immutable messages cache well.</li>       <li><strong>Real-time:</strong> REST for send + history; push delivery over <strong>WebSocket/SSE</strong> (or webhooks for bots). At-least-once → recipients dedup by message id.</li>       <li><strong>Auth + at 100×:</strong> object-level membership check on every read/write; shard by conversation id, fan-out via a queue, cap per-conversation message rate to limit abuse.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for a ride-hailing booking flow (request ride, match, track, complete).
      a: >-
          <ul>       <li><strong>Clarify:</strong> a ride is a long-lived state machine (requested → matched → en route → completed/cancelled); location updates are high-frequency; matching is async.</li>       <li><strong>Resources:</strong> <code>Ride</code> with an explicit <code>status</code>; <code>POST /rides</code> {pickup, dropoff} → <code>202 Accepted</code> + a ride in <code>requested</code> (matching can't be synchronous).</li>       <li><strong>Idempotency:</strong> <code>Idempotency-Key</code> on <code>POST /rides</code> so a double-tap or retry doesn't create two rides / two charges.</li>       <li><strong>State + async:</strong> poll <code>GET /rides/{id}</code> or subscribe to push for status; this is where HATEOAS earns its keep — links advertise legal transitions (cancel only while cancellable).</li>       <li><strong>Tracking:</strong> high-rate location via a streaming channel, not REST polling on the hot path; <code>POST /rides/{id}/cancel</code> with policy.</li>       <li><strong>Auth + at 100×:</strong> object-level checks (rider/driver own this ride); geo-shard the matching service, back-pressure on the dispatch queue.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for a multi-tenant SaaS (tenant isolation, roles, per-tenant config).
      a: >-
          <ul>       <li><strong>Clarify:</strong> hard tenant isolation is the cardinal requirement (cross-tenant leakage is a breach), role-based access within a tenant, per-tenant limits/config.</li>       <li><strong>Tenancy model:</strong> tenant derived from the token/subdomain, never from a client-supplied body field; every query is tenant-scoped server-side.</li>       <li><strong>Auth + object-level:</strong> OAuth2/OIDC with tenant + role claims; <strong>every</strong> object access checks tenant ownership <em>and</em> role — BOLA across tenants is the worst failure here.</li>       <li><strong>Resources:</strong> avoid putting tenant in the path (leaks it); keep it implicit in auth. Standard CRUD with cursor pagination, RFC 9457 errors.</li>       <li><strong>Limits/config:</strong> per-tenant rate limits and feature flags so a noisy tenant can't degrade others; per-tenant API version pinning eases migrations.</li>       <li><strong>At 100×:</strong> shard or pool-vs-silo by tenant tier, isolate the largest tenants, and watch for one tenant's hot keys becoming everyone's problem.</li>     </ul>
    - level: 'staff'
      q: >-
          Design the API for e-commerce cart → order → fulfilment.
      a: >-
          <ul>       <li><strong>Clarify:</strong> a multi-step flow with money and inventory; the order-placement step is the consistency-critical one; fulfilment is async and long-running.</li>       <li><strong>Resources:</strong> <code>Cart</code> (mutable), <code>Order</code> (immutable once placed, with a <code>status</code> state machine), <code>Shipment</code>. <code>POST /carts/{id}/items</code>; <code>POST /orders</code> {cart_id} → <code>201</code>.</li>       <li><strong>Idempotency:</strong> <code>Idempotency-Key</code> on <code>POST /orders</code> so a retried checkout doesn't place two orders / double-charge — non-negotiable.</li>       <li><strong>Concurrency:</strong> <code>If-Match</code> on cart edits; reserve inventory at order time with optimistic concurrency to avoid overselling.</li>       <li><strong>Async fulfilment:</strong> placement returns fast; payment capture + warehouse + shipping run async, statuses surfaced on the order and pushed via webhooks (<code>order.shipped</code>), at-least-once → consumers dedup.</li>       <li><strong>Auth + at 100×:</strong> object-level checks (user owns this cart/order); cursor-paginate order history off a read replica; decouple fulfilment via a durable queue.</li>     </ul>
    - level: 'staff'
      q: >-
          Consistency vs availability — how does CAP show up in API design?
      a: >-
          <p>Decide <em>per resource/operation</em>, not globally. Money, inventory, and bookings need <strong>strong consistency</strong> on the write path — refuse or fail (<code>409</code>/<code>503</code>) rather than risk a double-charge or oversell. Feeds, counts, search, and history can be <strong>eventually consistent</strong> and stay available. Make it visible in the contract: an explicit <code>status</code>/<code>pending</code> state, a "may be stale" note, read-your-writes only where it's promised. The senior move is naming which path is which and why.</p>
    - level: 'staff'
      q: >-
          When is it right to break REST conventions?
      a: >-
          <p>When the resource model genuinely doesn't fit and forcing it hurts clarity. Common justified breaks: <strong>action/RPC-style sub-resources</strong> for state transitions (<code>POST /orders/{id}/cancel</code>) where modeling a verb as a noun is contortion; <strong>batch endpoints</strong> to avoid chatty round-trips; a <strong>search endpoint</strong> with a body when query strings can't express the filter. The rule: break a convention deliberately, name the cost (less uniform, harder to cache), and keep the rest of the API consistent — don't break it out of laziness.</p>
    - level: 'staff'
      q: >-
          When does HATEOAS actually earn its cost?
      a: >-
          <p>Rarely, and naming that is the senior tell. It pays off with <strong>many third-party clients you can't redeploy</strong> (evolve URLs/workflows without breaking them), or a genuine <strong>state machine</strong> where legal actions change with state (a payment that's capturable/refundable/voidable, a ride that's cancellable only while pending) — links express the legal transitions. For version-pinned first-party clients it's dead weight, which is why most production "REST" stops at Level 2.</p>
    - level: 'staff'
      q: >-
          "What changes at 100×?" — how do you answer this generically?
      a: >-
          <p>Walk the bottlenecks in order: <strong>hot partitions / keys</strong> (a whale tenant or merchant — protect with per-entity caps), <strong>reads</strong> move to replicas + caching tiers + CDN, <strong>writes</strong> get sharded (and cursors must stay stable across shards), <strong>async work</strong> goes behind durable queues with back-pressure and DLQs, and the <strong>synchronous critical path</strong> gets kept narrow and wrapped in circuit breakers. Close with what you'd <em>measure</em> (p99 per route) to know when to act. This step is the headline senior signal.</p>
    - level: 'staff'
      q: >-
          How does caching / CDN change your API design at scale?
      a: >-
          <p>You design <em>for</em> cacheability: keep reads on <code>GET</code> (so proxies/CDNs can cache), set honest <code>Cache-Control</code> + <code>ETag</code>, and separate immutable resources (long TTL, content-addressed URLs) from volatile ones. Push large/static payloads to a CDN and serve bytes via signed URLs so the API never proxies them. Watch the trade-offs: aggressive caching fights read-your-writes and skews analytics (the 301-redirect problem), and cache invalidation becomes a real design surface.</p>
    - level: 'staff'
      q: >-
          How do read replicas / sharding leak into the API contract?
      a: >-
          <p>Replicas introduce <strong>replication lag</strong> → reads can be stale, so you must either not promise read-your-writes on replica-served reads or route the just-written read to the primary. Sharding constrains queries: <strong>cursors must encode the shard/sort key</strong> to stay stable, and cross-shard list/aggregation gets expensive — you may need to push filters into the contract or precompute. Surface an explicit <code>status</code> field rather than letting clients infer consistency.</p>
    - level: 'staff'
      q: >-
          How do you design rate limiting &amp; abuse protection at scale?
      a: >-
          <p>Multi-layer: per-key/tenant <strong>token buckets</strong> (distributed counters in Redis) returning <code>429</code> + <code>RateLimit-*</code> and <code>Retry-After</code>; per-endpoint <strong>cost weighting</strong> so an expensive call counts more; a separate <strong>burst tier</strong> so one abuser can't starve neighbors; and quota tiers by plan. Add edge defenses (WAF, bot detection) and per-account caps so a hot tenant degrades only itself. The fairness goal — protecting neighbors — is the staff framing.</p>
    - level: 'staff'
      q: >-
          What does backward-compatible evolution actually require?
      a: >-
          <p>Be <strong>additive</strong>: new optional fields, new endpoints, new enum values handled tolerantly (clients ignore unknown fields — "must-ignore"). Never remove/rename fields, tighten validation, change types, or repurpose a status code within a version — those are breaking and need a version bump. Defaults must preserve old behavior. Contract tests + a published OpenAPI spec catch accidental breaks before they ship.</p>
    - level: 'staff'
      q: >-
          How do you deprecate an endpoint or version at scale?
      a: >-
          <p>Announce, then measure, then sunset. Mark responses with a <code>Deprecation</code> header and a <code>Sunset</code> date + a link to the migration guide; <strong>instrument usage per client</strong> so you know who's still on it; reach out to the heaviest callers directly. Give third parties a long window (months), offer a migration path or shim, and only then turn it off — possibly with brownouts to surface stragglers. Never silently break; for internal-only clients the window can be much shorter.</p>
    - level: 'design'
      q: >-
          Design the API for a payments service &mdash; charge a card, refund, list transactions.
      a: >-
          <ul>         <li><b>A strong answer covers:</b> <strong>clarify first</strong> &mdash; untrusted merchant callers, write-heavy money path / read-heavy history, charges strongly consistent &amp; exactly-once from the caller's view, server-to-server auth, "never double-charge."</li>         <li><strong>Resources:</strong> <code>Charge</code>, <code>Refund</code> (sub-resource of a charge), <code>Customer</code>, <code>PaymentMethod</code>; money as integer minor units + currency, never floats.</li>         <li><strong>Endpoints:</strong> <code>POST /v1/charges</code> &rarr; <code>201</code> sync / <code>202</code> async capture; <code>POST /v1/charges/{id}/refunds</code> &rarr; <code>201</code> (<code>422</code> over captured total, <code>409</code> if fully refunded); <code>GET /v1/charges?customer=&amp;limit=</code> cursor-paginated.</li>         <li><strong>Idempotency &mdash; the headline:</strong> <code>Idempotency-Key</code> on every charge/refund, key&rarr;result stored ~24h; a replay returns the original response.</li>         <li><strong>Caching/concurrency:</strong> settled charges get long-lived <code>Cache-Control</code>+<code>ETag</code>; mutable metadata needs <code>If-Match</code> &rarr; <code>412</code> on a stale tag.</li>         <li><strong>Auth + object-level:</strong> OAuth2 client-credentials, scopes like <code>charges:write</code>; verify this account owns <code>{id}</code> or return <code>404</code> (don't leak existence); RFC 9457 errors; per-account <code>429</code> with limit headers.</li>         <li><strong>Async + at 100×:</strong> async capture &rarr; <code>202</code> + HMAC-signed at-least-once <code>charge.succeeded</code> webhook (consumer dedups); shard idempotency/charge stores by account, watch hot merchants, replica for list reads, queue + DLQ behind webhooks, circuit-break the card-network path.</li>       </ul>
    - level: 'design'
      q: >-
          The interviewer twists the constraint mid-answer: "now there are 10M untrusted third-party clients." Walk the deltas.
      a: >-
          <p><b>A strong answer covers:</b> don't restart &mdash; re-anchor to the framework and name what <em>changes</em> &rarr; <strong>authZ tightens</strong>: strict object-level (BOLA) checks on every access, scoped/least-privilege tokens, return <code>404</code> not <code>403</code> to avoid leaking existence &rarr; <strong>versioning becomes a product promise</strong>: additive-first evolution, version bump only on true breaking changes, <code>Deprecation</code>/<code>Sunset</code> headers + a long window because you can't redeploy their clients &rarr; <strong>abuse &amp; fairness</strong>: per-tenant token-bucket rate limits with <code>RateLimit-*</code>/<code>Retry-After</code>, per-endpoint cost weighting, a burst tier and per-account caps so one abuser can't starve neighbors &rarr; <strong>discoverability</strong>: a published OpenAPI spec, stable machine-readable error <code>code</code>s, and HATEOAS/links where you need to evolve URLs without breaking pinned clients &rarr; <strong>at 100×</strong>: distributed limit counters (Redis), edge caching/CDN, shard by tenant and isolate the largest, and watch one tenant's hot keys becoming everyone's problem &rarr; the signal is naming the constraint-sensitive deltas, not re-drawing the API.</p>

rubric:
    - 'Clarify requirements &amp; constraints: clients (1st vs 3rd-party), scale &amp; read/write ratio, consistency, auth, SLA &mdash; pin before drawing.'

    - 'Map methods + precise status codes: 201 vs 202, 200 vs 204, 409 vs 422.'
    - >-

    - 'Collections: cursor pagination, a filter/sort contract, bounded page sizes on every list.'
    - >-
        Caching &amp; concurrency: <code>ETag</code>+<code>Cache-Control</code> on reads, <code>If-Match</code> on mutables to stop lost updates.
    - 'Versioning &amp; evolution: a strategy, a definition of "breaking", a stated deprecation path.'
    - 'AuthN/authZ + object-level (BOLA): check ownership on every access.'
    - >-
        Errors, rate limits, observability: RFC 9457 problem+json, <code>429</code> + limit headers, request/trace IDs.
    - >-
        Async / webhooks where it matters: <code>202</code> + status resource, signed at-least-once webhooks.
