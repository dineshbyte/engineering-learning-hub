Interview questions on these concepts

Use each one twice: answer it yourself from memory (recall), and ask it of a candidate or vendor to test depth. The answer block is a checklist of what a strong answer hits — not a script.

1Walk me through exactly what happens when an agent "uses a tool".

The model emits a structured request (tool name + args); the runtime executes the real function; the result is fed back into the next call; the model never runs anything itself. Bonus: a tool's scope is a security boundary, like an API key.

2An LLM is stateless — so how does a chat assistant "remember" 20 turns of context?

The runtime re-sends the history every call; the context window is bounded RAM; durable memory is an external store (RAG); as the window fills you trim or summarise. Red-flag answer: "the model remembers."

3When would you NOT build an agent, and ship a workflow instead?

When the path is known and repeatable — you buy predictability, lower cost, lower latency, and easier debugging. Default to workflows; reach for agents only when the steps can't be predetermined.

4Your agent occasionally loops forever and burns huge cost. Where do you fix it?

The runtime, not the model — max-steps, timeouts, token budgets, and guardrails are the runtime's job. Anyone who blames "the model" missed Level 5.

5Explain MCP to a skeptical exec in one analogy, and name the problem it solves.

"USB-C / JDBC for tools" — an open standard so any agent can reach any tool. It turns N×M bespoke integrations into N+M; a server exposes tools, a client connects. Not a Claude-only feature.

6A team wants to add a second agent to "go faster." Talk them into it — or out of it.

Same trade-off as premature microservices. Multi-agent costs coordination, latency, and ~15× tokens; it only helps when work is context-bound or genuinely parallel and independent. Start single-agent; split with evidence.

7How does a "coding agent" differ from a "support agent" architecturally?

It barely does — same engine (LLM + loop + runtime + memory). What differs is the tool registry, the system prompt, and the guardrails. Same employee, different job description.

8Whiteboard: design a billing-dispute agent that's safe to ship.

Tools (look up invoice/payments, issue refund, notify); a bounded loop; guardrails — refund caps, human approval over a threshold, idempotent refunds, a full audit log, scoped permissions; per-case memory; a workflow for the common path with an agent escape-hatch for the messy ones.
