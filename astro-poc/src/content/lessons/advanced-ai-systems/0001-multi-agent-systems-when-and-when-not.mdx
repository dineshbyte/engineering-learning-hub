# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is a multi-agent system, in one breath?'
      a: >-
          <b>Hit these points:</b> multiple agents — each its own <b>LLM + tools + context window</b> — working on one goal and coordinating by passing messages or results &rarr; it's the <b>distributed-systems version of an agent</b>: you trade a single reasoning thread for several that must agree &rarr; the upside is throughput on genuinely parallel work; the downside is <b>coordination overhead</b> — the tokens, latency, and failure surface spent getting them to hand off and not duplicate work &rarr; so it is a scaling pattern, not a smarter brain: more agents mean more parallel hands, not more intelligence.
    - level: 'core'
      q: 'What is orchestrator-worker and what does the orchestrator actually own?'
      a: >-
          <b>Hit these points:</b> one lead (orchestrator) agent decomposes the goal into independent sub-tasks, delegates each to a worker, then <b>synthesizes</b> the results into the final answer &rarr; it's the <b>scatter-gather / fan-out-fan-in</b> pattern &rarr; the orchestrator owns planning (how to split), delegation (who does what), and integration (combining results) — workers own only their isolated sub-problem &rarr; that single control point is the win: one place to plan, one place to validate, one place to attribute a failure.
    - level: 'core'
      q: 'What is an agent swarm, and what does it trade away?'
      a: >-
          <b>Hit these points:</b> many <b>peer</b> agents with no central orchestrator; behavior <b>emerges</b> from local interactions and a shared environment or state &rarr; it maximizes parallelism and decoupling — no planning bottleneck, agents act on what they see locally &rarr; the trade is <b>predictability and attribution</b>: with no central plan you can't easily say why the system did what it did, or which agent caused a bad outcome &rarr; so swarms suit exploratory or simulation-style work where emergence is the point, not production tasks that need an audit trail and a synthesis step.
    - level: 'core'
      q: 'What is coordination overhead, and why is it the central anti-pattern of multi-agent?'
      a: >-
          <b>Hit these points:</b> coordination overhead = the extra <b>tokens, latency, and failure surface</b> spent getting agents to agree, hand off, and not duplicate work &rarr; every multi-agent design pays this tax whether or not the task benefits &rarr; if the task is genuinely parallel, the throughput gain can exceed the tax; if it's sequential or context-heavy, the tax is pure loss &rarr; that's the anti-pattern: adding agents to a task that isn't parallel buys coordination cost with no payoff &rarr; the discipline is to keep the default at one agent and make the multi-agent design justify the tax.
    - level: 'senior'
      q: 'When does a single well-tooled agent beat a fleet? Walk me through the call.'
      a: >-
          <b>Hit these points:</b> a single agent wins whenever the work is <b>sequential</b>, shares a lot of context, or isn't large enough to amortize coordination cost &rarr; one agent keeps all state in one context window, so there's no fragmentation, no handoff loss, and one place to debug &rarr; it's cheaper (no inter-agent chatter) and lower-latency (no fan-out / join) &rarr; the honest default: most production features are a single agent with good tools and a tight loop &rarr; reach for multiple agents only when you've shown the work is genuinely parallel and the value of the outcome outweighs the extra spend — separate that real case from the demo that just looks impressive.
    - level: 'senior'
      q: 'Your fan-out spawned far more workers than the task needed and the bill exploded. What happened and how do you prevent it?'
      a: >-
          <b>Hit these points:</b> the orchestrator <b>over-decomposed</b> — it spun up a worker per trivial sub-question because nothing bounded it &rarr; tokens scale with <b>agents &times; turns &times; retries</b>, so an unbounded fan-out multiplies spend with no quality gain &rarr; root cause is missing limits, not a model bug: no cap on worker count, no per-task budget, no floor on what's worth delegating &rarr; fix: cap concurrent workers, set a token/turn budget per run with a <b>kill switch</b>, and only delegate sub-tasks above a complexity threshold &rarr; add per-agent token attribution so you can see which branch ran away &rarr; the lesson: an autonomous decomposer needs the same guardrails you'd put on any process that can fan out work.
    - level: 'senior'
      q: 'Quality swings run to run and you suspect cascading errors. Diagnose and fix.'
      a: >-
          <b>Hit these points:</b> log each worker's intermediate output and the orchestrator's synthesis for a good and a bad run — you're auditing the <b>handoffs</b>, not the final answer &rarr; look for a worker that returned a wrong or low-confidence result that got trusted downstream and folded into a confident final answer &rarr; that's a <b>cascading error</b>: no validation at the join, so one bad slice poisons the synthesis &rarr; fix: validate or score worker outputs before integration, have the orchestrator reconcile disagreements instead of blending them, and let workers signal low confidence &rarr; add a check that contradictory results trigger a re-run or escalation, not a silent merge &rarr; the framing: in a pipeline of probabilistic steps, error compounds — you must inspect at the joins.
    - level: 'senior'
      q: "Two workers duplicated each other's work and a third contradicted both. Why, and how do you design against it?"
      a: >-
          <b>Hit these points:</b> this is <b>context fragmentation</b> — each agent saw only its slice, none saw the whole, so they overlapped and disagreed &rarr; root cause is an unmade decision: what context is <b>shared</b> across agents vs <b>isolated</b> to each &rarr; too much isolation duplicates and contradicts; too much sharing reintroduces the cost and bloat you split to avoid &rarr; fix: give the orchestrator authority over the global plan so sub-tasks are carved to be genuinely non-overlapping, share the minimal common context (the goal, constraints, what's already done), and isolate the rest &rarr; have the synthesis step reconcile, not concatenate &rarr; the principle: decide the context boundaries deliberately, the way you'd define service boundaries — fragmentation is a design smell, not bad luck.
    - level: 'staff'
      q: 'Reason about the trade-offs across coordination cost, latency, and reliability in a multi-agent design.'
      a: >-
          <b>Hit these points:</b> name the axes — <b>cost</b> (tokens scale with agents &times; turns), <b>latency</b> (fan-out can run workers in parallel, but the join waits for the slowest, and an orchestrator adds planning turns), <b>reliability</b> (more independent steps means more places to fail and compound) &rarr; parallelism can cut wall-clock latency for genuinely independent work, which is the main reason to pay the rest &rarr; but it raises cost (inter-agent chatter, retries) and lowers reliability (cascading errors, fragmentation) &rarr; sequence the call: prove the work is parallel first; if not, one agent dominates on all three axes &rarr; if it is, bound the blast radius — cap agents/turns/budget, validate at joins, attribute spend per agent &rarr; measure parallel speedup against the cost multiple; if you're not getting the speedup, you're paying the tax for nothing.
    - level: 'staff'
      q: 'A teammate wants every feature to be a swarm of specialist agents "because that''s how the smartest demos work." Tear it apart as a system design.'
      a: >-
          <b>Hit these points:</b> two flawed assumptions &rarr; <b>"more agents = smarter"</b> is false: agents add parallel hands, not intelligence; a swarm of weak reasoners doesn't out-think one strong agent on a sequential problem &rarr; <b>"demos generalize"</b> is false: demos optimize for looking impressive on a curated task; production optimizes for cost, latency, debuggability, and a clean audit trail &rarr; a swarm gives up what production needs most — predictability and failure attribution — for emergence you rarely want &rarr; map the failure modes you'd inherit: coordination cost, cascading errors, context fragmentation, cost blowup &rarr; the framing, same as microservices: don't distribute until the work is genuinely independent and at a scale that pays the tax; default to one well-tooled agent and make the fleet justify itself with measured parallel speedup, not vibes.
    - level: 'staff'
      q: 'Multi-agent demos look magical; production multi-agent often disappoints. Separate the real capability from the demo.'
      a: >-
          <b>Hit these points:</b> a demo is a curated happy path — one impressive trace on a task chosen to show emergence; it hides cost, tail latency, and the runs where coordination went wrong &rarr; production is the distribution, not the best trace: you pay for the average and the tail, you debug the failures, and you carry the bill every call &rarr; the real, durable capability is narrow: <b>genuinely parallel</b> work (gather from many independent sources, fan out independent checks) where the orchestrator can synthesize and validate &rarr; the fake capability is "more agents = smarter on a sequential problem" — that just adds coordination tax and failure surface &rarr; the test: does the fleet beat a single well-tooled agent on the metric that matters, measured, with the failure modes bounded? If you can't show that, you have a demo, not a system.
    - level: 'design'
      q: 'Design the architecture for a research assistant that must answer a broad question by gathering from many independent sources.'
      a: >-
          <b>A strong answer covers:</b> clarify whether the work is genuinely parallel — many independent sources to gather is the textbook case where fan-out pays, so this is a fair multi-agent candidate &rarr; choose <b>orchestrator-worker</b> over swarm: a lead agent plans the sub-questions, fans out a bounded set of worker agents to gather in parallel, then synthesizes — you keep one control point, an audit trail, and a place to validate &rarr; bound the blast radius: cap concurrent workers, set a per-run token/turn budget with a kill switch, only delegate sub-tasks above a complexity floor so you don't over-decompose &rarr; handle the failure modes: validate worker results at the join (cascading errors), share the goal + what's gathered and isolate the rest (context fragmentation), attribute tokens per worker (cost blowup) &rarr; compare honestly against the single-agent baseline: if one agent with a search tool and a tight loop gets close, ship that &rarr; measure parallel speedup vs the cost multiple, and treat the multi-agent design as something to justify, not assume (multi-agent token use is many times a single chat turn, illustrative).<sup><a href="#s1" id="r5">1</a></sup>
    - level: 'design'
      q: 'Design the guardrails and observability for a multi-agent system you have to run in production.'
      a: >-
          <b>A strong answer covers:</b> one principle drives the design — every multi-agent system is a <b>distributed system</b>, so bound it like one &rarr; limits: hard caps on concurrent agents, total turns, and per-run token budget, with a <b>kill switch</b> that stops a runaway fan-out &rarr; attribution: per-agent / per-task token and latency accounting so you can see which branch ran away and what the parallel speedup actually was &rarr; validation at the joins: score or check worker outputs before the orchestrator synthesizes, reconcile disagreements instead of blending them (defuses cascading errors) &rarr; context discipline: define shared vs isolated context explicitly so agents don't duplicate or contradict (defuses fragmentation) &rarr; safety: autonomy raises blast radius, so least privilege on tools, no arbitrary egress, confirmation on irreversible actions — map to OWASP LLM Top 10 (excessive agency) &rarr; measure against a single-agent baseline; if the fleet isn't beating it on the metric that matters, collapse it back to one agent.<sup><a href="#s5" id="r6">5</a></sup>

    <div class="lvltop"><span class="num">1</span><h2>Multi-Agent Systems</h2></div>
    <p class="tagline">Multiple agents — each its own <b>LLM + tools + context window</b> — working on one goal, coordinating by passing messages or results. It's the <b>distributed-systems version of an agent</b>: more throughput on parallel work, paid for with coordination.<sup><a href="#s3" id="r2">2</a></sup></p>

    <h3>One agent, or many? Start from the default</h3>

    <Fig2 />

    <h3>When does the work justify more than one agent?</h3>
    <table>
        <thead><tr><th>Property of the task</th><th>Single agent</th><th>Multi-agent</th></tr></thead>
        <tbody>
            <tr><td><b>Shape of the work</b></td><td>Sequential / one thread</td><td>Genuinely parallel, independent sub-tasks</td></tr>
            <tr><td><b>Shared context</b></td><td>Sub-steps share a lot of state</td><td>Sub-tasks need little of each other's state</td></tr>
            <tr><td><b>Scale</b></td><td>Small enough to not amortize overhead</td><td>Large enough that the coordination tax pays back</td></tr>
            <tr><td><b>What you optimize</b></td><td>Cost, latency, debuggability</td><td>Wall-clock throughput on parallel work</td></tr>
            <tr><td><b>Honest default</b></td><td>Most production features</td><td>The exception you must justify</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact"><b>Is</b> A multi-agent system is several agents (each its own LLM + tools + context) on one goal, coordinating by passing messages or results. A scaling pattern, not a smarter brain.</div>
        <div class="fact"><b>Why it exists</b> Some goals decompose into independent sub-tasks that can run in parallel. Splitting them across agents turns serial work into concurrent work — when the work is actually parallel.</div>
        <div class="fact blue"><b>Like (world)</b> A newsroom on a breaking story: an editor assigns reporters to separate angles in parallel, then stitches the pieces. It only helps because the angles are independent.</div>
        <div class="fact blue"><b>Like (code)</b> A monolith vs microservices: you split for independent scaling, not for fun. The split pays only when the boundary is real and the volume justifies the network cost.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "More agents make the system smarter."</div>
        <div class="t">&#10003; More agents add parallel <em>hands</em>, not a smarter brain. A fleet of weak reasoners doesn't out-think one strong agent on a sequential problem.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Multi-agent is the modern, default way to build."</div>
        <div class="t">&#10003; The default is one well-tooled agent in a loop. Multi-agent is the exception you reach for only when the work is genuinely parallel.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> <span>One well-tooled agent is the default.</span> A multi-agent system is the <span>distributed-systems version</span> of an agent — reach for it only when the work is genuinely parallel.</div>

<section class="lvl" id="m2">
    <div class="lvltop"><span class="num">2</span><h2>Orchestration: Orchestrator-Worker vs Swarm</h2></div>
    <p class="tagline">Two ways to wire a fleet. <b>Orchestrator-worker</b>: one lead decomposes, delegates, and synthesizes (scatter-gather). <b>Swarm</b>: peer agents, no central planner, behavior <em>emerges</em>. For production you almost always want the control point.<sup><a href="#s2" id="r3">3</a></sup></p>

    <h3>The two patterns, side by side</h3>

    <Fig3 />

    <h3>What each pattern buys and costs</h3>
    <table>
        <thead><tr><th>Dimension</th><th>Orchestrator-worker</th><th>Swarm</th></tr></thead>
        <tbody>
            <tr><td><b>Control</b></td><td>Central lead plans &amp; integrates</td><td>None — peers act on local state</td></tr>
            <tr><td><b>Predictability</b></td><td>High — one plan, one synthesis</td><td>Low — behavior emerges</td></tr>
            <tr><td><b>Failure attribution</b></td><td>Clear — trace to a worker or the join</td><td>Hard — no single owner of an outcome</td></tr>
            <tr><td><b>Parallelism</b></td><td>Bounded by the lead's fan-out</td><td>Maximal, fully decoupled</td></tr>
            <tr><td><b>Best for</b></td><td>Production tasks needing an audit trail</td><td>Exploration / simulation where emergence is the point</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact"><b>Is</b> Orchestrator-worker = a lead agent decomposes the goal, delegates sub-tasks to workers, and synthesizes results. Fan-out-fan-in with one owner of planning and integration.</div>
        <div class="fact"><b>Why it exists</b> Production needs a control point: somewhere to plan the split, validate the results, and attribute failures. The lead is that point; workers stay isolated on their sub-problem.</div>
        <div class="fact blue"><b>Like (world)</b> A general contractor: subcontractors do isolated jobs in parallel; the contractor scopes the work, checks it, and signs off the whole build.</div>
        <div class="fact blue"><b>Like (code)</b> Scatter-gather / map-reduce: a coordinator fans work to mappers and reduces their outputs. The reduce step is the synthesis — and the place bad partials get caught.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "A swarm is more advanced, so it's the better default."</div>
        <div class="t">&#10003; A swarm trades away the things production needs most — predictability and failure attribution — for emergence you rarely want. Default to orchestrator-worker.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "The orchestrator is just a router; the workers do the real work."</div>
        <div class="t">&#10003; The orchestrator owns planning, delegation, AND synthesis. The synthesis (and validating partials) is where quality is won or lost.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> Orchestrator-worker gives you <span>one control point</span> — plan, validate, attribute. A swarm trades that for <span>emergence you rarely want</span> in production.</div>

    <MemoryCheck items={mc2} />

    <details class="iq">
        <summary>M2 — Contrast orchestrator-worker with a swarm. When would you reach for each, and what do you give up?</summary>
        <div class="a">
            <b>Hit these points:</b> orchestrator-worker = one lead agent decomposes the goal, fans out independent sub-tasks to workers, then synthesizes — scatter-gather, the lead owns planning and integration &rarr; swarm = many peer agents, no central planner; behavior emerges from local interactions and shared state &rarr; orchestrator-worker buys a clear control point, easier debugging, and a place to attribute failures; you pay a planning bottleneck and a single coordination point &rarr; swarm buys maximum parallelism and decoupling; you give up predictability and the ability to attribute a bad outcome to any one agent &rarr; default to orchestrator-worker for production: you almost always want the audit trail and the synthesis step more than emergent behavior.
        </div>
    </details>

<section class="lvl" id="m3">
    <div class="lvltop"><span class="num">3</span><h2>Failure Modes &amp; Anti-Patterns</h2></div>
    <p class="tagline">Three failures appear <em>only</em> once agents must coordinate: <b>cascading errors</b>, <b>context fragmentation</b>, and <b>cost / latency blowups</b>. Each is a distributed-systems failure wearing an LLM costume — and each is why you bound a fleet like one.<sup><a href="#s4" id="r4">4</a></sup></p>

    <h3>The three multi-agent-specific failures</h3>

    <Fig4 />

    <h3>Failure mode &rarr; root cause &rarr; defense</h3>
    <table>
        <thead><tr><th>Failure mode</th><th>Root cause</th><th>Defense</th></tr></thead>
        <tbody>
            <tr><td><b>Cascading errors</b></td><td>A wrong partial is trusted and folded into the synthesis</td><td>Validate / score worker outputs at the join; reconcile, don't blend</td></tr>
            <tr><td><b>Context fragmentation</b></td><td>Each agent sees a slice; no one sees the whole</td><td>Decide shared vs isolated context up front; orchestrator owns the global plan</td></tr>
            <tr><td><b>Cost / latency blowup</b></td><td>Tokens scale with agents &times; turns &times; retries; unbounded fan-out</td><td>Hard caps on agents/turns/budget; per-agent token attribution; kill switch</td></tr>
            <tr><td><b>Excessive agency</b></td><td>Autonomous agents act with broad tool access</td><td>Least privilege, no arbitrary egress, confirmation on irreversible actions</td></tr>
        </tbody>
    </table>

    <div class="facts">
        <div class="fact"><b>Is</b> The multi-agent failure set: cascading errors, context fragmentation, and cost/latency blowup — failures that exist only because agents must coordinate. Add excessive agency once they can act.</div>
        <div class="fact"><b>Why it exists</b> Coordination introduces joins (where bad partials compound), boundaries (where context fragments), and amplification (where tokens multiply). Distribution creates them; a single agent has none.</div>
        <div class="fact blue"><b>Like (world)</b> A relay race: one dropped baton (cascading), runners who don't know the course (fragmentation), and a roster you keep growing for no speed gain (cost blowup).</div>
        <div class="fact blue"><b>Like (code)</b> Microservice sprawl: cascading failures across calls, state split awkwardly across services, and a bill that balloons with chatty inter-service traffic. Same lessons apply.</div>
    </div>

    <div class="myth">
        <div class="m">&#10007; "We'll add specialist agents because that's how the impressive demos work."</div>
        <div class="t">&#10003; Demos optimize for looking smart on a curated task; production optimizes for cost, latency, debuggability, and an audit trail. Don't generalize from the demo.</div>
    </div>
    <div class="myth">
        <div class="m">&#10007; "Adding agents is a low-risk way to improve quality."</div>
        <div class="t">&#10003; Each agent is a new failure surface. You inherit cascading errors, fragmentation, and cost blowups — a distributed-systems problem — for parallelism you may not need.</div>
    </div>

    <div class="rule"><span>&#128229; Memory rule:</span> <span>Don't add agents for the reasons you wouldn't add microservices.</span> Bound a fleet like a distributed system: <span>validate at joins, cap the blast radius, attribute spend</span>.</div>

    <MemoryCheck items={mc3} />

    <details class="iq">
        <summary>M3 — Your multi-agent feature's quality is unstable and the bill is 10x what you modeled. Diagnose the failure modes.</summary>
        <div class="a">
            <b>Hit these points:</b> separate the three multi-agent-specific failures &rarr; cascading errors: a worker's wrong intermediate result is trusted downstream and the orchestrator synthesizes confident nonsense — fix with validation at the join, not blind trust &rarr; context fragmentation: each agent sees a slice, no one sees the whole, so they duplicate work or contradict each other — fix by deciding shared vs isolated context up front &rarr; cost/latency blowup: tokens scale with agents &times; turns &times; retries; one chatty loop or a runaway fan-out multiplies spend fast (multi-agent can burn many times the tokens of a single chat turn, illustrative) — fix with hard caps on agent count, turns, and budget &rarr; measure: per-agent token attribution, a turn cap, and a kill switch; if you can't attribute the spend, you can't control it.
        </div>
    </details>

    <details class="iq">
        <summary>M3 — Make the explicit analogy: why is "don't add agents for the reasons you wouldn't add microservices" the governing rule?</summary>
        <div class="a">
            <b>Hit these points:</b> splitting a monolith into microservices trades in-process calls for network calls — you gain independent scaling and team autonomy but inherit partial failure, serialization cost, and distributed debugging &rarr; splitting one agent into many trades in-context reasoning for message-passing — you gain parallelism but inherit coordination overhead, cascading errors, and context fragmentation &rarr; in both cases the split only pays when the boundary is real (genuinely independent work) and the volume justifies the overhead &rarr; people add both for resume-driven or demo reasons, not load reasons; the result is a distributed system with all the failure modes and none of the benefit &rarr; so the test is identical: is the work truly independent and at a scale where the coordination tax pays for itself? If not, keep it in one process / one agent.
        </div>
    </details>
