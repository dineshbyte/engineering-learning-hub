# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: "What problem does consistent hashing solve, and what's the one property it buys?"
      a: >-
          <b>Hit these points:</b> it maps keys <em>and</em> nodes into one hash space (a ring,
          0&hellip;2&sup3;&sup2;&minus;1; Dynamo uses 2&sup1;&#8310;&#8304;) &rarr; a key is owned by
          the first node clockwise &rarr; the property: a membership change (add/remove a node) moves
          only ~<b>K/N</b> keys &mdash; one node's worth &mdash; instead of nearly all of them &rarr;
          that's the entire reason it exists, versus <code>mod N</code> which has no locality.
    - level: 'core'
      q: 'Why is <code>hash(key) mod N</code> a trap? Quantify what happens when you add a node.'
      a: >-
          <b>Hit these points:</b> <code>mod N</code> is balanced and O(1), so it's the default
          everyone reaches for &rarr; but it has no locality: when N &rarr; N+1, <code>mod (N+1) &ne;
          mod N</code> for almost every key &rarr; the expected fraction that stays put is only
          ~<code>1/(N+1)</code> &mdash; nearly every key remaps &rarr; for a cache that means a
          universal miss &rarr; DB stampede; for a sharded store it means re-shuffling almost the whole
          dataset over the network on every scaling event.
    - level: 'core'
      q: 'Walk me through a lookup on the ring and what its cost is.'
      a: >-
          <b>Hit these points:</b> a node sits at <code>hash(node_id)</code>; a key is owned by the
          first node clockwise from <code>hash(key)</code> &mdash; same hash space for both &rarr;
          lookup is a <b>successor search</b>: binary-search the sorted node positions for the next one
          clockwise, wrapping past the top of the ring &rarr; that's O(log N) in the number of (v)node
          positions &rarr; the wrap-around at 2&sup3;&#8322; is what makes it a ring rather than a line.
    - level: 'core'
      q: 'Exactly which keys move when a node joins, and which when one leaves?'
      a: >-
          <b>Hit these points:</b> on <b>add</b>, the new node claims only the arc between it and its
          counter-clockwise neighbour &mdash; ~K/N keys, taken from <em>one</em> node; every other key
          stays exactly where it was &rarr; on <b>remove</b>, that node's arc falls to its clockwise
          successor &mdash; again ~K/N keys, to <em>one</em> node &rarr; the locality (only one
          neighbour's arc is disturbed) is the whole win, and it's what <code>mod N</code> cannot give
          you.
    - level: 'senior'
      q: 'A bare ring with 5 nodes is hot-spotting. Diagnose it with the math, then fix it.'
      a: >-
          <b>Hit these points:</b> random placement does NOT give equal arcs &mdash; the largest arc is
          ~<code>&Theta;((log N)/N)</code>, so the hottest node can hold ~<code>log N</code>&times; the
          average; with few nodes one routinely owns 2&ndash;3&times; another &rarr; fix: virtual nodes
          &mdash; hash each <em>physical</em> node to many positions (V &asymp; 100&ndash;200),
          scattering N&middot;V points around the ring &rarr; a node's load becomes the <em>sum</em> of
          V independent little arcs, which concentrates at <code>1/N</code> with std dev
          ~<code>1/&radic;V</code> &mdash; within a few % of uniform &rarr; bonus dividends: a dead
          node's vnodes spread to many successors (graceful failure), and 2&times; capacity = 2&times;
          vnodes gives weighting for free.
    - level: 'senior'
      q: "Vnodes balanced my keyspace but one node is still on fire. What's the failure mode and the fix?"
      a: >-
          <b>Hit these points:</b> vnodes balance the <em>keyspace</em>, not <em>popularity</em> &mdash;
          a single viral key has exactly one home node, and no amount of vnode count splits one key
          &rarr; reach for consistent hashing with <b>bounded loads</b>: cap each node at
          <code>(1+&epsilon;)&times;</code> the average load and, when a key's home node is full,
          overflow it clockwise to the next node with spare capacity &rarr; this guarantees no node
          exceeds the cap while keeping the extra key movement bounded &rarr; &epsilon; is the knob:
          small &epsilon; = tighter balance but more churn/overflow, larger &epsilon; = looser cap but
          less movement &rarr; used in GCP's load balancer and HAProxy &rarr; if the hot key is
          read-heavy specifically, also replicate it across more nodes so reads fan out.
    - level: 'senior'
      q: "How do you place N replicas on a vnode ring, and what's the classic bug?"
      a: >-
          <b>Hit these points:</b> walk clockwise from the key's position and pick the next N
          <em>distinct physical</em> nodes &mdash; that ordered set is Dynamo's <b>preference list</b>
          &rarr; the classic bug: naively taking the next N ring positions can land several replicas on
          vnodes of the <em>same physical machine</em>, so one host failure loses the key entirely
          &mdash; you must skip vnodes whose physical node is already chosen &rarr; go further and prefer
          distinct racks/AZs so a rack power loss or AZ outage doesn't take all N copies &rarr; the
          preference list is exactly the set the <code>R + W &gt; N</code> quorum operates over.
    - level: 'senior'
      q: 'When would you pick rendezvous (HRW) or jump hash over a vnode ring?'
      a: >-
          <b>Hit these points:</b> <b>HRW</b> assigns a key to the node with the highest
          <code>hash(key, node)</code> (argmax) &mdash; it gives K/N-minimal movement and uniform
          balance with no vnode bookkeeping, so it's simpler and more balanced for tens-to-low-hundreds
          of nodes, at the cost of O(N) per lookup &rarr; <b>jump hash</b> is ~5 lines, zero memory,
          perfect balance &mdash; but buckets only grow/shrink at the high end and it has no weighting,
          so it fits numbered reshards (8 &rarr; 10) and is wrong for arbitrary node failure or
          heterogeneous capacity &rarr; the vnode ring wins when you need incremental, weighted,
          arbitrary membership changes &mdash; which is most real clusters.
    - level: 'staff'
      q: "Separate consistent hashing from quorum replication. Where's the boundary, and why does it matter?"
      a: >-
          <b>Hit these points:</b> they are two orthogonal decisions that people conflate &rarr;
          consistent hashing answers <em>which</em> nodes hold a key &mdash; placement, the ring,
          vnodes, and the preference list (routing + rebalancing) &rarr; the quorum answers <em>how
          many must agree</em> &mdash; <code>R + W &gt; N</code> guarantees a read overlaps the latest
          write &rarr; together they form the storage layer, but you can vary one without the other:
          tune R/W for the consistency/availability point (W=1 for always-writeable) while the ring
          stays fixed, or change vnode count/placement without touching the quorum &rarr; why it
          matters: a membership change is a <em>placement</em> event (re-replicate ~K/N keys to new
          owners), while a slow/lost node is a <em>quorum</em> event (hinted handoff, read-repair)
          &mdash; keeping them separate is how you reason about each failure cleanly.
    - level: 'staff'
      q: 'Argue for vnodes — then steelman why a system might deliberately avoid them.'
      a: >-
          <b>For:</b> a bare ring is a hot-spot generator (hottest node ~log N&times; average); vnodes
          turn each node into a sum of many arcs so load concentrates at 1/N with variance
          ~1/&radic;V, and they give graceful failure spread and free weighting &mdash; which is why
          Dynamo, Cassandra and Riak all use tokens. <b>Steelman against:</b> vnodes multiply ring
          metadata (N&middot;V positions) and gossip/membership overhead; range scans and ordered
          iteration fragment across many small arcs; rebuilding/streaming a node's data touches many
          peers; and at small, homogeneous, fixed-size clusters HRW gives equal-or-better balance with
          simpler reasoning. <b>Synthesis:</b> default to vnodes for large, heterogeneous,
          frequently-changing clusters; tune V to trade balance against metadata; consider HRW or fewer
          tokens when N is small and stable or when ordered range access dominates &mdash; pick by
          membership volatility, heterogeneity, and access pattern, not by reflex.
    - level: 'staff'
      q: 'A team caches with <code>mod N</code> and says "rehashing on scale-up is a one-time blip." Make the principal-level call.'
      a: >-
          <b>Hit these points:</b> reject the framing &mdash; it's not a one-time blip, it recurs on
          <em>every</em> membership change, planned or not: a deploy, an autoscale event, or a single
          node death all change N &rarr; each one re-scatters ~all keys, so the cache cold-starts and
          the full miss traffic stampedes the DB at exactly the moment you're already losing capacity
          (a node just died) &rarr; that couples a routine operational event to a correlated DB overload
          &mdash; a reliability hazard, not a performance footnote &rarr; the fix is cheap and
          well-understood: a consistent-hashing ring with vnodes limits movement to ~K/N and confines
          the miss blast radius to one node's keys &rarr; quantify it for them: with 50 nodes,
          <code>mod N</code> invalidates ~98% of keys on one change, the ring ~2% &rarr; principal move:
          mandate the ring (or HRW) for any cluster whose membership ever changes, and add stampede
          protection (single-flight + jittered TTLs) regardless, so even the ~K/N miss can't take the DB
          down.
    - level: 'design'
      q: 'Design the sharding/routing layer for a 100-node distributed cache.'
      a: >-
          <b>A strong answer covers:</b> clarify &mdash; membership changes often (deploys, autoscale,
          failures), nodes may be heterogeneous, items are hot/popular, replication optional &rarr; pick
          a consistent-hashing ring in a 2&sup3;&sup2;/2&sup1;&#8310;&#8304; space with V &asymp;
          128&ndash;256 vnodes/node for balance and capacity-weighting &rarr; lookup is an O(log N)
          successor search over sorted token positions &rarr; on membership change only ~K/N keys move
          (one node's worth), so the miss blast radius and the DB stampede are confined to one node's
          keyspace &rarr; because cache items are popular, add bounded loads with a small &epsilon; so no
          single node can be buried by a hot home arc, and replicate genuinely viral keys for read fan-out
          &rarr; if replicated, use a rack/AZ-aware preference list of the next N distinct physical nodes
          &rarr; protect the DB on every reshard with single-flight + jittered TTLs &rarr; failure modes:
          hot key, correlated rack loss, reshard streaming saturating the network (throttle it) &rarr;
          trade-off: mention HRW as a simpler alternative at ~100 nodes, but justify the ring for
          incremental, weighted, arbitrary membership.
    - level: 'design'
      q: 'Design partitioning + replication for a write-heavy distributed key-value store that must survive node and AZ loss.'
      a: >-
          <b>A strong answer covers:</b> clarify &mdash; durability and availability targets, the
          consistency bar (eventual + conflict resolution acceptable?), and that it's write-heavy &rarr;
          placement: consistent-hashing ring with vnodes so a membership change moves only ~K/N keys and
          load stays even; weight vnodes by node capacity for heterogeneity &rarr; replication: each key
          to the next N=3 <em>distinct physical</em> nodes clockwise (the preference list), spread across
          racks/AZs so an AZ loss doesn't take all N copies &rarr; consistency: leaderless tunable
          quorums, R + W &gt; N for overlap; W can be low for write-availability at the cost of more
          reconciliation &rarr; concurrent writes &rarr; vector clocks / version vectors return siblings
          to merge rather than blind last-write-wins &rarr; write-heavy storage engine: an LSM-tree per
          node &rarr; failure handling: hinted handoff for a briefly-down node, Merkle-tree anti-entropy
          + read-repair for divergence, gossip for membership/failure detection, throttled
          re-replication on permanent loss &rarr; rebalancing: adding capacity moves ~K/N keys to the new
          owners; stream in the background with backpressure &rarr; trade-off: this is an AP design (CAP)
          that accepts temporary inconsistency under partition &mdash; if strong consistency were
          required you'd shift to single-leader-per-shard with synchronous replication, losing some write
          availability.

    </div>
    <p class="tagline">
        Map nodes <em>and</em> keys into the same hash space. A key belongs to the first node
        clockwise. That locality is the whole win.
    </p>

    <Fig3 />

    <div class="facts">
        <div class="fact">
            <b>Placement</b> A node sits at <code>hash(node_id)</code>; a key is owned by the first
            node clockwise from <code>hash(key)</code>. Same hash space for both.
        </div>
        <div class="fact">
            <b>Hash space</b> A circle of <code>0 … 2³²−1</code> (Dynamo uses <code>2¹⁶⁰</code>).
            Wrapping around the top is what makes it a ring.
        </div>
        <div class="fact blue">
            <b>Add</b> Claims only the arc between it and its counter-clockwise neighbour — ~K/N keys
            from <em>one</em> node. Everyone else stays.
        </div>
        <div class="fact blue">
            <b>Remove</b> Its arc's keys fall to the clockwise successor — again ~K/N keys, to{' '}
            <em>one</em> node.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Adding a node reshuffles keys across the cluster"</div>
        <div class="t">✓ It touches exactly one neighbour's arc; locality is the entire point</div>
    </div>

    <div class="rule">
        <span>🧭 Memory rule:</span> Lookup is a clockwise <span>successor search</span> (O(log N));
        a membership change touches exactly <span>one neighbour's arc</span> — but those arcs are
        not equal.
    </div>

    <MemoryCheck items={mc2} />

</section>

{/* ================= SECTION 3 ================= */}

<section class="lvl" id="l3">
    <div class="lvltop">
        <span class="num">3</span>
        <h2>Virtual nodes — fixing the lopsided ring</h2>
    </div>
    <p class="tagline">
        Random placement does NOT give equal arcs. The most-loaded node can hold <code>log N</code>×
        the average. Vnodes are an averaging machine.
    </p>

    <Fig4 />

    <figure>
        <img
            src="diagrams/ch-02-vnodes.png"
            alt="Virtual nodes smooth the load. Left: four physical nodes placed once each carve the ring into wildly uneven arcs — one node is overloaded. Right: each physical node owns many small vnodes scattered around the ring, so every node's total load converges to 1/N."
        />
        <figcaption>
            <strong>Virtual nodes smooth the load.</strong> Each physical node owns many small
            vnodes scattered around the ring, so its total load — the sum of many independent arcs —
            converges to <code>1/N</code>.
        </figcaption>
    </figure>

    <div class="facts">
        <div class="fact">
            <b>The problem</b> Random placement → largest arc ≈ <code>Θ((log N)/N)</code>; the
            hottest node can hold <code>log N</code>× average. With few nodes, one routinely owns
            2–3× another.
        </div>
        <div class="fact">
            <b>The fix</b> Hash each <em>physical</em> node to many positions (<code>V ≈ 100–200</code>
            ). N nodes scatter <code>N·V</code> points around the ring.
        </div>
        <div class="fact blue">
            <b>Why it works</b> A node's load = the <em>sum</em> of its V little arcs. Averaging V
            independent pieces concentrates load at <code>1/N</code> with std dev ~ <code>1/√V</code>
            — within a few % of uniform.
        </div>
        <div class="fact blue">
            <b>Three dividends</b> ① balance (1/√V) · ② graceful failure (a dead node's vnodes
            spread to many successors, not one) · ③ heterogeneity (2× capacity = 2× vnodes —
            weighting for free).
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "N random nodes carve N roughly-equal arcs"</div>
        <div class="t">✓ Random placement is lopsided; only vnodes (summing many arcs) flatten it</div>
    </div>

    <div class="rule">
        <span>🌀 Memory rule:</span> A bare ring is a hot-spot generator. Vnodes turn each node into
        a <span>sum of many arcs</span> — variance shrinks like <span>1/√V</span>. Dynamo,
        Cassandra, Riak all use tokens, never a bare ring.
    </div>

    <MemoryCheck items={mc3} />

</section>

{/* ================= SECTION 4 ================= */}

<section class="lvl" id="l4">
    <div class="lvltop">
        <span class="num">4</span>
        <h2>Bounded loads — when one key still runs hot</h2>
    </div>
    <p class="tagline">
        Vnodes balance the <em>keyspace</em>, not <em>popularity</em>. A viral key has exactly one
        home. Cap every node and overflow clockwise.
    </p>

    <figure>
        <img
            src="diagrams/ch-03-bounded-loads.png"
            alt="Consistent hashing with bounded loads. Each node has a capacity cap of (1+ε)× the average. A key whose home node is full overflows clockwise to the next node with spare capacity, so no node can exceed the cap — at the cost of a little extra movement."
        />
        <figcaption>
            <strong>Consistent hashing with bounded loads.</strong> Cap each node at{' '}
            <code>(1+ε)×</code> average. A key whose home node is full overflows clockwise to the
            next node with room — so no node can exceed the cap.
        </figcaption>
    </figure>

    <Fig5 />

    <div class="facts">
        <div class="fact">
            <b>Why vnodes aren't enough</b> They balance the keyspace but assume keys are equally
            popular. A viral key (or live connections in a cache / LB) hammers its one home node.
        </div>
        <div class="fact">
            <b>The mechanism</b> Pick slack <code>ε &gt; 0</code>; cap each node at <code>(1+ε)×</code>{' '}
            average. A key whose home is at capacity walks clockwise to the next node with room.
        </div>
        <div class="fact blue">
            <b>The knob ε</b> Small ε → tight balance, more overflow reassignment on change. Large ε
            → looser cap, cheaper.
        </div>
        <div class="fact blue">
            <b>In the wild</b> Google Cloud load balancing and HAProxy <code>balance</code> modes use
            exactly this. The answer to "vnodes balanced my keyspace but node 7 is still on fire."
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Vnodes guarantee even load, so no node can overload"</div>
        <div class="t">✓ Vnodes balance keyspace, not popularity; bounded loads cap actual load</div>
    </div>

    <div class="rule">
        <span>🔥 Memory rule:</span> Bounded loads (Mirrokni–Thorup–Zadimoghaddam 2016)
        <sup>
            <a href="#s3" id="r3">3</a>
        </sup>{' '}
        caps each node at <span>(1+ε)× average</span> and overflows clockwise — guaranteeing the cap
        while keeping movement bounded.
    </div>

    <MemoryCheck items={mc4} />

</section>

{/* ================= SECTION 5 ================= */}

<section class="lvl" id="l5">
    <div class="lvltop">
        <span class="num">5</span>
        <h2>You may not need a ring</h2>
    </div>
    <p class="tagline">
        The ring is the famous answer, not the only one — and often not the best one. Two
        alternatives that drop the ring entirely.
    </p>

    <Fig6 />

    <table>
        <tr>
            <th>Scheme</th>
            <th>Lookup</th>
            <th>Balance</th>
            <th>Arbitrary removal</th>
            <th>Weights</th>
        </tr>
        <tr>
            <td>Ring + vnodes</td>
            <td>O(log N)</td>
            <td>good (needs vnodes)</td>
            <td>yes</td>
            <td>yes (more vnodes)</td>
        </tr>
        <tr>
            <td>Rendezvous (HRW)</td>
            <td>O(N)</td>
            <td>excellent</td>
            <td>yes</td>
            <td>yes</td>
        </tr>
        <tr>
            <td>Jump hash</td>
            <td>O(log N)</td>
            <td>perfect</td>
            <td>
                <strong>no</strong> (high-end only)
            </td>
            <td>
                <strong>no</strong>
            </td>
        </tr>
    </table>

    <div class="facts">
        <div class="fact blue">
            <b>Rendezvous (HRW) — for free</b> Removing a node only re-homes keys whose top weight
            was that node → exactly K/N move. Uniform argmax → no log N imbalance, no vnodes.
            <sup>
                <a href="#s4" id="r4">4</a>
            </sup>
        </div>
        <div class="fact blue">
            <b>HRW — the cost</b> <code>O(N)</code> per lookup. Fine for tens-to-low-hundreds of
            nodes (the common case); for huge N go back to a ring or hierarchy.
        </div>
        <div class="fact">
            <b>Jump hash — the gift</b> 5 lines, zero memory, very fast, perfect balance, minimal
            remap. Ideal for resharding a numbered range (8→10 shards).
            <sup>
                <a href="#s5" id="r5">5</a>
            </sup>
        </div>
        <div class="fact">
            <b>Jump hash — the trap</b> Grows/shrinks only at the <em>high end</em> and has no
            weights. Wrong for arbitrary cluster membership where any node may fail. Gorgeous and
            narrow.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Consistent hashing means the ring; reach for it by default"</div>
        <div class="t">
            ✓ For small/medium clusters HRW is often simpler &amp; more balanced; jump hash wins for
            numbered reshards
        </div>
    </div>

    <div class="rule">
        <span>🧰 Memory rule:</span> Ring+vnodes for arbitrary membership at scale ·{' '}
        <span>HRW</span> for simple, well-balanced small clusters · <span>jump hash</span> only for
        a numbered range that grows at the end.
    </div>

    <MemoryCheck items={mc5} />

</section>

{/* ================= SECTION 6 ================= */}

<section class="lvl" id="l6">
    <div class="lvltop">
        <span class="num">6</span>
        <h2>Replication — the preference list</h2>
    </div>
    <p class="tagline">
        A key isn't on one node but N. The replicas are the next N <em>distinct physical</em> nodes
        clockwise — skip duplicate vnodes, prefer distinct racks.
    </p>

    <Fig7 />

    <div class="facts">
        <div class="fact">
            <b>Replicas</b> The next N nodes clockwise — but you must reach N <em>distinct physical</em>{' '}
            nodes, skipping extra vnodes of a machine you already picked.
        </div>
        <div class="fact">
            <b>The vnode trap</b> Naïvely taking the next N vnodes can land all "replicas" on one
            machine — a single failure then loses the key.
        </div>
        <div class="fact blue">
            <b>Rack / AZ awareness</b> Prefer distinct racks / availability zones so a rack power
            loss doesn't take all N replicas at once.
        </div>
        <div class="fact blue">
            <b>How it composes</b> Hashing decides <em>which</em> nodes hold a key; the quorum (
            <code>R + W &gt; N</code>) decides <em>how many must agree</em>. Together: the whole
            storage layer.
        </div>
    </div>

    <div class="myth">
        <div class="m">✗ "Replicas = the next N vnodes clockwise"</div>
        <div class="t">
            ✓ The next N DISTINCT physical nodes (ideally distinct racks) — or one failure loses the
            key
        </div>
    </div>

    <div class="rule">
        <span>🗂️ Memory rule:</span> The <span>preference list</span> = next N distinct physical
        nodes clockwise (across racks). Hashing picks <em>which</em> nodes; quorum picks <em>how
        many agree</em>.
    </div>

    <MemoryCheck items={mc6} />

</section>

<h2 class="sec" id="cheat">
    ★ Cheat sheet — claim → engineering
</h2>

<div class="cheat">
    <p style="margin-top:0">
        <strong>Mental models:</strong> mod N = no locality (re-scatters all) · ring = first node
        clockwise (~K/N moves) · vnodes = sum of arcs (1/√V) · bounded loads = (1+ε) cap + overflow
        · HRW = argmax, O(N) · jump hash = 5 lines, high-end only · preference list = next N
        distinct physical nodes.
    </p>
    <h3>Translation table</h3>
    <table>
        <tr>
            <th>Term</th>
            <th>What it actually is</th>
        </tr>
        <tr>
            <td>Consistent hashing</td>
            <td>Same ring for keys + nodes so a change moves ~K/N keys</td>
        </tr>
        <tr>
            <td>Successor search</td>
            <td>Binary search of sorted node positions, clockwise — O(log N)</td>
        </tr>
        <tr>
            <td>Virtual node / token</td>
            <td>One of V ring positions per physical node; balances via 1/√V</td>
        </tr>
        <tr>
            <td>Bounded loads</td>
            <td>Per-node cap (1+ε)× average; full node overflows clockwise</td>
        </tr>
        <tr>
            <td>Rendezvous / HRW</td>
            <td>Assign key to node with highest hash(k,node); no ring</td>
        </tr>
        <tr>
            <td>Jump hash</td>
            <td>5-line key→bucket (0…N−1); zero memory; high-end resize only</td>
        </tr>
        <tr>
            <td>Preference list</td>
            <td>Next N distinct physical nodes/racks; the set R+W&gt;N runs over</td>
        </tr>
    </table>
    <h3>Three rules for the design round</h3>
    <p style="margin-bottom:0">
        ① Never <strong>mod N</strong> for cluster membership — it re-scatters everything. ② A bare
        ring is a <strong>hot-spot generator</strong>; always use vnodes (or HRW). ③ Match scheme to
        job: <strong>ring</strong> for arbitrary membership, <strong>HRW</strong> for small balanced
        clusters, <strong>jump hash</strong> only for numbered reshards.
    </p>
</div>
