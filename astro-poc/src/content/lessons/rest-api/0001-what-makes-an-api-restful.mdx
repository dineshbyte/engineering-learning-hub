# —— interview Q&A (rendered by <Interview>; also the FAQPage JSON-LD source) ——
interview:
    - level: 'core'
      q: 'What is REST, precisely?'
      a: >-
          <p>An <strong>architectural style</strong> defined by a set of constraints, coined by Roy
          Fielding — <em>not</em> a protocol, data format, or "JSON over HTTP." An API is "RESTful"
          only to the degree it honors the constraints.</p>
    - level: 'core'
      q: 'Name the six architectural constraints.'
      a: >-
          <p>Client–server, stateless, cacheable, uniform interface, layered system, and
          code-on-demand (the only optional one).</p>
    - level: 'core'
      q: 'What is HATEOAS?'
      a: >-
          <p>Hypermedia As The Engine Of Application State: responses include <strong>links
          advertising the valid next actions</strong>, so the client discovers what it can do rather
          than hard-coding URIs and workflow &rarr; the server can then change its URL structure
          without breaking clients.</p>
    - level: 'core'
      q: 'Describe the Richardson Maturity Model.'
      a: >-
          <p>A 0–3 ladder of how fully an API uses REST's ideas: <strong>L0</strong> one URI / one
          verb (RPC-over-HTTP, "Swamp of POX"); <strong>L1</strong> many resources, each a URI;
          <strong>L2</strong> proper HTTP verbs + status codes; <strong>L3</strong> hypermedia
          (HATEOAS). Most production "REST" APIs sit at L2.</p>
    - level: 'senior'
      q: 'Is REST the same thing as HTTP?'
      a: >-
          <p>No. REST is a <em>style</em>; HTTP is a <em>protocol</em> that fits it very well. You can
          apply REST principles over other protocols, and you can use HTTP <em>un</em>-RESTfully —
          e.g. tunnelling everything through one <code>POST</code> endpoint (Richardson Level 0).</p>
    - level: 'senior'
      q: 'Why does statelessness help you scale — and what does it <em>not</em> forbid?'
      a: >-
          <p><em>Any</em> node can serve <em>any</em> request — no sticky sessions, no affinity — so
          you scale horizontally behind a load balancer and a dying node loses no session &rarr; it
          also improves visibility (each request is self-contained). It does <em>not</em> forbid
          databases or cookies: it forbids <em>server-side session</em> state between requests. Auth
          moves into a token (Bearer/JWT) sent every request.</p>
    - level: 'senior'
      q: 'If HATEOAS is part of REST, why is it so rare in real APIs?'
      a: >-
          <p>Because the payoff rarely justifies the cost. Most clients are first-party and
          version-pinned — coded against known endpoints, not dynamically following links.
          Tooling/codegen expect fixed URLs, so the decoupling HATEOAS buys goes unused and teams stop
          at Level 2. The senior move is to <em>know</em> that and say so, not pretend everyone does
          L3.</p>
    - level: 'senior'
      q: 'REST vs GraphQL vs gRPC — when would you pick each?'
      a: >-
          <p><strong>REST</strong>: resource-oriented, leans on HTTP caching, broad reach, easy to
          evolve — great at the public edge. <strong>GraphQL</strong>: client shapes the query (kills
          over-/under-fetching), one endpoint for many varied clients — but HTTP caching is hard, plus
          query-cost/N+1 control. <strong>gRPC</strong>: low-latency internal service-to-service,
          typed contracts, streaming — but binary, less edge-friendly. Rule of thumb: REST at the
          edge, gRPC between internal services, GraphQL for rich multi-client read graphs.</p>
    - level: 'staff'
      q: 'Is a Level 2 API "really REST" by Fielding''s own definition — and how do you defend it?'
      a: >-
          <p>Strictly no — Fielding argues that without the hypermedia constraint it isn't truly REST.
          But Level 2 is what the industry pragmatically means by "REST." The strong answer names that
          tension explicitly, then defends the pragmatic choice for the given context (first-party
          version-pinned clients, codegen, fixed URLs) rather than dodging it.</p>
    - level: 'staff'
      q: 'When would you actually invest in HATEOAS / Level 3?'
      a: >-
          <p>When you have <strong>many third-party clients you can't redeploy</strong> and must
          evolve URLs/workflows without breaking them; when the API models a <strong>state
          machine</strong> where available actions genuinely change (a payment that can be
          captured/refunded/voided — links express the legal transitions); or where discoverability is
          itself a product feature.</p>
    - level: 'staff'
      q: 'A team says "let''s make our API RESTful." How do you advise them?'
      a: >-
          <p>First ask <em>why</em> — what outcome (evolvability, caching, DX, broad reach)? Then
          assess current maturity. For most, the goal is a clean Level 2: good resource modeling,
          correct verbs/status codes, real cache headers, consistent errors. Pursue L3/HATEOAS only if
          uncontrollable third-party clients or a genuine state machine demand it. Don't chase REST
          purity for its own sake — tie every constraint back to a concrete benefit.</p>
    - level: 'design'
      q: 'Design a RESTful API for a blogging platform (posts, comments, authors). Walk the resource model and maturity level.'
      a: >-
          <p><b>A strong answer covers:</b> resources as nouns — <code>/posts</code>,
          <code>/posts/{id}</code>, <code>/posts/{id}/comments</code>, <code>/authors/{id}</code> —
          nesting only where ownership is real, plus a flat <code>/comments/{id}</code> for direct
          access &rarr; verbs mapped cleanly (GET, POST, PATCH, DELETE) with correct status codes (201
          + <code>Location</code>, 404 vs 410, 409 on conflict) &rarr; statelessness: bearer token per
          request, per-resource authorization (an author edits only their own posts) &rarr; caching
          with <code>ETag</code>/<code>Last-Modified</code> and conditional requests on writes &rarr;
          pagination + filtering on collections (cursor-based, bounded size) &rarr; a consistent error
          envelope and a versioning/deprecation story &rarr; name the maturity call: deliberately
          Level 2, and say <em>when</em> you'd move to L3.</p>
    - level: 'design'
      q: 'You''re handed a Level 0 "JSON over HTTP" API (one <code>POST /api</code> with an action field) and asked to make it RESTful. How do you approach it?'
      a: >-
          <p><b>A strong answer covers:</b> diagnose first — it's RPC tunnelled through one endpoint,
          so verbs, status codes, and caching carry zero meaning &rarr; climb the Richardson ladder
          incrementally, not big-bang: L1 split into resource URIs, L2 map each action to the correct
          method + status code and expose real cache headers &rarr; keep the old endpoint behind a
          versioned facade so existing clients don't break; run both during migration &rarr; tie every
          change to a benefit (cacheability, idempotent retries, intermediary visibility) over purity
          &rarr; stop at L2 unless uncontrollable third-party clients or a genuine state machine
          justify HATEOAS &rarr; call out migration risks: dual-write/dual-read consistency, client
          cutover, deprecation timeline with metrics.</p>

# —— design-round rubric (rendered by <Interview> above the design questions) ——
rubricIntro: >-
    <b>REST design-round framework</b> — drive any "design a REST API" prompt through these, out
    loud:
rubric:
    - >-
          Model resources &amp; URIs — nouns not verbs, collections vs items (<code>/orders</code>,
          <code>/orders/{id}</code>), nesting only for true ownership.
    - >-
          Map methods &amp; status codes — GET/POST/PUT/PATCH/DELETE to the right semantics;
          2xx/4xx/5xx that mean what they say.
    - >-
          Idempotency &amp; caching — safe/idempotent verbs, idempotency keys for POST,
          <code>ETag</code>/<code>Cache-Control</code>/conditional requests.
    - >-
          Pagination, filtering &amp; sorting — cursor vs offset, bounded page sizes, consistent query
          params.
    - >-
          AuthN/AuthZ — bearer tokens on every request (stateless), scopes/roles, transport security,
          rate limits (OWASP API Top 10).
    - >-
          Versioning &amp; evolution — additive changes, deprecation policy, URL vs header versioning.
    - >-
          Errors &amp; contracts — a consistent error envelope (e.g. RFC 9457 problem+json),
          correlation IDs, documented schema.

## The six constraints (and the senior "why")

<p>
    Don't recite the list — pair each with the property it buys. That trade is the answer
    interviewers are actually grading.
</p>

<div class="facts">
    <div class="fact">
        <b>1 · Client–server</b> Split UI from data storage so each side evolves independently.
    </div>
    <div class="fact blue">
        <b>2 · Stateless</b> Every request carries all its context; no per-client session is kept.
        Any node serves any request, so you scale horizontally. <em>Probed hardest.</em>
    </div>
    <div class="fact">
        <b>3 · Cacheable</b> Responses declare whether they can be cached — the foundation under
        ETags and CDNs.
    </div>
    <div class="fact blue">
        <b>4 · Uniform interface</b> One generic contract: identified resources, manipulation via
        representations, self-descriptive messages, hypermedia. Decouples client from server.
    </div>
    <div class="fact">
        <b>5 · Layered system</b> A client can't tell origin from intermediary, so gateways and
        proxies slot in invisibly.
    </div>
    <div class="fact">
        <b>6 · Code-on-demand</b> The server may ship runnable code to the client. The only{' '}
        <em>optional</em> constraint.
    </div>
</div>

<p class="pull">
    <strong>Statelessness is the constraint to lead with.</strong> Trace it forward: no server
    sessions &#8594; auth rides a token on every request (Bearer/JWT) &#8594; any node serves any
    request &#8594; you scale by adding boxes, not by sticky-routing users. One constraint, a whole
    architecture.
</p>

## What "stateless" buys you

<p>
    Each request is self-contained — it carries its own auth and context. The server keeps no memory
    between calls, so the load balancer is free to route the next request anywhere.
    <sup>
        <a href="#s3" id="r3">
            3
        </a>
    </sup>
</p>
